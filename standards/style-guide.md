# 文风指南

> 书籍写作风格规范

---

## 核心原则

**好读、适度幽默、不废话、有启发**

---

## 语言风格

### 短句优先

- 单句不超过30字
- 复杂内容拆成多句
- 用句号代替逗号

```
❌ 这是一个非常重要而且十分关键的概念，产品经理必须深入理解才能做好需求设计。
✅ 这个概念很重要。理解它，需求设计才能落地。
```

### 准确术语

- 术语使用必须与术语表一致
- 新术语首次出现必须定义
- 避免未解释的技术术语

```
❌ 后端返回的API响应中包含状态码。
✅ 后端通过接口返回数据，其中包含状态码——就像餐厅的菜单，告诉你有什么菜、价格多少。
```

### 相对轻松

- 不刻意幽默
- 自然使用类比
- 让读者轻松理解复杂概念

```
❌ 下面我要用一个超级有趣超级搞笑的比喻来解释这个概念。
✅ 这就像你点外卖——选好菜、付款、等配送。
```

---

## 幽默度控制

### 适度幽默

幽默是调味品，不是主菜。

**目标**：让读者在理解复杂概念时，有一个轻松的记忆点。

### 正确做法

```
✅ 老李听完开发说"这个要做状态机"，心想：状态机？是不是卖咖啡的机器？
   → 自然的人物困惑，真实场景
```

### 错误做法

```
❌ 老李听完开发说"这个要做状态机"，笑得肚子疼，心想这开发是不是想吃咖啡了哈哈哈！
   → 过度夸张，刻意搞笑
```

### 幽默场景建议

- 人物对话中的误解（老李不懂术语、小想踩坑）
- 类比时的有趣对比
- 案例复盘时的反思

---

## 避免空洞表达

### 禁止表达

| 禁用 | 原因 |
|------|------|
| "非常重要" | 没说为什么重要 |
| "十分关键" | 没说关键在哪 |
| "极其重要" | 没说影响是什么 |
| "必不可少" | 没说缺失会怎样 |

### 替代方案

```
❌ 验收标准非常重要。
✅ 没有验收标准，开发做完你说不对，开发说你就这么说的，两边都不服。
```

## 人物设定一致性

### 老李

- **语气**：稳重、有经验感、偶尔吐槽但点到为止
- **口头禅**："我不需要知道怎么实现，但我要知道会在哪里出问题"
- **行为**：判断力展示、全局观、用业务语言翻译技术

```
✅ 老李听完方案，点点头："技术上我不挑，但这里有个坑——状态切换失败怎么处理？"
```

### 小想

- **语气**：年轻、有野心、敢用AI、偶尔踩坑后反思
- **口头禅**："这个需求我让AI先跑一版，然后我们再对齐？"
- **行为**：工具尝试、踩坑展示、快速学习

```
✅ 小想兴奋地说："我用AI生成了方案初稿，但好像漏了异常处理..."老李补充道："这正是AI的边界。"
```

## 章节节奏公式（Chapter Pacing Formula）

### 设计理念

面向非技术人员的技术概念教学，参考业界标杆文风：

| 标杆 | 核心写法 | 借鉴点 |
|------|---------|--------|
| Basecamp《Getting Real》 | 主张 → 反面案例（痛）→ 正面做法（爽）→ 结束 | 每章极短，一个观点、一个例子、一个行动建议 |
| Julia Evans（jvns.ca） | "你遇到过X？" → 本质是Y → 怎么做 → 完 | 极简语言，一篇只解决一个问题 |
| Will Larson《Staff Engineer》 | 概念定义 → 为什么重要 → 怎么做 → 常见坑 | 信息增量契约——没有新信息就不写 |
| Avinash Kaushik（Occam's Razor） | 直接、节奏快、"So What"测试每一段 | 如果读者读完会说"So What?"，就删掉 |

### 五段节奏（控制 2500-3000 字）

每章按以下节奏分配内容，10 要素映射进 5 个大段：

| 段 | 名称 | 字数 | 功能 | 写法要求 | 10 要素映射 |
|----|------|------|------|---------|------------|
| 1 | 痛点场景 | ~300 | 一个故事，制造共鸣 | 有人物、有冲突、让读者说"对对对就是这样" | ①问题定义 + ②业务场景 |
| 2 | 核心概念 | ~800 | 是什么、为什么重要 | 清晰定义 + 类比 + 误解纠正，不超过5个概念 | ③概念解释 + ④PM误解 |
| 3 | 开发怎么看 | ~400 | 拉齐认知差 | 换位思考，讲开发关注点和技术约束 | ⑤开发视角 |
| 4 | 怎么做 | ~600 | 模板+AI用法+自查，一次讲完 | 可执行的行动建议、AI辅助方法、checklist | ⑥PM行动 + ⑦AI辅助 + ⑧Checklist |
| 5 | 一句话总结 | ~50 | 提炼公式，不复述 | 一个公式或记忆点，绝不重复全文 | ⑨案例复盘要点 + ⑩总结 |

> **总字数预算**：2550 字（±20%弹性）。如果一章写到 4000+ 字，说明塞了太多，拆章。

### 信息增量契约（So What 测试）

每段必须经受的自检：

> "这一段相比上一段，读者多知道了什么？"
> 如果答案是"没有"，删掉。

**执行规则**：

1. 每段必须有且仅有一个明确的「信息增量」
2. 删除所有"正确但无新信息"的段落
3. 案例复盘不重述故事，只提炼新洞察
4. 总结不复述全文，只给公式或记忆点
5. 相邻两段如果可以合并且不损失信息，必须合并

### 节奏自检清单

写作完成后，对照检查：

- [ ] 全章字数在 2500-3000 字区间
- [ ] 痛点场景在前 300 字内建立共鸣
- [ ] 核心概念不超过 5 个
- [ ] "怎么做"段有可直接执行的 checklist
- [ ] 总结不超过 50 字，是公式不是复述
- [ ] 每段通过"So What"测试——删掉任何"读完没有新知"的段落
- [ ] 没有连续两段在讲同一件事

## 章节结构文风

每个章节按10要素结构写作（映射进上述五段节奏），各要素有特定文风要求：

### 1. 本章要解决什么问题

- **文风**：直接、简洁、一句话点明
- **长度**：100-500字

```
✅ 本章解决一个让无数产品经理头疼的问题：为什么开发总说"做不了"？
```

### 2. 真实业务场景开场

- **文风**：故事化、代入感、有人物和冲突
- **必须包含**：人物、情境、冲突

```
✅ 周一上午，产品评审会。老李刚说完需求，开发就说"这个做不了"。老李心里一沉...
```

### 3. 核心概念解释

- **文风**：清晰定义 + 类比 + 例子
- **概念数量**：2-5个

```
✅ **状态机**是什么？简单说，就是定义"状态怎么变"的规则。
   比如订单：待支付→已支付→发货中→已完成。每一步怎么跳、能不能跳，都写在规则里。
```

### 4. 为什么产品经理容易误解

- **文风**：共情、真实误解表现 + 原因分析
- **必须包含**：误解表现、误解原因

```
✅ 很多产品经理听到"状态机"会想：不就是几个状态吗，画个流程图就行。
   误解在哪？你没看到"跳转条件"。光有状态，没有跳转规则，系统就不知道怎么走。
```

### 5. 开发视角如何看这个问题

- **文风**：换位思考、开发关注点 + 技术约束
- **必须包含**：开发关注点、技术约束

```
✅ 开发看状态机，想的不是"有几个状态"，而是：
   - 状态怎么存储？
   - 跳转条件怎么判断？
   - 并发时状态冲突怎么办？
   这些才是真正要实现的东西。
```

### 6. 产品经理应该如何做

- **文风**：行动建议、可执行、有checklist
- **必须包含**：行动建议、checklist

```
✅ 下次写需求时，先把状态流转画出来。然后问开发：跳转条件是什么？冲突怎么处理？
   Checklist：
   - [ ] 状态定义完整
   - [ ] 跳转条件明确
   - [ ] 异常情况覆盖
```

### 7. AI可以如何辅助

- **文风**：实用、具体方法 + 应用场景
- **必须包含**：AI应用场景、具体方法

```
✅ AI可以帮你生成状态机初稿。你描述业务流程，AI画出状态图。
   但记住：AI生成的是"草稿"，跳转条件、异常处理需要你来补。
```

### 8. 本章 checklist

- **文风**：列表、可勾选、简洁
- **条目数量**：3-10条

```
✅ - [ ] 我能识别需求中的状态流转
   - [ ] 我知道开发关注什么
   - [ ] 我会用AI辅助方案设计
```

### 9. 案例复盘

- **文风**：回顾案例 + 提炼要点
- **必须包含**：案例回溯、要点提炼

```
✅ 回看老李和小想的对话：小想用AI生成方案，漏了异常处理。老李补上了关键问题。
   要点：AI是工具，判断力是你的。工具帮你提速，你把控质量。
```

### 10. 本章总结

- **文风**：一句话总结、启发感
- **长度**：不超过300字

```
✅ 开发说"做不了"，往往不是不想做，而是你没给够信息。
   学会从开发视角看问题，需求才能真正落地。
```

## 检查清单

写作完成后自查：

- [ ] 单句不超过30字
- [ ] 无空洞表达（非常重要、十分关键等）
- [ ] 幽默适度，不刻意
- [ ] 术语与术语表一致
- [ ] 新术语首次出现已定义
- [ ] 人物行为符合设定
- [ ] 10要素结构完整
- [ ] 各要素文风符合要求
