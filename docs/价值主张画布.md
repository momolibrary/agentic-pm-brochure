## 一、核心定位语

> **写给不懂技术、但不想再只做 PRD 的产品经理。**
>
> 本书帮助传统产品经理借助 AI 补齐最小必要软件工程能力，从需求传递者升级为 **Agentic 交付型产品经理**。

---

## 二、读者画像

### 主读者画像：传统交付型 PM

| 维度 | 描述 |
|------|------|
| **岗位** | 产品经理、业务产品经理、B端产品经理、数字化产品经理 |
| **年限** | 2~8 年为主 |
| **背景** | 非计算机背景居多 |
| **工作领域** | B端系统、内部数字化、业务系统、平台产品 |

#### 过去的核心能力
- 理解业务
- 整理需求
- 画原型
- 写 PRD
- 推动开发测试上线

#### 当前典型工作状态
- 对业务流程比较熟
- 对用户/内部客户沟通较熟
- 知道项目怎么推进
- 但对前后端、接口、数据库、权限、状态流转、系统耦合理解较弱
- 开发说"这个复杂""这个做不了""这个要改架构"，很难独立判断
- 方案讨论时容易停留在页面与交互层
- 面对 AI 编程浪潮，会有期待，也会有焦虑

#### 当前焦虑
- "以后是不是 PRD 不值钱了？"
- "如果 AI 能写需求、画原型、出页面，那我还剩什么？"
- "开发都能往前走做产品，我为什么不能往后走懂工程？"
- "我不想转程序员，但我必须更懂交付。"
- "我希望自己也能独立完成一些小系统/自动化/内部工具。"

#### 他们真正想要的
- 听懂技术语言
- 判断方案边界
- 与研发共创方案
- 借助 AI 先做出第一版
- 端到端完成简单数字化项目
- 在业务面前显得更专业、更可信

#### 他们不想要的
- 成为专业工程师
- 学完编程语言体系
- 深入算法和底层原理

## 三、状态转变地图

### 当前态：Traditional PM（传统文档型 PM）

| 能力维度 | 典型表现 |
|----------|----------|
| **工作方式** | 主要靠会议、PRD、原型推进工作 |
| **技术参与** | 对技术约束的理解依赖研发解释 |
| **方案设计** | 以页面和流程为主 |
| **验收方式** | 更多依赖测试和业务反馈 |
| **交付定义** | "推动别人完成"的过程 |
| **核心产物** | PRD、原型、排期 |

### 目标态：Agentic Delivery PM（交付型 PM）

| 能力维度 | 典型表现 |
|----------|----------|
| **工作方式** | 能把业务问题拆成规则、数据、流程、状态、权限、接口等结构化对象 |
| **技术参与** | 能借助 AI 预演方案、补齐异常流、生成验收清单 |
| **方案设计** | 能独立搭建简单工具、自动化流程、原型应用 |
| **验收方式** | 能和研发一起讨论方案优劣，而不是只提诉求 |
| **交付定义** | "自己能做一部分 + 能高质量编排他人/AI完成剩余部分" |
| **核心产物** | 可执行方案、规则定义、AI协作任务、验收标准、原型/脚手架 |

## 四、能力差异对比

### 传统 PM vs Agentic PM 七维对比

| 维度 | 传统 PM | Agentic PM |
|------|---------|------------|
| **核心定位** | 需求翻译者 | 问题求解者 / 交付编排者 |
| **主要产物** | PRD、原型、排期 | 可执行方案、规则定义、AI协作任务、验收标准、原型/脚手架 |
| **技术参与深度** | 被动听评审 | 主动预演、提出方案、识别约束 |
| **与研发关系** | 需求给到研发实现 | 与研发共创架构和实现路径 |
| **与业务关系** | 记录需求、推动上线 | 帮业务把问题转成可运行解决方案 |
| **个人交付能力** | 依赖团队串行配合 | 可独立完成简单场景端到端落地 |
| **核心竞争力** | 沟通协调、文档能力 | 业务理解 + 工程认知 + AI编排 + 交付能力 |

## 五、价值主张画布

### 客户任务（Jobs to be Done）

**功能性任务**
- [ ] 完成需求文档撰写
- [ ] 推动项目按时交付
- [ ] 与业务方沟通确认需求
- [ ] 协调前后端开发资源
- [ ] 处理需求变更和异常情况

**情感性任务**
- [ ] 在业务面前展现专业性
- [ ] 减少对研发解释的依赖
- [ ] 获得"能独立解决问题"的自信
- [ ] 在 AI 时代保持职业安全感

**社会性任务**
- [ ] 在团队中建立技术可信度
- [ ] 成为业务和技术之间的桥梁
- [ ] 提升在组织中的价值定位

---

### 客户痛点（Pains）

**知识焦虑**
- 不懂技术，担心被边缘化
- AI 能力崛起，担心 PRD 技能贬值
- 不知道如何系统学习技术知识

**工作困境**
- 开发说"做不了"，无法判断真假
- 方案停留在页面层，缺乏系统视角
- 依赖他人验证，自己无法预判风险
- 复杂需求容易遗漏边界情况

**能力瓶颈**
- 无法独立完成端到端交付
- 技术讨论只能"听"，不能"问"
- 对系统架构缺乏判断力
- 难以提出可落地的技术方案

---

### 客户收益（Gains）

**能力提升**
- 听懂技术语言，参与技术讨论
- 判断方案边界，识别技术风险
- 借助 AI 快速验证想法
- 独立完成简单数字化项目

**职业发展**
- 从"文档生产者"升级为"交付编排者"
- 在业务面前更专业、更可信
- 拓展职业发展路径
- 获得 AI 时代新竞争力

**工作改善**
- 减少与研发的沟通成本
- 提前识别问题，减少返工
- 更高效地推动项目落地
- 能够预判和规避技术风险

---

## 七、行业趋势支撑

### 2026 年行业背景

**趋势1：AI-native 开发平台推动"构建权"前移**
- 软件构建越来越能够通过自然语言、生成式 AI、自动化组件来完成
- 更多"非传统开发者"有机会参与构建
- PM 需要理解边界、数据、状态、质量、安全、集成

**趋势2：企业软件从"功能集合"变成"agent 协作体系"**
- 产品不再只是页面和按钮，而是任务流、上下文、规则、授权、反馈闭环
- PM 需要提升任务设计、规则设计、上下文组织、人机协作设计能力

**趋势3：AI 编码工具降低实现门槛，但没有消灭工程复杂性**
- 工程世界真正难的：问题定义、约束识别、系统边界、异常流、质量保证、上线风险、长期维护
- PM 不是不需要工程认知了，而是更需要
