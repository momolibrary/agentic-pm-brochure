# CH-41 Git 协作实操 — 工作流基础、Issue 驱动与代码评审

## 1. 痛点场景

周一下午，老李正在检查上周末上线的产品页面。

发现登录按钮的文案有问题：「点击登录」写成了「点击登登」。

老李心想：「这不就是改一个字的事吗？」

他走到开发的工位：「小明，帮我改一下文案。登录按钮写错了，就改一个字。」

小明抬起头：「好的，老李。不过这个要走完整流程。」

老李皱眉：「完整流程？改一个字也要走流程？直接改一下不就行了？」

小明叹了口气：「老李，我理解您的想法。但是我们的代码库是这样的——」

小明打开 Git 界面：「您看，main 是正式版本，不能随便改。」
「如果我直接在 main 上改，万一改错了，所有分支都受影响。上个月有个实习生直接改了一个字，结果把整个页面样式搞乱了。」
「因为那个文件里还包含关键配置，当时全站崩溃了半小时。」

老李脸色变了：「还有这种事？」

小明点头：「是的。所以任何改动，不管大小，都要走流程。」

---

另一个场景。

周三下午，运营团队找老李：「上周上线的产品功能有问题，需要马上改。」
「用户反馈验证码发送太慢。」
老李回忆了一下：「这个功能是上周上线的……」

他打开邮件找负责人。翻半天，问三个开发，都说不是自己做的。最后发现是实习生做的，但周二离职了，没交接。

组长打开 Git 搜索一小时，才找到代码。commit 消息只写「update」，改动人、时间都无法确定。

老李叹气：「如果当时创建了 Issue……」

---

你是不是也遇到过？

开发说「改一个字也要走流程」，你觉得是故意刁难。紧急需求找不到负责人，翻邮件问了一圈才知道实习生离职了。

本章解决三个问题：为什么改一个字也要走流程？Issue 驱动开发有什么好处？产品经理如何参与 PR 评审？

---

## 2. 核心概念：Branch-Commit-Merge-PR、Issue 驱动、代码评审

### 2.1 Git 分支 = 图书馆抄写本

想象一个古老图书馆，有一本珍贵的原版书。

抄写员想修改内容，不能直接在原版上涂改。他必须借出原版，抄写一份副本。在副本上修改、增删、润色。完成后请馆长审核。审核通过后，将副本与原版合并。

**Git 分支就是这个抄写副本。**

| 图书馆概念 | Git 对应 | 说明 |
|-----------|---------|------|
| 原版珍本 | main 分支 | 正式版本，不能直接改 |
| 抄写副本 | feature 分支 | 从原版复制，可以随便改 |
| 修改记录 | commit | 每次改动盖时间戳印章 |
| 审核请求 | Pull Request | 请馆长审核 |
| 合入原版 | merge | 审核通过后更新原版 |

### 2.2 Git 工作流五步

一个改动从提交到上线，要经过这五步：

| 步骤 | 名称 | 做什么 |
|------|------|--------|
| 1 | 创建分支 | 从 main 复制一份副本 |
| 2 | commit | 修改内容，保存记录 |
| 3 | push | 把副本推到服务器 |
| 4 | Pull Request | 提交审核请求 |
| 5 | merge | 审核通过后合入 main |

**为什么要这样？**

原因一：可追溯。改错了能回退到之前任何版本。

原因二：不影响他人。十几个开发同时在改不同的功能，各抄各的副本，不冲突。

原因三：有审核。改一个字可能改的不是字本身，而是改了包含这个字的配置文件。

### 2.3 Issue 驱动开发 = 餐厅订单系统

想象一家餐厅。

没有订单系统：服务员凭记忆记住客人要什么。漏单、错单、忘记客人需求。

有订单系统：每笔需求写成订单。可追踪、不遗漏、可复盘。

**Issue 驱动开发就是把软件开发变成「有订单系统的餐厅」。**

| 餐厅概念 | Issue 对应 | 说明 |
|---------|-----------|------|
| 订单小票 | Issue | 记录需求细节、责任人、状态 |
| 下单 | 创建 Issue | 需求入库 |
| 接单 | 指派 Assignee | 分配责任人 |
| 制作中 | In Progress | 开发中 |
| 出品 | Review | 待审核 |
| 送达 | Done | 已完成 |

**没有 Issue 的问题**：

- 需求忘记实现
- 需求理解偏差
- 多人做同一件事
- 出问题找不到责任人
- 新人离职后无法交接

### 2.4 PR 评审 = 出版社审稿流程

想象作者写完一本书，不能直接出版。

先提交稿件给编辑部。编辑审核内容、格式、逻辑。作者根据意见修改。审核通过后，稿件入档案库。正式出版发行。

**PR 评审就是这个审稿流程。**

| 出版社概念 | PR 对应 | 说明 |
|-----------|---------|------|
| 提交稿件 | Pull Request | 请审核我的改动 |
| 编辑审稿 | code review | 检查代码 |
| 稿件改动对比 | diff | 原文和新文对比 |
| 审稿意见 | review comment | 直接写在稿件上 |
| 入档案库 | merge | 审核通过后合入 main |

### 2.5 产品经理容易误解什么

**误解一：「改一个字的事，不用走流程」**

真相：任何改动都可能影响其他功能。流程大部分自动化，手动操作只需几分钟。不走流程的风险成本远超流程成本。

**误解二：「Git 是程序员的工具，跟产品没关系」**

真相：Git 是团队协作基础设施，产品经理是参与者。需求变更、文案改动、验收检查都在这个流程里。

**误解三：「需求口头说就行，不用写 Issue」**

真相：口头需求不可追溯。新人离职后口头承诺就没了。Issue 是协作基础设施。

## 3. 开发怎么看这个问题

产品说「改一个字的事」，开发脑子里自动展开：

**流程层面**：
- 这个改动在哪个分支？
- 分支从 main 创建了吗？
- commit 消息写清楚了吗？
- push 到远程了吗？
- PR 创建了吗？谁审核？

**Issue 层面**：
- 这个改动有 Issue 吗？
- Issue 状态是什么？
- Assignee 是谁？
- 验收标准写清楚了吗？

**评审层面**：
- 审稿人是谁？
- 自动测试跑了吗？
- Staging 验证过了吗？
- 产品经理确认了吗？

**技术约束层面**：
- 这个改动会影响哪些模块？
- 有没有共享配置被修改？
- 会不会影响其他分支的开发？
- 需不需要修改数据库？
- 有没有第三方依赖？

开发的关注点不是刁难，而是「防御性设计」。

不走流程的改动，上线出问题、影响用户、紧急回滚——都是开发在救。

没有 Issue 的需求，新人离职后找不到负责人、无法交接——都是开发在查。

没考虑技术约束的改动，改一个字却破坏了共享配置——都是开发在修。

开发看到的是流程安全性和可追溯性。产品一句话，开发要展开十几个问题。
不说清楚，开发只能按最严格的方式处理。

## 4. 怎么做：理解流程、创建 Issue、参与评审

### 4.1 如何理解 Git 流程

**第一步：记住五步流程**

创建分支 → commit → push → PR → merge。

**第二步：理解各阶段时间**

| 改动类型 | 预估时间 |
|---------|---------|
| 小改动（文案、配置） | 1-2 天 |
| 功能迭代 | 3-5 天 |
| 大版本发布 | 1-2 周 |

**第三步：合理安排上线时间**

不要周五下班前说「周一能上线吗」。周四就告诉开发：「这个改动预计下周二上线，来得及吗？」

### 4.2 如何创建 Issue

**Issue 模板示例**：

```
标题：[需求] 手机验证功能优化 - 添加进度提示

描述：
- 当前问题：验证码发送无进度提示，用户等待时不知道状态
- 需求目标：添加发送进度提示，告知用户「正在发送验证码」

验收标准：
- 用户点击发送后，立即显示「正在发送」提示
- 验证码发送完成后，提示消失
- 发送失败时，显示「发送失败，请重试」

优先级：P0（紧急）
Assignee：建议指派 @小明
```

**Issue 创建三要素**：

1. 描述：当前问题 + 需求目标
2. 验收标准：具体、可检查
3. Assignee：指派责任人

### 4.3 如何参与 PR 评审

**产品经理审核三要素**：

| 审核内容 | 产品经理审核 | 开发审核 |
|---------|-------------|---------|
| 需求完整性 | ✅ 审核 | ❌ 不审核 |
| 文案准确性 | ✅ 审核 | ❌ 不审核 |
| 业务逻辑正确性 | ✅ 审核 | ✅ 也审核 |
| 代码质量 | ❌ 不审核 | ✅ 审核 |

**审核技巧**：

1. 搜索关键词定位文案
2. 对照设计稿逐字对比
3. 看 if/else 分支判断逻辑
4. 有疑问直接评论，不要猜测

**审核方式**：

在 PR 页面看 diff。绿色行是新增内容，红色行是删除内容。产品经理看 diff，不用看全文，只看改动部分。

**PR 审核清单**：

```
□ 需求完整性：功能是否完整实现需求
□ 验收标准：验收标准是否全部满足
□ 文案准确性：文案是否与设计稿一致
□ 业务逻辑：流程是否与流程图一致
□ 异常处理：异常情况是否处理正确
□ 用户提示：用户提示是否友好易懂
```

### 4.4 AI 可以如何辅助

**生成 Issue 模板**

提示词示例：
> 请帮我生成一个 Issue 模板，需求是：新增用户注册手机验证功能。

AI 输出框架：标题、描述、验收标准、优先级建议。

**生成 PR 审核清单**

提示词示例：
> 请帮我生成一个产品经理 PR 审核清单，用于审核手机验证功能。

AI 输出：需求完整性检查项、文案检查项、业务逻辑检查项。

**注意**：AI 能给你框架，但验收标准、文案细节要你自己补充。AI 不知道你们团队的流程、设计稿内容。

## 5. 案例复盘

回看老李的两个场景：

**场景一：改一个字也要走流程**

老李想改文案，小明解释了流程的必要性。实习生改一个字导致全站崩溃半小时。

要点：流程不是官僚，是保护机制。任何改动都可能影响其他功能。

**场景二：找不到负责人**

运营找老李改功能，但实习生离职没交接。Git 搜索一小时才找到代码。

要点：没有 Issue，需求不可追溯。Issue 是协作基础设施，不是开发的工作日志。

## 6. 一句话总结

**分支是副本不乱改，Issue 是订单不遗漏，PR 是审核要参与。**

---

| 概念 | 一句话记住 |
|------|------------|
| Git 分支 | 抄写副本，不能直接改原版 |
| Issue | 订单小票，不遗漏、可追溯 |
| PR 评审 | 审稿流程，产品审核需求、文案、业务逻辑 |
