# CH-45 审批与权限实战

> 本章解决一个问题：审批和权限，看起来很简单，为什么上线后总是出问题？

---

## 1. 痛点场景：审批系统上线第一天就乱了

周一上午，采购审批系统刚上线。

小想兴奋地说："搞定！提交→领导审批→通过，三个状态，清清爽爽。"

老李刚喝了一口水，还没来得及回，销售经理的电话就来了。

"审批卡住了！我提交的采购申请三天没动静，领导出差了没人批！"

"我想撤回改一下金额，找不到撤回按钮！"

"财务驳回了，但我不知道能重新提交还是就这样算了？"

小想愣住了："这些...我没设计。"

老李放下水杯："审批不只是提交→通过。还有撤回、超时、驳回后怎么办、金额大了找谁批。这些漏掉，系统上线第一天就乱。"

---

## 2. 权限漏洞：销售经理看到了财务部的客户

下午，CRM系统的权限问题也被发现了。

销售经理点开客户管理，看到了全公司的客户——包括财务部跟进的客户。

财务经理投诉："销售经理怎么能看我们的客户信息？这不合适。"

小想检查配置："我设置了销售经理有客户管理权限，没问题啊。"

老李问："数据权限呢？销售经理能看到哪些范围的数据？"

小想愣住："数据权限？我只设置了功能权限——有没有这个菜单。没管数据范围。"

"这就是问题。"老李说，"权限不只是有和无。还有能看到哪些数据、能做什么操作。漏掉数据权限，销售经理就看到了不该看的。"

---

## 3. 核心概念：审批和权限到底是什么

### 审批流程

审批流程，本质上是**公司决策流程的数字化**。

公司里大大小小的事都要审批：预算、合同、报销、采购。以前是拿着纸质材料找领导签字，领导看了觉得没问题就签，觉得有问题就说不行。

现在有了系统，就在系统里提交，领导在线上看、在线上批。本质上是一样的——只是从线下搬到线上。

所以设计审批系统，首先要搞清楚：公司里有哪些决策流程？谁有权批什么？什么情况找大老板？

### 审批状态机

审批流程的每个阶段，对应一个**状态**。状态之间的转换，就像审批单的"生命周期"。

| 状态 | 类比 | 说明 |
|------|------|------|
| 草稿 | 申请人在写材料 | 还没提交，随时改 |
| 待审批 | 材料写好递给领导 | 等领导看 |
| 已通过 | 领导签字同意 | 可以执行了 |
| 已驳回 | 领导说不行 | 要修改再重新递 |
| 已撤回 | 申请人说算了 | 自己撤回，不批了 |

状态之间的转换，就是审批过程中可能发生的事：提交、通过、驳回、撤回。

### 权限系统

权限系统，本质上是**公司门禁卡的数字化版本**。

公司门禁卡决定你能进哪些门：普通员工能进自己的办公室、会议室；部门经理能进部门会议室、档案室；总经理能进所有区域。

系统权限也一样，决定你在系统里能做什么。普通员工能看自己的数据、提交申请；部门经理能看部门数据、审批申请；系统管理员能配置系统。

所以设计权限系统，先想清楚：系统里的"门"有哪些？谁有权限进？什么数据是"公共区域"，什么数据是"财务室"？

### 权限三维模型

权限不只是"有权限"和"无权限"两种，而是三维的：

| 维度 | 类比 | 说明 |
|------|------|------|
| 功能权限 | 能进哪些门 | 能看到哪些菜单、按钮、页面 |
| 数据权限 | 能看哪些资料 | 能看到全部、本部门、本人数据 |
| 操作权限 | 能做什么事 | 能查看、编辑、删除、审批 |

举例：销售经理的权限

| 维度 | 权限 | 类比说明 |
|------|------|----------|
| 功能权限 | 客户管理、订单管理、报表查看 | 能进客户部、订单部、报表部 |
| 数据权限 | 本部门客户、本部门订单 | 只能看自己部门的档案柜 |
| 操作权限 | 查看、编辑、审批折扣≤10% | 能看资料、能改资料、能批小事 |

## 4. 为什么产品经理容易误解

### 误解一："审批就是提交→通过/驳回，很简单"

很多产品经理第一次设计审批，以为三个状态就够了。

真实世界里的审批，要考虑：

| 要素 | 漏掉会怎样 |
|------|-----------|
| 撤回机制 | 申请人想改没入口，只能找开发删数据 |
| 超时处理 | 审批卡住不动，业务停滞 |
| 驳回后动作 | 申请人不知道能重新提交还是放弃 |
| 多级审批 | 大事小领导批了，出问题没人负责 |
| 会签 | 采购需要财务+采购+使用部门都同意，逻辑不清 |
| 审批记录 | 事后追溯找不到谁批的、什么时候批的 |

误解在哪？只看到"领导签字"这个动作，没看到整个决策流程里的各种情况。

### 误解二："权限就是管理员和普通用户两种"

很多产品经理设计权限，只分"管理员"和"普通用户"。

真实系统的权限是多维的：

| 角色 | 功能权限 | 数据权限 | 操作权限 |
|------|----------|----------|----------|
| 普通销售 | 客户管理 | 自己的客户 | 查看、编辑 |
| 销售经理 | 客户管理+报表 | 本部门客户 | 查看、编辑、审批折扣 |
| 销售总监 | 客户管理+报表+配置 | 全公司客户 | 查看、编辑、删除 |
| 财务人员 | 财务报表 | 全公司财务数据 | 查看 |

误解在哪？只看到"有没有权限"，没看到权限的三维结构——功能、数据、操作。

## 5. 开发视角：技术实现要考虑什么

### 审批流程的技术约束

开发看审批流程，想的不是"有几个状态"，而是：

| 关注点 | 技术实现 |
|--------|----------|
| 状态存储 | 每个审批单的当前状态怎么存？数据库表设计 |
| 状态转换规则 | 从待审批能跳到已通过，但不能跳到已撤回，规则怎么定义？ |
| 并发处理 | 两个人同时审批一个申请怎么办？锁机制 |
| 超时检测 | 怎么检测哪些审批超时了？定时任务扫描 |
| 审批记录 | 每次审批动作都要记录，表怎么设计？ |

产品经理给的状态机，要能回答这些问题。状态定义不完整，开发就要猜；猜错了，上线就出bug。

### 权限系统的技术约束

开发看权限系统，想的不是"有没有菜单"，而是：

| 关注点 | 技术实现 |
|--------|----------|
| 权限存储 | 角色-权限关系怎么存？RBAC表设计 |
| 权限判断 | 用户请求时怎么判断有没有权限？中间件拦截 |
| 数据过滤 | 数据权限怎么实现？SQL查询加WHERE条件 |
| 权限继承 | 销售经理继承销售权限，怎么算？递归查询 |
| 权限变更审计 | 谁改了权限、改成什么，怎么记录？变更日志表 |

产品经理给的权限矩阵，要能回答这些问题。数据权限边界不清，开发就要猜；猜错了，销售经理就看到了财务数据。

## 6. 产品经理应该如何做

### 审批流程设计要素清单

设计审批流程前，先填这张表：

| 要素 | 设计要点 | 示例（采购审批） |
|------|----------|-----------------|
| 审批角色 | 谁有审批权 | 部门经理、总经理、老板 |
| 审批层级 | 单级还是多级 | ≤5万部门经理批，>5万总经理批 |
| 审批规则 | 什么情况触发审批 | 金额、类型、紧急程度 |
| 驳回处理 | 驳回后能做什么 | 可重新编辑后再次提交 |
| 撤回机制 | 待审批时可撤回 | 提交后3天内可撤回 |
| 超时处理 | 自动升级或催办 | 待审批超3天催办，超7天升级 |
| 会签逻辑 | 多人审批逻辑 | 财务+采购+使用部门都通过才算 |
| 审批记录 | 记录审批动作 | 审批人、时间、意见、动作 |

### 权限设计六步法

设计权限系统，按这六步走：

| 步骤 | 内容 | 输出 |
|------|------|------|
| 1.识别角色 | 列出系统所有用户角色 | 角色清单：普通销售、销售经理、销售总监、财务人员、管理员 |
| 2.定义功能权限 | 每个角色能执行什么操作 | 功能权限表 |
| 3.定义数据权限范围 | 每个角色能看到哪些数据 | 数据权限表：全部/本部门/本人/无 |
| 4.设计权限继承规则 | 角色之间有无继承 | 销售经理继承普通销售权限+部门权限 |
| 5.处理特殊角色 | 临时授权、代理审批 | 临时访客权限、代理审批权限 |
| 6.输出权限矩阵 | 整合所有权限定义 | 权限矩阵文档 |

### 审批与权限的关联

审批流程和权限系统是耦合的：

| 审批节点 | 权限问题 |
|----------|----------|
| 谁有权审批 | 审批人角色有审批权限 |
| 能看到哪些申请 | 审批人能看到范围内的申请（数据权限） |
| 审批后能做什么 | 审批人能通过、驳回、撤回（操作权限） |

设计时必须一起考虑：审批流程设计 → 每个节点涉及谁审批 → 权限系统定义谁能审批。

## 7. AI可以如何辅助

### AI辅助审批状态机设计

给AI描述业务场景，让它生成状态机初稿：

```
场景：采购审批，金额≤5万部门经理批，>5万总经理批
要求：生成审批状态机，包含撤回、驳回、超时处理
```

AI会输出状态定义和转换规则。

但要检查：
- 状态是否完整（草稿、待审批、已通过、已驳回、已撤回）
- 转换条件是否明确（从哪能跳到哪）
- 异常处理是否覆盖（超时、驳回后动作）

### AI辅助权限矩阵设计

给AI描述角色和功能，让它生成权限矩阵初稿：

```
角色：普通销售、销售经理、销售总监、财务人员、管理员
功能：客户管理、订单管理、财务报表、系统配置
要求：生成权限矩阵，包含数据权限范围
```

AI会输出角色-功能-数据的权限表。

但要检查：
- 数据权限范围是否定义（全部/本部门/本人/无）
- 权限继承是否合理
- 特殊角色是否考虑（临时授权、代理审批）

## 9. 案例复盘

回看采购审批系统和CRM权限的两个问题：

采购审批系统上线第一天乱了，因为只设计了"提交→通过"。漏掉了撤回、超时、驳回后动作。小想以为审批很简单，没考虑到公司决策流程里的各种情况。

CRM权限漏洞被发现，因为只设置了功能权限，没设置数据权限。销售经理能看到全公司客户，包括不该看的财务部门客户。小想以为权限是"有和无"，没考虑到权限的三维结构。

**要点提炼**：

1. 审批流程 = 公司决策流程数字化。设计前先搞清楚线下怎么走。
2. 权限系统 = 门禁卡数字化。不只是有和无，还有功能、数据、操作三个维度。
3. AI能生成初稿，但状态完整性、数据权限边界需要你来检查。

---

## 10. 本章总结

审批流程是公司决策的数字化，权限系统是组织架构的映射。

设计前先问自己：线下怎么走？门禁卡怎么设计？漏掉这两个问题，系统上线第一天就出事。
