# CH-20 TOGAF 企业架构与 DevOps — 企业架构框架与持续交付

## 1. 痛点场景

周一上午，需求评审会。

小想刚入职一个月，兴奋地拿着需求文档去找开发。

"开发哥，这个需求我已经写好了，什么时候能开工？"

架构师老王看了一眼："先等等，这个需求要过架构评审。"

小想愣住："架构评审是什么？我需求都写完了，为什么还要过什么评审？"

老王："架构评审是看你的需求和企业架构是否对齐。"

小想一头雾水："企业架构又是什么？我只想改个功能……"

老王耐心解释："TOGAF 企业架构有四大域：业务架构、数据架构、应用架构、技术架构。你的需求涉及数据表结构变更，要走数据架构评审；调用新接口，要走应用架构评审。"

小想脱口而出："那我改个小功能也要走这么多评审？这不是官僚主义吗？"

老王没生气："不是官僚。你现在的需求要加一个新字段，但如果这个字段和现有数据模型不兼容，后期集成成本是现在的 10 倍。我们不评审，你上线了，后面数据口径不一致，报表全乱，谁来修？"

小想沉默了。

---

另一个场景。

周三下午，小想发现产品详情页有一处文案写错了。

"开发哥，这个文案有个错字，改一下就行，今天能上线吗？"

开发哥："改一行也要走发布流程，至少两天。"

小想惊讶："改一个字也要两天？这不是浪费时间吗？"

开发哥叹气："任何改动都要经过 CI/CD 流程。CI 是持续集成，CD 是持续交付。你的改动要先自动构建、自动测试，然后部署到测试环境验证，再部署到预发环境，最后才能上生产。"

小想不解："我改的是文案，不会影响代码逻辑啊……"

开发哥："你可能不会，但之前有个实习生「只改文案」，结果把变量名也改了，上线后整个页面空白。我们吃过亏，所以任何改动都要走全流程。"

---

你是不是也遇到过？

架构师说"要过架构评审"，你觉得是官僚主义在拖开工时间。开发说"改一行也要两天"，你觉得是故意刁难。

本章解决两个问题：为什么需求要过架构评审？为什么改一行文案也要走发布流程？

## 2. 核心概念：TOGAF 四大架构域、CI/CD Pipeline

### 2.1 TOGAF 是什么？

**TOGAF = 企业架构框架（The Open Group Architecture Framework）。**

听起来高大上？换个类比就明白了。

企业架构就像城市规划。

不能每栋楼随便盖，要按城市规划来。盖楼前要过规划审批——确认这块地能盖什么、怎么接入水电管网、怎么连通道路。

| 城市规划概念 | 企业架构对应 | 说明 |
|-------------|-------------|------|
| 总体规划 | 企业架构框架（TOGAF） | 定义整个企业的整体框架 |
| 功能分区 | 业务架构 | 商业区、住宅区各有规划 |
| 路网规划 | 应用架构 | 定义道路走向、路口设计 |
| 水电管网 | 数据架构 | 定义数据如何流动、存储 |
| 基础设施标准 | 技术架构 | 建筑材料标准、消防标准等 |
| 规划审批 | 架构评审 | 盖楼前要过规划审批 |
| 违章建筑 | 技术孤岛 | 不按规划建，后期问题一堆 |

### 2.2 四大架构域（BDAT）

TOGAF 把企业架构分成四个域，简称 BDAT：

| 架构域 | 中文名 |管什么 | 评审关注点 |
|-------|--------|--------|-----------|
| Business | 业务架构 | 业务流程、组织结构、业务功能 | 需求是否和现有业务流程对齐 |
| Data | 数据架构 | 数据资产、数据模型、数据流 | 需求是否和现有数据模型兼容 |
| Application | 应用架构 | 应用系统组合、应用交互、应用边界 | 需求是否和现有应用边界清晰 |
| Technology | 技术架构 | 技术基础设施、平台、中间件 | 需求是否符合技术选型标准 |

你的需求过了架构评审，就是在这四个层面都"对齐"了——不会和现有系统打架。

### 2.3 架构评审防什么？

**架构评审防止"技术孤岛"。**

半年前，有个产品经理小张的需求没过架构评审，觉得太麻烦，找关系直接让开发"绕过去"做了。

结果是什么？他加了一个"用户状态"字段，但没对齐数据架构。现有的用户表已经有"状态"字段了，他用的是另一个口径。

报表团队按新口径做报表，运营团队按老口径做报表。两边数据打架，谁也不认谁的。最后 CEO 问为什么两个报表不一样，团队花了两个月去修数据。

小张的需求本来只要两周开发，但后期修复花了两个月。成本是原来的 4 倍。

这就是技术孤岛——各团队自建系统，数据口径不一致，API 格式不统一，后期全是坑。

### 2.4 CI/CD 是什么？

**CI/CD = 持续集成/持续交付（Continuous Integration/Continuous Delivery）。**

听起来复杂？换个类比就明白了。

CI/CD 就像工厂流水线的质检系统。每个环节都有质检站，不合格不放行。

| 工厂流水线概念 | CI/CD 对应 | 说明 |
|---------------|-----------|------|
| 原料入库 | Git Commit/Push | 代码提交到仓库 |
| 原料检验 | Build + Unit Test | 确保代码能构建、基础逻辑正确 |
| 加工组装 | Integration Test | 确保模块能正确协作 |
| 半成品检验 | Deploy to Staging | 在接近生产的环境验证 |
| 成品检验 | UAT 测试 | 人工确认功能符合预期 |
| 出厂检验 | Pre-prod 环境 |最后一道关卡 |
| 产品上架 | Deploy to Production | 对用户可见 |

### 2.5 发布流程各阶段

一个改动从提交到上线，要经过这些阶段：

| 阶段 | 名称 | 时长 | 做什么 |
|------|------|------|--------|
| 1 | Git Commit/Push | 5 分钟 | 提交代码到仓库，触发流水线 |
| 2 | Build | 10 分钟 | 编译代码、打包镜像 |
| 3 | Unit Test | 15 分钟 | 运行单元测试 |
| 4 | Integration Test | 20 分钟 | 运行集成测试 |
| 5 | Deploy to Staging | 10 分钟 | 部署到测试环境 |
| 6 | UAT 测试 | 数小时 | 人工或自动验证功能 |
| 7 | Deploy to Pre-prod | 10 分钟 | 部署到预发环境 |
| 8 | Deploy to Production | 10 分钟 | 上线到生产环境 |

为什么至少两天？

**第一天**：上午提交代码 → 构建 → 自动测试（约1小时）→ 下午部署测试环境 → QA验证（可能半天）。

**第二天**：上午部署预发环境 → 验证 → 下午灰度发布或全量上线。

### 2.6 灰度发布是什么？

**灰度发布 = 先给部分用户使用新版本，逐步扩大范围。**

为什么不能直接全量上线？

去年有个功能全量上线，结果一个十年老用户的数据格式特殊，触发了没测到的 bug。那个用户打开个人主页直接白屏，投诉到客服。

如果灰度发布，那个用户在 1% 的首批用户中。他打开白屏，监控报警，立即回滚，只影响他一个人。其他 99% 用户不受影响。

实际全量上线，所有用户同时看到新版本。虽然只有一个用户触发 bug，但其他用户看到"系统异常"也很紧张。回滚需要时间，回滚期间所有人都在用有问题的版本。

灰度发布的流量递增：1% → 5% → 25% → 50% → 100%。每一步观察没问题，才继续扩大。

### 2.7 产品经理容易误解什么？

**误解一："架构评审是官僚主义"**

真相：架构评审确保新需求和现有系统兼容。不评审就上，后期集成成本可能是现在的 10 倍。就像城市规划审批——不是官僚，是确保你盖的楼能接入城市整体系统。

**误解二："小改动不需要走发布流程"**

真相：任何改动都可能引入问题。之前有个实习生"只改文案"，结果把变量名也改了，上线后整个页面空白。工厂流水线没有"小改动免检"这个概念，CI/CD 也没有。

## 3. 开发怎么看这个问题

产品说"这个需求很简单"，开发脑子里自动展开：

**架构层面**：

- 这个需求涉及哪些数据表？
- 数据模型兼容吗？会不会产生口径冲突？
- 调用哪些接口？接口边界清晰吗？
- 技术选型符合现有标准吗？

**发布层面**：

- 这个改动影响哪些模块？
- 构建会不会失败？
- 测试覆盖够吗？
- 哪些环境需要验证？
- 发布策略是什么？全量还是灰度？

开发的关注点不是刁难，而是"防御性设计"。

架构评审不通过的需求，后期数据打架、接口不统一、维护困难——都是开发在修。发布流程不走的改动，上线出问题、影响用户、紧急回滚——都是开发在救。

**核心认知差**：产品看到的是"功能上线"，开发看到的是架构兼容性和发布安全性。产品一句话，开发要展开十几个问题。不说清楚，开发只能按最严格的方式处理。

## 4. 怎么做：提前对齐、理解流程、合理安排

### 4.1 如何准备架构评审？

**第一步：判断需求涉及哪些架构域**

| 需求类型 | 涉及架构域 | 评审关注 |
|---------|-----------|---------|
| 新增业务流程 | 业务架构 | 流程和现有业务对齐 |
| 数据表结构变更 | 数据架构 | 数据模型兼容 |
| 新增接口调用 | 应用架构 | 应用边界清晰 |
| 技术选型变更 | 技术架构 | 符合技术标准 |

**第二步：准备架构对齐说明**

在需求文档中补充：
- 数据层面：涉及哪些表？字段是否新增？口径是否一致？
- 接口层面：调用哪些系统？接口是否已有？
- 技术层面：是否引入新技术？是否符合现有标准？

**第三步：自查清单**

- [ ] 需求涉及的数据表是否已有？口径是否一致？
- [ ] 调用的接口是否已有？是否需要新增？
- [ ] 技术选型是否符合现有标准？
- [ ] 是否有架构评审会？什么时候？

### 4.2 如何合理安排上线时间？

**第一步：了解团队发布节奏**

每个团队有自己的发布节奏。有的团队每周二、周四发布，有的每天都能发布。先问清楚。

**第二步：预留足够时间**

| 改动类型 | 预估时间 |
|---------|---------|
| 小改动（文案、配置） | 1-2 天 |
| 功能迭代 | 3-5 天 |
| 大版本发布 | 1-2 周 |

**第三步：提前沟通**

不要周五下班前说"周一能上线吗"。周四就告诉开发："这个改动预计下周二上线，来得及吗？"

### 4.3 AI 可以如何辅助？

**生成架构对齐检查清单**

提示词示例：
> 请帮我生成一个架构对齐检查清单，需求是：新增用户积分功能，积分可兑换优惠券。

AI 输出框架：
- 数据层面：是否新增积分表？积分字段口径？
- 接口层面：积分查询接口、兑换接口？
- 业务层面：积分规则和现有业务对齐？

**生成发布流程说明**

提示词示例：
> 请帮我写一个发布流程说明，给非技术背景的产品经理看。

AI 输出：各阶段说明、时间预估、注意事项。

**注意**：AI 能给你框架，但业务规则要你自己补充。AI 不知道你们团队的发布节奏、架构评审频率。

## 5. 一句话总结

**架构评审防孤岛，CI/CD 防翻车。**

需求前过架构评审，确保和现有系统兼容。上线前走发布流程，确保不引入问题。不是官僚，不是刁难——是防止后期爆炸。

---

| 概念 | 一句话记住 |
|------|------------|
| TOGAF | 企业架构框架，四大域（业务/数据/应用/技术） |
| 架构评审 | 确保需求和现有架构对齐，防止技术孤岛 |
| CI/CD | 流水线质检，每个环节有验证 |
| 灰度发布 | 先给部分用户，逐步扩大，控制风险 |
