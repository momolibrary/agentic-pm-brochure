# CH-50 异常与边界陷阱 — 异常流程和边界条件常见的坑

## 一、"完美流程"上线即崩溃

周一上午。小想在群里发了条消息："用户注册功能上线了！"

下午两点。客服群里炸了。

"填了半天信息，页面突然空白了！"
"点击验证码按钮，页面卡住了！"
"注册成功但没进入首页！"
"显示了一堆英文报错！"

小想懵了。"我设计的流程很完美啊，怎么上线就崩？"

老李翻出小想的设计稿。指着那张流程图：
"填写信息 → 验证手机号 → 设置密码 → 注册成功 → 进入首页。"

"你看，这是正常流程。很顺畅。但你漏了一件事。"

小想："漏了什么？"

老李："下雨怎么办？"

小想愣住了。"什么下雨？"

老李笑了。"你设计的这个流程，就像盖房子只考虑晴天。下雨怎么办？屋顶漏水怎么办？台风来了怎么办？"

"网络断了，你的房子就塌了。服务器挂了，你的房子也塌了。用户遇到异常，你的流程就崩了。"

**问题揭示**：只设计正常流程，相当于只给房子设计晴天使用。网络断开、服务器超时、接口失败这些"下雨天"，你的流程都没预案。

上线第一天，客服收到大量投诉。运维被迫手动处理异常。

## 二、异常陷阱与边界五虎

### 只考虑正常流程，后果有多严重？

很多产品经理有个误解：异常情况很少发生，不需要花太多精力。

**这是错的。**

工程经验表明，异常处理代码量往往超过正常流程代码量。用户投诉数据统计，80%的投诉源于异常场景处理不当。

用老李的话说："正常流程只占20%。异常场景占了80%。你设计的'完美流程'，只覆盖了那20%。"

### 异常流程有哪些坑？

老李拿出一张对比表：

| 正常流程（晴天设计） | 异常流程（下雨设计） |
|---------------------|---------------------|
| 用户登录 → 进入首页 | 网络断开 → 页面空白 → 用户不知道发生了什么 |
| 点击提交 → 保存成功 | 服务器超时 → 页面卡住 → 用户反复点击 |
| 搜索商品 → 显示列表 | 无结果 → 页面空空 → 用户不知道是没有还是系统错误 |

"你看，正常流程很顺畅。但每一步都可能'下雨'。"

网络断了怎么办？服务器挂了怎么办？数据不存在怎么办？这些都需要设计。

### 边界五虎：最常被忽略的边界条件

小想负责设计一个订单管理系统。老李验收时说："功能不错。但边界条件考虑了吗？"

小想："边界？什么边界？"

老李："并发、大数据量、极端输入、时区、精度。这五个我叫它们'边界五虎'。不防好它们，上线就出事故。"

老李用一个类比解释：

"忽略边界条件的产品设计，就像考驾照只考晴天路况。驾校只教平直道路、白天行驶、正常速度。但雨天湿滑路面、夜间灯光使用、拥堵时低速行驶——这些都不教。"

"新手拿到驾照，遇到雨天就出事故。产品不考虑边界，上线遇到边界就崩溃。"

**边界五虎详解**：

| 边界类型 | 典型问题 | 后果 |
|---------|---------|------|
| **并发问题** | 两人同时修改同一订单 | 数据覆盖、数据丢失 |
| **大数据量** | 导出10万条数据 | 内存溢出、超时崩溃 |
| **极端输入** | 用户输入5000字备注 | 存储失败、内容截断丢失 |
| **时区问题** | 海外用户下单时间显示凌晨3点 | 业务时间计算错误 |
| **精度问题** | 金额用浮点数计算 | 0.1×3=0.29元，总价错误 |

小想被逐一击中。

"两人同时编辑订单？我没想过。"
"导出10万条？我只测了几十条。"
"用户输入5000字？我以为没人会这样。"
"时区？我不知道海外用户。"
"精度？我以为0.1+0.1+0.1=0.3..."

老李叹气。"这就是边界五虎的连环陷阱。你不防，它们就咬你。"

### 产品经理容易误解什么？

**误解一：异常情况很少发生，不需要花太多精力**

正确理解：异常处理是系统稳定性的核心。Murphy's Law——能出错的一定会出错。上线后，异常场景才是真正考验系统的地方。

**误解二：边界条件是测试的事**

正确理解：边界条件应在需求设计阶段就识别和处理。测试只是验证。你设计时不定义，开发就不写。上线后用户用真实数据告诉你答案——崩了。

## 三、开发怎么看这个问题？

开发看需求，想的不是"流程有多顺畅"，而是：

- 网络断了怎么处理？
- 超时怎么提示？
- 并发冲突怎么办？
- 数据量大了会不会崩？

老李问张开发："你写这个注册功能，网络断开怎么处理？"

张开发挠挠头："我...没定义。网络断了页面就空白，用户也不知道发生了什么。"

老李又问："两人同时编辑订单，你怎么处理？"

张开发："我没想过。后保存的覆盖前保存的。数据可能丢失。"

"这就是验收标准缺失的后果。"老李说。"开发需要明确的异常处理规则。你没定义，他就按自己理解来。"

你定义了异常流程，他才写异常处理。你标注了边界条件，他才做边界校验。

## 四、产品经理应该怎么做？

### 异常处理 Checklist

老李给小想一张清单：

```
□ 网络断开时：页面是否有提示？是否有重试按钮？
□ 超时时：是否有加载动画？是否自动重试？
□ 权限不足时：是否有友好提示？是否引导解决？
□ 数据不存在时：是否有默认展示？是否有引导提示？
□ 第三方接口失败时：是否有降级方案？是否有错误上报？
```

**使用方法**：每个需求设计完成后，逐项检查。漏一项就补一项。

### 边界条件 Checklist

老李又给一张清单：

```
□ 并发问题：两人同时操作时是否加锁？是否提示冲突？
□ 大数据量：超大数据量时是否分页？是否异步处理？
□ 极端输入：空值、超长、特殊字符是否校验？
□ 时区问题：跨时区时间是否统一存储、本地显示？
□ 精度问题：金额计算是否用定点数？是否正确舍入？
```

**追问方法**：老李有个习惯。每个需求他都问三个问题：最大多少？最小多少？超出怎么办？

这三个问题能发现90%的边界隐患。

### AI可以如何辅助？

小想准备上线用户管理系统。老李让他做一次异常和边界检查。

小想："检查什么？我觉得流程都设计好了。"

老李："上次上线即崩溃，就是因为你靠'感觉'检查。这次用清单。"

小想把需求发给AI：

> 我有一个用户管理系统，包含登录、查询用户列表、编辑用户信息、批量导入用户。请帮我生成异常场景清单和边界条件清单。

AI返回了12项遗漏：

| 遗漏项 | 原设计 | 补充设计 |
|-------|-------|---------|
| 网络断开提示 | 页面空白 | 提示"网络异常" + 重试按钮 |
| 登录超时处理 | 页面卡住 | 显示加载动画 + 自动重试 |
| 查询无结果 | 空列表 | 显示"暂无数据" + 添加引导 |
| 用户名长度限制 | 无限制 | 前端限制50字符 |
| 特殊字符校验 | 无校验 | 前端实时校验 + 错误提示 |
| 并发编辑冲突 | 无处理 | 乐观锁 + 冲突提示 |
| 批量导入进度 | 无提示 | 进度条 + 完成邮件通知 |

上线后一周，客服投诉量比上次减少90%。

老李："这就是检查清单的作用。你不再靠感觉，而是靠系统。"

**记住**：AI生成的是"草稿"。具体边界值、预期表现，需要你来补。

### Checklist

- [ ] 异常处理清单逐项检查（网络/超时/权限/数据不存在/接口失败）
- [ ] 边界条件清单逐项检查（并发/大数据量/极端输入/时区/精度）
- [ ] 每个需求追问三个问题：最大多少？最小多少？超出怎么办？
- [ ] AI生成的异常场景已人工审核补充

## 五、案例复盘

回看开头场景：小想设计用户注册流程，只写了正常流程。上线第一天，客服收到大量投诉。网络断开、服务器超时、首页加载失败——这些异常都没处理。

再看边界五虎场景：小想设计订单管理系统，没考虑边界条件。两人同时编辑订单，数据丢失。导出10万条数据，系统崩溃。用户输入5000字备注，内容丢失。海外用户看到凌晨3点下单时间。金额计算0.29元而非0.3元。

再看清单拯救场景：小想用异常清单+边界清单逐项检查。发现12处遗漏，上线后投诉减少90%。

**三个案例的共同根因**：异常流程未设计，边界条件未标注。

**洞察**：系统性发现异常和边界遗漏，就像飞行员起飞前检查清单。异常清单+边界清单+AI辅助生成，让检查不再靠感觉，而是靠系统。

## 六、一句话总结

**正常流程是晴天设计，异常流程是下雨设计，边界条件是护栏设计。**

三者缺一，上线就崩。
