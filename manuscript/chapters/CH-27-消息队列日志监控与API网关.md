# CH-27 消息队列、日志监控与 API 网关 — 系统中间件认知

## 1. 痛点场景

周五下午，需求评审会。

老李刚说完订单系统的需求，开发主管就问："订单创建后要发消息给库存系统，用消息队列还是直接调接口？"

老李愣了。"什么消息队列？直接调接口不就行了吗？"

开发主管摇头："高峰期一万个订单同时创建，库存系统扛不住。用消息队列排队处理。"

老李心里犯嘀咕：调接口不是最快吗？为什么要排队？

会后，老李路过开发工位，看到小想盯着大屏幕上的曲线图发呆。

"这是什么？"

"Grafana 监控大屏。"小想指着屏幕上一条红色的线，"这个是错误率，刚上线五分钟就飙起来了。开发让我看日志定位问题。"

"日志在哪？"

小想打开一个黑底白字的终端窗口，密密麻麻的输出滚得飞快。

老李问："开发说'这个接口要过网关'，网关又是什么？"

小想摇头："我也不懂，开发说这是统一入口，所有外部请求都要过这道门。"

老李叹气：系统里这些东西，看不见摸不着，开发总挂在嘴边，产品经理却一头雾水。

**你是不是也有同样的困惑？**

本章解决三个问题：消息队列是什么、为什么要用；日志和监控有什么区别；API 网关是干什么的。

---

## 2. 核心概念：消息队列、日志监控、API 网关

### 2.1 消息队列

**消息队列是系统间异步通信的管道。**

类比：餐厅取号系统。

高峰期一百个顾客同时来吃饭。如果都直接往厨房挤，厨房会被堵死。餐厅的做法是：顾客取号排队，厨房按顺序一个个做。

消息队列就是这个"取号系统"。

生产者（订单系统）发出消息，消息排队进入队列。消费者（库存系统）按顺序一个个处理。高峰期的请求不会直接压垮下游系统，而是排队等待。

**两个核心作用：**

| 作用 | 说明 |
|------|------|
| 削峰 | 高峰期请求排队，不直接冲击下游 |
| 解耦 | 生产者发完就走，不用等消费者处理完 |

**什么时候用消息队列？**

| 场景 | 用消息队列 | 原因 |
|------|-----------|------|
| 发短信通知 | ✅ | 不需要即时，晚点发也行 |
| 订单支付确认 | ❌ | 需要立刻知道结果，用 API |
| 数据同步 | ✅ | 批量处理，不用实时 |

产品经理常犯的误解：以为"直接调接口最快，消息队列多此一举"。真相是：直接调接口在高并发时会压垮系统。消息队列用排队换稳定。

**常见消息队列产品：**

| 产品 | 特点 |
|------|------|
| Kafka | 高吞吐，适合大数据场景 |
| RabbitMQ | 功能丰富，适合业务场景 |
| RocketMQ | 阿里开源，适合电商场景 |

产品经理不需要记住这些产品细节，但需要知道：不同场景用不同队列，开发会根据业务特点选择。

### 2.2 日志与监控系统

**日志是记录系统运行过程中每一步发生的事的记录。**

类比：飞机黑匣子。

飞机出事故后，调查人员打开黑匣子，查看飞行过程中的每一秒发生了什么。日志就是系统的黑匣子。

出了 Bug，开发第一反应是"看日志"。因为日志记录了系统运行时的每一步细节——谁请求了什么、处理到哪一步、报了什么错。

**监控是持续观察系统运行状态的工具。**

类比：医院心电监护仪。

医生看心电监护仪，不需要知道病人每个心跳的具体波形，只需要知道心率是否正常、有没有异常波动。

监控系统（如 Prometheus + Grafana）实时展示系统的健康状况：请求量多少、响应时间多少、错误率多少。发现问题立刻告警。

**日志和监控的区别：**

| 工具 | 作用 | 使用场景 |
|------|------|----------|
| 日志 | 记录每一步细节 | 定位具体问题 |
| 监控 | 观察整体健康状况 | 发现系统异常 |

开头故事里的小想，看到 Grafana 大屏上错误率飙红，就需要去日志里找具体是哪个请求、哪一步出了错。

### 2.3 API 网关

**API 网关是所有外部请求进入系统的统一入口。**

类比：小区门卫。

所有访客进小区，都要先在门口登记。门卫验身份、记录来访信息、控制进出流量。

API 网关就是系统的"门卫"。所有外部请求（如手机 App、网页）要访问后端服务，必须先过网关。

**网关的三个职责：**

| 职责 | 说明 |
|------|------|
| 路由 | 把请求分发到对应的后端服务 |
| 鉴权 | 验证请求者身份，拒绝未授权请求 |
| 限流 | 控制请求流量，防止系统被压垮 |

为什么不让外部直接访问后端？

直接访问后端，每个后端服务都要自己验身份、控流量。漏了一个就出安全问题。统一入口便于管理——一处把关，全局安全。

产品经理常犯的误解：听到"要过网关"觉得是开发在添麻烦。真相是：网关保护系统安全，统一控制流量，不是多此一举。

## 3. 开发怎么看这个问题

产品说"订单创建后通知库存"，开发脑子里自动展开：

**关于消息队列：**
- 高峰期并发多少？直接调接口扛得住吗？
- 库存系统处理速度如何？会不会被压垮？
- 发消息失败怎么办？重试还是暂存？
- 消息丢了怎么处理？需要持久化吗？

**关于日志监控：**
- 关键操作有没有记日志？出了问题能定位吗？
- 日志格式统一吗？能不能搜索？
- 监控指标有哪些？能及时发现异常吗？
- 告警阈值是多少？什么时候触发告警？
- 告警发给谁？怎么处理？

**关于 API 网关：**
- 这个接口需要什么权限？谁可以调用？
- 限流阈值多少？高峰期会不会被拦截？
- 调用方是谁？需要走网关还是内部直接调用？
- 网关本身会不会成为瓶颈？

开发说的不是刁难，是"防御性设计"。消息队列防止高峰崩溃、日志监控帮助定位问题、网关保护系统安全。

**常见坑点：**

| 坑点 | 后果 |
|------|------|
| 高峰期没用消息队列 | 库存系统被压垮，订单全部失败 |
| 没记日志 | 出了问题只能靠猜，定位半天 |
| 没配监控告警 | 系统挂了半小时才发现，用户已经投诉 |
| 网关没配限流 | 一个恶意调用方把系统打满，正常用户访问不了 |

产品经理和开发的信息差：产品看到的是"功能能做吗"，开发看到的是"系统稳吗"。你只说功能，不说场景，开发只能自己判断技术方案。

## 4. 怎么做：异步场景识别、监控需求、网关需求

### 4.1 异步场景识别

判断一个场景是否适合用消息队列：

```
这个场景需要实时结果吗？
  └─ 是 → API调用（同步）
  └─ 否 → 用消息队列（异步）
```

需求中标注清楚：

| 功能 | 方式 | 说明 |
|------|------|------|
| 订单支付 | API | 需立刻确认结果 |
| 发货通知 | 消息队列 | 允许延迟 |
| 短信提醒 | 消息队列 | 不必即时 |

### 4.2 日志监控需求

需求中明确监控指标：

| 指标 | 建议值 | 说明 |
|------|--------|------|
| 响应时间 | <3秒 | 超过需告警 |
| 错误率 | <1% | 飙高需定位 |
| 并发上限 | 标明数值 | 高峰期参考 |

告诉开发：关键操作要记日志。出了问题能追溯。

### 4.3 API 网关需求

需求中说明：

| 内容 | 说明 |
|------|------|
| 调用方是谁 | App用户、第三方系统、内部服务 |
| 需要什么权限 | 登录用户、特定角色、开放接口 |
| 限流建议 | 高峰期预期流量 |

### 4.4 自查清单

**消息队列**
- [ ] 判断场景是否适合同步/异步
- [ ] 需求中标注通信方式

**日志监控**
- [ ] 明确关键操作的日志需求
- [ ] 定义监控指标和告警阈值

**API 网关**
- [ ] 说明调用方和权限要求
- [ ] 提供流量预期供开发参考

### 4.5 AI 辅助方法

让 AI 帮你判断场景适合同步还是异步：

提示词示例：
> 我有一个功能：用户下单后发送短信通知。请判断这个场景适合用同步 API 还是异步消息队列，并给出理由。

AI 给建议，你做判断。

**AI 还能帮你生成监控指标清单：**

提示词示例：
> 我要上线一个订单系统，请帮我列出需要监控的指标，包括响应时间、错误率、并发等，并给出建议阈值。

### 4.6 案例复盘

开头故事里，老李不理解消息队列、日志监控、网关的作用。

开发说"高峰期用消息队列排队"，老李才明白：直接调接口在高并发时会压垮系统。

小想看到 Grafana 大屏上的红色曲线，才知道：监控是系统的"心电图"，日志是系统的"黑匣子"。两者配合才能定位问题。

老李听到"接口要过网关"，以为是添麻烦。后来才明白：网关是系统的"门卫"，统一把守入口，一处把关全局安全。
