# CH-09 活动定义与流程架构 — 活动颗粒度与作业指导书

## 1. 痛点场景

"审批流程画好了。"

小想把流程图发给老李。三个框，两条线：提交 → 审批 → 完成。

老李看了一眼，眉头皱起来："你这叫流程？"

小想不解："不就是审批吗？提交了审一下，审完就完了。"

老李追问："谁提交？在哪提交？用什么系统？"

小想愣了。

"审批人在哪个系统操作？审批时要看什么？输出什么？是不是决策点？"

小想开始冒汗。

老李的追问没停："提交后数据存哪？审批时怎么判断通过？驳回后怎么处理？审批人和提交人是一个人吗？"

**你是不是也遇到过？**

开发总嫌流程图"不够细"，追着一连串问题不放。本章就来解决这个问题：为什么"三个框两条线"对开发来说根本不够用，以及如何按 TOGAF 标准把流程定义到开发能用的颗粒度。

---

## 2. 核心概念：活动、流程架构、作业指导书

要理解老李为什么追问，先搞清楚几个概念。

### 2.1 活动是什么

**活动是流程的最细颗粒度执行单元。**

就像菜谱不只是"切菜 → 炒菜 → 装盘"。每一步要写清楚：用什么刀、切什么形状、开多大火、放什么料、炒多久、什么时候出锅。

老李追问的那些问题，就是在逼小想把"审批"拆成真正的"活动"。

每个活动必须关联六要素：

| 要素 | 含义 | 示例 |
|------|------|------|
| 所属系统 | 在哪个系统操作 | OA审批系统 |
| 业务模块 | 属于哪个功能模块 | 费用审批模块 |
| 操作角色 | 谁来执行 | 财务专员 |
| 决策点 | 是否需要判断 | 是 — 审批通过/驳回 |
| 输入物 | 执行前需要什么 | 费用申请单、附件 |
| 输出物 | 执行后产生什么 | 审批记录、状态变更 |

"提交 → 审批 → 完成"这三步，每一步都是一个活动。每个活动都要把这六要素写清楚，开发才知道怎么写代码。

### 2.2 流程架构是什么

**流程不是一张图就完了。**

一本菜谱不只有一道菜的做法，而是整个厨房的运作体系。

流程有层级关系：

```
单个活动 → 详细流程 → 作业指导书 → 流程架构 → 业务域闭环
```

| 层级 | 内容 | 示例 |
|------|------|------|
| 活动 | 最小执行单元 | "审批员在OA系统点击通过" |
| 详细流程 | 多个活动按顺序组成 | 提交 → 初审 → 复审 → 归档 |
| 作业指导书（SOP） | 每个活动的详细操作步骤 | "打开OA → 找到待办 → 查看申请 → 点击通过/驳回" |
| 流程架构 | 多个流程的组织结构 | 费用审批流程、合同审批流程、报销流程拼成审批体系 |
| 业务域闭环 | 完整的业务运行体系 | 财务管理域 = 审批 + 记账 + 报表 + 对账 |

老板问"业务流程全景图在哪"，要的就是流程架构那一层。不是一张流程图，而是整个业务的运行体系。

### 2.3 TOGAF 是什么

**TOGAF 是企业架构框架，由 OpenGroup 发布。**

它把企业架构分成四层：业务架构、数据架构、应用架构、技术架构。流程建模属于业务架构层。

TOGAF 要求把活动定义到"可执行"的颗粒度，就是上面说的六要素。很多大公司（华为、一汽）都用这套方法做流程治理。

## 3. 开发怎么看这个问题

产品说"画个流程图"，开发脑子里自动展开：

**第一层：活动拆分**
- "审批"是一个活动还是多个？（初审、复审、终审）
- 每个活动谁来执行？（角色）
- 在哪个系统操作？（系统）

**第二层：活动要素**
- 输入什么数据？（申请单ID、申请人信息、费用金额）
- 输出什么数据？（审批结果、审批时间、审批意见）
- 是不是决策点？（要判断通过/驳回）
- 判断规则是什么？（金额小于500自动过，大于500要复审）

**第三层：系统实现**
- 数据怎么存？（审批记录表）
- 状态怎么流转？（待审批 → 已通过/已驳回）
- 异常怎么处理？（审批超时、审批人离职）
- 权限怎么控制？（只有审批人能看）

开发要的不是"流程图"，是"可执行的流程定义"。每个活动都要清楚到能写代码的程度。

## 4. 怎么做：从三步到完整活动定义

### Step 1：把步骤拆成活动

小想的"提交 → 审批 → 完成"拆成：

| 步骤 | 活动 | 是否决策点 |
|------|------|-----------|
| 提交 | 申请提交 | 否 |
| 审批 | 初审审批 | 是 |
| 审批 | 复审审批（金额>500） | 是 |
| 完成 | 审批归档 | 否 |

### Step 2：补齐每个活动的六要素

以"初审审批"为例：

| 要素 | 定义 |
|------|------|
| 所属系统 | OA审批系统 |
| 业务模块 | 费用审批 |
| 操作角色 | 部门主管 |
| 决策点 | 是 — 判断通过/驳回 |
| 输入物 | 费用申请单、附件材料 |
| 输出物 | 审批记录（含审批意见、审批时间） |

### Step 3：画出活动间的数据流转

```
┌─────────┐    申请单ID     ┌─────────┐    审批结果    ┌─────────┐
│ 申请提交 │ ───────────────→ │ 初审审批 │ ───────────────→ │ 归档记录 │
└─────────┘                  └─────────┘                  └─────────┘
                                  │
                                  │ 金额>500时
                                  ↓
                             ┌─────────┐
                             │ 复审审批 │
                             └─────────┘
```

### Step 4：建立流程-SOP 层级

流程定义完成后，还要有作业指导书（SOP）。告诉执行人具体怎么操作。

审批活动的 SOP 示例：
> 1. 登录OA系统
> 2. 进入"待办审批"列表
> 3. 点击申请单查看详情
> 4. 核对费用金额、附件材料
> 5. 判断：金额≤500且材料齐全 → 点击"通过"
> 6. 判断：金额>500或材料缺失 → 点击"驳回"，填写驳回原因
> 7. 审批结果自动通知申请人

### Step 5：AI 辅助方法

提示词示例：
> 请帮我拆解"费用审批流程"的活动定义。审批包含初审和复审两个环节。请按 TOGAF 活动六要素（系统、模块、角色、决策点、输入、输出）列出每个活动的定义。

AI 会给一个框架。你补业务规则：
- 判断条件（金额阈值、材料要求）
- 异常处理（审批超时、审批人离职）
- 权限边界（跨部门审批规则）

### Step 6：自查清单

**活动要素**
- [ ] 每个活动标注系统了吗？
- [ ] 每个活动标注角色了吗？
- [ ] 每个活动标注输入/输出了吗？
- [ ] 决策点标注判断条件了吗？

**流程层级**
- [ ] 活动之间的数据流转画了吗？
- [ ] 有对应的 SOP 吗？
- [ ] 能拼成流程架构吗？

**异常覆盖**
- [ ] 驳回后怎么处理？
- [ ] 审批人离职怎么办？
- [ ] 超时怎么办？

---

## 5. 一句话总结

**流程的精度决定了系统的精度。**

三个框两条线不是流程，是"大意"。按活动六要素定义，开发才能动手写代码。

---

| 概念 | 一句话记住 |
|------|------------|
| 活动 | 流程的最小执行单元，要写清六要素 |
| 六要素 | 系统、模块、角色、决策点、输入、输出 |
| SOP | 每个活动的详细操作步骤 |
| 流程架构 | 多个流程拼成的业务运行体系 |
