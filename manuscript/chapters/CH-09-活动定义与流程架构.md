# CH-09 活动定义与流程架构 — 活动颗粒度与作业指导书

## 1. 痛点场景

"审批流程画好了。"

小想把流程图发给老李。三个框，两条线：提交 → 审批 → 完成。

老李看了一眼，眉头皱起来："你这叫流程？"

小想不解："不就是审批吗？提交了审一下，审完就完了。"

老李追问："谁提交？在哪提交？用什么系统？"

小想愣了。

"审批人在哪个系统操作？审批时要看什么？输出什么？是不是决策点？"

小想开始冒汗。

老李的追问没停："提交后数据存哪？审批时怎么判断通过？驳回后怎么处理？审批人和提交人是一个人吗？"

**你是不是也遇到过？**

开发总嫌流程图"不够细"，追着一连串问题不放。本章就来解决这个问题：为什么"三个框两条线"对开发来说根本不够用，以及如何按 TOGAF 标准把流程定义到开发能用的颗粒度。

---

## 2. 核心概念：活动、流程架构、作业指导书

要理解老李为什么追问，先搞清楚几个概念。

### 2.1 活动是什么

**活动是流程的最细颗粒度执行单元。**

就像菜谱不只是"切菜 → 炒菜 → 装盘"。每一步要写清楚：用什么刀、切什么形状、开多大火、放什么料、炒多久、什么时候出锅。

老李追问的那些问题，就是在逼小想把"审批"拆成真正的"活动"。

每个活动必须关联六要素：

| 要素 | 含义 | 示例 |
|------|------|------|
| 所属系统 | 在哪个系统操作 | OA审批系统 |
| 业务模块 | 属于哪个功能模块 | 费用审批模块 |
| 操作角色 | 谁来执行 | 财务专员 |
| 决策点 | 是否需要判断 | 是 — 审批通过/驳回 |
| 输入物 | 执行前需要什么 | 费用申请单、附件 |
| 输出物 | 执行后产生什么 | 审批记录、状态变更 |

"提交 → 审批 → 完成"这三步，每一步都是一个活动。每个活动都要把这六要素写清楚，开发才知道怎么写代码。

### 2.2 流程架构是什么

**流程不是一张图就完了。**

一本菜谱不只有一道菜的做法，而是整个厨房的运作体系。

流程有层级关系：

```
单个活动 → 详细流程 → 作业指导书 → 流程架构 → 业务域闭环
```

| 层级 | 内容 | 示例 |
|------|------|------|
| 活动 | 最小执行单元 | "审批员在OA系统点击通过" |
| 详细流程 | 多个活动按顺序组成 | 提交 → 初审 → 复审 → 归档 |
| 作业指导书（SOP） | 每个活动的详细操作步骤 | "打开OA → 找到待办 → 查看申请 → 点击通过/驳回" |
| 流程架构 | 多个流程的组织结构 | 费用审批流程、合同审批流程、报销流程拼成审批体系 |
| 业务域闭环 | 完整的业务运行体系 | 财务管理域 = 审批 + 记账 + 报表 + 对账 |

老板问"业务流程全景图在哪"，要的就是流程架构那一层。不是一张流程图，而是整个业务的运行体系。

### 2.3 SOP（作业指导书）是什么

**SOP 是 Standard Operating Procedure 的缩写，译为"作业指导书"或"标准操作流程"。**

它是对单个活动或单个流程的详细操作说明。告诉执行人"具体怎么干"。

流程图告诉你"先做什么再做什么"，SOP 告诉你"每个步骤具体怎么操作"。

比如审批活动的 SOP：
> 1. 登录OA系统
> 2. 进入"待办审批"列表
> 3. 点击申请单查看详情
> 4. 核对费用金额、附件材料
> 5. 判断：金额≤500且材料齐全 → 点击"通过"
> 6. 判断：金额>500或材料缺失 → 点击"驳回"，填写驳回原因

有了 SOP，新员工也能照着操作。流程图是给设计者看的，SOP 是给执行者看的。

### 2.4 TOGAF 是什么

**TOGAF 是企业架构框架，由 OpenGroup 发布，全称 The Open Group Architecture Framework。**

它把企业架构分成四层：业务架构、数据架构、应用架构、技术架构。流程建模属于业务架构层。

TOGAF 要求把活动定义到"可执行"的颗粒度，就是上面说的六要素。很多大公司（华为、一汽）都用这套方法做流程治理。

## 3. PM 常见误解

### 误解一：画框连线就是流程

小想以为画三个框、连两条线，就"完成流程设计了"。

实际上，那只是流程的轮廓，不是流程的定义。

就像菜谱写"切菜 → 炒菜 → 装盘"，看起来完整，但实际操作的人不知道用什么刀、开多大火。真正的流程定义要细化到每个活动的六要素。

老李追问的那些问题，就是在逼小想把轮廓变成定义。

### 误解二：流程定义一次就够了

小想以为流程画完就"锁定"了，以后不用再改。

**流程会随业务演进，定义必须同步更新。**

业务规则变了（审批金额阈值从 500 改成 1000），流程的判断条件就要变。组织架构变了（审批人离职或换岗），流程的角色定义就要变。业务规模变了（从单部门审批变成跨部门审批），流程的架构就要调整。

很多线上 Bug 不是代码问题，而是流程变了、系统没跟上。

老李常说："流程不是一次性的事。你画完流程图的那天，只是流程演进的起点。"

## 4. 开发怎么看这个问题

产品说"画个流程图"，开发脑子里自动展开三层。

**第一层：活动拆分决策**

"审批"到底是一个活动还是多个？

开发要判断：
- 初审和复审是同一个人操作吗？→ 同一个人，可能合并；不同人，必须拆开
- 初审和复审的判断规则一样吗？→ 规则一样，可能合并；规则不同，必须拆开
- 初审和复审的系统入口一样吗？→ 同一个入口，可能合并；不同入口，必须拆开

这个判断影响代码结构。合并成一个活动，后台只要一个审批接口；拆成两个活动，要有两个接口、两种状态流转。

**第二层：活动要素展开**

每个活动要问六个问题：
- 输入什么数据？（申请单ID、申请人信息、费用金额、附件列表）
- 输出什么数据？（审批结果、审批时间、审批意见、下一审批人）
- 谁来执行？（角色ID、权限校验）
- 是不是决策点？（有判断逻辑 → 需要规则引擎或硬编码规则）
- 判断规则是什么？（金额阈值、材料完整性检查、跨部门规则）
- 在哪个系统？（影响接口路径、数据表选择）

这六要素直接影响数据库设计。输入物对应查询字段，输出物对应写入字段，决策点对应状态流转逻辑。

**第三层：系统实现约束**

活动要素确定后，开发要考虑：
- **状态流转**：审批通过 → 待复审/已归档；审批驳回 → 已驳回；审批超时 → 待处理 → 怎么设计状态机？
- **异常处理**：审批人离职 → 流程卡死，怎么自动转交？审批超时 → 怎么提醒？多人同时审批 → 怎么防冲突？
- **权限控制**：只有审批人能看申请单详情？申请人能看审批进度？驳回后申请人能修改再提交？
- **数据一致性**：审批结果写入审批记录表，同时更新申请单状态 → 两个表怎么保证同步？

这些实现约束，产品不提，开发也要考虑。但产品提前想到，开发就不用反复追问。

开发要的不是"流程图"，是"可执行的流程定义"。每个活动都要清楚到能写代码的程度。

## 5. 怎么做：从三步到完整活动定义

### Step 1：把步骤拆成活动

小想的"提交 → 审批 → 完成"拆成：

| 步骤 | 活动 | 是否决策点 |
|------|------|-----------|
| 提交 | 申请提交 | 否 |
| 审批 | 初审审批 | 是 |
| 审批 | 复审审批（金额>500） | 是 |
| 完成 | 审批归档 | 否 |

### Step 2：补齐每个活动的六要素

以"初审审批"为例：

| 要素 | 定义 |
|------|------|
| 所属系统 | OA审批系统 |
| 业务模块 | 费用审批 |
| 操作角色 | 部门主管 |
| 决策点 | 是 — 判断通过/驳回 |
| 输入物 | 费用申请单、附件材料 |
| 输出物 | 审批记录（含审批意见、审批时间） |

### Step 3：画出活动间的数据流转

```
┌─────────┐    申请单ID     ┌─────────┐    审批结果    ┌─────────┐
│ 申请提交 │ ───────────────→ │ 初审审批 │ ───────────────→ │ 归档记录 │
└─────────┘                  └─────────┘                  └─────────┘
                                  │
                                  │ 金额>500时
                                  ↓
                             ┌─────────┐
                             │ 复审审批 │
                             └─────────┘
```

### Step 4：编写 SOP

流程定义完成后，还要有作业指导书（SOP）。告诉执行人具体怎么操作。

初审审批的 SOP 示例：
> 1. 登录OA系统
> 2. 进入"待办审批"列表
> 3. 点击申请单查看详情
> 4. 核对费用金额、附件材料
> 5. 判断：金额≤500且材料齐全 → 点击"通过"
> 6. 判断：金额>500或材料缺失 → 点击"驳回"，填写驳回原因
> 7. 审批结果自动通知申请人

### Step 5：AI 辅助方法

提示词示例：
> 请帮我拆解"费用审批流程"的活动定义。审批包含初审和复审两个环节。请按 TOGAF 活动六要素（系统、模块、角色、决策点、输入、输出）列出每个活动的定义。

AI 会给一个框架。你补业务规则：
- 判断条件（金额阈值、材料要求）
- 异常处理（审批超时、审批人离职）
- 权限边界（跨部门审批规则）

### Step 6：自查清单

**活动要素**
- [ ] 每个活动标注系统了吗？
- [ ] 每个活动标注角色了吗？
- [ ] 每个活动标注输入/输出了吗？
- [ ] 决策点标注判断条件了吗？

**流程层级**
- [ ] 活动之间的数据流转画了吗？
- [ ] 有对应的 SOP 吗？
- [ ] 能拼成流程架构吗？

**异常覆盖**
- [ ] 驳回后怎么处理？
- [ ] 审批人离职怎么办？
- [ ] 超时怎么办？

## 6. 案例复盘：小想的成长

小想第一次交的"三步流程图"，老李追问了六个问题。

小想第二次交的版本，补了角色和系统，但漏了输入输出。

老李又追问："初审时看什么？复审时输出什么？"

小想第三次交的版本，完整定义了六要素。

老李终于点头："这回开发能动手了。"

小想复盘：
> "我以为画框就是流程，其实只是轮廓。真正的流程定义，要细化到每个活动在哪个系统、谁来操作、输入输出什么、怎么判断。开发要的不是流程图，是能写代码的流程定义。"

老李补充：
> "流程还会演进。今天定义完了，明天业务规则变了，还得同步更新。流程不是一次性的事，画完那天只是起点。"

## 7. 一句话总结

**流程的精度决定了系统的精度。**

三个框两条线不是流程，是"大意"。按活动六要素定义，开发才能动手写代码。

---

| 概念 | 一句话记住 |
|------|------------|
| 活动 | 流程的最小执行单元，要写清六要素 |
| 六要素 | 系统、模块、角色、决策点、输入、输出 |
| SOP | 每个活动的详细操作步骤，给执行者看 |
| 流程架构 | 多个流程拼成的业务运行体系 |
| TOGAF | 企业架构框架，要求活动定义到可执行颗粒度 |
