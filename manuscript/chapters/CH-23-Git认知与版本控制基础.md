# CH-23 Git 认知 — 版本控制基础

## 1. 痛点场景

小想接到一个需求修改任务。

花了三天把产品文档改了一遍，改了十几处内容。登录流程调整了，加了两个筛选条件，权限说明也改了。

周五评审会上，开发问："你这周改了哪些地方？我们要评估工作量。"

小想愣住了。

她记得改了登录流程、加了筛选条件、调整了权限说明。但具体哪些地方改了、哪天改的、为什么改，脑子里一片浆糊。

"你自己对比一下上次那版吧。"小想说。

开发叹了口气："上次那版在哪？你有没有存档？"

小想更慌了。她只在文档里改改改，没有保存历史版本。上上周的版本被覆盖了，上周的版本也没有备份。

老李在一旁插话："小想，这就是版本控制缺失的问题。如果你用 Git 管理需求文档，每一次改动都有记录：改了什么、谁改的、什么时候改的、为什么改。开发问你'改了哪里'，你打开 Git 日志就能告诉他。"

**你是不是也遇到过？**

改了需求却说不清改了什么，开发追问时慌得不行。本章就来解决这个产品经理的普遍困境：为什么你也需要版本控制思维。

---

## 2. 核心概念：Git、Commit、Branch、Merge、Conflict

老李看着小想困惑的表情，笑了笑说：

"小想，Git 就像一个时光机。"

### 2.1 Git 是什么

**Git 是版本控制系统，记录每次文件变更的历史。**

想象一下，你在写一份需求文档。写了一版，改了一点，又改了一点……如果不保存，你可能根本不知道昨天写了什么、上周删了什么。

但 Git 就像给你的每次修改都拍了一张照片。它记录了：谁改的、改了什么、什么时候改的、为什么改。你可以随时"穿越"回去，看到任何一个历史版本。

开发问你"你改了哪里"，你如果能打开 Git，就能告诉他：上次改了第3节，两周前改了标题，上个月删了那个功能点。这就是版本控制的力量——让一切变化有迹可循。

### 2.2 Commit（提交）

**Commit 是一次变更的快照。**

就像游戏存档点——你完成了一个任务，存档；又完成一个，再存档。每个存档点都记录了当前状态，你可以随时读取回到那个时刻。

Commit 也一样。每次你改完代码或文档，告诉 Git "帮我记一下这次改动"，Git 就创建一个 commit。这个 commit 包含：
- 改了哪些文件
- 改了什么内容
- 谁改的
- 改的时间
- 为什么改（你写的提交说明）

### 2.3 Branch（分支）

"分支就像平行宇宙。"老李继续解释。

**Branch 是独立的开发线。**

想象一下，你有一份产品需求文档。你想尝试一个新功能，但又怕改坏了原来的版本。怎么办？

你可以"开一个平行宇宙"——创建一个分支。在平行宇宙里，你大胆尝试新功能、修改逻辑、加新内容。这个宇宙和原来的主宇宙完全独立，你在平行宇宙的任何改动，都不会影响主宇宙。

等你在平行宇宙把功能做好了、测试过了，你可以把平行宇宙的成果"合并"回主宇宙。

### 2.4 Merge（合并）

**Merge 是把不同分支的变更合在一起。**

就像把平行宇宙的好东西带回主宇宙。开发在自己的分支做实验，做好了再合并回主线，这样主线永远是稳定的。

### 2.5 Conflict（冲突）

"但有时候，两个宇宙在同一位置有不同的东西——比如主宇宙第5页写着'功能A'，平行宇宙第5页写着'功能B'。合并时，Git 不知道该保留哪个，这就叫'冲突'。这时候就需要你人工决定：保留哪个，或者怎么融合。"

**Conflict 是 Git 无法自动决定保留哪个版本的情况。**

冲突不只在代码里发生。两个产品经理同时改同一份需求文档，改了同一节内容，也会产生冲突。这时候需要业务判断：哪个版本更符合目标？或者两个版本能否融合？

---

## 3. 为什么产品经理容易误解

很多产品经理听到"Git"会想：这是程序员用的工具，跟我有什么关系？

**误解在哪？你没看到"版本控制"的本质。**

Git 的版本管理思维适用于任何需要追踪变更的场景，不只是代码。产品管理需求也一样：
- 你改了需求，需要知道改了什么、什么时候改的
- 多人协作修改需求，可能产生冲突
- 发布时，需要知道当前版本对应哪个需求版本
- 上线后追溯问题，需要回到当时的需求版本

另一个常见误解："需求文档直接改就行了，存一份当前版本就够了。"

直接覆盖改是大忌。你改了什么、什么时候改的、为什么改，都没记录。以后追溯问题时，你找不回"当时的需求版本"。开发问你"改了哪里"，你答不上来。

正确做法：每次重要修改后，保留一个版本记录。就像写论文——每一稿都保留，不能直接覆盖上一稿。

---

## 4. 开发视角如何看这个问题

开发看 Git，想的不是"有哪些命令"，而是：

**追溯能力**
- 谁改了这行代码？
- 什么时候改的？
- 为什么改？
- 上次是什么样？

**协作基础**
- 多人同时开发怎么不冲突？
- 怎么追踪每个人的进度？
- 合并时遇到冲突怎么处理？

**版本管理**
- 当前版本对应哪个提交？
- 回滚到上个版本怎么做？
- 发布标记怎么打？

产品经理和开发的信息差：产品看到的是"当前文档"，开发看到的是"变更历史"。开发习惯追溯每一笔改动，产品习惯只看最新版本。这个差异导致沟通困难：开发问你"改了什么"，你只记得"改了"，但说不清细节。

---

## 5. 怎么做：用版本控制思维管理需求

把版本控制思维用到需求管理上，三件事就够了。

### Step 1：重要变更要记录

每次重要修改需求后，记录：
- 改了什么（改动点）
- 为什么改（变更原因）
- 谁要求改的（需求来源）
- 改的时间（变更时间）

不需要真的用 Git。但养成这个习惯：改需求不是"覆盖"，而是"追加一条变更记录"。

### Step 2：多人协作要同步

你和另一个 PM 同时改一份需求文档？
- 先约定各自改哪些部分，避免改同一处
- 改完后互相确认，不要直接覆盖对方的版本
- 如果改了同一处，坐下来讨论保留哪个

这是"冲突解决"的产品版本。

### Step 3：发布版本要标记

每次发布前，给需求文档打一个"版本号"或"时间戳"。

这样上线后发现问题，你能找到"当时的需求版本"，而不是一堆改了无数次的最新版本。

### AI 可以如何辅助

AI 能帮你理解 Git 命令的含义。

比如你看到开发说"merge conflict"，问 AI：
> "merge conflict 是什么意思？产品经理需要做什么？"

AI 会解释：合并时遇到冲突，Git 无法自动决定保留哪个版本。你需要做业务判断。

AI 也能帮你整理变更记录。你描述改动内容，AI 帮你生成变更说明：
> "请帮我整理这条需求变更记录：把登录流程从三步改成了两步，原因是用户反馈太繁琐，时间是今天，需求来源是运营部门。"

### Checklist

**基础认知**
- [ ] 我能解释 Git 是什么（版本控制系统）
- [ ] 我能解释 Commit（变更快照）
- [ ] 我能解释 Branch（独立开发线）
- [ ] 我能解释 Merge（合并分支）
- [ ] 我能解释 Conflict（无法自动合并的情况）

**实践习惯**
- [ ] 重要需求变更我会记录改动点、原因、时间
- [ ] 多人协作时我会先约定分工，避免改同一处
- [ ] 发布前我会给需求文档打版本号或时间戳
- [ ] 开发问"改了什么"我能追溯到具体改动

## 6. 案例复盘

回看小想的困境：开发问"改了哪里"，她答不上来。

如果她有版本控制思维，她能怎么应对？

**改需求时**：记录每次修改的改动点、原因、时间。比如"第3节登录流程从三步改成两步，原因是用户反馈繁琐，时间是4月15日"。

**开发追问时**：拿出变更记录，说清楚改了什么、为什么改。而不是翻遍脑子想半天，最后只能说"我记得改了……但具体说不清"。

**发布时**：给需求文档打版本号，比如"v2.3-登录优化"。上线后发现问题，能追溯到"当时的需求版本"。

要点：不是学 Git 命令，而是养成"每次改动有记录"的习惯。

---

## 7. 一句话总结

**版本控制思维 = 每次改动有记录、多人协作不冲突、发布版本可追溯。**

不是"学 Git 命令"，而是理解"变更应该被管理"。代码用 Git 管，需求用变更记录管——道理一样。

---

| 概念 | 一句话记住 |
|------|------------|
| Git | 时光机，记录每次改动的历史 |
| Commit | 游戏存档点，一个变更的快照 |
| Branch | 平行宇宙，安全做实验不影响主线 |
| Merge | 把平行宇宙的成果带回主宇宙 |
| Conflict | 两个宇宙在同一位置打架，需要人工决定 |
