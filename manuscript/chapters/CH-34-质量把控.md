# CH-34 质量把控 — 风险识别、验收清单、测试协同、发布把控与问题追溯

## 一、"上线了"为什么还出问题？

周五下班前。老李催着上线一个新功能："这个要今晚上线，周一业务要用。"

小想赶紧测试了一遍正常流程。"没问题，可以上线！"

上线后半小时，客服群里炸了："数据不对！旧数据和新数据不一致！"

老李脸色沉下来。打开日志一看："数据迁移脚本没跑，新旧数据没同步。"

小想愣住："我...我只测了新数据，没想到旧数据也要校验..."

老李叹气："上线不是按个按钮就完事。上线前要识别风险、设计验收清单、和测试对齐、做好发布把控。"

"你只测了正常流程，漏了一堆风险点。上线匆忙，检查不完整，没有预案。出问题是必然的。"

复盘会上，老李总结。"质量把控有五个环节：风险识别、验收清单、测试协同、发布把控、问题追溯。你漏了前四个，现在要用第五个来补救。"

**问题揭示**：只测正常流程，不识别风险、不设计验收清单、不和测试对齐、匆忙上线。出问题后手忙脚乱。

## 二、质量把控的五个关键环节

### 什么是质量风险识别？

**质量风险识别**是在上线前系统性识别可能出问题的地方。

老李问小想："如果这个功能上线后出问题，最可能是什么问题？"

小想想了想："数据不对吧？"

老李摇头："数据不对只是其中一种。你得想五个方向。"

| 风险类型 | 具体内容 | 典型表现 |
|---------|---------|---------|
| 数据一致性风险 | 数据在不同系统/时间点的一致性 | 新旧数据不一致、并发写入覆盖 |
| 并发性能风险 | 高并发场景下的系统稳定性 | 峰值流量崩溃、响应超时 |
| 权限遗漏风险 | 权限配置不完整 | 普通用户访问管理员功能 |
| 异常流程覆盖不足 | 只考虑正常流程 | 网络中断数据丢失、超时流程卡住 |
| 依赖系统可用性风险 | 外部服务不稳定 | 第三方支付接口超时 |

就像医生做术前检查。不会只看"手术能成功吗"，还要问一连串问题：麻药过敏吗？心脏能承受吗？有药物冲突吗？手术中出血怎么办？

风险识别不是"能不能上线"，而是"上线后可能出什么问题"。

### 验收清单和测试用例有什么区别？

**验收清单**是产品视角的检查项——功能是否符合预期。**测试用例**是技术视角的检查项——代码是否正确运行。

| 维度 | 验收清单 | 测试用例 |
|------|---------|---------|
| 视角 | 产品视角（业务价值） | 技术视角（代码逻辑） |
| 制定者 | 产品经理 | 测试工程师 |
| 关注点 | 用户行为、业务结果 | 边界条件、异常处理 |
| 颗粒度 | 功能模块级 | 代码逻辑级 |

小想问："验收清单不是测试写的吗？"

老李纠正："测试写的是测试用例，你写的是验收清单。验收清单是'签收快递前要检查什么'，测试用例是'快递运输中要保证什么'。两者互补，不能互相替代。"

就像快递签收验收：验收清单是检查包裹外包装是否完整、内部商品是否完好、数量是否正确。测试用例是保证快递运输过程中没有摔坏、没有丢失、没有延误。

验收清单关注最终呈现，测试用例关注过程质量。两者分工不同，目的一致——确保用户收到正确的东西。

### 产品经理容易误解什么？

**误解一："质量是测试团队的事"**

正确理解：产品经理是质量的第一责任人。需求不清晰导致的质量问题，测试再多也发现不了。

测试是专业把关者，但产品是最终责任人。

就像装修监理和业主：监理检查施工达标，但业主验收最终效果。监理是专业把关者，业主是效果责任人。

**误解二："上线就结束了"**

正确理解：上线是质量把控的中点。上线后验证、监控告警、复盘改进才是闭环。

就像航班起飞：起飞不是结束，到达目的地、乘客安全下机才是完成。

## 三、开发怎么看这个问题？

开发看质量把控，想的不是"测完了就行"，而是：

- 风险点在哪些地方？
- 验收标准是什么？
- 测试覆盖了哪些场景？
- 上线时机选对了吗？
- 出问题怎么快速定位？

老李问张开发："这次上线，你最担心什么风险？"

张开发想了想："数据一致性。这次有数据迁移，新旧系统数据要同步。"

老李点头："那验收清单里，数据校验这一条你写了吗？"

张开发摇头："没写。我以为测试会覆盖..."

老李叹气："这就是问题所在。产品不定义验收标准，测试不知道测什么，开发不理解风险在哪。三个角色都没对齐。上线后一定出问题。"

开发需要明确的验收标准才知道"做到什么程度"。你定义了验收清单，测试才知道测什么。

## 四、产品经理应该怎么做？

### 识别风险：列出五大类型

每次上线前，列出风险清单：

| 风险类型 | 检查方法 |
|---------|---------|
| 数据一致性 | 有数据迁移吗？新旧数据同步了吗？ |
| 并发性能 | 峰值流量预估多少？压测了吗？ |
| 权限遗漏 | 新增权限了吗？临时角色考虑了吗？ |
| 异常流程 | 网络中断、超时、失败怎么处理？ |
| 依赖系统 | 第三方服务稳定吗？有备选方案吗？ |

**追问方法**：老李的习惯是问三个问题——数据量多大？并发多少？超出预期怎么办？

这三个问题能发现90%的风险隐患。

### 设计验收清单：覆盖五个维度

验收清单应该在开发前就定义好，不是上线前临时列。

验收清单模板：

| 维度 | 检查项示例 |
|------|-----------|
| 正常流程 | 用户按标准流程能否完成目标 |
| 异常流程 | 操作失败、中断后处理是否正确 |
| 边界条件 | 最大数据量、极端输入的处理 |
| 权限控制 | 不同角色看到的内容和操作权限 |
| 数据校验 | 数据格式、范围、唯一性是否符合约束 |

### 测试协同：四个关键动作

**测试协同**是产品与测试团队对齐质量标准的过程——确保验收清单和测试用例互补覆盖。

| 动作 | 具体内容 |
|------|---------|
| 需求评审带验收标准 | 需求评审时提交验收清单，测试据此设计用例 |
| 参与测试用例评审 | 关键功能测试用例评审，产品经理必须参与 |
| 建立问题沟通机制 | 测试发现需求不清晰，及时通知产品澄清 |
| 上线前验收检查 | 产品执行验收清单，测试执行测试用例 |

测试说"测不透"时，你要帮助。解释业务场景、说明角色差异、描述数据流转、提供真实案例。

就像装修监理协同：业主和监理要沟通哪些是"必须达标"，哪些是"可以接受"。监理按标准检查，业主验收最终效果。

### 发布把控：上线前检查清单

**发布把控**是上线前的最后一道检查——确认功能验收、数据准备、配置确认、监控准备、应急准备都到位。

上线前检查：

| 检查项 | 具体内容 |
|------|---------|
| 功能验收 | 核心功能、异常流程、权限验收完成 |
| 数据准备 | 数据迁移脚本、数据校验脚本、数据备份 |
| 配置确认 | 系统配置、权限配置、第三方依赖可用 |
| 监控准备 | 关键指标监控启用、告警规则配置 |
| 应急准备 | 回滚方案确认、应急联系人确认 |

上线时机选择：避开业务高峰、预留验证时间、团队就绪确认。

就像航班起飞前检查：机长不会只看"引擎能启动吗"，还要确认乘客安全带系好、货物固定、天气条件、应急预案。起飞前检查清单是最后一道门。

### 问题追溯：5W1H方法

**问题追溯**是上线出问题后系统性定位根因的方法——从现象到本质，找到真正原因。

**5W1H**是结构化问题分析工具——从六个维度追问问题详情，挖清全貌。

上线出问题后，用5W1H分析：

| 维度 | 问题 |
|------|------|
| What | 问题是什么？具体表现是什么？ |
| When | 何时发生的？何时被发现？ |
| Where | 在哪个系统/模块/环境发生？ |
| Who | 谁发现的？谁负责？谁受影响？ |
| Why | 问题为什么会发生？根本原因是什么？ |
| How | 如何发生的？如何复现？如何解决？ |

再追问5个"为什么"，挖到根因。

就像侦探破案：不会只看"案件发生了"，还要追问案发时间、案发地点、嫌疑人、作案手法、作案动机。5W1H就是把问题当案件来侦破。

### AI可以如何辅助？

AI可以帮你生成风险清单初稿、验收清单模板、测试场景补充。

> 提示词示例：我有个支付功能上线，请帮我列出可能的质量风险点，包括数据一致性、并发性能、权限遗漏等方向。

但记住：AI生成的是"草稿"。具体风险点、验收条件、边界值，需要你来补。

### Checklist

- [ ] 上线前列出五大风险类型检查项
- [ ] 验收清单覆盖正常/异常/边界/权限/数据五个维度
- [ ] 参与关键功能测试用例评审
- [ ] 上线前检查清单全部确认
- [ ] 上线时机避开高峰、预留验证时间
- [ ] 出问题用5W1H分析根因

## 五、一句话总结

回看开头场景：小想只测正常流程，漏了数据一致性风险、验收清单缺失、测试协同不足、发布把控缺失。

上线时机选错了——周五下班前，没有验证时间，没有应急预案。

**根因**：风险识别不到位，验收清单缺失，测试协同不足，发布把控缺失。

**改进**：上线前列风险清单、设计验收清单、参与测试评审、做好发布检查。

**质量不是测出来的，是从风险识别到问题追溯全流程把控出来的。**

上线不是终点，闭环才是。
