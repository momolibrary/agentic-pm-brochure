# CH-36 排期评审与 Git-like 协作 — 排期协商、评审准备与版本管理思维

## 一、"做不了那么快"、"评审被怼"、"需求什么时候改的？"

周一上午。产品评审会。

小想刚说完需求，开发老王就开口了："下周五上线？这不可能。"

小想心里一沉。"业务方催得很紧，两周太长了..."

老王摇头："不是我不想快，是客观工作量决定的。积分功能涉及四个模块。还要做前端展示、接口联调、测试回归。至少两周。"

小想不知道怎么回应。只能硬撑："那我...再去和业务方商量一下。"

下午。另一个评审会。

小想准备了PRD（需求文档，Product Requirement Document）和原型图，觉得准备充分。但评审会刚开十分钟就被怼了。

开发问："有数据支撑吗？有多少用户提过？"

小想懵了："这个...没有具体数据，业务方说用户反馈多。"

测试问："边界条件是什么？用户输入超长文本会怎样？"

小想更懵："PRD里没写..."

运营问："上线后怎么推广？需要配合什么？"

小想继续懵："推广方案还没想..."

主持人说："今天的评审先暂停，小张回去补充材料后再开第二轮。"

第二天。开发来找小想。

"这个需求的逻辑怎么变了？上周评审不是说要显示原价吗？"

小想支吾："业务方上周五说要改成折扣价，我就改了PRD..."

开发追问："那你通知我们了吗？我代码已经按原价写的了。"

小想："我直接改了文档，以为你们会看..."

开发无奈："我们怎么知道你改了？你又没发通知。现在改要重写代码，影响排期。"

**问题揭示**：排期协商无方法、评审准备不充分、需求变更无记录。这三个场景是产品经理日常最常见的痛点。

## 二、排期协商、评审准备、Git-like 工作方式

### 排期协商是什么？

**排期协商**是产品经理与开发就开发时间达成一致的过程。

不是开发说多久就是多久。也不是业务方说什么时候就要什么时候。

就像建筑施工。业主说"下周五入住"。施工队说"主体施工两周，水电检测一周，竣工验收三天"。

业主不理解为什么砌墙很快，但水电检测要一周。

产品经理不理解为什么编码很快，但联调测试要三天。

**工作量组成**：

| 阶段 | 占比 | 说明 |
|------|------|------|
| 编码开发 | 30-40% | 前端、后端、单元测试 |
| 联调测试 | 20-30% | 接口联调、Bug修复 |
| 发布部署 | 5-10% | 环境配置、上线验证 |
| 项目管理 | 10-15% | 会议、文档、进度跟踪 |

开发说"两周"，不是不想做，是工作量确实那么多。

### 评审准备是什么？

**评审准备**是需求评审会前的系统性准备工作。

就像考试前复习清单。考试现场，考官一问就答不上来。因为没复习。

评审现场，开发、测试、运营一问就懵。因为没准备。

**准备清单**：

| 准备事项 | 具体内容 |
|---------|---------|
| 需求背景 | 需求来源、业务场景、解决什么问题 |
| 数据支撑 | 用户数据、行为数据、竞品分析 |
| 方案完整 | 业务流程、功能模块、正常+异常流程 |
| 风险评估 | 技术风险、运营风险、兼容性问题 |
| 依赖确认 | 外部系统接口、第三方服务、其他团队 |
| 预演方案 | 会议流程、时间分配、问题预案 |

准备不充分，评审会就变成"被怼会"。

### Git-like 工作方式是什么？

**Git-like 工作方式**是将Git版本管理思维应用到需求管理中。

每次改动有记录。改动原因有说明。改动内容可追溯。改动需要评审。

就像文档修订留痕。你改了文档，谁改的？什么时候改的？为什么改？改了什么？都要有记录。

**Git-like 需求管理**：

| Git概念 | 需求类比 |
|--------|---------|
| Issue | 需求变更请求单（像工单，每次改动都开一张） |
| PR Review | 变更评审（改动需团队确认才能合并） |
| Commit Log | 变更记录（日期、内容、原因、影响范围） |

需求变更无记录，开发问"什么时候改的？"你答不上来。代码白写了，排期白花了。

### 产品经理容易误解什么？

**误解一：排期是开发说了算**

正确理解：排期是双向协商。产品经理需要理解工作量组成，提出分期交付或整体延期方案。

开发说"两周"，你可以问：哪些模块可以分期交付？核心功能先上线，后续迭代补充？

**误解二：评审准备就是写好PRD**

正确理解：PRD只是准备的一部分。还需要数据支撑、风险评估、依赖确认、预演方案。

只写PRD不准备其他，评审会必被怼。

**误解三：需求文档改了不用记录**

正确理解：Git-like变更管理是协作基础。每次改动需要Issue追踪，改动需要评审确认，改动记录是追溯依据。

改了PRD不通知，开发代码白写。这是协作的常见断裂点。

## 三、开发怎么看这些问题？

### 开发看排期

开发说"两周"，想的是：

- 编码需要几天？
- 聁调测试需要几天？
- Bug修复预留几天？
- 部署上线需要几天？

开发不是不想快。是每个阶段都有客观工作量。

老李问老王："积分功能，编码需要几天？"

老王回答："前端一天，后端两天。"

老李追问："联调测试呢？"

老王说："至少两天，加上Bug修复可能三天。"

老李算了一下："总计六七天。下周三做完开发测试，周四发布，周五上线。"

老王补充："前提是需求不变、没有其他紧急插入、测试环境稳定。"

这就是工作量拆解。不是开发说多久就是多久。是一起拆解，看看有没有优化空间。

### 开发看评审准备

开发评审需求，关注的是：

- 技术可行性？
- 工作量评估？
- 技术难点在哪？
- 有哪些依赖？

如果PRD只写功能列表，没写技术风险、没标注依赖，开发评审时就会追问。

追问是为了评估风险。不是为了怼产品经理。

但准备不充分的产品经理，会被追问问懵。

### 开发看需求变更

开发问"需求什么时候改的？"，想的是：

- 代码白写了？
- 排期白花了？
- 改动影响哪些模块？
- 还要多久能做完？

没有变更记录，开发不知道改动范围。不知道改了什么，就只能重新看一遍PRD。

浪费时间，影响排期。

## 四、产品经理应该怎么做？

### 排期协商：三步走

**第一步：理解工作量组成**

不要直接否定开发的判断。

错误做法：
- "不可能那么久，我见过别人做得很快"
- "你就不能加班赶一下吗？"

正确做法：
- "咱们一起拆解一下工作量，看看有没有优化空间"

**第二步：提出分期交付方案**

分期交付的核心逻辑：
- 找出核心价值功能（MVP）
- 先交付核心功能验证价值
- 后续迭代补充完整功能

判断方法：

| 问题 | 判断依据 |
|------|---------|
| 哪个功能是业务最急需的？ | 业务方痛点分析 |
| 哪个功能可以单独上线？ | 功能依赖关系分析 |
| 哪个功能工作量最小？ | 开发工作量评估 |

**第三步：探讨资源协调可能性**

如果分期交付不可行，可以：
- 申请增加开发资源
- 申请延期（与业务方协商）
- 申请降低需求复杂度（砍掉非核心功能）

### 评审准备：检查清单

会前对照检查：

- [ ] 能用一句话说清楚需求背景？
- [ ] 有具体数字的数据支撑？
- [ ] PRD覆盖异常流程？
- [ ] 技术风险标注？
- [ ] 依赖方确认排期？
- [ ] 会前预演？

**不同角色关注点**：

| 参会角色 | 关注重点 |
|---------|---------|
| 开发 | 技术可行性、工作量评估、技术难点 |
| 测试 | 测试点覆盖、边界条件、异常场景 |
| 运营 | 上线效果、推广配合、用户反馈 |

会前与技术骨干沟通方案。准备测试用例要点。准备运营推广方案。

### Git-like 变更管理：Issue驱动

每次需求变更创建Issue记录：

**Issue模板**：

```markdown
## 变更请求

**变更标题**: 价格显示从原价改为折扣价
**变更原因**: 业务方反馈用户更关注折扣力度
**变更来源**: 业务运营团队
**变更时间**: 2026-04-21
**影响范围**:
- 商品详情页价格展示
- 购物车价格展示
- 订单确认页价格展示
```

变更后评审确认：
- 开发评估技术可行性和排期
- 测试评估测试范围
- 运营评估上线推广

评审通过后记录变更日志。

### AI可以如何辅助？

**排期协商**：AI可以帮你分析功能复杂度，生成工作量拆解初稿。

提示词示例：我有个积分功能，包含积分规则引擎、积分账户、积分流水、积分兑换四个模块。请帮我拆解工作量组成。

**评审准备**：AI可以帮你生成常见追问预答清单。

提示词示例：我要开需求评审会，功能是历史订单查询。请帮我列出开发、测试、运营可能的追问，并给出预答。

**变更管理**：AI可以帮你生成Issue模板和变更日志格式。

但记住：AI生成的是草稿。具体工作量、边界条件、变更影响范围，需要你来确认。

### Checklist

**排期协商**：
- [ ] 理解工作量组成（编码30-40%、联调20-30%、测试25-35%）
- [ ] 提出分期交付或整体延期方案
- [ ] 与业务方沟通权衡取舍

**评审准备**：
- [ ] 需求背景一句话能说清
- [ ] 数据支撑有具体数字
- [ ] PRD覆盖正常+异常流程
- [ ] 技术风险已标注
- [ ] 依赖方已确认排期
- [ ] 会前预演

**Git-like 变更管理**：
- [ ] 每次变更创建Issue
- [ ] 变更后评审确认
- [ ] 变更记录有日志

## 五、案例复盘

三个场景的共同根因：协作方法缺失。

**要点提炼**：
- 排期要点 → 拆解工作量 + 提出分期交付
- 评审要点 → 准备清单逐项检查 + 会前预演
- 变更要点 → Issue记录 + 评审确认 + 变更日志

---

## 六、一句话总结

**理解工作量组成，充分准备评审，变更有记录。**

这三个方法，让协作顺畅，让需求落地。
