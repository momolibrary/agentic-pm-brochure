# CH-24 云服务、前后端与数据库缓存 — 系统技术栈认知

## 1. 痛点场景

"部署方案：阿里云 ECS。"

周一上午，上线讨论会。小想第一次参与系统上线，会议纪要上写着这几个字。

小想好奇地问："我们公司有服务器机房吗？为什么不用自己的服务器？"

开发老李笑了："自建机房？一年三百万起步，还要养运维团队。上云一年一百多万，省心省钱。"

小想追问："那 ECS 是什么？上云就是把服务器搬到别人那吗？"

老李摆摆手："没那么简单。上云不只是租服务器，是租整套基础设施能力——弹性扩容、自动备份、安全防护。ECS 是 IaaS 层，基础设施即服务，就像租一个空仓库，里面放什么、怎么管理，你自己决定。"

小想恍然大悟：原来上云不只是"搬服务器"，还有这么多层次——IaaS、PaaS、SaaS，每个层次管的东西不一样。

**你是不是也遇到过？**

开发说的那些词——上云、IaaS/PaaS/SaaS、前端后端、缓存——到底是什么？为什么改了数据前端还是旧数据？为什么"改前端"有时也要改后端？本章就来回答这些问题。

## 2. 核心概念：云服务、前后端、缓存

### 2.1 云服务三层模型：IaaS、PaaS、SaaS

**上云 = 租云厂商的基础设施能力。**

类比：上云就像租仓库，自建机房就像自己盖仓库。

| 现实世界 | IT 世界 |
|---------|---------|
| 自己盖仓库 | 自建机房 |
| 租仓库 | 上云 |
| 仓库租金 | 云服务费用 |
| 仓库管理员 | 运维团队 |

自建仓库的痛点：
- 盖仓库周期长（1-3个月）
- 盖多了闲置浪费，盖少了不够用
- 管理员要养着，不管有没有货
- 出问题要自己修

租仓库的优势：
- 随租随用，不需要提前盖
- 用多少租多少，不浪费
- 管理员由仓库公司提供
- 出问题仓库公司修

**云服务的三层模型**

| 类型 | 类比 | 你管理的 | 云厂商管理的 |
|------|------|---------|-------------|
| **IaaS** | 租空仓库 | 货架布局、货物管理 | 仓库建筑、电力、消防 |
| **PaaS** | 租带货架的仓库 | 货物上架、库存管理 | 货架、仓库管理工具 |
| **SaaS** | 租全托管仓库 | 只管取货使用 | 一切 |

具体来说：

**IaaS（基础设施即服务）**：租空仓库。

云厂商提供虚拟机、存储、网络，你装操作系统、部署应用。典型产品：阿里云 ECS、AWS EC2。

**PaaS（平台即服务）**：租装修好的仓库。

云厂商提供运行平台（数据库、中间件），你只管应用代码。典型产品：阿里云函数计算、Heroku。

**SaaS（软件即服务）**：住酒店。

云厂商提供完整软件，你直接用。典型产品：钉钉、飞书、Salesforce。

**典型误解纠正**：上云就是把服务器搬到别人那？

错了。上云不只是"租服务器"，是租整套基础设施能力——弹性扩容、自动备份、安全防护、持续技术更新。上云让你专注于业务，而非基础设施。

类比理解：
- 自建机房 = 自己盖仓库、雇管理员、买消防设备、自己修电路
- 上云 = 租仓库，消防、安保、维护全由仓库公司管，你只管放货取货

### 2.2 前端和后端：界面和逻辑

**前端是用户看到的界面。**

网页、App 的页面布局、按钮、输入框——用户直接操作的东西。

**后端是服务器上的逻辑和数据。**

业务规则、数据处理、数据库操作——用户看不到但系统在运行的东西。

类比：前端是餐厅的菜单和餐桌，后端是厨房。

| 现实世界 | IT 世界 |
|---------|---------|
| 菜单展示 | 前端界面 |
| 厨房菜品 | 后端逻辑 |
| 服务员传菜 | API 接口 |

**前端和后端通过接口通信。**

前端发送请求（"我要一份红烧肉"），后端返回数据（"红烧肉一份，价格38元"）。

这就是为什么"改前端"有时也要改后端——接口返回的数据结构变了。

小想遇到的真实案例：

业务需求："用户列表页面，把'姓名'改成'姓名+手机号'显示。"

小想评估："这个改动很小，就改个前端显示，应该半天能做完吧？"

老李看了需求后说："这个涉及前后端两个地方。接口原来只返回'姓名'，现在要返回'姓名+手机号'。后端接口要改，数据库查询要改，前端显示也要改。"

小想恍然大悟：原来前端和后端不是两个独立的世界，而是通过接口连接的上下游。改显示字段，意味着数据传递链条每一层都要改。

### 2.3 缓存：用空间换时间

**缓存是暂存常用数据以提高访问速度的机制。**

类比：缓存就像桌上的几本常用书，数据库就像图书馆的书架。

| 现实世界 | IT 世界 |
|---------|---------|
| 图书馆书架 | 数据库（存储全部数据） |
| 桌上的常用书 | 缓存（存储热点数据） |
| 查书架的书 | 查数据库（慢） |
| 查桌上的书 | 查缓存（快） |

**为什么需要缓存？**

| 查询位置 | 类比 | IT 对应 |
|---------|------|---------|
| 查桌上书 | 伸手就拿，1秒 | 查缓存，微秒级（0.1ms） |
| 查书架书 | 走到书架、找书、拿书，10秒 | 查数据库，毫秒级（10ms） |

速度差异：缓存比数据库快10-100倍。

高峰期每秒几千次请求，全查数据库系统扛不住。有了缓存，大部分请求查缓存，少数请求查数据库，系统才能流畅运行。

**典型误解纠正**：缓存导致数据不准就不要用缓存？

错了。缓存是正常的设计选择，不是bug。几分钟延迟在大多数场景完全可以接受。用户更关心"响应快"，不关心"数据实时"。用微小延迟换取巨大性能提升是正确的权衡。

类比理解：
- 桌上书不是最新版本，但查起来快
- 大多数场景，旧版本也够用
- 你查字典在乎的是"快找到"，不在乎字典是不是最新版

小想遇到的真实案例：

上线后，业务反馈："后台改了商品价格，前端页面还是旧价格？等了几分钟才更新，数据不准！"

小想找老李："是不是有bug？改了数据应该马上显示才对。"

老李解释："数据存在数据库里，但查数据库慢。我们加了缓存，把常用数据放在内存里，查起来快100倍。缓存的数据是'快照'，几分钟刷新一次，所以改了数据要等一会才能看到。"

小想问："那可以改成实时更新吗？"

老李："可以，但代价很大。实时更新意味着每次修改都要同时更新缓存，写操作变慢，还可能出现数据不一致。"

小想恍然大悟：原来缓存是用空间换时间——花钱买内存，换取查询速度。几分钟延迟是正常的权衡，不是bug。

## 3. 开发怎么看这个问题

开发老李说的那些词，背后是他的技术视角：

**云服务选择**

- IaaS/PaaS/SaaS 怎么选？看业务需求和团队能力。
- 有运维团队、需要高度控制？选 IaaS，自己管系统。
- 想省运维、专注应用？选 PaaS，平台托管中间件。
- 直接用现成软件？选 SaaS，零运维。

老李的判断："我们团队有运维能力，选 IaaS，自己管系统，成本可控、灵活性高。"

**前后端分工**

- 前端负责：页面布局、交互逻辑、数据渲染
- 后端负责：业务规则、数据处理、接口设计
- 接口约定：数据字段、格式、错误处理

老李的提醒："改需求时，先问自己：改的是'显示'还是'数据'？改显示只改前端，改数据前后端都要动。"

**缓存设计**

- 哪些数据需要缓存？热点数据、高频查询、变化不频繁的数据
- 缓存过期时间（TTL）多久？看业务容忍延迟的程度
- 缓存更新策略：定时过期、主动更新、写时更新

老李的经验："商品价格这类重要数据，缓存5分钟过期。商品浏览数这类不重要数据，缓存1小时。核心原则：业务能容忍多久延迟，缓存就设多久。"

**开发视角和产品视角的信息差**

产品经理看到的是"改个显示"，开发看到的是"数据链条每一层都要改"。

产品经理看到的是"数据不准"，开发看到的是"缓存是正常设计，延迟是权衡结果"。

理解开发的视角，产品经理就能准确评估改动影响、理解技术决策背后的权衡。

## 4. 怎么做：技术栈认知 Checklist

### 4.1 云服务理解 Checklist

**识别云服务类型**

- [ ] 我能区分 IaaS/PaaS/SaaS
- [ ] 我知道各类型的典型产品（阿里云ECS、Heroku、钉钉）
- [ ] 我知道各类型"谁管什么"

**评估上云决策**

- [ ] 我知道为什么上云比自建机房划算（成本、弹性、运维）
- [ ] 我知道什么情况可能自建机房（超大规模、强合规）

### 4.2 前后端区分 Checklist

**判断改动涉及前端还是后端**

- [ ] 改页面布局、样式 → 只改前端
- [ ] 改显示字段、数据结构 → 前后端都要改
- [ ] 改业务规则、计算逻辑 → 只改后端

**理解接口的作用**

- [ ] 我知道前端和后端通过接口通信
- [ ] 我知道接口返回数据结构变了，前后端都要改

### 4.3 缓存认知 Checklist

**理解缓存的作用和利弊**

- [ ] 我知道缓存是"用空间换时间"
- [ ] 我知道缓存数据可能有几分钟延迟
- [ ] 我知道延迟是正常的权衡，不是bug

**评估业务对延迟的容忍度**

- [ ] 商品价格、订单状态 → 延迟容忍低，缓存时间短
- [ ] 商品浏览数、点赞数 → 延迟容忍高，缓存时间长

### 4.4 AI辅助用法

**用AI生成技术栈认知框架**

提示词示例：
> 请帮我解释 IaaS、PaaS、SaaS 三种云服务模式的区别，用类比说明。

**用AI评估改动影响**

提示词示例：
> 需求：用户列表页面增加手机号显示。请帮我分析这个改动涉及前端还是后端。

AI会给一个框架：
```
分析：
- 前端：页面增加手机号列
- 后端：接口返回数据增加手机号字段
- 数据库：查询增加手机号字段

结论：前后端都要改。
```

**关键提醒**：AI给的是分析框架，具体改动范围要和开发确认。
