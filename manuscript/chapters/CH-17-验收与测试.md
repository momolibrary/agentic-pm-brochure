# CH-17 验收与测试 — 验收标准、测试场景与边界测试

## 一、"做完了"为什么还出Bug？

周五下午。张开发在群里发了条消息："登录功能做完了，下周可以上线。"

小想看到消息。转头问老李："登录功能做完了，我怎么知道它没问题？"

老李笑了："你这话问得好。张开发说'做完了'。你知道他做完什么了吗？"

"是能登录了？还是所有场景都覆盖了？"

小想挠挠头："我...我只写了'用户可以登录'，没写别的。"

老李翻开需求文档。指着小想写的需求："你看，你写的是'用户可以登录'。"

张开发理解为：输入账号密码，跳转到首页。

但还有一堆问题没定义。密码错误怎么办？账号不存在怎么办？网络断了怎么办？连续输错几次怎么办？

张开发正好进会议室。听到这话："李总说得对。我理解的'做完'就是'能登录'。"

密码错误我写了提示。但账号不存在我没处理——我以为这是运营的事。

老李拍了拍小想的肩膀："这就是验收标准。你得告诉开发什么情况算'做对了'。"

"就像签收快递，你得有一张检查清单。"

**问题揭示**：需求只写"可以登录"，没写具体条件。开发理解"能登录"，产品理解"全场景覆盖"。

上线后账号不存在场景未处理。用户反馈报错。

## 二、验收标准、测试场景、边界测试

### 验收标准是什么？

**验收标准**是确认需求完成的具体条件。

就像签收快递的检查清单：

| 检查项 | 为什么检查 |
|--------|-----------|
| ✓ 外包装完好 | 确认运输中没摔坏 |
| ✓ 数量对 | 一个手机，一套配件 |
| ✓ 型号对 | 是我下单的那个型号 |
| ✓ 配件齐全 | 充电器、数据线、说明书都有 |
| ✓ 开机正常 | 手机能正常启动 |

快递员说"送到了"（做完了）。但你发现外包装凹陷、配件少了数据线。你会签收吗？不会。

因为"送到了"不等于"送对了"。

同理，开发说"做完了"，不代表"做对了"。你得有验收标准，一项一项检查。

### 测试场景要覆盖哪些？

**测试场景**是验证功能的各种情况组合。

很多产品经理只测正常流程：选商品→支付→成功。但90%的线上Bug出在异常流程和边界条件。

| 类型 | 说明 | 占Bug比例 |
|------|------|----------|
| 正常流程 | 用户正常操作路径 | ~10% |
| 异常流程 | 错误处理、网络异常 | ~60% |
| 边界条件 | 临界值、极端数据 | ~20% |
| 权限控制 | 无权限、跨权限操作 | ~10% |

只测正常流程，相当于只看了冰山的10%。

### 什么是边界测试？

**边界测试**验证极端条件下系统表现。

双十一晚上8点。订单量突然飙升。监控屏幕红了："订单服务响应超时！数据库连接池耗尽！"

张开发看日志。"这个订单列表查询，一次查10万条数据...内存爆了。"

老李问："上线前测过大数据量吗？"

张开发摇头："测试环境数据量很小。测了百条数据没问题。没想过会到10万条。"

小想一脸懵。"我...我只写了'用户可以查看订单列表'。没说数据量多大..."

这就是边界测试缺失。你得问：数据量多大时还能正常？超过怎么办？

### 产品经理容易误解什么？

**误解一：测试是测试人员的事**

正确理解：产品定义验收标准、提供测试场景是质量保障的第一步。测试人员执行，但标准由产品定义。

**误解二：正常流程走通了就行**

正确理解：90%的线上Bug出在异常流程和边界条件。只测正常流程，上线后最容易出问题的地方都没覆盖。

## 三、开发怎么看这个问题？

开发看验收标准，想的不是"有几条"，而是：

- 每条标准怎么验证？
- 异常情况怎么处理？
- 边界值是多少？

开发说"做完了"，往往是"正常流程写完了"。如果你没定义异常处理、没标注边界条件，开发就不会写。

老李问张开发："支付失败的时候，系统应该怎么处理？"

张开发挠挠头："我...没定义。支付失败就抛了个通用错误。"

老李叹气："这就是验收标准缺失的后果。开发需要明确的验收标准才知道'做到什么程度'。"

你定义了，他才写。你没定义，他就按自己理解来。上线后用户就用真实数据告诉你答案——崩了。

## 四、产品经理应该怎么做？

### 写验收标准：至少5条

每个需求至少列出5条验收条件：

| 验收条件 | 示例 |
|---------|------|
| 正常场景 | 用户输入正确账号密码，跳转首页 |
| 异常场景 | 密码错误，提示"密码错误，请重试" |
| 边界场景 | 连续输错5次，锁定账号15分钟 |
| 权限场景 | 已登录用户不能再次登录 |
| 数据场景 | 登录成功后记录登录时间和IP |

**为什么要5条？**少于5条意味着遗漏。正常场景1条，异常至少2条，边界至少1条，权限至少1条。

### 列测试场景：覆盖四类

系统性设计测试场景：

| 类型 | 示例场景 |
|------|---------|
| 正常流程 | 正常支付成功 |
| 异常流程 | 余额不足、网络超时、重复支付 |
| 边界条件 | 订单列表10万条数据、评论输入500字 |
| 权限控制 | 未登录用户访问订单页 |

**怎么列？**先写正常流程，然后逐个追问"如果失败会怎样"。余额不足？网络超时？重复点击？并发支付？退款怎么处理？

### 标注边界条件：追问极端情况

每次写需求时追问：

| 问题 | 示例回答 |
|------|---------|
| 数据量多大时还能正常？ | 预期10万条以内响应<2秒 |
| 超过预期怎么办？ | 分页查询，每页100条 |
| 并发用户多少？ | 预期5000并发，超了排队等待 |
| 输入最长多长？ | 评论限500字，超了截断提示 |

**追问方法**：老李有个习惯。每个需求他都问三个问题：最大多少？最小多少？超出怎么办？

这三个问题就能发现90%的边界隐患。

### AI可以如何辅助？

AI可以帮你生成测试场景初稿。

你描述业务流程。AI列出可能遗漏的场景：

> 提示词示例：我有个支付功能，用户选商品→支付→成功。请帮我列出所有可能遗漏的测试场景，包括异常流程和边界条件。

AI输出示例：余额不足、网络超时、支付失败重复扣款、并发支付、退款场景等。

但记住：AI生成的是"草稿"。具体边界值、预期表现，需要你来补。

### Checklist

- [ ] 每个需求有至少5条验收条件
- [ ] 测试场景覆盖正常/异常/边界/权限四类
- [ ] 边界条件已追问并记录（数据量/并发/输入长度）
- [ ] 异常处理已定义（不是"报错"而是具体提示和引导）
- [ ] AI生成的测试场景已人工审核补充

## 五、案例复盘

回看开头的登录场景：小想只写"用户可以登录"，没定义验收标准。张开发理解成"能登录就行"。上线后账号不存在场景漏处理，用户反馈报错。

再看双十一场景：小想只写"用户可以查看订单列表"，没说数据量多大。测试环境测百条数据没问题。上线后10万条数据压进来，系统崩溃。

**两个案例的共同根因**：验收标准缺失，边界条件未定义。

**洞察**：开发说"做完了"，你需要追问：做完什么？什么算做对了？极端情况会怎样？

这三个追问，能避免90%的上线事故。

## 六、一句话总结

**质量不是测出来的，是定义出来的。**

验收标准定义到位，上线才稳。
