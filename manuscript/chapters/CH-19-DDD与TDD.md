# CH-19 DDD 与 TDD — 领域驱动设计与测试驱动开发

## 1. 痛点场景

"这两个模块属于不同限界上下文。"

开发说完这句话，产品经理小想愣住了。限界上下文？什么意思？

另一场需求评审会上，小想提了新需求。开发听完，说："你的需求我没法写测试用例。"

小想心想：写不出测试用例？这不是找借口吗？

老李看出了小想的困惑，递给他一杯茶："这两个场景，都不是开发在刁难你。"

**你是不是也遇到过？**

开发说了一堆术语，你听不懂。开发说需求没法测试，你以为他在推脱。本章就来解决这两个问题：什么是限界上下文、什么是通用语言、为什么写不出测试用例反映需求有问题。

---

## 2. 核心概念：DDD、限界上下文、通用语言、TDD

### 2.1 DDD 是什么

**DDD（领域驱动设计）是一种让技术和业务说同一种语言的方法。**

传统开发模式里，业务说"客户"，开发说"用户"，数据库叫"member"。三个人说的是同一个东西，但名字不一样。沟通成本高，理解容易出偏差。

DDD 的核心：让团队对同一个概念使用统一的名称和定义。代码里的类名、变量名、数据库字段名，都用业务语言。

### 2.2 限界上下文

**限界上下文是 DDD 中划分业务边界的概念。**

开发说"两个模块属于不同限界上下文"，意思是：这是两块不同的业务，不能混在一起设计。各有各的数据模型、业务规则和术语。

**类比：国界。**

"chips"这个词：
- 在美国 = 薄薯片（potato chips）
- 在英国 = 炸薯条（French fries）

同一个词，在不同国家含义不同。你不能说"英国人说的 chips 是错的"。因为在英国的边界内，chips 就是炸薯条。

同样，电商系统里"Product"这个词：

| 上下文 | "Product"的含义 | 核心关注点 |
|--------|----------------|-----------|
| 商品目录 | 展示的商品信息（标题、图片） | 用户浏览、搜索 |
| 库存 | 物理库存项（数量、位置） | 库存管理、补货 |
| 订单 | 订单中的商品项（价格、数量） | 订单处理 |
| 营销 | 需推广的商品概念（卖点） | 营销策划 |

每个上下文有自己的"官方语言"。跨上下文需要"翻译机制"，就像过海关。

### 2.3 通用语言

**通用语言是团队统一使用的术语和定义。**

DDD 要求：开发人员、业务专家、产品经理对同一个概念使用同一个词。

典型问题场景：
- 产品说"客户"
- 开发说"用户"
- 业务说"买家"
- 数据库叫"member"

结果：大家说的可能是不同的东西，或者同一个东西叫了四个名字。

**实践要点**：
- 建立词汇表：每个术语有明确定义
- 代码即文档：类名、方法名都用通用语言
- 持续对话：新概念出现时，先统一命名再设计

### 2.4 TDD 是什么

**TDD（测试驱动开发）要求先写测试再写代码。**

传统方式：先写代码，写完再测试。问题是代码写完了才发现需求不清晰。改起来代价大。

TDD 方式：
1. 先写测试（定义期望行为）
2. 再写代码让测试通过
3. 重构优化代码

**类比：考试大纲。**

传统教学：老师先教课，期末出考试题。结果考题可能和教学内容不匹配。学生抱怨"你没讲这个"。

TDD 教学：老师先写考试大纲，明确期末考什么。然后根据大纲设计课程。教的内容和考的内容完全对齐。

**关键洞察**：开发说"你的需求我没法写测试用例"，就像老师说"你的大纲我写不出考试题"。这说明大纲本身有问题。不够清晰、不够具体、不够可验证。

## 3. 产品经理容易误解的地方

### 误解一："DDD 是开发的架构方法，跟产品无关"

**正确理解**：DDD 的核心是让技术和业务说同一种语言。限界上下文的划分直接影响产品的模块划分和功能边界。

你理解了限界上下文，就能：
- 判断两个功能是否应该放在一起
- 预判系统拆分的复杂度
- 和开发讨论业务边界时说同一种语言

### 误解二："开发说写不出测试用例是在找借口"

**正确理解**：如果需求足够清晰，测试用例自然能写出来。写不出说明需求本身有问题。

测试用例需要明确的输入输出。你的需求说"体验友好"，开发怎么测？说"响应要快"，快是多快？

**可验证的需求 vs 模糊的需求**：

| 模糊需求 | 可验证需求 |
|----------|-----------|
| "响应要快" | "响应时间 < 200ms" |
| "体验友好" | "成功率 ≥ 99.9%" |
| "给用户反馈" | "点击提交后显示确认弹窗" |
| "输入异常时处理" | "输入为空时提示'请填写必填项'" |

## 4. 开发怎么看这个问题

开发听到"限界上下文"，脑子里想的不只是"边界在哪"。还有一串技术问题要解决。

**边界划分后的技术考量**

| 问题 | 开发在想什么 |
|------|-------------|
| 数据怎么存？ | 每个上下文有自己的数据库表。商品目录的 Product 表和库存的 Product 表是两张不同的表。 |
| 跨上下文怎么通信？ | 需要"翻译机制"（DDD 里叫 Anti-Corruption Layer）。订单上下文拿到的商品信息要转换成订单项。 |
| 并发怎么办？ | 两个上下文同时修改同一个商品的库存，怎么保证不冲突？分布式事务还是事件驱动？ |
| 状态一致性？ | 库存上下文说"有货"，订单上下文下单后库存变"无货"，中间状态怎么处理？ |
| 接口设计？ | 上下文之间调用，是同步 API 还是异步消息？失败了怎么重试？ |

开发说"这是不同限界上下文"，其实是在说："你别把它们混在一起设计。它们的数据、规则、接口都不一样。强行合并，后期维护会很痛苦。"

**开发看到的需求测试问题**

开发看到"没法写测试"的需求，脑子里也在展开：

| 问题 | 开发在想什么 |
|------|-------------|
| 输入是什么？ | 需求说"用户提交订单"，具体提交哪些字段？必填的有哪些？格式要求是什么？ |
| 输出是什么？ | 需求说"系统给反馈"，具体返回什么？弹窗？跳转？消息提示？ |
| 边界条件？ | 购物车为空时能提交吗？商品下架了怎么办？库存不足怎么提示？ |
| 异常处理？ | 支付失败怎么处理？网络超时怎么处理？用户取消订单怎么处理？ |
| 可量化吗？ | "体验友好"怎么测？有具体指标吗？ |

开发说"写不出测试"，不是不想做，而是没法做。测试需要明确的"给定什么条件，执行什么操作，期望什么结果"。你给的需求没有这三要素，测试就写不出来。

## 5. 怎么做：推动通用语言 + 写可验证需求

### 5.1 推动 DDD 通用语言

**Step 1：建立术语表**

每次需求评审前，把涉及的核心术语列出来：

| 术语 | 定义 | 代码名称 |
|------|------|----------|
| 客户 | 购买商品的人 | Customer |
| 订单 | 客户下单产生的记录 | Order |
| 商品 | 在目录中展示的物品 | Product（Catalog上下文） |

**Step 2：发现术语不一致时立即纠正**

会议上有人说"用户"，有人说"客户"，立刻问：这两个是同一个东西吗？如果是，统一用一个词。

**Step 3：检查代码命名**

让开发确认代码里的类名、方法名是否和术语表一致。不一致的地方就是风险点。

**场景示例**

需求评审会上，开发说："用户下单后，系统要更新会员积分。"

老李立刻问："你说的'用户'和'会员'是同一个概念吗？"

开发想了想："对，是同一个东西。我们数据库里叫 member。"

老李："那就统一叫'会员'。产品文档、需求描述、代码命名都用会员。'用户'这个词不要再出现。"

小想记下了这个规则。

### 5.2 写可验证的需求

**用 Given-When-Then 模式**

把需求写成三段式：

```
Given（给定）：用户已登录且有购物车商品
When（当）：用户点击"提交订单"
Then（那么）：订单创建成功，购物车清空，跳转到订单详情页
```

**把模糊描述改成可量化描述**

| 原描述 | 改后 |
|--------|------|
| "搜索要快" | "搜索结果在 1 秒内返回" |
| "支持多人" | "支持最多 1000 人同时在线" |
| "自动处理异常" | "异常时记录日志并发送告警邮件给 admin@example.com" |
| "用户体验好" | "页面加载时间 < 2 秒，错误率 < 1%" |

**补充边界条件**

需求评审时，主动问开发这些问题：

| 问题 | 示例 |
|------|------|
| 输入边界是什么？ | 购物车为空时能提交吗？ |
| 异常怎么处理？ | 支付失败后订单状态是什么？ |
| 并发场景？ | 两个人同时买同一件商品，库存怎么处理？ |
| 数据一致性？ | 库存显示有货，下单后库存变成 0，中间状态怎么处理？ |

### 5.3 AI 辅助

**AI 可以帮你**

| 任务 | AI 能做什么 |
|------|------------|
| 翻译需求格式 | 把模糊需求翻译成 Given-When-Then 格式 |
| 发现边界条件 | 提示你可能漏掉的边界场景 |
| 生成术语表初稿 | 根据需求描述列出核心术语 |

**提示词示例**

示例一：把需求翻译成 Given-When-Then 格式

```
请把这个需求改写成 Given-When-Then 格式：
用户提交订单后，系统应该给用户反馈。
要求：
- Given 部分列出所有前置条件
- When 部分明确用户操作
- Then 部分列出所有期望结果
```

示例二：发现边界条件

```
这个需求的 Given-When-Then 写完了。请帮我检查：
1. 有哪些边界条件可能漏掉？
2. 有哪些异常场景需要补充？
3. 有哪些并发场景需要考虑？

需求：
Given 用户已登录且有购物车商品
When 用户点击"提交订单"
Then 订单创建成功，购物车清空，跳转到订单详情页
```

示例三：生成术语表初稿

```
请帮我生成这个需求的术语表初稿：
需求涉及：用户、商品、购物车、订单、支付。
每个术语需要：中文名、定义、建议的代码命名。
```

**AI 不能帮你**

| 任务 | 为什么 AI 不能做 |
|------|-----------------|
| 决定术语选择 | "客户"还是"用户"是正确的术语，需要你根据业务判断 |
| 判断验收标准 | 某个验收标准是否符合业务规则，AI 不知道你们的业务规则 |
| 补充业务约束 | 手机号必须 11 位、一个会员只能绑定一个手机，这些规则 AI 不知道 |

这些需要你的判断力和业务知识。

### 5.4 Checklist

**术语一致性**

- [ ] 核心术语有明确定义吗？
- [ ] 会议上所有人用同一个词描述同一个概念吗？
- [ ] 代码命名和术语表一致吗？
- [ ] 新术语首次出现时有定义吗？

**需求可测试性**

- [ ] 每个需求有明确的输入输出吗？
- [ ] 边界条件写清楚了吗？（空值、极值、并发）
- [ ] 验收标准可量化吗？（有数字、有具体结果）
- [ ] 异常处理规则写了吗？
- [ ] Given-When-Then 三要素完整吗？

## 6. 案例复盘

回看小想和老李的对话。

场景一：开发说"这两个模块属于不同限界上下文"。

小想当时不理解。现在知道了：这是开发在表达业务边界。两个上下文有各自的数据、规则、术语。强行合并会带来技术复杂度。

场景二：开发说"你的需求我没法写测试用例"。

小想当时以为是借口。现在知道了：这是需求本身不够清晰。测试需要明确的输入输出、边界条件、异常处理。这些都没写清楚，测试就写不出来。

**要点提炼**：
- 限界上下文是业务边界，不是技术边界
- 写不出测试是需求问题，不是开发问题
- AI 能帮你翻译格式、发现边界条件，但业务判断要靠你

---

## 7. 一句话总结

**DDD 让团队说同一种语言，TDD 倒逼需求足够清晰。**

下次开发说"限界上下文"，你就知道：这是在讨论业务边界。下次开发说"写不出测试"，你就知道：需求还不够具体。

---

| 概念 | 一句话记住 |
|------|------------|
| DDD | 让技术和业务说同一种语言 |
| 限界上下文 | 业务边界，同一词在不同边界含义不同 |
| 通用语言 | 团队统一使用的术语 |
| TDD | 先写测试再写代码，倒逼需求清晰 |
