# CH-40 Agentic Design 基础 — AI辅助设计协作、结构级大改与精准反馈

## 1. 痛点场景

「改了五版，越改越离谱。」

小想让AI帮他生成一个注册表单页面，反馈时只会说「感觉不太行」。AI改了五版——从蓝色改到绿色、从居中改到左对齐、从卡片改到边框——最后改回第一版的样子。

老李看到后说：「你对人说『感觉不太行』都不行，更别说对AI了。」

本章解决一个让无数产品经理头疼的问题：为什么你的AI反馈越改越乱，以及如何一句话让AI精准执行。

周一上午，产品评审会。

周三下午，开发小王做完页面。小想验收时发现：颜色跟设计稿不一样，按钮位置不对，间距也有偏差。

周四，小想找小陈确认。小陈：「这颜色完全不对啊，品牌色不是这个。」

下周三，验收新版本，颜色还是不对——小王用了深色模式的颜色。

下周周五，终于上线。小陈：「一个页面折腾了两周。」

小想夹在中间苦笑：「下次我组织你们一起看吧。」

## 2. 核心概念：Agentic Design协作模式、大改与小改、精准反馈

### 2.1 Agentic Design协作模式

**Agentic Design协作模式**：设计师、开发、产品经理三个人同时看着AI画布，实时讨论，当场定方案。

传统模式是接力棒：设计师出稿 → 产品转给开发 → 开发做完 → 产品验收 → 设计师看问题 → 再反馈。每一轮传递都增加延迟，每一轮理解都可能偏差。

新模式是同步协作：所有人坐在一起，看着AI实时生成。遇到技术限制，当场讨论；遇到设计细节，当场确认。一场会（2小时）定稿，替代两周的来回传递。

**类比**：传统模式像去餐厅点菜，厨师在厨房做，你等菜，发现问题再反馈；新模式像站在开放式厨房前，厨师当面做，你当场指点——一顿饭一次到位。

### 2.2 结构级大改 vs 细节微调

**大改**：换布局、新增模块、整体风格调整。影响范围大，牵一发动全身。

**小改**：颜色、间距、字号、具体组件替换。影响范围小，局部调整。

区分的原因：大改影响多个元素，必须整体规划；小改只影响单个元素，标注清楚就行。

**类比**：建筑师处理楼层功能重新规划（大改），要开正式会议、拿完整图纸；处理一面墙挪10cm（小改），直接在图纸标注就行。建筑师不会为一面墙开大会，也不会「随手」改楼层布局。

| 改动类型 | 特征 | 正确方式 |
|---------|------|---------|
| 大改 | 换布局、新增模块、整体风格 | 聊天界面 + 明确约束 |
| 小改 | 颜色、间距、字号、具体组件 | 元素标注 + 具体参数 |

### 2.3 给AI精准反馈

**精准反馈**：用具象参数描述需求，让AI能直接执行。

**模糊反馈**：用抽象情绪描述需求，AI无法理解。

AI的本质：擅长处理具象参数（数字、颜色值、尺寸），不擅长揣摩抽象审美（感觉、不对劲、不舒服）。

**类比**：让厨师做菜，说「感觉不好吃」，厨师不知道该调甜还是调咸、该大火还是小火；说「盐加一勺，辣椒减半」，厨师照着做一次到位。厨师不知道你喜欢咸还是淡，AI不知道你喜欢密集还是松散。

| 模糊感觉 | 翻译成参数 |
|---------|-----------|
| 「颜色太亮」 | 「饱和度降到60%以下」 |
| 「间距太大」 | 「字段间距改成16px」 |
| 「太松散」 | 「内容区宽度600px，居中」 |
| 「没有品牌感」 | 「使用品牌色 #FF6B35」 |
| 「看着累」 | 「字号16px，行高1.5」 |

## 3. 为什么产品经理容易误解

### 误解一：「传统接力模式效率够高」

很多产品经理觉得：设计师出稿、开发实现、产品验收，这个流程没问题啊。

误解在哪？你没看到**信息损耗**。设计稿 → 产品转述 → 开发理解，每一轮传递都有偏差。开发说「这个颜色代码里没法用」，设计师说「这颜色完全不对」，两边都没错，错的是中间的传递。

真实痛点：一个页面改了五轮，设计师和开发都快吵起来了。

### 误解二：「AI能自动理解我的审美」

很多产品经理觉得：AI这么智能，我说「感觉不太行」，它应该能理解吧。

误解在哪？AI没有审美直觉。它知道「间距16px」是什么意思，但不知道「太松散」是什么感觉。你说「看着不对劲」，AI只能随机尝试——改颜色、改布局、改样式——越改越偏离预期。

真实痛点：小想改了五版，最后改回第一版的样子。

### 误解三：「大改小改随便说就行」

很多产品经理觉得：不就是让AI改个东西吗，怎么说都行。

误解在哪？方式不匹配影响范围。小改用大改方式（聊天界面），一句话导致整个页面重新生成；大改用小改方式（元素标注），改了局部但忘了整体约束。

真实痛点：小想让AI改按钮颜色，AI把整个导航栏都重新设计了。

## 4. 怎么做

### 4.1 开发视角看这个问题

开发看AI设计工具，想的不是「好不好看」，而是：

- 这颜色在代码里怎么定义？
- 这个间距用什么单位？
- 这个布局在不同屏幕上会怎样？
- 这个改动会影响哪些组件？

开发需要的是**可执行的参数**，不是抽象的感觉。

当你说「感觉不对」时，开发只能猜。猜错了，返工；猜对了，运气。

### 4.2 产品经理应该怎么做

**第一步：区分大改和小改**

问自己：这个改动会影响多少元素？
- 只改一个元素的属性 → 小改，用元素标注
- 会影响多个元素或整体布局 → 大改，用聊天界面

**第二步：把感觉翻译成参数**

问自己：AI需要知道什么才能执行？
- 颜色 → 具体的颜色值（如 #FF6B35）
- 间距 → 具体的像素值（如 16px）
- 位置 → 具体的定位方式（如 居中、宽度400px）

**第三步：用正确方式下指令**

小改：点击元素 → 在标注里写具体参数
大改：在聊天界面写 → 加「保持其他元素不变」约束

### 4.3 案例复盘

**反面案例**：小想让AI改按钮颜色，说「把登录按钮改成红色」。AI理解成「重新设计登录相关元素」，把整个导航栏都改了。

**正确做法**：点击登录按钮元素，在标注里写「背景色改成 #FF0000」。AI只改这一个属性，其他元素不动。

### 4.4 检查清单

| 检查项 | 通过标准 |
|-------|---------|
| 改动类型判断 | 大改用聊天，小改用标注 |
| 参数具体化 | 颜色有值、间距有数、位置有定义 |
| 约束完整性 | 大改时加「保持其他元素不变」 |
| 术语一致性 | 使用术语表定义的术语 |

---

## 5. 一句话总结

把感觉翻译成参数，AI才能干活。

---

**延伸思考**：下次用AI生成设计时，先问自己三个问题：这是大改还是小改？AI需要什么参数？我要加什么约束？
