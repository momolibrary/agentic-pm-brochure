# CH-07 流程基础 — 业务流程与系统流程

## 1. 痛点场景

周五下午，老李画完审批流程图，发给开发。

"用户提交→审批人审批→通过→归档。完成！"

周一评审会，开发一句话让老李愣住：

"驳回后怎么处理？回到哪一步？能重新提交吗？"

老李盯着流程图，发现只有一条"一路顺风"的线——提交、审批、通过、结束。驳回？没画。驳回后回到发起人还是上一节点？没想。驳回后能再次提交吗？不知道。

"流程不完整，做不了。"开发把图退回来。

老李心里一沉：流程图不是画完了就行？

**你是不是也遇到过？**

开发总说流程"不完整"，但你明明把正常步骤都画了。本章解决一个问题：为什么你画的流程开发总说"漏东西"——以及怎么一次性画完整。

---

## 2. 核心概念：业务流程、系统流程、正常流程

### 2.1 业务流程

**业务流程是用户完成目标的操作序列。**

用户视角的步骤：提交审批→审批人审批→收到通知。每一步都是用户能感知的操作。

就像餐厅里**顾客点菜**：
- 看菜单
- 下单
- 等菜
- 吃饭
- 结账

顾客只关心这些。顾客不关心厨房怎么做菜、厨师怎么分工。

### 2.2 系统流程

**系统流程是系统处理请求的内部逻辑步骤。**

系统视角的步骤：接收请求→数据校验→写入数据库→更新状态→发送通知。用户看不见这些，但每一步都在发生。

就像餐厅里**厨房备菜**：
- 接收订单
- 检查食材库存
- 分配厨师
- 烹饪制作
- 质量检查
- 出菜通知

顾客的"下单"只是一个动作。厨房要做六步。

### 2.3 为什么要区分？

**业务一步操作，系统可能要做多步处理。**

用户点击"提交审批"，系统要：
1. 接收 HTTP 请求（用户和网络之间的数据传输）
2. 验证用户身份（确认是谁在操作）
3. 验证数据格式（检查表单是否完整）
4. 检查业务规则（如审批条件是否满足）
5. 写入数据库（保存审批记录）
6. 更新状态（标记为"待审批"）
7. 分配审批人（确定谁来审批）
8. 发送通知（消息/邮件提醒）
9. 记录操作日志（留痕审计）
10. 返回响应（告诉用户结果）

用户看到一个"提交成功"提示。后台做了十步。

**区分两者的实际意义**：
- 产品设计业务流程时，容易忽略系统流程的复杂性
- 开发实现时，每一步都可能出错，都需要异常处理
- 如果只画业务流程，漏掉的就是系统要处理的细节

**一个真实案例**：

小想设计了一个"提交报销"功能。PRD 里只写了：用户填写报销单→提交→等待审批→审批通过→收到款项。

上线后，用户投诉："我提交了，怎么没反应？"

排查发现：提交成功后，系统要分配审批人。但审批人池里有 3 个人，系统没有规则决定分配给谁——代码写不下去了。

小想漏掉的是：业务流程画完了，但系统流程里"分配审批人"这一步没定义清楚。

### 2.4 正常流程（Happy Path）

**正常流程是预期的标准操作路径。**

用户提交→审批通过→归档。这是一切顺利时的路径。

但流程不只是正常路径。还有：
- **逆向路径**：驳回、撤回、取消
- **异常路径**：超时、错误、冲突

老李画的审批流程，只有正常路径。这就是开发说"不完整"的原因。

### 2.5 产品经理容易误解什么？

**误解一："流程就是画个流程图"**

流程图只是可视化手段。关键要把每一步的**输入、输出、角色、判断条件**都定义清楚。

老李的流程图写了"审批"，但没写：
- 审批人是谁？
- 审批时看什么信息？
- 审批通过的条件是什么？
- 审批不通过怎么处理？

画图是第一步。定义每一步的细节才是流程设计。

**误解二："用户提交了就完了，后台自动处理"**

用户的一个操作可能触发系统内部多个处理步骤。每一步都可能出错：

- 身份验证失败怎么办？
- 数据格式不对怎么办？
- 写数据库失败怎么办？
- 发送通知失败怎么办？

如果你只设计"提交成功"的提示，没考虑失败时用户看到什么，上线后用户会投诉："我提交了，怎么没反应？"

## 3. 开发怎么看这个问题

开发看流程，想的不是"有几步"，而是：

**每一步的输入输出**
- 这一步需要什么数据？
- 这一步产出什么？
- 数据从哪来、到哪去？

**每一步的判断条件**
- 什么情况往下走？
- 什么情况跳到其他路径？
- 条件怎么判断？

**逆向路径和异常路径**
- 驳回回到哪一步？
- 超时怎么办？
- 两个人同时操作怎么办？
- 数据校验失败怎么处理？

**状态怎么变**
- 每一步后状态是什么？
- 状态跳转规则是什么？
- 状态冲突怎么处理？

开发说"流程不完整"，不是刁难。是"防御性设计"。没定义清楚就上线，驳回没处理、超时没兜底、状态没约束——上线就是 bug。

产品经理和开发的信息差：
- 产品看到的是"用户操作"
- 开发看到的是"系统处理"

产品画一条线"审批→通过"，开发要问十个问题。

---

## 4. 怎么做：先画业务流程，再补系统流程

### Step 1：画出业务流程（正常路径）

用 BPMN（一种标准化的流程图画法，就像流程图的"普通话"）或泳道图，画出用户视角的操作步骤。

老李的审批流程：
- 发起人提交申请
- 审批人审批
- 通过后归档

### Step 2：补逆向路径

问自己：正常路径之外，还有什么？

| 逆向路径 | 要定义的 |
|----------|----------|
| 驳回 | 回到哪一步？能重新提交吗？状态怎么变？ |
| 撤回 | 发起人能撤回吗？撤回后状态？ |
| 取消 | 能取消吗？取消后数据怎么处理？ |

老李补充：
- 驳回 → 回到发起人，状态改为"待修改"，可重新提交
- 撤回 → 发起人可撤回待审批的申请，状态改为"已撤回"

### Step 3：补异常路径

| 异常路径 | 要定义的 |
|----------|----------|
| 超时 | 审批超时怎么办？自动通过？自动驳回？提醒？ |
| 并发 | 两个人同时审批怎么办？ |
| 错误 | 系统处理失败怎么办？用户看到什么？ |

**具体场景举例**：

**超时场景**：审批人在 48 小时内没有审批。系统怎么处理？
- 方案 A：自动通过（风险：审批人没看就通过了）
- 方案 B：自动驳回（风险：审批人忙，用户无辜被驳回）
- 方案 C：提醒审批人 + 延长等待时间（推荐）
- 方案 D：升级给审批人的上级

老李选择方案 C：超时 24 小时提醒一次，48 小时再提醒一次，72 小时升级上级。

**并发场景**：审批人 A 和审批人 B 同时点击"通过"。
- 如果是单人审批：只认第一个，第二个提示"已被他人处理"
- 如果是多人会签：记录两人都通过，状态变为"会签完成"

**错误场景**：发送通知失败。
- 方案：记录发送失败，后台重试 3 次；仍失败则标记"通知异常"，运营人工跟进

### Step 4：定义每一步的要素

用 TOGAF 活动定义标准（一种企业架构方法论，就像建筑工程的施工规范）把每一步写清楚：

| 要素 | 说明 |
|------|------|
| 操作角色 | 谁来执行 |
| 输入物 | 需要什么信息 |
| 输出物 | 产出什么 |
| 决策点 | 是否需要判断 |
| 系统位置 | 在哪个系统执行 |

老李补充审批步骤：

| 步骤 | 角色 | 输入 | 输出 | 决策点 | 系统 |
|------|------|------|------|--------|------|
| 提交申请 | 发起人 | 申请表单 | 审批记录 | 否 | OA系统 |
| 审批 | 审批人 | 审批记录、申请内容 | 审批结果 | 是 | OA系统 |
| 归档 | 系统自动 | 审批结果 | 归档文件 | 否 | 档案系统 |

### Step 5：AI 辅助生成流程初稿

提示词示例：
> 请帮我生成请假审批的业务流程图，包含：提交申请、审批人审批、通过归档、驳回退回修改。

AI 会给出框架。但你要补充：
- 驳回后回到哪一步？
- 超时怎么处理？
- 每一步的输入输出是什么？

AI 出框架，你补细节。

### Checklist：流程完整性自查

**正常路径**
- [ ] 画出了用户视角的完整操作步骤

**逆向路径**
- [ ] 驳回回到哪一步已定义
- [ ] 撤回规则已定义
- [ ] 取消规则已定义

**异常路径**
- [ ] 超时处理已定义
- [ ] 并发冲突已定义
- [ ] 系统错误时用户提示已定义

**每一步定义**
- [ ] 操作角色已标注
- [ ] 输入物已说明
- [ ] 输出物已说明
- [ ] 决策点已标注

### 案例复盘：老李的流程图为什么被退回？

回头看老李的审批流程，问题出在哪？

他只画了"一路顺风"的正常路径：提交→审批→通过→归档。

漏掉的是：
- **逆向路径**：驳回后回到哪一步？
- **异常路径**：审批超时怎么办？
- **每一步定义**：审批人是谁？输入什么？输出什么？

开发退回来的原因不是"刁难"，而是：这些没定义清楚，代码写不了。

老李后来怎么改的？按四步法：
1. 先画正常路径（提交→审批→通过）
2. 补逆向（驳回→回到发起人→待修改→可重提交）
3. 补异常（超时48小时→自动提醒审批人）
4. 定义每一步的输入输出角色

改完再发开发，一次过。

---

## 5. 一句话总结

**流程不只是正常路径，逆向路径才是上线后的雷区。**

画流程：先画正常 → 补逆向 → 补异常 → 定义每一步的输入输出角色。

---

| 概念 | 一句话记住 |
|------|------------|
| 业务流程 | 用户视角的操作步骤，像顾客点菜 |
| 系统流程 | 系统内部的处理步骤，像厨房备菜 |
| 正常流程 | 一切顺利时的路径（Happy Path） |
| 逆向流程 | 驳回、撤回、取消等非正常路径 |
