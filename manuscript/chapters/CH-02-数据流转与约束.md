# CH-02 数据流转与约束 — 数据怎么流动 & 数据有什么规则

## 1. 痛点场景

周三下午，老李在画数据流图。

"用户下单→后端接收→写入数据库→通知仓库。"他边画边念叨。

画完一看，总觉得哪里不对。

小想凑过来："看起来挺完整的啊。"

老李眉头一皱："等等。如果后端接收成功，但写入数据库失败，怎么办？"

小想愣住。

"如果写入成功，但通知仓库失败，怎么办？"

小想开始冒汗。

"如果通知成功，但仓库系统返回错误，怎么办？"

老李叹了口气："这就是**异常数据处理**——我漏了。"

另一个场景。

上周小想写用户注册PRD，没写"手机号不能重复"。

上线三天，客服收到投诉：两个用户用同一个手机号注册，一个登录成功，另一个提示"密码错误"。

开发老张来找小想："你没写唯一性约束，我默认允许重复。"

小想一脸无辜："我以为你知道手机号不能重复啊……"

老张摇摇头："我不知道。**不写约束，就是允许。**"

---

你是不是也遇到过？

业务说"数据从A到B"，开发说"要经过三个系统"。你以为数据搬过去就行，开发却追着一堆问题不放。

本章解决一个问题：为什么"数据流转"不是搬过去那么简单？为什么"不写约束"会出脏数据？

---

## 2. 核心概念：数据流转与数据约束

### 2.1 数据流转是什么？

**数据流转 = 数据从输入到输出经过的完整路径。**

听起来简单？换个类比就明白了。

快递包裹从发货到收货，不是直接"搬过去"。它要经过：商家仓库→物流中转站→干线运输→末端配送→用户手中。每一站都有扫描、分拣、状态更新。任何一站出错，包裹就丢了。

数据也一样。

用户提交表单，数据要经过：前端校验→后端接口→业务处理→数据库写入→下游系统通知。每一站都有格式检查、逻辑判断、状态记录。

```
用户输入
   ↓
[前端校验] ──格式对不对？
   ↓
[后端接口] ──权限够不够？
   ↓
[业务处理] ──逻辑通不通？
   ↓
[数据库]   ──数据存没存？
   ↓
[下游系统] ──通知发没发？
   ↓
结果返回用户
```

**为什么不是直接搬过去？**

因为每个节点都有自己的规则。

前端只校验格式，不知道业务逻辑。后端知道业务逻辑，但不知道数据库有没有锁。数据库只管存，不知道下游系统能不能收。

每个节点都是一个"关卡"。关卡之间还有异步处理——数据不是立刻传，而是排队等。还有消息队列——多个系统间的数据传递管道。还有批处理——攒一批数据再一起处理。数据不是"一步到位"，而是"接力传递"。

### 2.2 异常数据怎么处理？

**正常路径只是冰山一角。真正的复杂度在于异常处理。**

老李画漏的部分，就是这些问题：

| 异常场景 | 不处理会怎样 |
|---------|-------------|
| 前端校验失败 | 用户填错格式，提交无响应，体验差 |
| 后端接口超时 | 用户不知道成功还是失败，重复提交 |
| 数据库写入失败 | 数据丢了，用户以为成功了 |
| 下游通知失败 | 仓库没收到发货指令，订单卡住 |
| 部分成功部分失败 | 订单创建成功，但库存扣减失败，数据不一致 |

不处理异常数据，会带来三种后果：

1. **脏数据**：格式错误、重复、缺失的数据进入系统
2. **流程卡断**：某个节点失败，后续节点无法继续
3. **系统崩溃**：异常数据堆积，处理队列溢出

所以老李画的"正常路径"只是起点。真正的数据流转图，必须包含异常分支。

### 2.3 数据约束是什么？

**数据约束 = 数据的规则限制：格式、范围、唯一性等。**

CH-01讲过，每个字段有名称、类型。但光有类型不够，还需要约束。

常见五种约束：

| 约束类型 | 限制什么 | 例子 |
|---------|---------|------|
| 格式约束 | 数据长什么样 | 手机号11位数字，邮箱包含@ |
| 范围约束 | 数值在什么区间 | 年龄0-150，评分1-5 |
| 唯一性约束 | 某个值不能重复 | 手机号不能重复，身份证号不能重复 |
| 非空约束 | 字段必须填写 | 用户名必填，订单号必填 |
| 外键约束 | 引用的数据必须存在 | 部门ID必须在部门表存在 |

### 2.4 脏数据是怎么产生的？

**脏数据 = 不符合预期规则的数据。**

小想的案例里，脏数据是这样产生的：

1. 用户A用手机号13800138000注册
2. 用户B用同一个手机号再次注册
3. 系统没校验唯一性，两个用户都创建成功
4. 用户B登录，系统匹配到用户A的记录
5. 用户B输入自己的密码，验证失败
6. 用户B投诉"登录不了"

**根本原因**：PRD没写"手机号唯一性约束"，开发默认允许重复。

开发不知道"手机号应该唯一"，这是业务规则，不是技术常识。

### 2.5 产品经理容易误解什么？

很多产品经理对数据流转和数据约束有三个常见误解。

### 误解一："数据从A到B就是搬过去嘛"

真相：数据流转不是复制粘贴。每个节点有格式转换、校验、持久化、异步回调。任何环节出错都可能数据丢失或不一致。

### 误解二："约束开发自己加就行了，不需要产品写"

真相：约束是业务规则，不是技术规则。"手机号能不能重复"、"年龄上限是120还是150"——这些是产品决策，不是技术决策。不写清楚，开发只能猜。

### 误解三："数据流转图是开发的事"

**误解表现**：产品说"数据怎么流转是架构师定的，我只写功能就行了"。

**为什么误解**：产品以为数据流转图是技术设计的一部分，不知道自己画数据流图能发现需求遗漏。

**正确理解**：产品画数据流转图不是为了替代架构设计，而是为了自己想清楚需求链条、发现遗漏。能画流转图的产品，需求完整度远高于只写功能描述的产品。你画出来的不是"最终架构"，而是"需求视角的流转路径"——让开发知道你考虑了哪些环节、漏了什么。

## 3. 开发怎么看这个问题

产品说"数据存起来"，开发脑子里自动展开：

**第一层：数据从哪来、到哪去？**
- 数据起点在哪？（用户输入、外部接口、定时任务）
- 数据终点在哪？（数据库、文件、下游系统）
- 中间经过几个系统？

**第二层：每个节点做什么？**
- 格式转换？（前端传JSON，后端要对象）
- 校验逻辑？（手机号格式、金额范围）
- 状态记录？（日志、流水表）

**第三层：出错了怎么办？**
- 校验失败怎么提示用户？
- 写入失败怎么回滚？
- 下游失败怎么重试？
- 部分成功部分失败怎么补偿？

**第四层：约束规则是什么？**
- 必填字段有哪些？
- 唯一性要求有哪些？
- 格式要求有哪些？
- 范围限制有哪些？

开发的关注点不是刁难，而是"防御性设计"。每个问题背后都是踩过的坑。没想清楚就上线，后期返工、数据清洗、用户投诉。

**核心认知差**：产品看到的是"功能"，开发看到的是数据路径和错误处理。产品一句话，开发要展开十几个问题。不说清楚，开发只能按最宽松的方式实现。

## 4. 怎么做：画数据流转图、写约束规则

### 4.1 数据流转图怎么画？

产品画数据流转图，不需要画到代码级别。画清三件事就够了：

**第一步：标注起点终点**

数据从哪来？到哪去？

```
起点：用户提交订单表单
终点：订单写入数据库，通知仓库发货
```

**第二步：标注中间节点**

数据经过哪些系统/服务？

```
前端 → 后端接口 → 业务处理 → 数据库 → 仓库通知
```

**第三步：标注输入输出**

每个节点，数据进去是什么样，出来是什么样？

```
前端输入：表单数据（订单号、商品、金额）
后端输出：订单对象（已校验、已计算）
数据库输出：订单记录（已持久化）
仓库通知：发货指令（订单号、商品列表）
```

**第四步：标注异常分支**

每个节点如果失败，数据怎么处理？

```
前端校验失败 → 返回错误提示，不提交
后端超时 → 返回"请稍后重试"
数据库失败 → 返回"订单创建失败"
仓库通知失败 → 记录异常，人工跟进
```

画完这四步，数据流转图就完整了。

### 4.2 数据约束怎么写？

用这个模板，每个字段写清约束：

| 字段名 | 中文名 | 类型 | 格式 | 范围 | 必填 | 唯一 | 外键 |
|--------|--------|------|------|------|------|------|------|
| phone | 手机号 | 文本 | 11位数字 | 无 | 是 | 是 | 无 |
| age | 年龄 | 整数 | 无 | 0-150 | 否 | 否 | 无 |
| dept_id | 部门ID | 整数 | 无 | 无 | 否 | 否 | 部门表.id |

### 4.3 AI可以怎么辅助？

**生成数据流转图初稿**

提示词示例：
> 请帮我画"用户下单"的数据流转图，标注起点终点、中间节点、异常处理分支。

AI会给一个框架。你需要补充的是：业务节点的具体逻辑、异常处理的业务决策（比如"仓库失败是人工跟进还是自动重试"）。

**检查约束遗漏**

提示词示例：
> 这是我的字段表[粘贴]，请检查是否遗漏了格式、范围、唯一性、非空、外键约束。

AI会指出"手机号没写唯一性约束"、"年龄没写范围限制"。你根据业务规则决定是否采纳。

### 4.4 自查清单

**数据流转检查**
- [ ] 数据起点终点标注了吗？
- [ ] 中间经过的节点列出了吗？
- [ ] 每个节点的输入输出说明了吗？
- [ ] 异常处理分支画了吗？

**数据约束检查**
- [ ] 格式约束写了吗？（手机号、邮箱）
- [ ] 范围约束写了吗？（年龄、金额）
- [ ] 唯一性约束写了吗？（手机号、身份证）
- [ ] 必填字段标注了吗？
- [ ] 外键关系说明了吗？

---

回看小想的案例：她用AI生成字段表，漏了"手机号唯一性约束"。上线后出现脏数据，客服接到投诉。

教训：AI出框架，你补业务规则。约束是业务决策，不是技术默认值。

---

## 5. 一句话总结

**数据不只是存起来，还要"流得通、管得住"。**

画数据流转图：起点→节点→终点→异常分支。
写数据约束：格式→范围→唯一→必填→外键。

---

| 概念 | 一句话记住 |
|------|------------|
| 数据流转 | 快递包裹的路径，每一站都可能出错 |
| 数据约束 | 表格的验证规则，不写就是允许 |
| 脏数据 | 没遵守规则的数据，用户投诉的源头 |
| 异常处理 | 正常路径只是冰山一角 |
