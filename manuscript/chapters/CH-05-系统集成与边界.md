# CH-05 系统集成与边界 — 系统依赖、集成方式与边界条件

## 1. 痛点场景

周一上午，排期评审会。

老李正得意洋洋地汇报："用户中心项目，预计4月20号上线。"

开发主管皱着眉头问："支付网关什么时候上线？"

老李愣了。"什么支付网关？"

"你的用户中心要接入支付系统，支付系统依赖新的支付网关。网关还在开发，预计5月中旬上线。"

会议室安静了三秒。

老李盯着排期表，发现上面只有"用户中心"的开发时间。他没问过依赖系统什么时候就绪。

结果：用户中心4月20号能做完，但上不了线。要等支付网关5月中旬上线，才能联调。

排期做完了，上线时间却不对。

**你是不是也遇到过？**

功能开发完了，却说"上不了线"。本章解决这个问题：为什么排期要考虑依赖系统、不同场景用什么集成方式、边界条件怎么定义。

---

## 2. 核心概念：系统依赖、集成方式、边界条件

### 2.1 系统依赖

**系统依赖是一个系统需要另一个系统先就绪才能运行的关系。**

装修时，水电必须先做好。水电是"隐蔽工程"，埋在墙里地板下，做完就看不见。贴砖贴完了发现水管要改，得把砖撬开重来，成本极高。

系统也一样。基础设施（数据库、API、架构）必须先就绪，后面的功能模块才能跑。

```
用户中心 → 支付系统 → 支付网关（第三方）
```

用户中心依赖支付系统，支付系统依赖支付网关。网关没上线，后面两个都跑不起来。

**产品经理常犯的错误**：排期只算自己系统的开发时间，忘了问依赖系统什么时候就绪。

### 2.2 集成方式：API、消息队列、文件传输

系统之间怎么连接？三种常见方式：

| 方式 | 特点 | 适用场景 |
|------|------|----------|
| API调用（同步） | 直接调，等返回 | 实时交易、支付确认 |
| 消息队列（异步） | 发完就走，排队处理 | 发短信通知、削峰解耦 |
| 文件传输（批量） | 定时批量导入 | 每日数据同步、报表生成 |

**同步调用**：调用方等待被调用方返回结果才继续执行。
**异步调用**：调用方发出请求后不等待结果，继续执行其他任务。

**类比：打电话 vs 发微信**

API调用就像打电话。你打过去，对方不接你就没法说话。双方必须同时在线，高峰期一千人同时打客服电话，客服接不过来就全堵住了。

消息队列就像发微信。你发出去就可以继续干别的事，对方有空时再看。不用等着，解耦。

老李解释给产品小想听："直接调接口就像打电话。高峰期一千人同时打，客服接不过来全堵住了。消息队列就像餐厅取号系统，高峰期不让所有人往厨房挤，排好队一个个来。"

**什么时候用哪种？**

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 支付确认 | API调用 | 需要立刻知道结果，不能等 |
| 发短信通知 | 消息队列 | 不需要即时，晚点发也行 |
| 每日数据同步 | 文件传输 | 数据量大，定时批量处理 |

### 2.3 边界条件

**边界条件是系统运行的限制范围。**

任何系统都有边界。超过边界，系统就会出错或崩溃。

常见边界条件：

| 边界类型 | 具体指标 | 没定义会怎样 |
|---------|---------|-------------|
| 并发上限 | 同时在线用户数、QPS | 高峰流量超出系统承载，崩溃 |
| 数据量上限 | 单表数据量 | 数据量太大，查询慢甚至失败 |
| 响应时间 | 超时阈值 | 用户等太久，体验差 |
| 存储上限 | 存储容量 | 存满了，新数据写不进去 |

**产品经理常犯的错误**：需求里写了功能，没写"并发多少"、"数据量多大"。上线后高峰期系统扛不住，开发说"你没告诉我并发上限是多少"。

## 3. 开发怎么看这个问题

产品说"这个功能很简单，就调个接口"，开发脑子里自动展开：

**关于系统依赖：**
- 依赖哪些系统？强依赖还是弱依赖？
- 依赖系统什么时候上线？
- 依赖系统挂了怎么办？有降级方案吗？

**关于集成方式：**
- 这个场景用同步还是异步？
- 高峰期流量多少？扛得住吗？
- 失败怎么处理？重试？回滚？

**关于边界条件：**
- 并发上限多少？QPS多少？
- 数据量最大多少？单表会不会太大？
- 超时阈值多少？超限怎么处理？

开发问的不是刁难，是"防御性设计"。秒杀活动崩溃、12306售票高峰宕机、大促活动服务器过载——这些都是边界条件没定义清楚的教训。

产品经理和开发的信息差：产品看到的是"功能能不能做"，开发看到的是"系统能不能扛"。你只说功能，不说边界，开发只能按默认值设计。默认值扛不住高峰流量，上线就崩。

## 4. 怎么做：依赖排查、集成选型、边界定义

### 4.1 依赖排查

写需求时，先问三个问题：

1. **这个功能依赖哪些系统？**
2. **依赖系统什么时候就绪？**
3. **依赖系统挂了，我们怎么办？**

用依赖清单模板：

| 依赖系统 | 依赖类型 | 就绪时间 | 降级方案 |
|---------|---------|---------|---------|
| 支付系统 | 强依赖 | 5月中旬 | 提示用户稍后重试 |
| 短信服务 | 弱依赖 | 已上线 | 队列暂存，恢复后补发 |
| 第三方物流 | 弱依赖 | 待确认 | 手动录入物流信息 |

**强依赖**：没它就跑不了，必须等。
**弱依赖**：没它也能跑，只是功能受限。

排期必须包含依赖系统的就绪时间。开头故事里的老李，就是漏了支付网关的上线时间，导致排期无效。

### 4.2 集成方式选型

用决策树判断：

```
这个场景需要实时结果吗？
  └─ 是 → API调用（同步）
  └─ 否 → 数据量大吗？
           └─ 大（万级以上）→ 文件传输（批量）
           └─ 不大 → 消息队列（异步）
```

需求里标注清楚集成方式：

| 功能 | 推荐方式 | 说明 |
|------|----------|------|
| 订单支付 | API调用 | 需立刻确认支付结果 |
| 发货通知 | 消息队列 | 不需要即时，允许延迟 |
| 每日订单报表 | 文件传输 | 批量生成，定时处理 |

### 4.3 边界条件定义

用边界条件清单，一次性说清楚：

| 边界类型 | 预期值 | 超限处理 |
|---------|--------|---------|
| 并发上限 | 1000 QPS | 排队等待，提示稍后重试 |
| 数据量上限 | 单表100万条 | 分表存储 |
| 响应时间 | 3秒内返回 | 超时提示，用户可重试 |
| 存储上限 | 100GB | 定期清理过期数据 |

**AI辅助方法**：给AI你的功能描述，让它帮你生成边界条件检查清单初稿。

提示词示例：
> 请帮我生成"秒杀活动"功能的边界条件检查清单，包括并发、数据量、响应时间、存储等维度。

AI给初稿，你补业务数值。

**依赖排查**

### 4.5 案例复盘：小想的教训

上线前一周，小想兴奋地给老李看秒杀活动的需求文档。

老李问："并发上限多少？"

小想愣了："没写。我以为...做好了就行？"

老李叹气："秒杀活动，高峰期流量可能是平时的100倍。你没告诉开发并发上限，开发按默认值设计。高峰来了，系统扛不住，直接崩溃。"

上线当天，秒杀活动开始5秒后，服务器挂了。

复盘时，小想在需求文档里补了一行：**并发上限：5000 QPS，超限排队等待**。

教训：边界条件不是"锦上添花"，是系统的护栏。

## 5. 一句话总结

**系统不是孤岛，依赖没理清排期就是赌运气。**

写需求时：列依赖 → 选集成方式 → 定边界条件。

| 概念 | 一句话记住 |
|------|------------|
| 系统依赖 | 装修水电先行，依赖系统先就绪 |
| 集成方式 | 打电话=同步，发微信=异步，批量=文件传输 |
| 边界条件 | 系统的护栏，超过就出问题 |
