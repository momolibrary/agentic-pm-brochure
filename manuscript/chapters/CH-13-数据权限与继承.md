## 1. "经理看全部"到底是什么意思？

周一上午，某企业ERP系统升级项目的需求评审会。

业务代表说："经理嘛，当然要看全部数据。"

小想点点头，在PRD里写下一条：`经理角色可以查看全部数据`。

张开发扫了一眼PRD，皱眉："经理看全部？具体指什么全部？"

小想愣了一下："就是...全部啊？所有数据都能看。"

张开发追问："跨部门的数据能看吗？下属的薪资数据能看吗？离职员工的历史数据能看吗？敏感字段比如手机号、身份证号能看吗？"

小想傻眼了，转头看向老李。

老李叹了口气："张开发问的是'组织边界'和'数据类型边界'。你写的'全部'太模糊了。数据权限必须按组织边界和数据类型精确定义。"

## 2. 数据权限、操作权限、权限继承

### 数据权限：你能看什么数据

**数据权限**控制用户可访问的数据范围。

就像图书馆的借书证级别——普通借书证只能进普通阅览室，古籍阅览室、期刊阅览室不能进。借书证级别决定你能进哪些阅览室，这就是数据权限。

数据权限必须定义两个边界：

- **组织边界**：哪些部门/层级的数据能看。比如"经理可看本部门及下属部门数据"，就定义了组织边界。
- **数据类型边界**：哪些字段能看。比如"薪资、身份证等敏感字段需脱敏显示"，就定义了数据类型边界。

"全部"是个危险的词。它没有定义边界。开发追问"跨部门能看吗"，就是在追问组织边界。追问"薪资能看吗"，就是在追问数据类型边界。

### 操作权限：你能做什么操作

**操作权限**控制用户可执行的功能操作。

继续图书馆类比：普通读者只能看书，不能上架新书、不能修改书籍位置。这些操作只有工作人员能做。工作人员身份决定你能执行哪些操作——上架、下架、修改位置。这就是操作权限。

小想写的"经理可以审批申请"，就漏了操作权限的细项。张开发追问："审批是批准还是驳回？撤回呢？转审呢？批量审批呢？"

小想打开AI生成的PRD，发现都没写。张开发说："批量操作是高风险操作，必须有独立权限控制。"

**关键区别**：数据权限和操作权限是两个独立的东西。你可能能进所有阅览室（数据权限高），但只能看书不能上架（操作权限低）。两者分开配置。

### 权限继承：上级自动有下级权限？

**权限继承**是权限沿组织层级或角色关系自动传递的机制。

业务代表说："上级自动有下级的所有权限，不用单独配置。"小想在PRD里写：`上级自动继承下级所有权限`。

张开发皱眉："财务专员能操作财务系统，总经理也要继承这个操作权限？万一误操作把账目改乱了怎么办？"

张开发继续："还有权限冲突——审批人不能审批自己提交的申请。如果上级继承了下级的'申请人'权限，上级能审批自己的申请吗？"

老李总结："权限继承不能一刀切。必须明确——继承哪些？不继承哪些？"

### 产品经理最容易踩的两个坑

**误解一**："数据权限和功能权限是一回事"

正确理解：数据权限控制范围（能看什么），操作权限控制功能（能做什么），两者独立配置。

**误解二**："上级权限一定大于下级"

正确理解：不同角色有不同维度的权限。财务专员能看财务数据，部门经理不一定能看。上级在某个维度权限高，不代表所有维度都高。

## 3. 开发怎么看权限设计

开发看权限设计，想的不是"经理能看全部"，而是：

- 数据范围规则怎么写？SQL查询条件怎么拼接？
- 操作权限矩阵怎么实现？每个角色能执行什么操作？
- 权限继承怎么计算？继承哪些、排除哪些、冲突怎么处理？

开发关注的具体要素：

| 开发关注点 | 产品经理应提供的 | 为什么重要 |
|-----------|-----------------|-----------|
| 数据范围规则 | 组织架构 + 数据归属规则 | 决定SQL查询条件怎么写 |
| 操作权限矩阵 | 角色 × 操作 × 权限的表格 | 决定权限校验代码怎么写 |
| 继承规则 | 继承哪些、排除哪些、冲突处理 | 决定权限计算逻辑怎么写 |
| 边界条件 | 各类边界场景的处理规则 | 决定测试用例怎么写 |

开发常见追问清单（产品经理应提前准备答案）：

**数据权限追问**：
- "跨部门数据能看吗？"
- "敏感字段要脱敏吗？脱敏规则是什么？"
- "历史数据（离职员工）能看吗？"
- "上级能看下级所有数据还是部分数据？"

**操作权限追问**：
- "审批包括驳回吗？驳回后能重新提交吗？"
- "撤回权限有吗？撤回后状态变成什么？"
- "批量操作有独立权限吗？批量操作有上限吗？"
- "删除需要审批吗？删除后数据保留吗？"

**继承规则追问**：
- "全部继承还是部分继承？"
- "操作权限继承吗？还是只继承数据权限？"
- "审批人不能审批自己的申请，这个规则怎么处理？"

## 4. 怎么做：权限设计三步法

### 第一步：定义数据权限边界

画出组织架构，标注每个层级的数据可见范围。

```
平台级（超级管理员）→ 可看全平台数据
部门级（部门经理）   → 可看本部门及下属部门数据
小组级（组长）       → 可看本小组数据
个人级（普通员工）   → 只看自己的数据
```

然后追问数据类型边界：
- 哪些字段敏感？
- 敏感字段是否脱敏？
- 脱敏规则是什么？

### 第二步：画出操作权限矩阵

列出所有角色 × 所有操作，标注每个格子是否有权限。

```
        │ 查看 │ 编辑 │ 删除 │ 审批 │ 驳回 │ 批量审批 │ 批量删除 │
────────┼──────┼──────┼──────┼──────┼──────┼──────────┼──────────┤
普通员工│  ✓   │  ✓   │  ✗   │  ✗   │  ✗   │    ✗     │    ✗     │
组长    │  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │    ✗     │    ✗     │
部门经理│  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │    ✓     │    ✗     │
管理员  │  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │    ✓     │    ✓     │
```

**注意**：批量操作必须有独立权限列。批量审批、批量删除是高风险操作，不能和单条操作混在一起。

### 第三步：明确继承规则和例外

继承规则建议：
- **数据查看权限**：通常可以继承。上级能看下级看的数据。
- **操作权限**：通常不建议继承。上级不一定需要下级的操作能力。
- **审批类权限**：遵循"不能审批自己提交"原则，需特殊处理。

例外场景：
- 某敏感数据上级也不能看？
- 临时授权给上级看某数据？
- 权限冲突时如何处理？

### AI可以怎么辅助

**场景一**：生成权限矩阵初稿

你可以描述业务场景，让AI生成角色 × 操作的权限矩阵。

但记住：AI生成的是"草稿"。批量操作、边界条件需要你来补。

**场景二**：发现遗漏操作

你可以把需求发给AI，让AI帮你列出可能遗漏的操作权限点。

比如你写"经理可以审批"，AI可以追问：驳回？撤回？转审？批量审批？

---

## 5. 总结

权限 = 数据范围 × 操作功能 × 继承规则。三者缺一不可。
