# CH-35 协作沟通基础 — 开发语言翻译、技术方案讨论与异议处理

## 1. 痛点场景

业务方说："这个功能上线要快，用户等着呢。"

小想把这句话直接写进 PRD："上线时间要求：快。"

开发看了后问："快是多快？一周？三天？还是今天就要？"

小想无言以对。她根本没问清楚业务的真正诉求。

另一场景。小想第一次参加技术方案评审会。

开发在讲："我们准备用微服务架构，把订单模块拆成独立服务，用 Kafka 做消息队列，Redis 做缓存..."

小想完全听不懂，全程没发言。

会后老李问："你怎么不说话？"

小想："我听不懂他们在说什么，怕说错话被当成外行。"

还有一次。业务要求："订单列表要支持多条件组合筛选。"

开发说："这个做不了。"

小想直接把这个需求删除了，没追问原因。

后来复盘发现：开发其实是说"现在做不了，需要两周开发周期"。

如果追问了，可以安排两周后开发，而不是直接放弃。

**你是不是也遇到过类似场景？**

业务说模糊的话，你不知道怎么翻译给开发。

开发说"做不了"，你不知道是真技术限制还是只是不想做。

参加技术评审，你插不上话，怕说外行话。

本章就来解决这三个问题。

---

## 2. 核心概念：翻译官、医生诊断、外行参议

### 2.1 翻译官类比：PM 的核心角色

老李说："产品经理就是业务和开发之间的翻译官。"

用翻译官类比来理解：

| 翻译官场景 | 产品经理角色 | 说明 |
|-----------|---------|------|
| 外国客人（说英语） | 业务方 | 提出需求的人，用业务语言表达 |
| 本地居民（说中文） | 开发团队 | 接收需求的人，用技术语言理解 |
| 翻译官 | 产品经理 | 把业务语言翻译成技术语言 |
| 英语原话 | "要快"、"用户体验好" | 模糊的业务表达 |
| 中文翻译 | "首屏加载 < 2秒"、"核心流程优先上线" | 具体的技术指标 |

翻译官不只是换一种说法。翻译官要把模糊的诉求变成具体的、可执行的指令。

**翻译对照表**：

| 业务语言（模糊） | 技术语言（具体） | 翻译要点 |
|-----------------|-----------------|---------|
| "要快" | "核心流程优先上线"、"首屏加载 < 2 秒" | 把"快"翻译成性能指标或优先级 |
| "用户体验好" | "操作步骤 ≤ 3 步"、"错误提示友好" | 把抽象体验拆成具体指标 |
| "像 XX 竞品那样" | 列出具体功能对比表 | 竞品对标需要具体化 |
| "数据要安全" | "敏感数据加密存储"、"操作日志可追溯" | 安全需求拆成具体措施 |

### 2.2 医生诊断类比：异议处理

当开发说"做不了"，就像医生听到病人说"肚子疼"——这不是诊断，只是症状描述。

用医生诊断类比来理解：

| 医生场景 | 产品经理-开发场景 | 说明 |
|---------|-------------|------|
| 患者说"肚子疼" | 开发说"做不了" | 表面表述，需要追问 |
| 医生追问"哪里疼？多久了？" | 产品经理追问"具体哪个环节做不了？" | 深入诊断 |
| 真实病因（胃炎、阑尾炎） | 真实原因（技术限制、排期问题、需求不清） | 确诊后对症下药 |

"做不了"的三种真实原因：

| 原因类型 | 具体情况 | 应对策略 |
|---------|---------|---------|
| 真技术限制 | 系统架构不支持、硬件限制 | 探讨替代方案、降低复杂度 |
| 资源约束 | 排期不够、人力不足 | 拆解需求、调整优先级 |
| 需求问题 | 描述不清、边界缺失 | 补充细节、重新梳理 |

### 2.3 外行参议类比：技术方案讨论

老李说："参加技术讨论，你不需要懂地基怎么打，但要能问预算够吗、有什么风险。"

用外行参议类比来理解：

| 外行场景 | 产品经理技术讨论 | 说明 |
|---------|-------------|------|
| 市长参加城市规划会 | 产品经理参加技术方案评审 | 不懂技术细节，但要能问关键问题 |
| 市长问"预算够吗？有什么风险？" | 产品经理问"这个方案有什么风险？" | 不问实现细节，问决策依据 |
| 市长不问"地基怎么打" | 产品经理不问"代码怎么写" | 避免外行话 |

问对了问题，就不会被当成外行。

### 2.4 误解纠正

**误解一**："产品只需要提需求，技术方案是开发的事。"

正确理解：产品需要理解技术方案才能判断方案是否满足业务目标。否则上线后才发现不对，为时已晚。

**误解二**："开发说做不了就真做不了。"

正确理解：可能是资源不够、需求不清，或只是"不想做"。需要追问具体原因。

## 3. 开发怎么看这个问题

开发听到"要快"，脑子里自动展开：

- 快是多快？具体时间是什么？
- 核心功能是什么？哪些可以后做？
- 技术上有什么限制？

开发说"做不了"，脑子里在想：

- 哪个具体环节做不了？
- 是技术限制还是排期问题？
- 如果要做，需要什么条件？

开发讲技术方案，脑子里有完整的链路：

- 这个方案有什么风险？
- 扩展性如何？
- 对现有系统有什么影响？
- 开发周期多久？

产品看到的是"一句话"。开发看到的是"一条链路"。信息差就在这儿。

### 3.1 技术评审中开发的真实心理

老李说："开发在技术评审会上，其实比你更紧张。"

为什么？

**开发担心的事**：

| 开发担心什么 | 为什么担心 |
|-------------|-----------|
| 方案被否决 | 前期调研白费，得重新设计 |
| 遗漏风险点 | 上线后出问题，责任在开发 |
| 范围蔓延 | 需求越聊越多，工期越来越紧 |
| 承诺不现实 | 被逼着说"能做"，结果做不完 |

开发讲方案时，心里在想："产品能不能问对问题？别问代码怎么写，问我风险在哪、时间够不够。"

**如果产品问外行话**：

开发心里会想："这人不懂技术，随便应付一下就行。"

结果：方案评审走过场，风险点没暴露，上线后出问题。

**如果产品问决策问题**：

开发心里会想："这人懂边界在哪，我得认真讲风险。"

结果：方案评审真讨论，风险点暴露出来，上线前有预案。

### 3.2 开发说"做不了"的内心挣扎

开发说"做不了"，不一定是真的技术限制。

**开发说"做不了"的真实心理**：

| 表面说法 | 内心想法 | 实际情况 |
|---------|---------|---------|
| "技术上做不了" | "这需求太复杂，我不想接" | 资源约束（不想投入） |
| "现在做不了" | "排期满了，没时间" | 排期冲突 |
| "做不了" | "你说的不清楚，我不知道要做什么" | 需求问题 |
| "真的做不了" | "架构不支持，改架构代价太大" | 真技术限制 |

产品如果不追问，就不知道是哪种情况。可能把可以做的需求删掉了。

产品追问了，开发才会说实话："其实可以做，但需要两周时间。"

这时候产品才有选择权：两周能接受吗？不能接受的话，简化版本能先上线吗？

## 4. 怎么做：翻译、讨论、异议处理

### 4.1 翻译技巧清单

| 技巧 | 说明 | 示例问题 |
|------|------|----------|
| 追问具体场景 | 把模糊诉求还原到具体使用场景 | "用户在什么场景下会遇到这个问题？" |
| 量化指标 | 把定性描述变成定量指标 | "快是多快？能接受多久？" |
| 拆解优先级 | 把"全部要"变成"先要哪些" | "如果分批上线，第一批最需要什么？" |
| 对标具体参照 | 把"像 XX"变成具体功能对比 | "竞品具体有哪些功能？我们要实现哪些？" |

### 4.2 技术讨论问题清单

**可以问的问题**：

| 类型 | 问题示例 |
|------|----------|
| 风险类 | "这个方案最可能出问题的地方是什么？" |
| 扩展性类 | "如果业务量翻 10 倍还能撑住吗？" |
| 替代方案类 | "有没有更简单的实现方式？" |
| 影响类 | "这个改动会影响现有哪些功能？" |
| 成本类 | "开发需要多长时间？" |

**不要问的问题**：

| 外行话 | 为什么不问 |
|--------|-----------|
| "代码怎么写？" | 这是实现细节 |
| "用什么框架？" | 这是技术选择，让开发决定 |
| "数据库表怎么设计？" | 这是架构细节 |

### 4.3 异议处理框架

**追问三步法**：

第一步："具体是哪个环节做不了？"
第二步："是什么原因导致做不了？"
第三步："如果要实现，需要什么条件？"

**对话模板**：

> 产品："具体是哪个环节做不了？"
> 开发："后端接口不支持这种查询。"
> 产品："是什么技术限制？"
> 开发："跨表复杂查询性能差。"
> 产品："如果时间不够，能不能先做简化版本？"
> 开发："可以，简化版两天能做完。"
> 产品："好，先做简化版上线。"

### 4.4 AI 辅助

**翻译辅助**：

提示词："业务说'要快'，请帮我列出可能的翻译方式，包括性能指标和优先级策略。"

AI 会给出：
- 性能指标：首屏加载 < 2 秒、接口响应 < 500ms
- 优先级策略：核心流程优先上线、MVP 一周内交付

**异议处理辅助**：

提示词："开发说'这个功能做不了'，请帮我列出三种可能的原因和对应的追问问题。"

AI 会给出：
- 技术限制 → 追问具体技术瓶颈
- 资源约束 → 追问需要多少时间/人力
- 需求问题 → 追问哪里描述不清

### 4.5 Checklist

**翻译检查**：
- [ ] 业务诉求已翻译成具体技术指标
- [ ] 指标有量化数字或明确优先级
- [ ] 边界条件已确认

**技术讨论检查**：
- [ ] 问的是决策问题，不是实现细节
- [ ] 至少问了风险、成本、替代方案三类问题

**异议处理检查**：
- [ ] 追问了具体原因
- [ ] 区分了技术限制 vs 资源约束 vs 需求问题
- [ ] 探讨了替代方案或简化版本

### 4.6 案例复盘：如果重来怎么做

回顾开头的三个场景，如果重来，小想该怎么做？

**场景一：业务说"要快"，翻译失败**

| 错误做法 | 如果重来 |
|---------|---------|
| 把"快"直接写进 PRD | 问业务三个问题：用户等的是什么场景？第一批最需要什么功能？能接受的上线时间是多久？ |
| 开发问"快是多快"，无言以对 | 翻译成具体指标：核心流程一周内上线、首屏加载 < 2 秒 |

复盘教训：模糊诉求必须追问具体场景和量化指标，不能把业务原话直接丢给开发。

**场景二：技术评审被当成外行**

| 错误做法 | 如果重来 |
|---------|---------|
| 听不懂全程不发言 | 问决策问题：这个方案有什么风险？如果业务量翻 10 倍还能撑住吗？ |
| 怕说错话被当成外行 | 问对了问题，开发就把你当成能参与决策的人 |

复盘教训：不懂技术细节没关系，但要能问风险、成本、替代方案。问对了问题，就不会被当成外行。

**场景三：开发说"做不了"，直接删除需求**

| 错误做法 | 如果重来 |
|---------|---------|
| 听到"做不了"直接放弃 | 追问三步：具体哪个环节做不了？是什么原因？需要什么条件才能做？ |
| 没追问，白白放弃需求 | 问完后发现：其实可以做，只是需要两周时间 |

复盘教训："做不了"不是终点，是追问起点。追问后可能找到替代方案或简化版本。

## 5. 一句话总结

**产品经理 = 业务与开发之间的翻译官。翻译对了，协作才能顺畅。**
