# CH-21 MVC 架构模式与 CQRS — 分层架构与读写分离

## 1. 痛点场景

"就改个页面显示。"

产品经理小想在评审会上轻描淡写地说完这句话，以为这个需求很简单。

开发老李却皱起眉头："这个要改后端接口。"

小想不解："不就是改个显示格式吗？前端加个逻辑就行了，为什么要改后端？"

老李开始解释："你说的改显示，有两种情况。如果只是改样式——字体变大、颜色变红、排版换位置——那确实只要改前端。但你这个需求是改数据内容，手机号要脱敏显示成 `138****1234`，脱敏逻辑必须放后端。"

小想还是疑惑："前端也能做脱敏啊，拿到完整手机号，在页面上处理一下不就行了吗？"

"问题就在这儿。"老李说，"如果前端脱敏，完整手机号还是会传到前端。有心的人通过开发者工具能看到完整数据。真正的脱敏是后端返回脱敏后的数据，前端根本拿不到完整手机号。这不是'改页面'，是'改数据接口'。"

小想愣住了。原来"改显示"不等于"改前端"。

**你是不是也遇到过类似场景？**

开发说"这个要改后端"，你心里想"不就是改个页面吗"。本章就来解决这个认知差：理解分层架构，才能准确判断改动的真实成本。

---

## 2. 核心概念：MVC、前后端分离、CQRS、Event Sourcing

### 2.1 MVC：三层分工

**MVC 把系统分成三层，各司其职。**

用一个类比来理解：餐厅。

| MVC 组件 | 餐厅角色 | 职责 |
|---------|---------|------|
| **Model（模型层）** | 厨房 | 管"食材"（数据）和"菜谱"（业务逻辑）。厨房不关心菜怎么摆盘，只关心菜做得对不对。 |
| **View（展示层）** | 菜单/摆盘 | 顾客看到的"呈现方式"。同样的菜可以摆成中式或西式。数据也一样：网页、App、小程序都可以呈现。 |
| **Controller（控制层）** | 服务员 | 接顾客订单，传达给厨房，把做好的菜端给顾客。 |

**为什么改显示不等于改前端？**

回到餐厅类比：
- 改菜单字体 = 改 View（纯前端）
- 改菜名 = 改 Model（厨房的食材要换）

小想的需求是"手机号脱敏显示"。这不是改菜单字体，是改"菜的内容"——后端要返回不同的数据，前端只是展示这个新数据。

### 2.2 前后端分离：分工更清晰

**前后端分离是前端和后端独立开发、独立部署的架构方式。**

以前没有前后端分离，程序员一个人写 HTML、CSS，还要写数据库查询、服务器逻辑。就像一个餐厅，厨师不仅炒菜，还要自己端盘子、自己写菜单。

后来发现效率低：厨师炒菜是核心能力，端盘子、写菜单是辅助能力。为什么不分工？

分工之后：
- **后端**：管数据、管业务逻辑、管服务器——就像厨房
- **前端**：管界面、管交互、管用户体验——就像菜单和摆盘
- **接口**：前端和后端沟通的"窗口"——就像出菜口

**前后端分离的好处**：
- 后端只管数据，一个接口服务多个前端（网页、App、小程序）
- 前端各自展示，网页可以详细版，App 可以精简版

就像同一个厨房，可以服务三个窗口：柜台展示详细菜品，外卖展示精简菜单，手机点餐展示卡片式菜单。厨房只管做菜，不管菜怎么展示。

**产品经理为什么要理解前后端分离？**

三个理由：
1. **评估改动影响范围**：知道是改前端还是改后端，能更准确评估工作量
2. **沟通更顺畅**：开发说"要改后端接口"，你不会误以为他在推脱
3. **规划产品演进**：知道后端接口可以服务多个前端，产品规划更合理

### 2.3 CQRS：读写分离

**CQRS（命令查询职责分离）是读和写走不同路径的设计。**

再用一个类比：银行。

| 操作类型 | 客户行为 | 银行处理方式 |
|---------|---------|-------------|
| **写操作（Command）** | 存款、取款、转账 | 必须到柜台，柜员核验身份、签字确认、更新账本 |
| **读操作（Query）** | 查余额、查交易明细 | 手机 App、ATM 随时查，秒级返回 |

**为什么读写要分开？**

银行的读写特点：
- 一天可能有 10 万次"查余额"，但只有 100 次"存取款"
- 查余额要"快"，存取款要"准"
- 一个系统很难同时"又快又准"

分开之后：
- 查余额走 App（轻量、快）
- 存取款走柜台（严谨、准）
- App 挂了不影响柜台，柜台升级也不影响 App

**什么时候适合用 CQRS？**

| 场景 | 读写特点 | CQRS 价值 |
|------|---------|----------|
| 电商商品展示 | 浏览量巨大，下单量相对小 | 读模型可缓存，写模型保精确 |
| 金融交易系统 | 查余额频繁，转账相对少 | 读模型轻量，写模型严谨审计 |
| 社交媒体 | 浏览帖子频繁，发帖相对少 | 读模型可分布式缓存 |

**什么时候不适合用 CQRS？**

- 读写比例接近 1:1 的系统：分开增加复杂度，收益不大
- 小型应用：用户量少，性能压力小，过度设计

### 2.4 Event Sourcing：存事件不存状态

**Event Sourcing（事件溯源）不存"当前状态"，存"所有变化事件"。**

用银行账本来类比：

**账本模式（Event Sourcing）**：
- 账本上记录每一笔交易：存入 1000、取出 200、存入 500
- "当前余额"是通过加总计算出来的
- 账本一旦写入，不能修改
- 可以追溯任意时间点的余额

**余额模式（传统方式）**：
- 只记录"当前余额 = 1300"
- 看不到历史交易
- 如果有人改了余额数字，无法发现

**为什么有些系统要用 Event Sourcing？**

金融合规要求："所有交易必须保存完整的审计日志，可追溯、不可篡改。"

用 Event Sourcing：
- 每个事件都有时间戳，监管可以看完整流程
- 事件一旦写入，不能修改（符合监管要求）
- 可以回溯任意时间点的状态

## 3. 开发怎么看这个问题

开发说"要改后端"，脑子里自动展开：

**MVC 视角**：
- 改动影响哪一层？
- 只改 View → 前端样式调整，工作量小
- 改 Model → 数据结构变化，要改接口、改数据库
- 改 Controller → 请求处理逻辑变化

**前后端视角**：
- 后端接口要返回新数据吗？
- 数据字段要新增吗？
- 数据格式要变化吗？

**CQRS 视角**：
- 这个改动是读操作还是写操作？
- 读操作 → 可以用缓存优化
- 写操作 → 要走严谨通道，考虑事务、审计

**Event Sourcing 视角**：
- 需要追溯历史吗？
- 需要审计日志吗？

开发看到的是"分层影响"。产品看到的是"业务需求"。信息差就在这儿：产品说"改显示"，开发想到的是改哪一层、影响什么数据、要不要审计。

## 4. 怎么做：判断改动影响的层级

### 4.1 判断前端还是后端

**需求改动是否涉及后端？**

| 改动类型 | 是否涉及后端 | 示例 |
|---------|-------------|------|
| 界面样式调整 | ❌ 不涉及 | 字体变大、颜色变红、排版换位置 |
| 交互流程优化 | ❌ 不涉及 | 点击按钮跳转逻辑、表单验证提示 |
| 显示字段变化 | ✅ 涉及 | 增加显示字段、字段脱敏、字段格式化 |
| 数据来源变化 | ✅ 涉及 | 从 A 表改到 B 表、增加数据过滤条件 |
| 业务逻辑变化 | ✅ 涉及 | 计算规则变化、状态判定规则变化 |

**判断方法**：问自己一个问题——"前端能独立完成吗？"

如果前端需要新的数据、新的字段、新的计算结果，就必须改后端。

### 4.2 判断是否需要 CQRS

当开发说"要用 CQRS"，问三个问题：

| 问题 | 如果答案是"是" | 产品判断 |
|------|-----------------|----------|
| 读写比例差距很大吗？（比如读是写的 10 倍以上） | 可能需要 | 读写差异大，分开有价值 |
| 读和写对"准确性"的要求不一样吗？ | 可能需要 | 读可以容忍短暂不一致，写必须精确 |
| 需要完整的操作历史审计吗？ | 可能需要 | 配合 Event Sourcing，价值更大 |

### 4.3 AI 辅助判断

提示词示例：
> 我有个需求：用户信息页面把手机号改成脱敏显示。请帮我分析这个改动涉及前端还是后端，以及具体影响范围。

AI 会给出分析框架：
```
这个改动涉及后端：
- 数据脱敏逻辑需在后端实现（安全考虑）
- 后端接口需返回脱敏后的手机号
- 前端只是展示后端返回的数据

影响范围：
- 后端：修改接口返回字段格式
- 前端：调整显示逻辑（如果脱敏格式变化）
- 数据库：无需修改（存储仍是完整手机号）
```

**记住**：AI 给的是分析框架，业务约束（脱敏规则、安全要求）是你补充的。

### 4.4 Checklist

**改动影响判断**
- [ ] 我能区分纯样式改动和数据内容改动
- [ ] 我知道"前端独立完成"还是"需要后端配合"
- [ ] 我能判断改动影响 MVC 的哪一层

**读写分离判断**
- [ ] 我知道读写比例差距大的场景适合 CQRS
- [ ] 我知道审计需求场景适合 Event Sourcing
- [ ] 我能用三个问题框架判断 CQRS 适用性

## 5. 一句话总结

**改显示 ≠ 改前端。改样式是前端，改数据内容是后端。**

判断公式：前端能独立完成吗？能 → 纯前端；不能 → 涉及后端。

---

| 概念 | 一句话记住 |
|------|------------|
| MVC | 餐厅分工：厨房（数据）、菜单（展示）、服务员（协调） |
| 前后端分离 | 一个厨房服务多个窗口，后端只管数据 |
| CQRS | 银行双通道：柜台存取款（严谨）、App 查余额（快捷） |
| Event Sourcing | 账本模式：存每一笔交易，不存当前余额 |
