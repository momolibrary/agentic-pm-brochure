# CH-04 接口基础 — 接口是什么 & 接口参数

## 1. 痛点场景

"调接口。"

开发老李在评审会上脱口而出。产品小想一脸困惑："什么接口？"

老李说："用户列表的接口，前端要调。"

小想追问："接口是什么意思？"

老李换了个说法："就像餐厅的服务窗口。你看着菜单点菜，服务员把需求传给厨房，厨房做好端给你。"

小想似懂非懂地点点头。

评审会继续，老李问："这个用户列表接口，传什么参数？"

小想说："就传用户ID过滤一下。"

老李追问："分页呢？每页多少条？排序规则？"

小想愣住了。需求只写了"展示用户列表"，没想过这些。

上线后，列表只显示前20条用户。后面的看不到。小想才发现漏了分页参数。

**你是不是也遇到过？**

开发说"调接口"，你听得云里雾里。需求写好了，上线才发现漏了分页、排序。本章来解决：接口到底是什么、参数漏了会怎样、产品该写清楚什么。

---

## 2. 核心概念：接口与参数

要理解"调接口"，先搞清楚两个概念。

### 2.1 接口

**接口是组件间交互的约定方式。**

就像餐厅的服务窗口：你提需求，系统给你结果。

| 餐厅类比 | 接口对应 | 点菜单类比 | 参数对应 |
|---------|----------|-----------|----------|
| 顾客 | 用户/前端 | 菜名 | 接口路径/方法 |
| 菜单 | API文档 | 份量 | 数量参数 |
| 服务员 | 接口本身 | 辣度 | 筛选参数 |
| 厨房 | 服务器 | 备注 | 其他参数 |
| 端上来的菜 | 响应数据 | - | - |

老李说的"调接口"，就是顾客向服务员点菜。你告诉接口要什么，接口传给后台，处理好把结果给你。

### 2.2 输入参数 vs 输出参数

**输入参数**：你传给接口的数据。比如筛选条件、分页规则。

**输出参数**：接口返回给你的数据。比如用户列表、总数。

示例：用户列表接口

| 类型 | 输入 | 输出 |
|------|------|------|
| 分页 | page=1, pageSize=20 | total=1000 |
| 筛选 | status=active | 用户列表 |

### 2.3 产品经理的常见误解

**误解一："接口就是一个网址"**

网址只是地址。接口还包括：请求方式、输入参数、输出格式、错误码。就像餐厅不止有门牌号，还有菜单和点单规则。

**误解二："开发自己定参数就行"**

接口参数决定前端能展示什么。产品不说清楚筛选条件，接口就会缺参数。开发问"分页怎么处理"，是想知道你的业务规则，不是技术决策。

## 3. 开发怎么看这个问题

产品说"展示用户列表"，开发脑子里自动展开三层。

**第一层：基本参数**
- 筛选条件有哪些？
- 分页怎么处理？
- 每页多少条？
- 排序规则是什么？

**第二层：返回结构**
- 返回哪些字段？
- 字段格式？（手机号脱敏吗）
- 分页信息要返回吗？

**第三层：边界情况**
- 没数据时返回什么？
- 数据量太大怎么处理？

开发问的不是刁难，是"防御性设计"。每个问题背后都有教训。

漏了分页参数，要么一次返回全部（性能爆炸），要么只返回默认数量（数据不全）。

## 4. 案例复盘：分页遗漏的三条教训

小想的用户列表只显示20条数据。复盘后提炼出三条教训：

**教训一：列表需求必写三件事**
分页规则、每页条数、默认排序。缺了任何一项，开发都只能按默认值设计，结果可能不是你想要的。

**教训二：默认值不是你的意图**
开发会假设"每页20条"是行业标准。但订单列表可能需要每页10条，商品列表可能需要每页50条。默认值不等于业务需求。

**教训三：上线才发现，成本翻倍**
分页遗漏导致用户投诉、数据不全、紧急修复。如果在需求评审时就写清楚，这些问题都不会发生。

## 5. 怎么做：参数清单模板

把接口参数写清楚，一套模板就够了。

### Step 1：列出参数清单

| 类型 | 参数名 | 说明 | 必填 | 默认值 |
|------|--------|------|------|--------|
| 分页 | page | 页码 | 是 | 1 |
| 分页 | pageSize | 每页条数 | 是 | 20 |
| 筛选 | status | 状态筛选 | 否 | 无 |
| 排序 | sortBy | 排序字段 | 否 | createTime |

### Step 2：列出返回字段

| 字段名 | 说明 |
|--------|------|
| id | 用户ID |
| name | 用户名 |
| status | 状态 |
| createTime | 创建时间 |
| total | 总数量 |

### Step 3：用AI辅助生成初稿

提示词示例：
> 请帮我生成"订单列表接口"的参数清单初稿，包含筛选、分页、排序。

AI会给框架，你补业务规则：订单列表每页10条，默认按下单时间倒序。

### Step 4：自查

**基础参数**
- [ ] 分页参数标注了吗？
- [ ] 每页条数有默认值吗？
- [ ] 排序规则写清楚了吗？

**筛选条件**
- [ ] 支持哪些筛选维度？
- [ ] 筛选参数格式？（精确还是模糊）

**返回字段**
- [ ] 返回哪些字段？
- [ ] 分页信息要返回吗？

**边界情况**
- [ ] 没数据时怎么展示？
- [ ] 数据量上限是多少？

---

**AI辅助的正确姿势**：AI出框架，你补规则。业务规则AI不知道，必须你自己补充。

---

## 6. 一句话总结

**接口 = 系统交流的约定；参数 = 约定的词汇。**

写需求时：列出参数清单 → 补业务规则 → 用模板交付。
