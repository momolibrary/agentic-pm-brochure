# CH-04 接口基础 — 接口是什么 & 接口参数

## 1. 痛点场景

"调接口。"

开发老李在评审会上脱口而出。

产品小想一脸困惑："什么接口？"

老李说："用户列表的接口，前端要调。"

小想追问："接口是什么意思？"

老李愣了一下，发现小想是真不懂。

于是他换了个说法："就像餐厅的服务窗口。你（前端）看着菜单点什么菜，服务员把你的需求传给厨房，厨房做好端给你。"

小想似懂非懂地点点头。

但评审会继续往下走时，问题又来了。

老李问："这个用户列表接口，传什么参数？"

小想说："就传用户ID过滤一下。"

老李追问："分页呢？每页多少条？排序规则？"

小想愣住了——需求里只写了"展示用户列表"，完全没想过这些。

上线后，列表只显示前20条用户。后面的用户看不到。小想才发现漏了分页参数。

**你是不是也遇到过？**

开发说"调接口"、"传参数"，你听得云里雾里。需求写好了，上线才发现漏了分页、排序。本章就来解决：接口到底是什么、参数漏了会怎样、产品该在需求里写清楚什么。

---

## 2. 核心概念：接口、参数、输入、输出

要理解老李说的"调接口"，先搞清楚几个基本概念。

### 2.1 接口

**接口是组件间交互的约定方式。**

就像餐厅的服务窗口：你提需求，系统给你结果。

| 餐厅组件 | 接口对应 | 解释 |
|---------|----------|------|
| 顾客 | 用户/前端 | 提出需求的人或系统 |
| 菜单 | API文档 | 展示可用的服务和数据 |
| 服务员 | 接口本身 | 接收请求、传递响应 |
| 点菜单 | 请求参数 | 你传给接口的数据 |
| 厨房 | 服务器/数据库 | 实际处理逻辑的地方 |
| 端上来的菜 | 响应数据 | 接口返回的结果 |

老李说的"调接口"，就是顾客向服务员点菜。你告诉接口要什么数据、要什么功能，接口传给后台系统，后台处理好，接口把结果给你。

### 2.2 参数

**参数是调用接口时传入/返回的数据项。**

就像点菜单上的每一项：

| 点菜单选项 | 参数对应 | 解释 |
|-----------|----------|------|
| 菜名 | 接口路径/方法 | 要什么功能 |
| 份量 | 数量参数 | 要多少条数据 |
| 辣度 | 筛选参数 | 按什么条件筛选 |
| 备注 | 其他参数 | 特殊需求（如排序） |

如果你点菜时没说份量，服务员可能默认给你一份标准份。接口也一样——参数没写清楚，系统就按默认值处理。漏了分页参数，可能只给你默认20条数据。

### 2.3 输入参数 vs 输出参数

**输入参数**：你传给接口的数据。比如筛选条件、分页规则、排序方式。

**输出参数**：接口返回给你的数据。比如用户列表、总数、分页信息。

一个用户列表接口的参数示例：

| 参数类型 | 输入参数 | 输出参数 |
|---------|----------|----------|
| 分页 | page=1, pageSize=20 | total=1000, page=1 |
| 筛选 | status=active | 用户列表数据 |
| 排序 | sortBy=createTime, sortOrder=desc | - |

### 2.4 为什么产品经理容易误解

**误解一："接口就是一个网址"**

网址只是接口地址的一部分。接口还包括：请求方式、输入参数、输出格式、错误码、权限。

就像餐厅不止有门牌号，还有菜单、点单规则、付费方式。

**误解二："开发自己定接口参数就行"**

接口参数决定了前端能展示什么、用户能操作什么。产品不说清楚筛选条件和展示字段，接口就会缺参数或多参数。

开发问"分页怎么处理"，不是为了刁难。他们需要知道传什么参数、返回什么字段、如何分页。这些是产品的业务规则，不是开发的技术决策。

## 3. 开发怎么看这个问题

产品说"展示用户列表"，开发脑子里自动展开：

**第一层：基本参数**
- 筛选条件有哪些？（状态、关键词、日期范围）
- 分页怎么处理？（page/pageSize 还是 offset/limit）
- 默认每页多少条？
- 排序规则是什么？

**第二层：返回结构**
- 返回哪些字段？（用户ID、姓名、状态、时间）
- 字段格式？（手机号脱敏还是完整）
- 分页信息要不要返回？（总数、当前页）

**第三层：边界情况**
- 没数据时返回什么？
- 数据量太大怎么处理？
- 并发请求怎么处理？

开发问的不是刁难，是"防御性设计"。每个问题背后都有教训。

漏了分页参数，要么一次返回全部数据（性能爆炸），要么只返回默认数量（数据不全）。

**案例：小想的分页遗漏**

上线后，用户列表只显示20条。后面的数据用户看不到。小想才发现：需求只写了"展示用户列表"，没写分页规则。开发按默认逻辑设计，每页20条。

老李复盘时说："列表页需求要写清楚三件事：分页规则、每页条数、默认排序。不然开发没法设计接口。"

## 4. 怎么做：参数清单与AI辅助

把接口参数写清楚，一套模板就够了。

### Step 1：列出参数清单

用这个模板，一次写全：

| 参数类型 | 参数名 | 说明 | 必填 | 默认值 | 示例值 |
|---------|--------|------|------|--------|--------|
| 分页 | page | 页码，从1开始 | 是 | 1 | 1 |
| 分页 | pageSize | 每页条数 | 是 | 20 | 20 |
| 筛选 | status | 状态筛选 | 否 | 无 | active |
| 筛选 | keyword | 关键词搜索 | 否 | 无 | 张 |
| 排序 | sortBy | 排序字段 | 否 | createTime | updateTime |
| 排序 | sortOrder | 排序方向 | 否 | desc | asc |

### Step 2：列出返回字段

| 字段名 | 说明 | 示例值 |
|--------|------|--------|
| id | 用户ID | 12345 |
| name | 用户名 | 张三 |
| status | 状态 | active |
| createTime | 创建时间 | 2026-04-19 10:00:00 |
| total | 总数量 | 1000 |
| page | 当前页码 | 1 |
| pageSize | 每页条数 | 20 |

### Step 3：用AI辅助生成初稿

提示词示例：
> 请帮我生成"订单列表接口"的参数清单初稿，包含筛选条件、分页、排序。

AI会给一个框架，你补业务规则：

| 参数 | AI给的 | 你要补的 |
|------|--------|----------|
| status | 状态筛选 | 枚举：pending/paid/cancelled |
| pageSize | 每页条数 | 业务规则：订单列表每页10条 |
| sortBy | 排序字段 | 默认按下单时间倒序 |

### Step 4：自查（checklist）

**基础参数**
- [ ] 分页参数标注了吗？（page/pageSize）
- [ ] 每页条数有默认值吗？
- [ ] 排序规则写清楚了吗？

**筛选条件**
- [ ] 支持哪些筛选维度？
- [ ] 筛选参数的格式？（精确匹配还是模糊搜索）

**返回字段**
- [ ] 返回哪些字段列清楚了吗？
- [ ] 分页信息（总数）要返回吗？
- [ ] 字段格式有要求吗？（手机号脱敏）

**边界情况**
- [ ] 没数据时前端怎么展示？
- [ ] 数据量上限是多少？

---

**AI辅助的正确姿势**：AI出框架，你补规则。

AI能帮你列出常见参数。但业务规则（订单列表每页10条、默认按下单时间排序）AI不知道，必须你自己补充。

---

## 5. 一句话总结

**接口 = 系统交流的约定；参数 = 约定的词汇。**

写需求时：列出参数清单 → 补业务规则 → 用模板交付。

---

| 概念 | 一句话记住 |
|------|------------|
| 接口 | 餐厅服务窗口，你点菜它端饭 |
| 参数 | 点菜单上的选项，菜名、份量、辣度、备注 |
| 输入参数 | 你传给接口的数据 |
| 输出参数 | 接口返回给你的数据 |
