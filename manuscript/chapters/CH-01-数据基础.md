# CH-01 数据基础 — 数据是什么 & 数据结构

## 1. 痛点场景

"加个字段。"

产品经理小王在需求文档里写下这句话，发给开发。

半小时后，开发老李的消息来了："什么类型？多长？必填吗？格式要求？"

小王一脸懵：不就是个字段吗？

老李的追问没停："长度限制呢？默认值？要不要唯一？和其他表有关系吗？"

需求评审会上，小张提了类似的需求："加个推荐人字段。"

老李习惯性地追问：类型、长度、必填、格式、推荐人是否要校验存在、一个人能被多少人推荐……

小张额头冒汗，需求评审延期。

**你是不是也遇到过？**

开发总嫌需求里的"字段"信息不够，追着一连串问题不放。本章就来解决这个问题：为什么"加个字段"四个字对开发来说意味着一堆追问，以及如何一次性把话说清楚。

---

## 2. 核心概念：数据、字段、类型、表、关系

要理解老李为什么追问，先搞清楚几个基本概念。

### 2.1 数据

**数据是信息的载体。**

用户名字"张三"、手机号"13800138000"、注册时间"2024-01-15"——这些都是数据。但数据不能乱存，每条数据都有**类型**（是什么）、**格式**（长什么样）、**约束**（有什么规则）。

### 2.2 字段

**字段是数据的最小存储单元，有名称和类型。**

就像 Excel 表格里的一列。"姓名"列存名字，"手机号"列存手机号——每一列就是一个字段。

字段必须说清楚两件事：**叫什么**（字段名）、**存什么**（数据类型）。你只说"加个字段"，没说叫什么、存什么，开发当然要追问。

### 2.3 数据类型

常见的四种基本类型：

| 类型 | 存什么 | 前端怎么展示 |
|------|--------|--------------|
| 文本 | 字符串 | 输入框 |
| 数字 | 整数/小数 | 数字键盘 |
| 日期时间 | 时间值 | 日期选择器 |
| 布尔 | 是/否 | 开关/复选框 |

**类型决定了字段能存什么、怎么校验、前端怎么展示。**

类型选错了会出问题。比如"年龄"存成文本，用户填"二十五岁"，程序就没法算年龄了。

### 2.4 表

**表是一组相关字段的集合。**

用户表可能包含：用户ID、姓名、手机号、邮箱、注册时间、是否会员。

就像 Excel 的一个 Sheet，每个 Sheet 存一类信息。

### 2.5 关系

**关系是表之间的关联方式。**

用户表和订单表通过"用户ID"关联——一个用户可以有多个订单。就像 Excel 里用 VLOOKUP 把两个 Sheet 的数据关联起来。

```
┌─────────────┐         ┌─────────────┐
│   用户表     │         │   订单表     │
├─────────────┤         ├─────────────┤
│ 用户ID (PK) │◄────────│ 用户ID (FK) │
│ 姓名        │         │ 订单ID      │
│ 手机号      │         │ 下单时间    │
└─────────────┘         └─────────────┘
```

> **PK** = 主键（唯一标识一条记录）
> **FK** = 外键（指向另一个表的主键）

## 3. 开发怎么看这个问题

产品说"加个字段"，开发脑子里自动展开：

**第一层：基本属性**
- 字段叫什么？（英文命名）
- 数据类型？（文本/数字/日期/布尔）
- 长度限制？
- 必填还是可选？
- 默认值？

**第二层：约束规则**
- 格式要求？（手机号11位、邮箱格式）
- 取值范围？（年龄0-150）
- 唯一性？（手机号不能重复）
- 参照关系？（部门ID必须在部门表存在）

**第三层：性能考虑**
- 要建索引吗？
- 数据量大吗？
- 需要排序分页吗？

开发问的不是刁难，而是"防御性设计"。每个问题背后都是经验教训——没想清楚就上线，后期返工、数据清洗、用户投诉。

**产品经理和开发的信息差**：产品看到的是"业务需求"，开发看到的是"存储细节"。产品一句话，开发要思考十几个问题。不说清楚，开发只能猜，猜错了就是线上 bug。

## 4. 怎么做：从 AI 初稿到可交付字段表

把字段定义写清楚，一条流水线就够了。

### Step 1：用 AI 生成初稿

提示词示例：
> 请帮我生成"订单表"的字段定义初稿，订单包含：订单号、用户、商品列表、总金额、下单时间、支付状态。

AI 会给一个框架：
```
订单表字段：
- order_id: 订单号，文本，唯一
- user_id: 用户ID，整数，非空
- total_amount: 总金额，小数
- order_time: 下单时间，日期时间
- payment_status: 支付状态，整数
```

### Step 2：人工补业务约束

AI 不知道你的业务规则，必须你来补：

| 字段 | AI给的 | 你要补的 |
|------|--------|----------|
| order_id | 文本，唯一 | 格式：YYYYMMDD+6位随机数 |
| total_amount | 小数 | 精度2位，范围0.01-999999.99 |
| payment_status | 整数 | 枚举：0-待支付，1-已支付，2-已取消，3-已退款 |

### Step 3：整理成字段清单

用这个模板，一次写全：

| 字段名 | 中文名 | 类型 | 长度 | 必填 | 默认值 | 约束 |
|--------|--------|------|------|------|--------|------|
| order_id | 订单号 | 文本 | 20 | 是 | 无 | YYYYMMDD+6位随机，唯一 |
| user_id | 用户ID | 整数 | - | 是 | 无 | 外键关联用户表 |
| total_amount | 总金额 | 小数 | 2位 | 是 | 无 | 0.01-999999.99 |
| order_time | 下单时间 | 日期时间 | - | 是 | 当前时间 | |
| payment_status | 支付状态 | 整数 | - | 是 | 0 | 0-待支付，1-已支付，2-已取消，3-已退款 |

### Step 4：自查（checklist）

**基础信息**
- [ ] 每个字段有中文名吗？
- [ ] 每个字段标注类型了吗？
- [ ] 文本字段有长度限制吗？

**约束规则**
- [ ] 必填/可选标注了吗？
- [ ] 非必填字段有默认值吗？
- [ ] 格式要求写清楚了吗？（手机号、邮箱）
- [ ] 取值范围说明了吗？（年龄、评分）
- [ ] 唯一性要求标注了吗？（手机号、用户名）

**关系说明**
- [ ] 和其他表的关联关系说明了吗？

---

**AI 辅助的正确姿势**：AI 出框架，你补规则。

AI 能帮你列出常见字段、给出基本类型。但业务约束（手机号必须11位、一个用户只能属于一个部门）AI 不知道，必须你自己补充。

开头故事里的小想，就是用 AI 生成了字段表，漏了"手机号必须11位"的约束，上线后用户填10位也能注册，短信验证码发不出，用户投诉。

现在你知道小想该怎么做了。

---

## 5. 一句话总结

**每个字段 = 名称 + 类型 + 格式 + 约束。**

写需求时：AI 出初稿 → 你补业务规则 → 用字段清单模板交付。

---

| 概念 | 一句话记住 |
|------|------------|
| 数据 | 有类型、格式、约束的信息载体 |
| 字段 | Excel 的一列，最小存储单元 |
| 数据类型 | 决定能存什么、怎么展示 |
| 表 | Excel 的一个 Sheet |
| 关系 | VLOOKUP 关联两张表 |
