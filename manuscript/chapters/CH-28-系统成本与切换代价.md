# CH-28 系统成本与切换代价 — 开门成本、迭代运维与切换成本

## 1. 痛点场景

"做个小系统应该很便宜吧？"

周一上午，预算审批会。老板盯着老李的报价单，眉头皱起来。

"一个简单的内部审批系统，几页表单、几个按钮。你们报八万？"

老李还没开口，小想抢着解释："老板，这个预算可能不够。光是基础设施就要不少钱。"

"基础设施？不就是一台服务器吗？云服务器很便宜的。"

小想打开成本清单："服务器确实不贵，但不止服务器。给您算算：云服务器一年三千，域名五百，SSL证书一千，数据库两千，监控告警一千，日志系统一千，数据备份五百……加起来一万多，这还没写一行代码。"

老板愣住："一万多？还没写代码就花了这么多？"

老李补了一句："这叫开门成本。还没开门营业就要花的钱。"

---

## 2. 核心概念：开门成本、运维成本、切换成本

### 2.1 开门成本

**开门成本是系统上线前的最低基础设施投入。**

还没写代码，这些钱已经花出去了。

| 项目 | 说明 | 一年费用（参考） |
|------|------|-----------------|
| 云服务器 | 运行代码的基础 | ¥3,000+ |
| 域名 | 让用户能找到系统 | ¥500+ |
| SSL证书 | HTTPS安全访问 | ¥1,000+ |
| 数据库 | 存业务数据 | ¥2,000+ |
| 监控告警 | 出问题能排查 | ¥1,000+ |
| 日志系统 | 记录运行状态 | ¥1,000+ |
| 数据备份 | 数据不能丢 | ¥500+ |

**类比**：开餐厅不只是买食材。

租金、装修、证件、设备都是开门就要花的钱。还没卖出一道菜，这些钱已经花出去了。IT系统也一样——服务器、域名、证书、监控、备份，还没写代码就已经花钱了。

### 2.2 运维成本

**运维成本是系统上线后的持续维护投入。**

系统上线不是结束，而是开始。

| 维护类型 | 具体内容 | 频率 |
|----------|----------|------|
| 安全补丁 | 操作系统、依赖库的安全更新 | 每月1-2次 |
| 依赖升级 | 框架版本升级 | 每季度1次 |
| Bug修复 | 用户反馈的问题 | 按需 |
| 性能优化 | 数据量增长后的查询优化 | 按需 |
| 监控巡检 | 检查日志、告警、资源使用 | 每周 |

**关键数据**：软件维护成本占总生命周期成本的**60%-80%**（开发成本仅占20%-40%）。

开发花两月，运维可能要五年。

### 2.3 切换成本

**切换成本是更换系统或供应商的真实代价。**

换系统不只是买新软件。

| 成本项 | 说明 |
|--------|------|
| 数据迁移 | 三年积累的业务数据迁移到新系统 |
| 数据清洗 | 旧数据格式不标准，需要清洗转换 |
| 接口重对 | 与其他系统的对接需要重新开发 |
| 流程适配 | 新系统流程与现有流程不匹配 |
| 人员培训 | 全公司员工重新学习新系统 |
| 业务中断 | 切换期间业务停滞损失 |
| 历史兼容 | 旧数据在新系统中的兼容问题 |

**类比**：换系统就像搬家。

你以为只是把东西搬过去？要整理打包、改水电、通知所有人、重新熟悉环境。搬家成本远比想象中贵。换系统也一样——数据迁移、接口重对、人员培训加起来，比重新开发还贵。

### 2.4 产品经理容易误解的地方

| 误解 | 正确理解 |
|------|----------|
| "系统上线就做完了" | 上线是开始，维护是持续的 |
| "不出问题就不用维护" | 安全补丁、依赖升级是预防性维护 |
| "小系统成本低" | 开门成本相对固定，小系统也要这些基础 |
| "换系统就是买新软件" | 切换成本可能比重新开发还贵 |
| "不好用就换一个" | 切换成本被严重低估 |

## 3. 开发怎么看这个问题

开发/运维眼中的系统成本，和产品经理看到的不一样。

**产品经理看到的是**：功能列表、上线时间、用户价值。

**开发看到的是**：服务器配置、依赖版本、安全漏洞、数据库备份、日志轮转……

半年前有个审批系统上线，老板觉得"做完了"，不再投入维护资源。三个月后，运维发现一个严重安全漏洞——依赖的开源库被爆出CVE（安全漏洞编号），攻击者可能拿到系统权限。

老板问：多久能修好？

运维回答：顺利一周，不顺利两周。因为半年没维护，积累的问题要一起处理。

老板不解："这系统才花五万做，修个漏洞要花这么久？"

这就是运维成本的真实。系统不是"有事才叫"的模式，需要持续关注。每个系统平均每月至少投入**0.5-1人天**的维护时间。

### 4.2 运维预算规划

运维预算 = 开发成本 × (60%-80%) / 运维年数

例：开发成本10万，运维5年，运维预算约12-16万。

| 项目 | 频率 | 估算投入 |
|------|------|----------|
| 安全补丁 | 每月1-2次 | 每年___人天 |
| 依赖升级 | 每季度1次 | 每年___人天 |
| Bug修复 | 按需 | 每年___人天 |
| 监控巡检 | 每周 | 每年___人天 |

### 4.3 切换成本评估清单

换系统前先评估：

| 成本项 | 估算工作量 | 是否可控 |
|--------|------------|----------|
| 数据迁移 | ___月 | ☐ |
| 数据清洗 | ___月 | ☐ |
| 接口重对 | ___月 | ☐ |
| 流程适配 | ___月 | ☐ |
| 人员培训 | ___周 | ☐ |
| 业务中断风险 | ___天 | ☐ |
| **总估算** | **___月** | |

**决策公式**：切换成本 > 改造现有系统成本 → 优先改造，不换。

### 4.4 AI 辅助生成成本清单

提示词示例：

> 请帮我生成一个审批系统的开门成本清单，包含服务器、域名、证书、数据库、监控、备份等项目，给出阿里云参考定价。

AI 会给出框架，但你需要补充：
- 实际配置需求（服务器规格、数据库容量）
- 公司已有的基础设施（是否可复用）
- 合规要求（SSL证书级别、数据备份频率）

## 5. 一句话总结

**IT系统不是一次消费品，是持续运营的资产。**

开门成本是门票，运维成本是续费，切换成本是搬家。
