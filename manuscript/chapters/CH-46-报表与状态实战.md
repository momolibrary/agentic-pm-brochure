# CH-46 报表与状态实战 — 数据查询设计与状态机完整实现

> 本章解决两个问题：报表设计不只是"展示数据"，状态机不只是"画几个圈"。

---

## 一、痛点场景

### 场景一：报表上线三天被叫停

周一上午，产品评审会。

运营团队上线了一个"订单报表"功能。报表上线三天后，客服小王在处理投诉时，看到了VIP客户的消费记录：客户姓名、手机号、50万元的订单金额、高档住宅区收货地址。

VIP客户投诉到公司高管。安全部门紧急介入。

安全负责人问："普通客服能看到VIP客户完整信息，数据权限是谁设计的？"

小想愣住："我以为报表就是展示数据，设为'全员可见'……"

老李叹气："报表权限不是'能登录就能看'。客服只能看自己负责的客户，经理才能看全部。这是'行级权限'——同一张报表，不同角色看到的行数不同。"

### 场景二：促销活动导出崩溃

周三下午，运营经理导出促销活动全部订单数据（约50万条）。

页面显示"导出中……请稍候"。

五分钟后，超时报错"导出失败"。

同时，其他用户反馈系统访问变慢。服务器CPU飙升到100%。

技术同事说："导出功能把服务器内存吃满了，其他请求都在排队。"

小想问："为什么会这样？我只设计了导出按钮……"

老李回答："导出不是无限导出。50万条数据一次性导出，服务器撑不住。你要设计数量限制、异步导出方案。"

### 场景三：同一订单发了两次货

周五下午，仓库反馈：订单#12345发了两次货，用户投诉"为什么给我发了两份"。

排查发现：两个操作员同时看到订单状态"已支付"，同时点击"发货"。系统没有并发保护，两个请求都成功了。

老李问："小想，你的订单状态机有并发锁吗？"

小想："并发？我以为状态机就是画几个圈和箭头……"

老李："状态机不只是画图。并发冲突、超时处理、异常路径，都要设计。"

## 二、核心概念：报表设计六要素

设计一个数据报表，就像设计餐厅的点菜系统。

顾客（用户）需要从众多菜品（数据）中找到自己想要的。餐厅需要提供清晰的菜单（报表界面），让顾客快速定位、筛选、选择。

| 报表设计要素 | 餐厅点菜系统 | 常见错误 |
|------------|-------------|---------|
| **展示字段** | 菜品展示信息（名称、价格、口味） | 把数据库所有字段都搬上去，表格臃肿 |
| **筛选条件** | 菜品分类筛选（菜系、价格区间） | 使用数据库字段名（status_code=1），用户不理解 |
| **排序规则** | 菜品排序（推荐、价格、人气） | 没定义默认排序，用户找不到重点 |
| **分页策略** | 菜品翻页/分类展示 | 一次加载全部数据，页面卡顿 |
| **数据权限** | VIP专属菜/普通顾客可选菜 | 只设计"能看/不能看"，忽略行级权限 |
| **导出功能** | 打包带走/外卖清单 | 无数量限制，大数据量导出崩溃 |

### 展示字段：用户视角，不是数据库视角

菜单不会列出每道菜的原料、烹饪时间、厨师信息。只展示顾客关心的：名称、价格、口味。

报表也一样。不要把数据库所有字段都列出来。站在用户视角，选择"用户关心什么"，而非"数据库有什么"。

### 筛选条件：业务语言，不是技术语言

顾客不会用"食材产地"或"厨师编号"筛选菜品。他们用"菜系""价格区间""辣度"。

报表筛选条件要转化为"用户能理解的业务语言"，而非"数据库字段名"。

### 数据权限：三个层次

报表权限分三个层次：

| 权限层次 | 说明 | 示例 |
|---------|------|------|
| **功能权限** | 能不能访问这个报表 | 经理能进"财务报表"，客服不能进 |
| **行级权限** | 能看到哪些数据行 | 经理看全部订单，客服只看自己负责的客户 |
| **列级权限** | 能看到哪些字段 | 普通员工看订单信息，薪资字段脱敏 |

产品经理最容易忽略的是"行级权限"——同一张报表，不同角色看到的行数不同。

### 导出功能：必须有限制

导出不是"无限导出"：

- Excel文件有物理限制（行数、文件大小）
- 服务器内存有上限
- 用户等待时间有容忍度

设计要点：设置单次导出上限（如10万条），超过上限提示用户筛选或分批导出。大数据量考虑异步导出方案。

### 产品经理容易误解什么？

**误解一："报表就是查数据库再展示出来"**

正确理解：报表涉及数据权限、查询性能、分页策略、导出限制等多个维度。不是简单的"查→展示"。

**误解二："数据权限就是'能登录就能看'"**

正确理解：数据权限有功能权限、行级权限、列级权限三个层次。最容易被忽略的是行级权限。

**误解三："导出就是简单转成文件"**

正确理解：导出需要数量限制、异步处理、性能保护。无限制导出会导致服务器崩溃。

## 三、核心概念：状态机设计六步骤

设计一个订单状态机，就像设计餐厅的出餐流程。

每张订单从"顾客下单"到"结账离开"，经历多个环节。每个环节有特定状态，状态之间有固定流转规则。

| 状态机要素 | 餐厅出餐流程 | 常见错误 |
|-----------|-------------|---------|
| **状态定义** | 出餐环节（下单、备菜、烹饪、上菜） | 只列出状态名称，不定义跳转规则 |
| **流转路径** | 环节衔接顺序 | 设计成"任何状态都能跳到任何状态" |
| **流转条件** | 环节触发条件 | 漏了前置条件（如支付成功才能发货） |
| **并发冲突** | 多桌同时点单 | 忽略并发场景，导致同一订单被多次操作 |
| **异常状态** | 退菜、换菜、投诉 | 只画正常流程，忽略异常路径 |
| **超时处理** | 菜品超时未上 | 没设计超时状态（待支付超时自动取消） |

### 状态机设计六步骤

**步骤一：列出所有状态**

不只是正向状态（待支付→已支付→已发货），还要列出：
- 逆向状态（取消、退款）
- 异常状态（失败、争议）
- 终态（已完成、已取消）

**步骤二：定义合法流转路径**

哪些状态可以跳到哪些状态？待支付只能跳到已支付或已取消，不能直接跳到已发货。

**步骤三：标注流转条件**

每个跳转的触发条件是什么？支付成功触发待支付→已支付。商家发货触发已支付→已发货。

**步骤四：处理并发冲突**

多操作同时触发怎么办？加版本号/时间戳，拒绝过期请求。关键操作加锁，防止并发执行。

**步骤五：设计异常状态**

超时怎么处理？待支付超时30分钟自动取消。失败怎么处理？支付失败留在待支付状态。

**步骤六：输出状态流转图**

画出全流程，包含正常路径、异常路径、超时路径。终态用双圈标注。

### 产品经理容易误解什么？

**误解一："状态就几个，画个图就完了"**

正确理解：真实状态机需要考虑并发冲突、超时处理、异常回退。不只是画几个圈。

**误解二："状态流转开发自己设计就行"**

正确理解：订单能不能取消、退款怎么走——这是产品决策。不定义清楚，开发按最宽松实现。

**误解三："只画正常流程，异常路径不用管"**

正确理解：异常状态（取消、退款、超时）是真实会发生的场景。不设计异常路径，上线后遇到异常无法处理。

## 四、开发视角

### 开发看报表设计

开发想的不只是"展示什么字段"，而是：

- 数据权限规则怎么写？SQL查询条件怎么拼接？
- 大数据量查询怎么优化？要不要建索引、预计算？
- 导出任务怎么处理？同步还是异步？任务队列怎么设计？
- 分页怎么实现？前端传页码，后端怎么查询？

开发常见追问：

- "行级权限规则是什么？客服看哪些客户的订单？"
- "大数据量查询有索引吗？没索引查询会很慢。"
- "导出上限是多少？超过上限怎么处理？"

### 开发看状态机设计

开发想的不只是"有几个状态"，而是：

- 状态怎么存储？数据库字段还是状态机？
- 跳转条件怎么判断？支付成功怎么判断？签收怎么判断？
- 并发时状态冲突怎么办？两个发货请求同时到达怎么处理？
- 超时怎么触发？定时任务扫描还是事件监听？

开发常见追问：

- "并发发货怎么处理？两个操作员同时点发货。"
- "超时取消怎么触发？定时任务还是延迟队列？"
- "终态有哪些？终态不能再跳转对吧？"

## 五、怎么做

### 报表设计三步法

**第一步：定义字段表、筛选条件表**

字段表：
```
| 字段名称 | 数据来源 | 用户可见 | 说明 |
|---------|---------|---------|------|
| 订单编号 | order.id | ✓ | 主键，不可筛选 |
| 订单状态 | order.status | ✓ | 转化为业务语言：待支付/已支付/已发货 |
| 下单时间 | order.create_time | ✓ | 默认排序字段 |
| 客单价   | order.amount | ✓ | 用户关心 |
```

筛选条件表：
```
| 筛选项 | 对应字段 | 筛选类型 | 说明 |
|--------|---------|---------|------|
| 订单状态 | status | 下拉选择 | 待支付/已支付/已发货/已完成/已取消 |
| 下单时间 | create_time | 时间范围 | 开始时间-结束时间 |
| 客单价   | amount | 数值范围 | 最小值-最大值 |
```

**第二步：定义权限规则表**

```
| 角色 | 数据范围 | 敏感字段 | 导出权限 |
|------|---------|---------|---------|
| 经理 | 全部订单 | 可看全部字段 | 可导出，上限10万条 |
| 主管 | 本部门订单 | 手机号脱敏 | 可导出，上限5万条 |
| 客服 | 自己负责的客户 | 手机号脱敏、地址脱敏 | 可导出，上限1万条 |
```

**第三步：定义导出策略**

- 单次导出上限：10万条
- 超过上限提示："数据量过大，请筛选后导出"
- 大数据量异步导出：点击导出→后台生成任务→完成后发送下载链接

### 状态机设计三步法

**第一步：列出所有状态（含异常）**

```
正向状态：待支付 → 已支付 → 已发货 → 已签收 → 已完成
逆向状态：已取消、退款中、已退款
异常状态：支付失败、发货失败
终态：已完成、已取消、已退款
```

**第二步：画出流转路径和条件**

```
正向路径：
待支付 ─[支付成功]→ 已支付 ─[商家发货]→ 已发货 ─[用户签收]→ 已签收 ─[确认完成]→ 已完成(终态)

逆向路径：
待支付 ─[用户取消]→ 已取消(终态)
已支付 ─[申请退款]→ 退款中 ─[退款成功]→ 已退款(终态)

超时路径：
待支付 ─[30分钟未支付]→ 已取消(终态)
已发货 ─[7天未签收]→ 自动签收 → 已签收

约束标注：
❌ 待支付 → 已发货（禁止跳过支付）
❌ 已发货 → 待支付（禁止回退）
```

**第三步：定义并发和超时处理**

并发处理：
- 加版本号：状态变更时检查版本号，版本已变则拒绝
- 幂等设计：同一请求重复执行不产生副作用

超时处理：
- 待支付超时：30分钟未支付自动取消，释放库存
- 发货超时：7天未签收自动签收

### AI可以如何辅助

**报表设计AI提示词：**

```
请帮我设计一个订单报表，要求：

1. 用户角色：经理、主管、客服
2. 权限差异：经理看全部，客服只看自己负责的客户
3. 输出格式：
   - 字段表（用户视角，不是数据库视角）
   - 筛选条件表（业务语言）
   - 权限规则表（行级权限）
   - 导出策略（数量限制）

4. 请特别关注：
   - 敏感字段如何处理？
   - 大数据量导出如何处理？
```

AI会生成初稿，但你需要检查：
- AI可能漏了"行级权限"的细项
- AI可能没有设计导出数量限制
- AI可能没有追问敏感字段脱敏规则

**状态机AI提示词：**

```
请帮我设计一个电商订单状态机，要求：

1. 场景：下单→支付→发货→签收→完成，含取消、退款
2. 输出格式：
   - 状态列表（含逆向状态、终态）
   - 流转路径和触发条件
   - 禁止跳转的路径
   - 超时处理规则

3. 请特别关注：
   - 并发冲突如何处理？
   - 超时状态如何定义？
```

AI会生成骨架，但你需要补充：
- 并发保护机制（版本号、锁）
- 超时触发方式（定时任务还是延迟队列）
- 异常路径的完整定义

## 六、案例复盘

回看三个事故场景：

**事故一**：客服看到VIP客户完整信息。问题根源：只设计"能登录就能看"，忽略行级权限。教训：权限分三层——功能权限、行级权限、列级权限。

**事故二**：导出50万条数据崩溃。问题根源：导出无数量限制，服务器撑不住。教训：导出必须有上限，大数据量用异步方案。

**事故三**：同一订单发两次货。问题根源：状态机无并发保护，两个发货请求同时成功。教训：状态机必须设计并发保护（版本号、锁）。

一句话总结：报表设计六要素（字段、筛选、排序、分页、权限、导出），状态机六步骤（状态、流转、条件、并发、异常、图）。漏一个，上线就踩坑。
