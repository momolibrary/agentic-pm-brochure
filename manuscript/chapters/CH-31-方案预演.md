# CH-31 方案预演 — AI辅助预演、边界检查与方案对比

## 1. 痛点场景

上线一周，周五复盘会。

测试小刘报告："这周用户反馈了30个问题，5个严重bug——订单状态流转异常、支付超时没提示、并发高时系统卡住..."

老李问小想："需求评审时，这些场景讨论过吗？"

小想摇头："开发说方案没问题，我就让他开工了..."

老李又问张开发："你说方案没问题，这些场景考虑过吗？"

张开发有点尴尬："说实话，我只考虑了正常流程。支付超时、并发高这些边界场景...确实没想过。"

老李叹气："开发说的'没问题'是指'正常流程没问题'，不代表'所有场景没问题'。不做预演，上线后用真实用户帮你发现问题。"

**本章解决一个问题**：怎么在开发动手前，把方案"预演"一遍，发现遗漏和风险。

---

## 2. 核心概念：方案预演、AI辅助预演、边界检查、方案对比

### 2.1 方案预演是什么？

**方案预演是在开发实现前，先模拟方案效果，提前发现问题。**

就像买房看样板间。开发商说"这套户型满足你需求"，你不会直接下单。你会先看样板间：
- 实际走进房间感受空间
- 看装修效果符不符合预期
- 发现厨房太小、储物空间不够

看完样板间，你才决定下单或换户型。

**如果不看样板间直接下单，装修完发现"和我想的不一样"，改起来要花多少钱？**

| 发现时机 | 时间成本 | 金钱成本 |
|---------|----------|----------|
| 看样板间发现问题 | 2小时 | 0元 |
| 装修完发现问题 | 2周 | 几万块 |

方案预演就是软件的"样板间参观"——开发动手前，先模拟效果，发现问题。

### 2.2 AI 辅助预演

以前预演要靠人工想象。现在可以用 AI 模拟。

**AI辅助预演就像用AI生成装修效果图——不用真正装修，就能看到效果。**

你告诉 AI："我要装修客厅，北欧风格，沙发靠墙，电视柜在对面。"

AI几秒钟生成效果图。你看到整体风格、空间布局。发现问题直接改描述，AI重新生成。

**改效果图几秒，改真实装修几周。**

软件也一样。你告诉 AI："订单流程是下单→支付→发货→完成。帮我预演这个流程，看看有没有遗漏。"

AI 回复：
- 漏了"取消订单"场景
- 漏了"支付失败"状态
- 漏了"发货超时"处理

你补充这些场景，需求才完整。

### 2.3 边界检查

**边界检查就像汽车上市前的碰撞测试。**

车企不只测"正常驾驶"。还要测正面碰撞、侧面碰撞、滚翻、极端温度。

因为汽车上路后会遇到各种意外：高速追尾、冰雪路面失控、极寒环境启动。

**如果只测正常驾驶，不测碰撞和极端场景，上市后会怎样？**

出了事故，乘客伤亡，车企要赔钱、召回、品牌受损。

软件也一样。只测正常流程，不测边界场景——并发5000人同时操作、数据量涨到100万条、网络断了、服务器宕了——上线后系统崩溃。

**什么是并发？**

并发就像餐厅高峰期——中午12点，100个顾客同时进来，5个服务员要同时服务这100桌。

如果服务员处理不了，顾客要么排队等，要么直接走人。

软件的并发也一样：多少人同时操作时系统还能正常响应？超了怎么办？排队还是拒绝？

**边界检查就是软件的碰撞测试。**

### 2.4 方案对比

**方案对比就像装修时比较多个设计方案的优劣。**

设计师给了3套方案：现代简约15万、欧式复古25万、北欧风18万。

你会直接选第一个吗？不会。你会逐项对比：预算差距、施工周期、后期维护、风格喜好、风险点。

对比完，才决定选哪个。

**如果不对比直接选第一个，装修完发现：家人不喜欢、预算能支持更好的方案、后来发现C方案更适合生活习惯。**

改起来要花多少钱？推倒重装？

技术方案也一样。开发说"有三种方案都能实现"，你要对比：功能完整性、开发成本、开发周期、技术风险、性能影响、维护复杂度。

对比完，选最优方案。

### 2.5 产品经理容易误解什么？

**误解一："方案预演是浪费时间，直接开发更快"**

预演花1小时发现的问题，开发中可能花1周才能修复。

不做预演，上线后发现问题的修复成本是预演时的10-100倍。

**误解二："AI预演的结果可以直接用"**

AI预演是发现遗漏的工具。AI说"漏了取消订单场景"，你要判断：取消订单必须加吗？退货可以二期再做吗？

AI是工具，判断力是你的。工具帮你提速，你把控质量。

## 3. 开发怎么看这个问题

开发说"方案没问题"，在想什么？

在想"正常流程没问题"。不是"所有场景没问题"。

你写"用户点击按钮，系统处理请求"，开发脑子里已经在跑十个问题：

| 开发追问 | 为什么问这个 |
|---------|-------------|
| 点击时传什么参数？ | 参数格式不对，系统报错 |
| 参数格式怎么校验？ | 用户输入乱七八糟的内容，系统要能处理 |
| 校验失败返回什么错误？ | 用户看到错误才知道哪里错了 |
| 并发5000人点击会怎样？ | 高峰期系统扛不住就崩溃 |
| 网络断了怎么处理？ | 用户点了没反应，会反复点，造成重复请求 |
| 服务器宕了怎么恢复？ | 数据丢了怎么办？状态乱了怎么办？ |
| 超时多久算失败？ | 等太久用户会以为卡死了 |
| 失败后可以重试吗？ | 重试会不会造成重复下单？ |
| 用户中途取消怎么办？ | 取消后状态怎么变？数据怎么清？ |
| 权限怎么控制？ | 谁能点？谁不能点？ |

一个"点击"动作，开发要问十个问题。每个问题没答案，代码就写不下去。

**开发说"需求不完整"，不是刁难。**

这是防御性设计的思维惯性。开发经历过太多"需求没定义清楚就开工"的惨痛教训：

| 遗漏项 | 上线后后果 |
|--------|-----------|
| 并发没定义 | 双十一高峰，5000人同时抢购，系统崩溃，订单丢失，客服接到几百个投诉 |
| 边界没考虑 | 数据量涨到100万条，查询超时，用户投诉"系统卡死了" |
| 异常没处理 | 网络断了，用户反复点击，造成重复下单，退款处理一个月 |
| 超时没定义 | 用户等了30秒没反应，以为系统卡了，打电话投诉，其实是后台处理慢 |

开发退回来，是在帮你避开这些坑。他问的每一个问题，都是他踩过的坑。

老李对小想说：

> "开发问的这些问题，不是在刁难你，是在帮你。他问的每一个问题，都是他曾经踩过的坑。你不回答，他只能猜。猜错了，上线后就是 bug。"

## 4. 怎么做：预演清单 + AI辅助 + 方案对比框架

### Step 1：预演清单

拿到需求，先问四个问题：

| 预演维度 | 问题 |
|---------|------|
| 状态预演 | 有哪些状态？取消怎么处理？回退怎么处理？ |
| 异常预演 | 网络断了怎么办？支付失败了怎么办？服务器宕了怎么办？ |
| 边界预演 | 数据量多大？并发多少？输入最长多长？超时多久算失败？ |
| 依赖预演 | 依赖的系统挂了怎么办？ |

四个问题答完，遗漏点就暴露了。

### Step 2：AI 辅助预演

把流程和状态告诉 AI，让 AI 帮你预演。

提示词示例：
> "订单流程：下单→支付→发货→完成。请帮我预演这个流程，看看有没有遗漏的场景。"

AI 会指出遗漏：取消订单、支付失败、发货超时、退货退款...

你逐项判断：哪些必须处理，哪些可以二期。

### Step 3：边界检查清单

用边界检查清单自查：

| 边界类型 | 需要追问的问题 |
|---------|---------------|
| 数据量边界 | 数据量多大时还能正常？超了怎么办？ |
| 并发边界 | 多少人同时操作还能正常？超了排队还是拒绝？ |
| 输入边界 | 输入最长多长？最短多短？超了怎么处理？ |
| 时间边界 | 超时多久算失败？失败怎么处理？ |
| 异常边界 | 输入乱七八糟的内容怎么办？ |
| 依赖边界 | 依赖的系统挂了怎么办？网络断了怎么办？ |

### Step 4：方案对比框架

开发提出多个方案时，用对比框架：

| 评估维度 | 权重示例 | 对比内容 |
|---------|---------|---------|
| 功能完整性 | 30% | 方案能覆盖所有需求吗？ |
| 开发成本 | 20% | 需要多少人天？ |
| 开发周期 | 15% | 多久能上线？ |
| 技术风险 | 15% | 有不确定的技术难点吗？ |
| 性能影响 | 10% | 对系统性能有什么影响？ |
| 维护复杂度 | 10% | 上线后维护难度如何？ |

每个方案在每个维度打分，加权求和，选出最优方案。

**关键风险点单独看：** 总分最高的方案，有没有致命风险？比如并发上限5000，预期并发1万——这个风险能不能接受？

## 5. 案例复盘

回看上线复盘会的故事：张开发只考虑正常流程，老李点出关键——"开发说的'没问题'≠'所有场景没问题'"。

这个故事告诉我们什么？

**要点一：开发说的"没问题"是有边界的。**

开发验收的是他理解的正常流程。你没说的边界场景，他不会主动帮你想。

**要点二：预演是补齐这个认知差的工具。**

你做预演，就是在主动追问边界场景。追问到位，开发才能实现到位。

**要点三：不做预演，上线后用真实用户帮你发现问题。**

真实用户会遇到各种边界场景——高峰期并发、网络断了、输入奇怪内容、超时等待。这些都是"碰撞测试"场景。不做预演，就是用真实用户当碰撞假人。

## 6. 一句话总结

**方案预演是开发前的样板间参观——模拟效果、发现遗漏、对比优劣、再开工。**

|------|------------|
| 方案预演 | 开发前先看样板间，发现问题成本低 |
| AI辅助预演 | 告诉AI流程，AI帮你发现遗漏 |
| 边界检查 | 软件的碰撞测试——测极端场景 |
| 方案对比 | 不选第一个，对比后选最优 |
