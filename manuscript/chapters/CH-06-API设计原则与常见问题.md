# CH-06 API 设计原则与常见问题 — DTO、接口粒度与 N+1

## 1. 痛点场景

"用户表有什么字段，接口就返回什么字段，不就查一下数据库吗？"

产品小想在评审会上说完这句话，开发老王摇头："这样不行，接口要按前端需要的数据结构设计。"

小想不理解："为什么？一个表对一个接口，多简单啊。"

这时老李介入了。

老李问小想："你点外卖的时候，餐厅会把你点的菜的原材料——生的青菜、没切的肉、整袋的米——直接端到你桌上吗？"

"当然不会。"小想笑了。

"为什么不会？"

"因为我要吃的是做好了的菜，不是原材料。原材料是厨房里的事。"

老李点头："数据库里的表，就像厨房里的原材料——它是系统的内部结构，不是给调用者看的。接口返回的数据，要像端上桌的菜一样，按调用者需求裁剪组合，而不是直接把表结构暴露出去。"

**一句话：API 是给调用者用的，不是给数据库用的。**

**你是不是也遇到过？**

开发说"接口不能直接返回表结构"，你不理解为什么。本章解决这个问题：为什么"一个表对一个接口"是错误的直觉，以及 N+1 问题为什么会让列表页加载 10 秒。

## 2. 核心概念：DTO、接口粒度、N+1、幂等性

### 2.1 DTO（数据传输对象）

**DTO 是专门为接口调用者设计的数据结构。**

它不是数据库表的直接映射，而是根据调用者需求"裁剪组合"后的结果。

用餐厅类比理解：

| 餐厅概念 | 软件概念 | 解释 |
|---------|---------|------|
| 厨房原材料 | 数据库表 | 原材料在厨房里；数据库表是系统内部结构 |
| 菜单订单 | API 请求 | 顾客通过菜单点菜；前端通过 API 请求数据 |
| 服务员装盘 | DTO | 从厨房取原材料，按顾客需求加工组合装盘 |
| 端上桌的菜 | API 响应 | 顾客只看到成品菜；前端只收到干净的 DTO 数据 |

关键点：
- 原材料≠成品菜 → 数据库表≠API 返回
- 同一份原材料可做成不同菜品 → 同一张表可返回不同 DTO（给移动端、PC端、第三方）
- 顾客不需要知道厨房有什么 → 前端不需要知道数据库结构

### 2.2 接口粒度

**接口粒度是指一个接口返回多少数据、涵盖多少功能。**

粒度设计有两个极端错误：

| 错误 | 问题 | 后果 |
|------|------|------|
| 太粗 | 一个接口返回所有数据 | 传输浪费、暴露敏感字段、安全隐患 |
| 太细 | 每个字段一个接口 | 调用次数爆炸、N+1 风险、前端拼接复杂 |

用点餐类比：

- 太粗 = 一道菜包含所有食材，你只想吃一碗饭，却拿到了整桌菜
- 太细 = 一个食材一道菜，你想吃一碗饭，菜单上只有"一粒米"
- 合适 = 按顾客需求设计菜品，荤素搭配，分量适中

### 2.3 N+1 问题

**N+1 问题是指查列表时逐条查详情，导致查询次数爆炸。**

典型场景：列表页需要显示 100 条订单，每条订单要关联客户名、商品名。

错误做法：
1. 查订单表，拿 100 条订单（1 次查询）
2. 对每条订单，查客户表拿客户名（100 次查询）
3. 对每条订单，查商品表拿商品名（100 次查询）

总共：1 + 100 + 100 = **201 次查询**

正确做法：
1. 查订单表，拿 100 条订单（1 次查询）
2. 批量预加载客户名和商品名（1 次 JOIN 查询）

总共：**2 次查询**

用快递员类比：

| 快递场景 | 软件场景 |
|---------|---------|
| 给一栋楼送 100 个包裹 | 查 100 条订单的关联数据 |
| 每次只拿 1 个包裹上楼，跑 100 次 | 逐条查详情，查 201 次 |
| 用大袋子一次带 100 个包裹上楼 | 批量预加载，只查 2 次 |
| 累死快递员 | 拖垮数据库 |

### 2.4 幂等性

**幂等性是指同一操作执行多次，结果和执行一次相同。**

典型场景：用户点击"提交订单"，页面卡住了，用户又点了一次。

没有幂等设计 → 创建了两个订单
有幂等设计 → 只创建一个订单，第二次请求被识别为重复

用 ATM类比：

你在 ATM 取款，按了一次取款按钮，机器卡住了。你按了第二次。

ATM 会扣两次钱吗？不会。

因为 ATM 有幂等设计——同一笔取款请求提交多次，系统只处理一次。

## 3. 为什么产品经理容易误解

### 误解一："一个表对一个接口"

**误解表现**：需求里写"用户列表接口"，以为就是查用户表返回。

**误解原因**：直觉上认为数据库和接口是一一对应的。

**正确理解**：接口是给调用者用的，调用者需要的可能是跨表数据。用户列表页可能需要：用户名（用户表）、部门名（部门表）、最后登录时间（登录日志表）。一个接口要聚合三张表的数据。

### 误解二："列表页加载慢是服务器不行"

**误解表现**：页面加载 10 秒，投诉"服务器太慢了"。

**误解原因**：以为性能问题是硬件问题。

**正确理解**：很可能是 N+1 问题导致查询次数爆炸。100 条数据查了 201 次数据库，每次 20 毫秒，201 次就是 4 秒。这不是服务器不行，是设计问题。

### 误解三："重复提交是用户的问题"

**误解表现**：用户点了两次"提交"，创建了两个订单，运营认为是用户操作失误。

**误解原因**：没意识到系统应该保护用户。

**正确理解**：网络波动时用户重复点击是常见场景。系统应该有幂等设计，防止重复操作产生重复结果。

## 4. 开发视角如何看这个问题

开发看接口设计，想的不是"有几个表"，而是：

### DTO 设计层面
- 前端需要哪些字段？
- 哪些字段需要跨表关联？
- 哪些字段是敏感信息（不能暴露）？
- DTO 的命名和结构是否清晰？

### 性能层面
- 这个列表页会不会触发 N+1？
- 关联查询会不会拖垮数据库？
- 是否需要批量预加载？
- 分页参数有没有传？

### 安全层面
- 密码 hash、内部标识符会不会暴露？
- 接口粒度是否合理（太粗暴露太多数据）？
- 幂等性有没有设计？

### 标准化层面
- 分页参数是统一的吗？（page/pageSize 还是 offset/limit）
- 排序字段是前端传还是后端定义？
- 过滤条件是否标准化？

开发问的不是刁难，而是"防御性设计"。每个问题背后都是踩坑经验——没想清楚就上线，后期 N+1 性能问题、敏感数据泄露、重复订单投诉。

## 5. 产品经理应该如何做

### 5.1 需求中标注数据聚合

写列表页需求时，明确标注：

> 用户列表页需显示：用户名、部门名、最后登录时间。
> **请批量预加载关联数据，避免 N+1 问题。**

这一句话，开发看到就知道要用 JOIN 查询或批量预加载。

### 5.2 需求中标注 DTO 设计要点

写接口需求时，用表格标注：

| 字段 | 来源 | 说明 |
|------|------|------|
| userName | 用户表 | 用户姓名 |
| departmentName | 部门表 | 关联查询 |
| lastLoginTime | 登录日志表 | 最新一条 |
| **不返回** | - | password、internalId、createTime |

### 5.3 需求中标注分页排序过滤规则

写列表接口需求时，标注：

| 参数 | 类型 | 说明 |
|------|------|------|
| page | int | 页码，从 1 开始 |
| pageSize | int | 每页条数，默认 20，最大 100 |
| sortBy | string | 排序字段，默认 createTime |
| sortOrder | string | 排序方向，asc/desc |
| filterByDepartment | string | 按部门过滤 |

### 5.4 需求中标注幂等要求

写"提交"类需求时，标注：

> **需支持幂等，防止重复提交。**
> 建议使用订单号作为幂等键，同一订单号重复请求只处理一次。

### 5.5 Checklist

**数据聚合**
- [ ] 列表页需关联的数据标注了吗？
- [ ] 是否标注了"批量预加载"？
- [ ] 哪些字段不返回标注了吗？

**分页排序过滤**
- [ ] 分页参数定义了吗？
- [ ] 排序字段和方向定义了吗？
- [ ] 过滤条件定义了吗？

**幂等性**
- [ ] 提交类需求标注了幂等要求吗？
- [ ] 幂等键是什么标注了吗？

## 6. AI 可以如何辅助

AI 可以帮你生成接口规格初稿。

### 提示词示例

> 请帮我生成"用户列表接口"的规格初稿。
> 需返回：用户名、部门名、最后登录时间。
> 需支持：分页、排序（按创建时间）、过滤（按部门）。

AI 会给一个框架：

```
接口：GET /api/users
参数：
  - page: 页码
  - pageSize: 每页条数
  - sortBy: 排序字段
  - sortOrder: 排序方向
  - departmentId: 部门过滤
返回：
  - userName: 用户名
  - departmentName: 部门名
  - lastLoginTime: 最后登录时间
```

### 你要补的内容

AI 不知道你的业务规则，必须你来补：

| AI 给的 | 你要补的 |
|--------|----------|
| page | 从 1 开始，默认 1 |
| pageSize | 默认 20，最大 100 |
| sortBy | 默认 createTime，可选 userName |
| 不返回字段 | password、internalId |

### AI 辅助的正确姿势

AI 出框架，你补业务规则。

AI 能帮你列出常见参数、给出基本结构。但业务约束（pageSize 最大值、哪些字段不返回、幂等键是什么）AI 不知道，必须你自己补充。

## 7. 案例复盘：小想的列表页 10 秒加载

小想负责设计一个订单管理后台的列表页。需求很简单：显示最近 100 条订单，每条订单显示订单号、客户名、商品名、金额、状态、创建时间。

上线第二天，客服投诉："订单列表打开要 10 秒，根本没法用！"

开发老王打开日志监控："100 条订单，后台查了 **501 次数据库**。你的需求里每条订单要显示客户名和商品名，但订单表只存了 ID，开发逐条去查客户表和商品表——这就是 N+1 问题。"

老李路过，听完后给小想一个行动要点：

"以后写列表页需求，记得标注一句话：**'关联数据请批量预加载'**。开发看到这个提示，就会用正确的方式实现。"

小想学会了。

## 8. 一句话总结

**API 是为调用者服务的，不是为数据库服务的。**

写需求时：标注数据聚合 → 标注分页规则 → 标注幂等要求。
