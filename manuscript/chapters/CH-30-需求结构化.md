# CH-30 需求结构化 — 需求拆解、需求元素、需求模板与完整性检查

## 1. 痛点场景

小想收到业务需求："财务部要一个报销审批功能。"

他想用 AI 辅助写需求。打开 Cursor，输入：

```
帮我写一个报销审批功能的 PRD
```

AI 输出了一份看起来很完整的 PRD。小想满意，直接发给开发评审。

评审会上，开发问了四个问题：

> 开发1："审批金额有没有上限？超过多少要走二级审批？"
> 开发2："审批人怎么确定？是固定的人还是按部门动态分配？"
> 开发3："如果审批人离职了，流程怎么处理？"
> 开发4："驳回后申请人能修改重新提交吗？还是只能重新申请？"

小想愣住了。翻遍 PRD，这四个问题都没写。

开发组长把 PRD 退回来："需求不完整，做不了。"

小想不明白：AI 生成的 PRD 看起来什么都写了，为什么开发还说"做不了"？

**本章解决一个问题**：怎么把"做个审批功能"变成开发能开工的结构化需求。

---

## 2. 核心概念：需求拆解、需求元素、需求模板

### 2.1 需求拆解是什么？

**需求拆解是把模糊需求拆成结构化元素的过程。**

业务说"做个审批功能"，就像顾客说"我想吃红烧肉"。

厨师不会直接开火炒菜。他会先列菜谱：
- 用什么肉（五花肉）
- 多少油（适量）
- 什么调料（酱油、糖、料酒）
- 炖多久（40分钟）
- 火候如何（先大火烧开，再小火慢炖）

产品经理也一样。"做审批功能"这句模糊需求，要拆解成：
- 审批什么数据？（食材是什么）
- 审批流程几步？（烹饪步骤）
- 谁有权限审批？（谁来做饭）
- 审批有哪些状态？（火候变化）

菜谱不写清楚，厨师炒不出你要的味道。需求不拆清楚，开发做不出你要的功能。

### 2.2 四维度元素框架

拆解需求用什么框架？用**四维度元素框架**：

| 维度 | 核心问题 | 典型遗漏 |
|------|----------|----------|
| **数据** | 审批什么数据？每个字段什么类型和约束？ | 金额上限没定义、附件格式没约束 |
| **流程** | 审批经过哪些步骤？异常怎么处理？ | 审批人离职没处理、驳回流程不清晰 |
| **权限** | 谁有权限审批？权限怎么分配？ | 审批人规则没定义、跨部门审批没说明 |
| **状态** | 申请有哪些状态？状态怎么流转？ | 驳回后状态定义不清晰 |

这四维度就像**四面镜子**，照出需求的全貌：

- **数据镜子**：看到需求背后有哪些数据在流转
- **流程镜子**：看到需求经过哪些操作步骤
- **权限镜子**：看到谁有权限做什么操作
- **状态镜子**：看到数据在不同状态间如何跳转

只照一面镜子，只能看到一面。开发要开工，需要四面都照清楚。

**为什么四元素缺一不可？**

用一个类比：需求元素就像房子的四个部件。

| 元素 | 房子类比 | 缺了会怎样 |
|------|----------|------------|
| 数据 | 地基 | 没地基房子塌 |
| 流程 | 楼梯 | 没楼梯上不去楼 |
| 权限 | 钥匙 | 没钥匙进不去门 |
| 状态 | 房间状态 | 不知道房间是空闲还是占用 |

开发问你"审批金额上限是多少"，是在问地基深度。你没定义，地基就没打稳。

开发问你"驳回后怎么处理"，是在问楼梯怎么走。你没定义，楼梯就断了。

开发问你"谁来审批"，是在问钥匙给谁。你没定义，门就锁不住。

开发问你"有哪些状态"，是在问房间什么情况。你没定义，状态就乱了。

### 2.3 产品经理容易误解什么？

**误解一："需求写清楚功能描述就行了"**

小想的 PRD 写了："用户提交申请，审批人审批，通过后付款。"

功能描述写清楚了。但开发问了四个问题，小想都答不上。

因为功能描述只是需求的**冰山一角**：

```
        需求的冰山模型

        ╱╲
       ╱  ╲  ← 功能描述（你能看到的）
      ╱────╲
     ╱      ╲ ← 数据定义（开发需要的）
    ╱        ╲
   ╱          ╲ ← 流程设计（开发需要的）
  ╱            ╲
 ╱              ╲ ← 权限分配（开发需要的）
╱                ╲
────────────────── ← 状态流转（开发需要的）
```

水面下面，还有数据、流程、权限、状态四个维度。开发问你这么多问题，就是在问水面下面的东西。

**误解二："开发会自己补全需求遗漏的部分"**

小想以为："审批人离职这种边界情况，开发应该会自己处理吧？"

开发摇头："我们不是业务专家，我们按需求实现。你漏了，我们就漏了。"

这不是开发偷懒。这是**分工边界**。

产品经理懂业务规则——审批人离职后怎么分配。

开发懂技术实现——代码怎么写、接口怎么调。

开发不懂业务规则，不可能替你补全。你漏了审批人离职的处理，开发要么不做，要么按自己理解做——结果可能和业务要求完全相反。

## 3. 开发怎么看这个问题

开发拿到一份需求，在想什么？

**在想"怎么做"，不是"做什么"。**

你写"用户提交申请"，开发想：
- 提交时传什么参数？
- 参数格式怎么校验？
- 校验失败返回什么错误？
- 成功后存到哪个表？
- 存完要触发什么操作？

你写"审批人审批"，开发想：
- 审批人是谁？怎么确定的？
- 审批时要看什么信息？
- 通过和驳回分别做什么？
- 并发审批怎么处理？
- 审批超时怎么办？

你写"通过后付款"，开发想：
- 付款调哪个接口？
- 付款失败怎么处理？
- 付款状态怎么更新？
- 付款记录怎么留存？

一个"审批"动作，开发要问十个问题。每个问题没有答案，代码就写不下去。

**开发说"需求不完整"，不是刁难。**

这是**防御性设计**。没定义清楚就开工：
- 驳回没处理 → 用户投诉"审批被驳回，但我不知道"
- 权限没定义 → 普通员工看到其他部门的敏感数据
- 状态没约束 → 两个人同时审批，状态冲突，系统报错
- 边界没考虑 → 并发请求压垮数据库

这些上线就是 bug。开发退回来，是在帮你避开这些坑。

## 4. 怎么做：四维度拆解法 + 标准模板 + 完整性检查

### Step 1：用四维度拆解需求

拿到"做个审批功能"，先问四个问题：

| 维度 | 问题 | 你的答案 |
|------|------|----------|
| 数据 | 审批什么数据？ | 报销申请表：申请人、金额、类型、理由、附件 |
| 流程 | 怎么审批？ | 提交 → 一级审批 → 二级审批（>5000元） → 通过/驳回 |
| 权限 | 谁来审批？ | 一级：部门经理；二级：财务总监 |
| 状态 | 状态怎么变？ | 待提交 → 待一级审批 → 待二级审批 → 已通过/已驳回 |

四个问题答完，需求的骨架就有了。

### Step 2：用标准模板写需求

骨架有了，用模板把细节填进去：

```markdown
# 报销审批功能需求

## 一、数据定义

| 字段名 | 类型 | 必填 | 约束规则 |
|--------|------|------|----------|
| 申请金额 | Decimal(10,2) | 必填 | >0，上限50000 |
| 申请类型 | Enum | 必填 | 差旅/办公/招待 |
| 申请理由 | Text | 必填 | 最少50字 |
| 附件 | File | 可选 | PDF/JPG/PNG，≤5MB |

## 二、流程定义

### 主流程
1. 提交申请
2. 一级审批（部门经理）
3. 二级审批（金额>5000时，财务总监）
4. 通过 → 付款；驳回 → 返回申请人修改

### 异常流程
- 审批人离职：自动转上级审批人
- 审批超时（7天）：自动提醒审批人

## 三、权限定义

| 角色 | 操作权限 | 数据权限 |
|------|----------|----------|
| 申请人 | 提交、撤回、查看自己申请 | 仅看自己的申请 |
| 部门经理 | 一级审批、驳回 | 看本部门所有申请 |
| 财务总监 | 二级审批、驳回 | 看所有申请 |

## 四、状态定义

| 状态 | 说明 | 可跳转状态 |
|------|------|------------|
| 待提交 | 申请草稿 | 待一级审批 |
| 待一级审批 | 等待部门经理审批 | 已通过（≤5000）、待二级审批、已驳回 |
| 待二级审批 | 等待财务总监审批 | 已通过、已驳回 |
| 已通过 | 审批完成 | 已付款 |
| 已驳回 | 审批被驳回 | 待一级审批（重新提交） |
```

模板的好处：
- 开发一眼能看清数据、流程、权限、状态四个维度
- 不用追问，直接能开工
- 格式标准，评审效率高

### Step 3：用完整性检查清单自查

写完需求，用检查清单自查：

**数据维度**
- [ ] 每个字段的类型、长度、约束是否定义？
- [ ] 字段的边界条件（上限、下限、格式）是否定义？
- [ ] 字段之间的关系（如 A 必填时 B 也必填）是否定义？

**流程维度**
- [ ] 正常流程的每一步是否定义？
- [ ] 驳回、撤回、取消等逆向流程是否定义？
- [ ] 异常流程（超时、并发、失败）是否处理？

**权限维度**
- [ ] 每个角色能执行什么操作是否定义？
- [ ] 每个角色能看什么数据是否定义？
- [ ] 特殊权限（如管理员跳过审批）是否定义？

**状态维度**
- [ ] 有哪些状态是否定义？
- [ ] 状态怎么流转是否定义？
- [ ] 状态流转的约束（不能直接跳转的情况）是否定义？

每项打勾，需求才算完整。

### Step 4：AI 辅助拆解和检查

AI 能帮你做两件事：

**帮拆解**：把模糊需求拆成四维度框架

提示词示例：
> 请帮我拆解"报销审批功能"需求，按数据、流程、权限、状态四个维度输出。

AI 会给出框架。但你要补充：
- 金额上限是多少？
- 审批人怎么分配？
- 驳回后怎么处理？

AI 出骨架，你补细节。

**帮检查**：检查需求有没有遗漏

提示词示例：
> 请检查这份报销审批需求，按数据/流程/权限/状态四维度检查是否有遗漏。

AI 会指出遗漏点。你逐项补齐。

记住：AI 是工具，判断力是你的。AI 帮你提速，你把控质量。

---

## 5. 一句话总结

**需求不是一句话描述，是一张结构化地图。**

拆需求：问数据/流程/权限/状态四个问题 → 用模板填细节 → 用清单查遗漏。

---

| 概念 | 一句话记住 |
|------|------------|
| 需求拆解 | 把模糊需求拆成结构化元素，像写菜谱 |
| 四维度框架 | 数据、流程、权限、状态——四面镜子照全貌 |
| 需求模板 | 用标准格式把四维度写清楚，开发直接开工 |
| 完整性检查 | 用清单逐项核对，确保四面都不漏 |
