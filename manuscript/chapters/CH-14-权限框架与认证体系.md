# CH-14 权限框架与认证体系 — RBAC/ABAC、SSO/IDaaS 与隐私合规

## 1. 痛点场景

"权限用 RBAC 还是 ABAC？"

周一上午，需求评审会。开发老李刚坐下，抛出一个问题。

产品小想愣住了："RBAC？ABAC？这俩是什么？"

老李解释道："权限框架。你需求里写了'经理可以审批'、'只有工作日能访问'。一个是角色判断，一个是属性判断。框架选型会影响我们能做什么权限规则。"

小想更懵了："我以为权限就是'谁能干什么'..."

老李点点头："是'谁能干什么'。但框架决定你能设计什么样的'能'。RBAC 按角色分，简单适合大多数场景。ABAC 按属性分，灵活但复杂。你得知道差异，不然设计权限规则时开发会说'这个做不了'。"

**你是不是也遇到过？**

开发问权限框架选型，产品完全听不懂。本章就来解决一个问题：为什么"权限规则设计"不只是"谁能干什么"，框架选型影响你能设计什么规则。

---

## 2. 核心概念：权限框架、认证授权、SSO/IDaaS、隐私合规

### 2.1 RBAC vs ABAC：两种权限框架

**RBAC（基于角色的访问控制）** — 按角色分配权限。

就像公司的工牌系统：员工工牌、经理工牌、高管工牌。工牌类型决定能进哪些区域。不需要每次进门时判断"你是什么部门、现在几点"。

特点：
- 简单直观：给角色分配权限，给用户分配角色
- 易管理：新员工入职 → 分配角色 → 自动获得权限包
- 有局限：做不到动态判断。比如"工作日才能访问"、"异地登录需要额外验证"，RBAC 做不了。

**ABAC（基于属性的访问控制）** — 按属性动态判断权限。

就像智能门禁：不只看工牌，还看时间、地点、部门。能实现"工作日才能进"、"异地需要额外验证"。还能做"临时授权过期自动收回"。

特点：
- 灵活强大：能基于时间、地点、部门、项目等属性组合判断
- 复杂度高：规则多了难维护、难理解
- 适合复杂场景：金融、政务、跨组织协作、合规审计要求高的系统

| 对比 | RBAC | ABAC |
|------|------|------|
| 判断依据 | 角色（静态） | 属性组合（动态） |
| 复杂度 | 低 | 高 |
| 适用场景 | 大多数企业内部系统 | 金融、政务、跨组织协作 |
| 能做什么 | 角色权限矩阵 | "工作日+本部门+敏感数据=只读" |

**什么时候用哪个？**

先问三个问题：
1. 权限规则有没有"时间/地点/条件"的动态判断？
2. 用户会不会频繁跨角色、跨项目？
3. 合规审计有没有复杂要求？

如果都是"否"，RBAC 够用。如果有"是"，考虑 ABAC。

### 2.2 认证 vs 授权：不是一回事

**认证（Authentication）** — 解决"你是谁"。

登录时输入账号密码，系统验证你就是你。就像进门时出示身份证。

**授权（Authorization）** — 解决"你能干什么"。

登录成功后，系统判断你能看什么数据、做什么操作。就像进门后，身份证决定你能进哪些房间。

| 对比 | 认证 | 授权 |
|------|------|------|
| 问题 | 你是谁？ | 你能干什么？ |
| 发生时机 | 登录时 | 认证后 |
| 依赖关系 | 先认证 | 后授权（依赖认证结果） |

容易混淆的场景：
- 产品说"做个登录功能"，以为登录就完了。但登录只是认证，后面还有授权。
- 开发问"认证方案用什么？"，产品以为在问界面长什么样。其实开发在问：自建还是接统一登录？

### 2.3 SSO 与 IDaaS：登录不该自己造

**SSO（单点登录）** — 一处登录，多处使用。

就像万能通行证：刷一次卡，所有大门都能进。不需要每个门单独出示证件。

好处：
- 用户方便：一次登录，全系统通行
- 管理方便：账号统一管理，离职一处收回
- 安全可控：集中监控登录行为

**IDaaS（身份即服务）** — 开箱即用的身份服务。

就像租用银行的保险柜系统：不用自己造保险柜、不用自己设计安全规则、不用自己搞合规审计。专业服务商帮你搞定一切。

主流产品：Okta、Auth0、Azure AD、阿里云身份服务等。

**为什么登录不该自己造？**

自建登录系统的坑：
- 安全风险：密码存储、传输加密、防攻击，每项都是技术活
- 合规审计：个人信息保护法要求，自己搞很难过审计
- 维护成本：账号体系迭代、安全漏洞修复、用户支持，长期投入

老李说："登录是基础设施，不是业务功能。每个系统自己造一套，成本和安全风险都会爆炸。"

### 2.4 隐私合规：法律要求，不是可选

**个人信息保护法** 对 IT 系统的硬性约束：

| 约束类型 | 具体要求 |
|----------|----------|
| 存储约束 | 手机号等敏感信息不能明文存储 |
| 传输约束 | 传输必须加密 |
| 展示约束 | 展示需脱敏（手机号显示 138****5678） |
| 用户权利 | 用户有权要求删除数据、查看数据 |
| 审计要求 | 数据访问要有日志记录 |

**这也是为什么认证要基础设施化**。

自建登录系统，过不了合规审计。用 IDaaS，公司的平台帮你搞定合规。

## 3. 产品经理容易误解的

### 误解一："权限框架是开发选的，跟产品没关系"

权限框架选型直接影响你能设计什么权限规则。

RBAC 做不了基于时间/地点的动态权限。如果产品设计了"工作日才能审批"的规则，开发用 RBAC 框架就做不了。

产品需要参与框架选型讨论：告诉开发你要什么权限规则，开发才能判断框架够不够用。

### 误解二："每个系统做自己的登录就行"

各自建登录导致：
- 账号密码管理成本爆炸
- 安全风险分散难以控制
- 合规审计根本过不了

大公司都在搞 SSO 和 IDaaS，不是因为"酷"。是因为分散登录的成本和风险都扛不住。

### 误解三："隐私合规是法务的事"

隐私合规是法律要求，直接影响：
- 存储方案（敏感数据加密）
- 接口设计（脱敏返回）
- 前端展示（脱敏显示）

产品需求中必须体现隐私约束，不能甩给法务去查。

## 4. 开发怎么看这个问题

开发问"RBAC 还是 ABAC"，脑子里在展开：

**第一层：权限规则复杂度**
- 有没有"时间/地点/条件"的动态判断？
- 用户会不会频繁跨角色、跨项目？
- 规则会不会经常变化？

这些问题决定框架选型。如果规则简单稳定，RBAC 夠用。如果规则复杂多变，ABAC 更合适。开发不想选错了框架，后面一改改半年。

**第二层：认证方案**
- 自己做还是接公司统一登录？
- 接统一登录要不要适配？
- 现有账号体系怎么迁移？

开发最怕的是"又要接统一登录，又要保留旧账号"。这种需求会让技术方案变得很复杂。迁移过程容易出问题：数据怎么同步？登录体验怎么统一？用户投诉怎么处理？

**第三层：合规审计**
- 有没有敏感数据存储？
- 登录日志要不要留存？
- 用户数据删除请求怎么处理？

合规审计是硬性要求，不是可选。开发需要知道敏感数据有哪些，才能设计加密方案。需要知道日志留存多久，才能设计存储容量。需要知道用户删除数据的流程，才能设计接口。

开发问的不是刁难，而是"防御性设计"。框架选错了后面要重构。登录自建了合规过不了。隐私要求漏了上线被警告。

## 5. 怎么做：框架选型、认证确认、合规自查

### Step 1：权限规则清单

用这个模板梳理你要的权限规则：

| 规则 | 判断依据 | 框架影响 |
|------|----------|----------|
| 经理可以审批 | 角色 | RBAC 够用 |
| 只有工作日能访问 | 时间 | 需要 ABAC 或扩展 RBAC |
| 异地登录需要额外验证 | 地点 | 需要 ABAC 或扩展 RBAC |
| 项目成员只能看本项目数据 | 项目归属 | RBAC + 数据权限 |
| 临时授权过期自动收回 | 时间+临时角色 | ABAC 或临时角色机制 |

如果"框架影响"列有多个"需要 ABAC"，考虑用 ABAC 框架。如果都是"RBAC 够用"，就用 RBAC。

### Step 2：认证方案确认

| 场景 | 建议方案 |
|------|----------|
| 公司已有统一登录 | 接统一登录（SSO） |
| 新公司、无统一登录 | 用 IDaaS（Auth0、Okta 等） |
| 外部用户（客户、供应商） | IDaaS 或社交登录 |
| 内部员工+外部用户混合 | IDaaS + 多身份源 |

### Step 3：隐私合规自查

| 检查项 | 是否涉及 | 处理方式 |
|--------|----------|----------|
| 手机号存储 | 是/否 | 加密存储 |
| 手机号展示 | 是/否 | 脱敏显示 |
| 登录日志 | 是/否 | 留存审计 |
| 用户数据删除请求 | 是/否 | 设计删除流程 |
| 数据访问审计 | 是/否 | 记录访问日志 |

### Step 4：用 AI 辅助框架对比

提示词示例：
> 我要设计一个权限系统，规则包括：经理可以审批、只有工作日能访问、异地登录需要额外验证。请帮我判断用 RBAC 还是 ABAC 更合适。

AI 可能分析：
- 经理审批 → RBAC 够用
- 工作日判断 → 需要 ABAC 或时间条件扩展
- 异地验证 → 需要 ABAC 或地理条件扩展
- 建议：如果只有这两条动态规则，可以用 RBAC + 扩展条件；如果规则多，考虑 ABAC

### Step 5：自查（checklist）

**权限框架**
- [ ] 权限规则梳理完成了吗？
- [ ] 有没有动态判断（时间/地点/条件）？
- [ ] 框架选型有结论了吗？

**认证方案**
- [ ] 公司有没有统一登录系统？
- [ ] 确认接统一登录还是自建（建议 IDaaS）？
- [ ] 账号迁移方案考虑了吗？

**隐私合规**
- [ ] 涉及哪些敏感数据？
- [ ] 存储方案加密了吗？
- [ ] 展示方案脱敏了吗？
- [ ] 用户数据删除流程设计了吗？

## 6. 案例复盘：手机号明文存储被警告

小想做的系统上线三个月，法务发来警告：用户手机号不能明文存储。

开发一脸懵："产品没说要加密啊..."

老李复盘时指出："隐私合规不是可选的，是法律要求。个人信息保护法出来之后，敏感数据必须加密存储、加密传输、脱敏展示。这不是你做不做的事，是必须做的事。"

小想恍然大悟："原来隐私合规直接影响技术方案..."

老李补充："这也是为什么登录要用基础设施。IDaaS 服务商已经帮你把合规审计、加密存储都搞好了。自己造，这些都要自己做。还容易过不了审计。"

**教训**：隐私合规是法定义务，产品需求中必须标注敏感数据及其处理方式。

## 7. 一句话总结

**认证走基础设施，权限选对框架，隐私是法定义务。**

| 概念 | 一句话记住 |
|------|------------|
| RBAC | 工牌系统——按角色分权限 |
| ABAC | 智能门禁——按属性组合判断 |
| 认证 vs 授权 | "你是谁" vs "你能干什么" |
| SSO | 万能通行证——一次登录全通行 |
| IDaaS | 租保险柜——专业服务商搞定合规 |
| 隐私合规 | 银行硬性要求——法律规定不是可选 |
