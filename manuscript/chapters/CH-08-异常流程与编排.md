## 一、为什么流程上线后总「卡住」？

周一早上九点，老李刚坐下准备看周报，客服主管就冲进来：「老李，周末用户投诉炸了，全是『流程卡住了』，你快看看！」

小想（22岁实习生）一脸紧张：「我上周上线那个审批流程，测试没问题啊...」

老李打开后台日志，看了两分钟，抬头问小想：「你测试的什么场景？」

小想：「正常流程啊——提交、审批、通过。」

老李叹了口气：「那驳回呢？超时呢？审批人不在呢？网络断了呢？」

小想愣住：「这些...我没想...」

## 二、异常流、状态机、流程编排：三个关键概念

### 异常流：备用路线

「流程设计最容易漏的是异常流。」

你画的流程图，从提交到审批到通过，一路顺到底。但这只是理想场景。

真实世界里，审批人可能驳回申请、超时不处理、离职了没人接替、系统出故障。这些都是异常流。你没设计，用户就「卡住了」——不知道怎么办，系统也没给出路。

异常流就是备用路线。主路堵了（驳回、超时、故障），用户得有别的路走。没设计备用路线？用户撞墙了只能原地投诉。

**常见异常类型**：

| 异常类型 | 表现 | 处理方式 |
|---------|------|---------|
| 驳回 | 审批不通过 | 回到申请人 + 通知 + 可修改重提 |
| 超时 | 长时间无处理 | 自动催办 / 转备选审批人 |
| 审批人缺失 | 审批人离职/请假 | 显示备选人 / 转上级 |
| 系统故障 | 网络/服务中断 | 提示稍后重试 + 自动保存进度 |

### 状态机：轨道切换器

需求评审会上，小想刚说完订单流程：「用户下单 → 支付 → 发货 → 收货 → 完成，就这几步。」

开发组长听完，直接说：「你这个流程需要状态机，不然做不了。」

小想一脸茫然：「状态机？那是...什么？咖啡机吗？」

老李笑了笑：「不是咖啡机。状态机是控制流程怎么『走』的规则系统。」

状态就是系统在某一时刻的数据情况。比如订单现在是「待支付」，过一会变成「已支付」。状态机定义状态怎么变的规则：有哪些状态、状态怎么跳、跳转条件是什么。

状态机就像红绿灯。红灯 → 黄灯 → 绿灯，按顺序切换。不能从红灯直接跳到绿灯，必须先过黄灯。状态机也是这样：待支付 → 已支付 → 发货中 → 已完成。有些跳可以，有些跳不行。规则写在状态机里。

**流程推进 = 状态变化**。没有状态变化，流程就没推进。

| 用户操作 | 状态变化 |
|---------|---------|
| 下单 | → 待支付 |
| 支付成功 | → 已支付 |
| 商家发货 | → 发货中 |
| 用户收货 | → 已完成 |

开发说「要状态机」，意思是：你流程图画了，但没告诉系统怎么跳。支付失败，订单状态怎么变？用户取消订单，状态怎么变？超时未支付，状态怎么变？这些跳转规则，开发需要知道才能写代码。

### 流程编排：并行时的冲突处理

需求评审会上，业务方提出：「这个大订单，我们想拆成多个子订单同时审批——不同部门审批不同的子订单，最后汇总。这样能快很多！」

小想觉得这个需求很合理，画了一张图：

```
大订单拆分 → 子订单A审批 → 汇总结果
           → 子订单B审批
           → 子订单C审批
```

「三个子订单同时审批，审批完了汇总。搞定！」

开发组长看完，问了一个问题：「冲突怎么处理？」

小想愣住：「什么冲突？」

老李接过话：「开发问的不是『会不会打架』，而是：多个并行流程同时修改同一个东西，怎么处理？你这个场景里，『同一个东西』是什么？大订单的状态。子订单A审批通过，要更新大订单状态。子订单B审批驳回，也要更新大订单状态。三个人同时改同一个数据，这就是冲突风险。」

并行流程像投票。三个人同时投票（审批），结果怎么算？全票通过才算通过？过半数通过就算通过？有人反对就驳回？这些规则，你要提前定好。

### 产品经理容易误解的两件事

**误解一：「异常情况很少发生，不用特别处理」**

很多人觉得异常是「小概率事件」，不值得花时间。但上线后用户投诉「流程卡住了」，几乎都是异常流没设计好。

**误解二：「流程图画好了，状态就自然对了」**

流程图描述的是「操作步骤」，状态需要单独定义。流程的每一步对应什么状态？能不能跳过？能不能回退？这些你没告诉系统，系统就不知道。

## 三、开发怎么看这些问题

### 异常流：开发关注「怎么处理」

开发看异常流，想的不是「会不会出问题」，而是「出问题了怎么办」。

每个异常场景，开发需要知道：
- 用户看到什么提示？
- 数据怎么保存？
- 流程怎么继续？

你没设计异常流，开发就没法写代码。上线后用户投诉「卡住了」，其实是系统不知道怎么处理异常情况。

### 状态机：开发关注「跳转规则」

开发看状态机，想的不是「有几个状态」，而是：
- 状态怎么存储？
- 跳转条件怎么判断？
- 并发时状态冲突怎么办？

你以为状态「自然就对了」，但系统不知道：待支付能不能跳到已完成？发货中能不能跳回已支付？这些「能跳」「不能跳」，你要明确告诉系统。

### 流程编排：开发关注「冲突策略」

开发看并行流程，想的不是「效率高不高」，而是：
- 加锁还是优先级规则？
- 部分通过怎么算？
- 中途取消怎么处理？

你没定义冲突处理策略，开发就要问你。问不到答案，系统就不知道怎么写。

## 四、产品经理应该怎么做

### 异常流设计清单

写需求时，先把正常流程画完。然后问自己：每一个节点，可能出什么问题？

| 步骤 | 可能异常 | 处理方式 |
|------|---------|---------|
| 提交 | 网络中断、数据格式错误 | 提示 + 保存草稿 |
| 审批 | 驳回、超时、审批人离职 | 回到申请人 / 催办 / 转办 |
| 通过 | 系统故障 | 提示稍后查看 |

**异常流设计公式**：正常流程画完 → 每个节点列异常 → 每个异常定处理方式。

### 状态机设计方法

画状态流转图，回答三个问题：
1. 有哪些状态？
2. 状态怎么跳？顺序是什么？
3. 跳转条件是什么？

| 状态机要素 | 产品经理要定义 |
|-----------|---------------|
| 状态集合 | 有哪些状态 |
| 初始状态 | 流程开始时是什么状态 |
| 终态 | 流程结束时是什么状态 |
| 跳转规则 | 状态怎么跳、条件是什么 |
| 禁止跳转 | 哪些跳转不允许 |

### 并行流程冲突处理策略

设计并行流程时，回答三个问题：
1. 谁是主流程？
2. 子流程如何汇总？（全部通过？三中二？有驳回就驳回？）
3. 冲突怎么处理？（加锁？优先级规则？）

| 策略 | 适用场景 | 优缺点 |
|------|---------|--------|
| 加锁 | 安全优先、效率不重要 | 简单安全，但效率低 |
| 优先级规则 | 效率优先、逻辑明确 | 效率高，但规则要清晰 |
| 乐观锁 | 并发少、冲突罕见 | 性能好，但冲突时要重试 |

### AI 可以如何辅助

**异常场景列举**：把流程图发给 AI，让它列举可能遗漏的异常场景。

> 提示词示例：「这是一个审批流程：提交 → 审批 → 通过。请列举至少 5 种可能导致流程『卡住』的异常场景，并给出处理建议。」

**状态流转图生成**：描述业务流程，让 AI 生成状态机图。

> 提示词示例：「订单流程：下单 → 支付 → 发货 → 收货。请生成订单状态流转图，包含状态定义、跳转规则、异常状态。」

## 五、Checklist + 案例复盘 + 总结

### Checklist

- [ ] 我能识别流程中的异常路径并补齐（给出只有正常流程的设计，能列出至少 3 种异常场景）
- [ ] 我能理解流程和状态的关系（为一个简单流程画出状态流转图）
- [ ] 我能识别并行流程的冲突风险（给出两个并行流程，能指出可能的冲突点）

### 案例复盘要点

并行流程设计，核心要回答三个问题：谁是主流程？子流程如何汇总？冲突怎么处理？这三个问题不定义清楚，开发就要问你。

**教训**：效率高 ≠ 规则清晰。规则不清晰，效率越高，混乱越大。

### 一句话总结

**正常流程 + 异常分支 + 状态规则 + 并行策略 = 完整流程设计。**
