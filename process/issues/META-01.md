## 背景

项目基础设施已全部就绪（质量标准、术语表、lint 规则、Issue 协议、Git 规范、质量门禁、multica 智能体系统），但内容层面零产出。需要制定从「知识点地图 → TDD Issue → 写作 → 验收」的标准化闭环，并用 CH-01 样章验证可行性。

## 目标

1. 确认 TDD Issue 创建标准（基于现有 `tdd-template.md` + 知识点地图的映射规则）
2. 确认章节交付物的完整定义
3. 用 CH-01 作为样章试跑完整工作流，验证标准可执行

## 决策记录

| 决策项 | 结论 | 依据 |
|-------|------|------|
| Issue 粒度 | **一章一个 Issue（CH-XX）** | 知识点地图已有 50 章映射（ch-01 ~ ch-50），每章覆盖 2-5 个知识点 |
| Issue 编号 | 保持 `CH-XX` 协议不变 | 与 `process/issue-protocol.md` 一致 |
| 文件存放 | 直接写进 `manuscript/chapters/ch-XX.md` | 一章一文件，不拆知识点独立文件 |
| 交付物 | 正文 + 验收全套 | 包含术语覆盖、故事锚点、考核问答 |
| 启动策略 | 先 CH-01 样章跑通，再批量铺开 | 验证 TDD 工作流可行性 |
| 章节模板 | 不创建独立 chapter-template.md | 作者按 `style-guide.md` 10 要素结构 + TDD Issue 验收标准写作 |

## TDD Issue 创建标准（从知识点地图到 Issue 的映射规则）

### 知识点地图 → Issue 字段映射

| Issue 字段 | 数据来源 |
|-----------|---------|
| **章节编号与名称** | 知识点地图「八、章节映射建议」表 |
| **必须回答的问题** | 从覆盖的各 KP 的「核心概念」和「典型场景」提炼，每章 3-5 个 |
| **读者能做到的事** | 从所属模块的「模块验收标准」中选取本章覆盖项，每章 3 个 |
| **必须纠正的误解** | 从覆盖的各 KP 的「故事锚点」中提炼反直觉认知，每章 2-3 个 |
| **必须准确的概念** | 从覆盖的各 KP 的「核心概念」交叉比对 `term-glossary.md`，列出本章涉及术语 |
| **素材需求** | 从知识点地图「故事锚点」确定需要的老李/小想场景、类比、图示 |
| **目标读者** | 默认 P0（0-3 年产品经理），除非知识点地图标注深度为深 |
| **技术深度** | 认知域 = 浅/适中，方法域 = 适中，实践域 = 适中/深 |

### 章节交付物清单

每个 CH-XX Issue 关闭时，必须产出：

- [ ] `manuscript/chapters/ch-XX.md`：正文（遵循 10 要素结构）
- [ ] TDD 验收报告（嵌入 Issue 评论或独立文件）
- [ ] 术语覆盖确认（本章涉及术语已在 `term-glossary.md` 中定义）
- [ ] lint-check 零 error
- [ ] review-quality 通过率 100%
- [ ] 人类终审确认

## 验收标准（本 META Issue 自身的验收标准）

- [ ] CH-01 Issue 已创建，包含完整 TDD 验收标准
- [ ] CH-01 已完成写作并通过全部质量门禁
- [ ] 写作过程中发现的模板/流程问题已记录并修正
- [ ] 确认现有 `tdd-template.md` 无需修改 / 或已同步更新
