## 一、核心概念

**验收清单**：产品视角的系统化验收检查项，用于确认功能是否符合预期。

### 验收清单 vs 测试用例的区别

| 维度 | 验收清单 | 测试用例 |
|------|---------|---------|
| **视角** | 产品视角（功能是否符合预期） | 技术视角（代码是否正确运行） |
| **制定者** | 产品经理 | 测试工程师 |
| **关注点** | 业务价值、用户体验 | 边界条件、异常处理 |
| **颗粒度** | 功能模块级 | 代码逻辑级 |
| **目的** | 确认需求实现 | 发现技术缺陷 |
| **关系** | 互补而非替代 | 互补而非替代 |

**核心观点**：验收清单和测试用例是互补的，两者缺一不可。

## 二、验收清单设计方法

### 方法 1：按验收维度分类

| 维度 | 检查项示例 |
|------|-----------|
| **正常流** | 用户按标准流程操作能否完成目标 |
| **异常流** | 操作失败、中断、超时后的处理是否正确 |
| **边界条件** | 最大数据量、最长等待时间、极端输入的处理 |
| **权限控制** | 不同角色能否看到正确的内容和操作权限 |
| **数据校验** | 数据格式、范围、唯一性是否符合约束 |

### 方法 2：验收清单模板结构

```markdown
## 验收清单：[功能名称]

### 1. 正常流程验收
- [ ] 核心流程完整性
- [ ] 用户操作顺畅度
- [ ] 结果展示正确性

### 2. 异常流程验收
- [ ] 失败提示清晰度
- [ ] 重试机制有效性
- [ ] 数据恢复完整性

### 3. 边界条件验收
- [ ] 最大承载量测试
- [ ] 极端输入处理
- [ ] 超时处理正确性

### 4. 权限验收
- [ ] 角色权限正确性
- [ ] 数据权限边界
- [ ] 操作权限限制

### 5. 数据验收
- [ ] 数据格式校验
- [ ] 数据范围校验
- [ ] 数据唯一性校验
```

## 三、类比建议

### 类比 1：快递签收验收

**类比内容**：
验收清单就像快递签收时的检查清单：
- 包裹外包装完整吗？（正常流）
- 内部商品完好吗？（数据验收）
- 数量对吗？（边界条件）
- 快递单信息正确吗？（权限验收）
- 如果商品破损怎么处理？（异常流）

**产品经理理解要点**：
验收清单是"签收快递前要检查什么"，测试用例是"快递运输过程中要保证什么"。两者分工不同但目的一致——确保用户收到正确的东西。

---

### 类比 2：餐厅菜品验收

**类比内容**：
验收清单就像餐厅菜品出厨房前的验收：
- 菜品完整吗？（正常流）
- 口味对吗？（数据验收）
-份量够吗？（边界条件）
- 卫生吗？（权限验收）
- 如果菜品有问题怎么处理？（异常流）

**产品经理理解要点**：
验收清单是"菜端上桌前要检查什么"，测试用例是"烹饪过程中要保证什么"。验收清单关注最终呈现，测试用例关注过程质量。

---

### 类比 3：装修工程验收

**类比内容**：
验收清单就像装修工程验收：
- 水电正常吗？（正常流）
- 材料对吗？（数据验收）
- 规格达标吗？（边界条件）
- 安全吗？（权限验收）
- 如果有问题怎么保修？（异常流）

**产品经理理解要点**：
验收清单是"交房前业主要检查什么"，测试用例是"施工过程中要保证什么"。验收清单是产品视角，测试用例是工程视角。

## 四、设计验收清单的关键要点

### 关键要点 1：验收标准先于开发

> **老李经验**：验收清单应该在开发开始前就定义好，而不是上线前临时列。

**为什么**：
- 验收标准不清晰 → 开发不知道做到什么程度才算完成
- 上线前临时列 → 容易遗漏关键验收点
- 需求评审时就定义验收 → 开发和测试都有明确目标

### 关键要点 2：验收清单要覆盖全流程

**检查清单**：
- [ ] 用户从哪里进入？（入口验收）
- [ ] 操作过程中有什么提示？（交互验收）
- [ ] 操作完成后展示什么？（结果验收）
- [ ] 操作失败怎么处理？（异常验收）
- [ ] 不同角色看到什么？（权限验收）

### 关键要点 3：验收清单要与测试用例对齐

**对齐方法**：
- 验收清单的每条检查项，对应至少一条测试用例
- 测试用例覆盖的技术细节，验收清单可以不写
- 两者共同覆盖完整的质量保障范围

## 六、故事锚点素材

**场景：老李教小想设计验收清单**

> 老李："你的PRD写完了，验收清单呢？"
>
> 小想："验收清单不是测试写的吗？"
>
> 老李："测试写的是测试用例，你写的是验收清单。验收清单是产品视角——功能做出来是不是你想要的样子。测试用例是技术视角——代码跑起来有没有bug。两个要互补，不能互相替代。"

**场景：小想设计验收清单后遗漏验收导致问题**

> 上线后才发现一个关键功能漏了验收，用户投诉"操作完成后看不到结果"。
>
> 老李复盘："验收清单里没有'结果展示'这条。你只验收了'功能能做'，没验收'做完后用户能看到什么'。验收清单要覆盖全流程——从入口到结果到异常，每个环节都要有验收点。"
