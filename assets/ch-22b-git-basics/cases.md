## 一、知识点地图故事锚点（必须使用）

### 故事场景：开发问"你改了哪里？"产品说不上来

**原始场景（来自知识点地图 A8-02）**：

> 开发问"你改了哪里？"产品说"你自己对比一下上次那版"

**完整场景脚本**：

> 小想接到一个需求修改任务，花了三天把产品文档改了一遍，改了十几处内容。
>
> 周五评审会上，开发问："你这周改了哪些地方？我们要评估工作量。"
>
> 小想愣住了。她记得改了登录流程、加了两个筛选条件、调整了权限说明……但具体哪些地方改了、哪天改的、为什么改，她脑子里一片浆糊。
>
> "你自己对比一下上次那版吧。"小想说。
>
> 开发叹了口气："上次那版在哪？你有没有存档？"
>
> 小想更慌了。她只在文档里改改改，没有保存历史版本。上上周的版本被覆盖了，上周的版本也没有备份。
>
> 老李在一旁插话："小想，这就是版本控制缺失的问题。如果你用 Git 管理需求文档，每一次改动都有记录：改了什么、谁改的、什么时候改的、为什么改。开发问你'改了哪里'，你打开 Git 日志就能告诉他。"

**教训要点**：
- 需求变更应该有版本记录，不应该直接覆盖
- 产品经理应该能追溯"改了什么、什么时候改的、为什么改"
- Git 的版本控制思维适用于需求管理，不只是代码

## 二、产品经理与 Git 协作案例

### 案例 1：需求追溯困境

**场景描述**：

> 上线后发现一个功能逻辑不对，开发说"这是按你上周的需求文档做的"。
> 小想翻出上周的文档，发现已经被修改覆盖了，找不到"当时写了什么"。
> 开发追问："那时候的需求到底怎么写的？"
> 小想无法回答，因为版本被覆盖了，历史没了。

**教训要点**：
- 需求文档应该保留历史版本，不能随意覆盖
- 上线后追溯问题时，需要能回到"当时的需求版本"
- Git 的版本控制能解决这个问题

### 案例 2：多人协作需求冲突

**场景描述**：

> 小想和另一个产品经理小王同时修改同一个需求文档。
> 小想改了第 3 节的逻辑，小王改了第 3 节的流程图。
> 两人都提交了修改，结果文档出现两个版本，不知道该用哪个。
> 开发问："你们到底要哪个版本？需求文档都不一致！"

**教训要点**：
- 多人协作时需要版本管理机制
- 同一文档被多人修改可能产生"冲突"
- Git 的分支和合并机制能管理多人协作

### 案例 3：发布版本混淆

**场景描述**：

> 小想在管理产品发布，但不知道"当前版本对应哪个需求版本"。
> 开发说："你最近改了太多需求，我不知道这期发布该做哪些功能。"
> 小想发现：上周改了一版，上上周改了一版，上周又改了一版……不知道哪个版本对应"发布计划"。
> 老李说："如果你给每次需求变更打上版本标签，就不会混淆了。Git 的 tag 功能就是干这个的。"

**教训要点**：
- 每次发布应该对应明确的需求版本
- 需求变更频繁时，需要标记版本节点
- Git 的 tag（标签）能标记重要版本节点

## 三、开发问产品的 Git 相关场景

### 场景 1：分支开发进度追踪

**对话脚本**：

> 开发："我在 feature/login 分支上开发登录功能，预计下周合并到 main。"
> 小想："什么是分支？什么是合并？"
> 开发解释："feature/login 是一个开发分支，我在这个分支上做登录功能，不影响主线。做好了再合并回 main（主线）。"
> 老李："小想，你可以把分支理解为'正在开发的功能'。你问开发进度时，可以问：'这个功能在哪个分支？什么时候合并？' 这样你就知道进度了。"

### 场景 2：冲突解决需要产品决策

**对话脚本**：

> 开发："合并时遇到冲突了。你和另一个 PM 都改了同一处需求，我不知道该保留哪个版本。"
> 小想："冲突是什么？"
> 开发解释："两个人改了同一处内容，Git 不知道该用谁的版本，需要你决定。"
> 小想追问："那我该保留哪个？"
> 老李："这时候你需要做业务判断：哪个版本更符合业务目标？或者两个版本能否融合？你不能说'都可以'，必须给出明确决策。"

### 场景 3：版本回滚需求

**对话脚本**：

> 上线后发现问题，开发说："需要回滚到上个版本。"
> 小想："回滚是什么？"
> 开发解释："把代码退回到上一个版本，相当于撤销这次发布。"
> 老李："小想，回滚是紧急情况下的应急操作。你需要知道：我们上一个版本是什么？如果回滚，会影响什么功能？这些需要你判断。"

## 四、误解纠正场景

### 误解 1："Git 是程序员的工具，跟产品无关"

**纠正场景脚本**：

> 小想问："Git 是开发用的工具，我为什么要学？"
>
> 老李回答：
> "小想，Git 的版本控制思维适用于任何需要追踪变更的场景，不只是代码。"
> "产品管理需求，一样需要版本控制：
> - 你改了需求，需要知道改了什么、什么时候改的
> - 多人协作修改需求，可能产生冲突
> - 发布时，需要知道当前版本对应哪个需求版本
> - 上线后追溯问题，需要回到当时的需求版本"
>
> "这不是'学 Git 命令'，而是理解'版本控制思维'。你不需要会写代码，但你需要知道：变更应该被记录、版本应该被管理。"

### 误解 2："需求文档直接改就行了"

**纠正场景脚本**：

> 小想："需求文档我就直接改改改，存一份当前版本就行了。"
>
> 老李纠正：
> "小想，直接覆盖改是大忌。你改了什么、什么时候改的、为什么改，都没记录。
> 以后追溯问题时，你找不回'当时的需求版本'。
> 开发问你'改了哪里'，你答不上来。
>
> 正确做法：每次重要修改后，保留一个版本记录。
> 用 Git 或者其他版本管理工具，给每次修改打一个'快照'。
> 这就像你写论文——每一稿都保留，不能直接覆盖上一稿。"

## 五、GitHub/GitLab 平台使用场景

### 场景 1：用 GitHub Issue 管理需求

**场景描述**：

> 小想发现 GitHub 上有 Issue 功能，可以用来管理需求。
> 她创建了一个 Issue："新功能：用户登录优化"。
> 开发在 Issue 下评论："已开始开发，分支 feature/login-optimization"。
> 小想能实时看到进度：Issue 状态从 Open 变成 In Progress，开发评论记录进展。

**价值要点**：
- GitHub Issue 可以管理需求，不只是 Bug
- Issue 状态追踪开发进度
- Issue 评论记录沟通历史

### 场景 2：用 GitLab Roadmap 规划版本

**场景描述**：

> 小想在 GitLab 上创建 Roadmap，规划 Q2 的产品路线图。
> 每个 Epic（史诗）对应一个大功能模块，每个 Issue 对应具体需求。
> 开发看到 Roadmap，就知道优先级和里程碑时间。
> 小想和开发用共同的平台沟通，减少口头传达的误解。

**价值要点**：
- GitLab Roadmap 是产品规划工具
- Epics 和 Issues 形成需求层级结构
- 产品和开发用共同平台协作
