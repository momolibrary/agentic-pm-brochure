## 场景一：改一行文案为什么要等两天

### 背景

小想发现产品详情页有一处文案写错了，心想「改一个字而已，应该马上能上线吧」。

### 对话

**小想**：「开发哥，这个文案有个错字，改一下就行，今天能上线吗？」

**开发哥**：「改一行也要走发布流程，至少两天。」

**小想**：「？？？改一个字也要两天？这不是浪费时间吗？」

**开发哥**：「你不懂。任何改动都要经过 CI/CD 流程。」

**小想**：「CI/CD 是什么？」

**开发哥**：「CI 是持续集成，CD 是持续交付。你的改动要先自动构建、自动测试，然后部署到测试环境验证，再部署到预发环境，最后才能上生产。」

**小想**：「为什么要这么麻烦？」

**开发哥**：「因为任何改动都可能引入问题。你不测试，万一改错字的时候手抖删了别的代码？或者改错字影响了别的地方显示？我们不验证，上线出问题影响所有用户。」

**小想**：「那我改的是文案，不会影响代码逻辑啊...」

**开发哥**：「你可能不会，但之前有个实习生「只改文案」，结果把变量名也改了，上线后整个页面空白。我们吃过亏，所以任何改动都要走全流程。」

**小想**：「好吧...那两天是什么流程？」

## CI/CD 流程详解

### 流水线示意图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          CI/CD Pipeline                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   第一天                        第一天晚上                    第二天       │
│   ┌──────────────┐              ┌──────────────┐           ┌──────────────┐
│   │   Git Commit │ ─────────→ │   Build      │ ─────────→│   Staging    │
│   │   Push       │              │   Unit Test  │           │   环境       │
│   │              │              │   Integration│           │   UAT 测试   │
│   └──────────────┘              │   Test       │           └──────────────┘
│                                 └──────────────┘                    │     │
│                                                                     ↓     │
│   第二天下午                     第二天下午                    第三天      │
│   ┌──────────────┐              ┌──────────────┐           ┌──────────────┐
│   │   Production│ ←─────────→ │   Pre-prod    │ ←─────────│   验证通过    │
│   │   生产环境   │              │   预发环境   │           │   发布       │
│   └──────────────┘              └──────────────┘           └──────────────┘
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 各阶段说明

| 阶段 | 名称 | 时长 | 做什么 | 为什么需要 |
|------|------|------|--------|-----------|
| 1 | Git Commit/Push | 5 分钟 | 提交代码到仓库 | 触发流水线 |
| 2 | Build | 10 分钟 | 编译代码、打包镜像 | 确保代码能构建成功 |
| 3 | Unit Test | 15 分钟 | 运行单元测试 | 确保基础逻辑正确 |
| 4 | Integration Test | 20 分钟 | 运行集成测试 | 确保模块之间能正确协作 |
| 5 | Deploy to Staging | 10 分钟 | 部署到测试环境 | 在接近生产的环境验证 |
| 6 | UAT 测试 | 数小时 | 人工或自动验证 | 确保功能符合预期 |
| 7 | Deploy to Pre-prod | 10 分钟 | 部署到预发环境 | 最后一道关卡 |
| 8 | Deploy to Production | 10 分钟 | 上线到生产 | 对用户可见 |

### 为什么至少两天

**第一天**：
- 上午：提交代码 → 构建 → 自动测试（约 1 小时）
- 下午：部署到测试环境 → QA/产品验证（可能要半天）
- 晚上：测试环境验证通过

**第二天**：
- 上午：部署到预发环境 → 验证
- 下午：灰度发布或全量上线

**时间消耗的关键点**：
- 自动测试是必须的，不能跳过
- 人工验证（UAT）是必须的，需要人确认
- 每个环境都有验证，确保不引入问题

## 场景二：灰度发布的故事

### 背景

老李给小想讲灰度发布的故事。

### 对话

**老李**：「小想，你知道为什么我们不能直接全量上线吗？」

**小想**：「因为要测试？」

**老李**：「不只是测试。即使测试过了，也不能直接给所有用户上新版本。」

**小想**：「那怎么做？」

**老李**：「灰度发布。先给 1% 的用户用新版本，观察没问题再给 5%、25%、50%、最后 100%。」

**小想**：「为什么要这么慢？」

**老李**：「因为测试环境不可能覆盖所有真实情况。有些问题只有真实用户真实数据才会触发。如果我们直接全量上线，一出问题所有用户都受影响。」

**小想**：「有真实案例吗？」

**老李**：「去年有个功能全量上线，结果一个老用户的账号数据格式特殊，触发了我们没测到的 bug。那个用户投诉，我们紧急回滚，但已经影响了几百个用户。」

**小想**：「那灰度发布怎么避免？」

**老李**：「先给 1% 用户用。如果出问题，只影响 1%，我们发现后立即回滚，不影响其他 99%。这就是「灰度」的意义——控制影响范围。」

---

## 灰度发布 vs 蓝绿部署

### 灰度发布（Canary Release）

**定义**：先给部分用户使用新版本，逐步扩大范围。

**流程**：
```
新版本 → 1% 用户 → 观察 → 5% 用户 → 观察 → 25% → 50% → 100%
```

**特点**：
- ✅ 渐进式，风险可控
- ✅ 可以随时回滚
- ✅ 真实用户验证
- ⚠️ 需要流量分配能力

**适用场景**：
- 新功能上线
- 大版本升级
- 有风险的功能

### 蓝绿部署（Blue-Green Deployment）

**定义**：同时维护两个完全相同的环境（蓝和绿），切换流量。

**流程**：
```
蓝环境（旧版本）─────→ 绿环境（新版本）
     ↑                     │
     │                     ↓
  用户流量 ←─────── 突然切换
```

**特点**：
- ✅ 切换瞬间完成（零停机）
- ✅ 回滚简单（切回去就行）
- ⚠️ 需要两套完整环境（成本高）

**适用场景**：
- 需要零停机的升级
- 大版本切换
- 有足够资源的环境

### 对比

| 维度 | 灰度发布 | 蓝绿部署 |
|------|----------|----------|
| 切换方式 | 渐进式 | 瞬间切换 |
| 影响范围 | 逐步扩大 | 全量切换 |
| 回滚难度 | 较简单 | 非常简单 |
| 资源需求 | 一般 | 两套环境 |
| 适用场景 | 大部分上线 | 需要零停机 |

## 场景三：之前全量上线出问题的案例

### 背景

小想知道去年全量上线出问题的具体案例。

### 对话

**老李**：「去年那个全量上线的案例是这样的：」

**小想**：「什么功能？」

**老李**：「用户个人主页改版。我们测试了所有场景，但有个用户的数据很特殊。」

**小想**：「什么特殊情况？」

**老李**：「那个用户是十年老用户，他的个人主页有一条记录，数据格式是早期系统的遗留格式。我们新代码只处理了当前格式，没考虑十年前的格式。」

**小想**：「结果是什么？」

**老李**：「那个用户打开个人主页，直接报错白屏。他投诉到客服，客服转给我们。我们紧急排查，发现是数据格式问题。」

**小想**：「如果灰度发布呢？」

**老李**：「如果灰度发布，那个用户在 1% 的首批用户中。他打开白屏，我们监控报警，立即回滚，只影响他一个人。其他 99% 用户不受影响。」

**小想**：「但实际全量上线...」

**老李**：「实际全量上线，所有用户同时看到新版本。虽然只有一个用户触发 bug，但其他用户看到「系统异常」的提示也很紧张。而且我们回滚需要时间，回滚期间所有人都在用有问题的版本。」

**小想**：「那之后我们就开始用灰度发布？」

**老李**：「对。这是学费。」
