## 图示需求

### 需求描述

一张「CI/CD 流水线示意图」（代码提交→构建→测试→部署→上线）

### 图示内容建议

**CI/CD Pipeline 流水线**

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          CI/CD Pipeline                                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   开发者提交代码          构建与测试            部署到各环境              │
│   ┌──────────────┐      ┌──────────────┐      ┌──────────────┐         │
│   │   Git Commit │ ──→ │   Build      │ ──→ │   Staging    │         │
│   │   Push       │      │   Unit Test  │      │   环境       │         │
│   └──────────────┘      │   Integration│      │   UAT 测试   │         │
│                         │   Test       │      └──────────────┘         │
│                         └──────────────┘               │               │
│                                                        ↓               │
│   ┌──────────────┐      ┌──────────────┐      ┌──────────────┐         │
│   │   Production│ ←── │   Pre-prod    │ ←── │   验证通过    │         │
│   │   生产环境   │      │   预发环境   │      │   自动/手动   │         │
│   └──────────────┘      └──────────────┘      └──────────────┘         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

标注要点：
- 每个环节都有验证，不合格不放行
- Staging 环境 ≈ 生产环境配置
- 预发环境是最后一道关卡
- 发布策略：全量 / 灰度 / 蓝绿

### 图示风格建议

- 使用流水线布局：从左到右流动
- 使用颜色区分：CI 部分用蓝色（构建测试），CD 部分用绿色（部署上线）
- 使用箭头标注流向
- 整体简洁清晰，适合 PM 理解

## 验收标准对照

### 必须回答的问题

1. **TOGAF 的四大架构域是什么？为什么需求要"过架构评审"？**
   - 素材支撑：`story-architecture-review.md`（架构评审对话）
   - 类比支撑：`analogy-togaf-city-planning.md`（城市规划类比）
   - 概念解释：四大架构域（BDAT）— 业务/数据/应用/技术

2. **"改一行文案为什么要等两天上线"——发布流程是怎么回事？**
   - 素材支撑：`story-release-cycle.md`（改一行走全流程案例）
   - 类比支撑：`analogy-cicd-assembly-line.md`（流水线质检类比）
   - 概念解释：CI/CD 各阶段（构建→测试→部署→验证）

3. **什么是灰度发布？为什么不能直接全量上线？**
   - 素材支撑：`story-release-cycle.md`（灰度发布失败案例）
   - 概念解释：灰度发布 vs 蓝绿部署 vs 全量发布

### 读者能做到的事

1. **能理解企业架构框架对产品设计的约束**
   - 素材支撑：城市规划类比（不能想做就做）
   - 验证方式：能解释为什么要先过架构评审

2. **能理解 CI/CD 发布流程**
   - 素材支撑：流水线质检类比（每个环节有验证）
   - 验证方式：能解释一个改动从提交到上线经过哪些步骤

3. **能合理预期上线节奏**
   - 素材支撑：改一行走全流程案例
   - 验证方式：理解发布流程的时间需要

### 必须纠正的误解

1. **误解："小改动不需要走发布流程"**
   - 素材支撑：`story-release-cycle.md`（改一行文案的完整流程）
   - 正确理解：任何改动都可能引入问题，发布流程每步都有存在的理由

2. **误解："架构评审是官僚主义"**
   - 素材支撑：`story-architecture-review.md`（不评审导致技术孤岛）
   - 正确理解：架构评审确保新需求和现有系统兼容，不评审后期成本 10 倍
