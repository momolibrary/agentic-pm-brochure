# 故事：上线即崩溃的"完美流程"

> 素材类型：故事/案例
> 知识点：C3-03 异常陷阱（只考虑 happy path）
> 适用章节：CH-50 Section 1

---

## 故事内容

### 背景

小想负责设计一个"用户注册流程"。他画了一个完美的流程图：

```
填写信息 → 验证手机号 → 设置密码 → 注册成功 → 进入首页
```

流程顺畅，用户体验设计精美，UI同事也称赞不已。老李看了却说：
> "你这个流程很完美，但就像盖房子只考虑晴天。下雨怎么办？"

### 问题暴露

上线第一天，客服收到大量投诉：

| 用户投诉 | 对应异常场景 |
|---------|-------------|
| "填了半天信息，页面突然空白了" | 网络断开，无提示 |
| "点击验证码按钮，页面卡住了" | 短信接口超时，无重试 |
| "注册成功但没进入首页" | 服务器负载高，首页加载失败 |
| "显示了一堆英文报错" | 异常信息直接暴露给用户 |

### 复盘分析

老李带着小想复盘：

> "你看，正常流程只占 20%，异常场景占了 80%。你设计的'完美流程'，只覆盖了那 20%。"

小想：
> "那异常场景怎么设计？"

老李拿出一张异常流程图：

```
正常流程                    异常流程（需要补充）
─────────────────────────────────────────────────
填写信息                    填写信息
  ↓                          ↓（网络断开？→ 提示"网络异常，请检查网络"）
验证手机号                  验证手机号
  ↓                          ↓（短信发送失败？→ 提示"发送失败，点击重试"）
                            ↓（短信接口超时？→ 提示"等待中..." + 自动重试）
设置密码                    设置密码
  ↓                          ↓（密码格式错误？→ 提示具体要求）
注册成功                    注册成功
  ↓                          ↓（服务器响应慢？→ 显示"注册成功，正在跳转..."）
进入首页                    进入首页
                            ↓（首页加载失败？→ 显示"首页加载异常，请刷新"）
```

### 核心教训

1. **正常流程 ≠ 完整流程**：正常流程只是"晴天设计"，异常流程才是"下雨设计"
2. **异常占比高**：工程经验表明，异常处理代码量往往超过正常流程代码量
3. **用户投诉源于异常**：80% 的投诉来自异常场景处理不当
4. **运维被迫手动处理**：系统异常时无自动恢复，运维被迫人工介入

### 解决方案

老李给小想一个**异常处理 Checklist**：

```
□ 网络断开时：页面是否有提示？是否有重试按钮？
□ 超时时：是否有加载动画？是否自动重试？
□ 权限不足时：是否有友好提示？是否引导解决？
□ 数据不存在时：是否有默认展示？是否有引导提示？
□ 第三方接口失败时：是否有降级方案？是否有错误上报？
```

- 用户体验崩坏
- 运维被迫手动处理
- 异常流程往往占系统设计量的60-80%
