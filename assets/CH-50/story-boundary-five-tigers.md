# 故事：边界五虎的连环陷阱

> 素材类型：故事/案例
> 知识点：C3-04 边界陷阱
> 适用章节：CH-50 Section 2

---

## 故事内容

### 背景

小想负责设计一个"订单管理系统"，包含以下功能：
- 创建订单
- 编辑订单
- 导出订单
- 批量导入商品

老李验收时说：
> "功能不错，但边界条件考虑了吗？"

小想：
> "边界？什么边界？"

老李：
> "并发、大数据量、极端输入、时区、精度——这五个我叫它们'边界五虎'。不防好它们，上线就出事故。"

### 问题暴露（逐一击中）

**第一虎：并发问题**

场景：两个客服同时编辑同一个订单

```
客服A 打开订单 → 修改收货地址 → 点击保存
客服B 打开订单 → 修改备注信息 → 点击保存

结果：客服A的修改被客服B覆盖，收货地址丢失
投诉：客户投诉"地址怎么变了？"
```

老李：
> "这就是并发问题。两人同时修改，谁后保存谁覆盖。解决方案：乐观锁——保存时检查版本号，冲突时提示用户。"

**第二虎：大数据量**

场景：运营人员导出10万条订单数据

```
点击"导出全部订单" → 页面卡住 → 5分钟后报错 → 系统崩溃

技术原因：一次性加载10万条数据到内存，内存溢出
投诉：运营投诉"导出功能不能用"
```

老李：
> "这就是大数据量问题。解决方案：分页导出或异步导出——告诉用户'正在导出，稍后邮件发送'。"

**第三虎：极端输入**

场景：用户在备注栏输入了5000字

```
用户粘贴了一份完整合同 → 点击保存 → 系统报错"内容过长"

问题：备注字段设计长度500字，前端未限制输入，后端直接报错
投诉：用户投诉"我写了一大段备注，结果全丢了"
```

老李：
> "这就是极端输入问题。解决方案：前端限制输入长度 + 实时提示剩余字数。"

**第四虎：时区问题**

场景：海外用户下单，显示时间为凌晨3点

```
美国用户下单 → 系统记录北京时间 → 用户看到"下单时间 03:00"

问题：系统用北京时间存储，前端未转换时区
投诉：用户投诉"我明明是下午3点下单的，怎么显示凌晨？"
```

老李：
> "这就是时区问题。解决方案：后端统一存UTC时间，前端按用户时区显示。"

**第五虎：精度问题**

场景：金额计算出现错误

```
商品价格 0.1 元 × 数量 3 件 → 总价显示 0.29 元（应为 0.3 元）

问题：系统用浮点数计算金额，0.1 + 0.1 + 0.1 ≠ 0.3
投诉：用户投诉"总价算错了！"
```

老李：
> "这就是精度问题。浮点数计算有精度误差，金额计算必须用定点数（如 BigDecimal）。"

### 复盘总结

老李给小想一张**边界条件 Checklist**：

```
□ 并发问题：两人同时操作时是否加锁？是否提示冲突？
□ 大数据量：超大数据量时是否分页？是否异步处理？
□ 极端输入：空值、超长、特殊字符是否校验？
□ 时区问题：跨时区时间是否统一存储、本地显示？
□ 精度问题：金额计算是否用定点数？是否正确舍入？
```
