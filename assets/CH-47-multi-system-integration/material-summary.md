## 四、产品需求常见遗漏场景

### 遗漏类型汇总

| 遗漏类型 | 具体表现 | 开发追问 |
|---------|----------|----------|
| **依赖梳理遗漏** | 只写"对接某某系统"，没画依赖关系图 | "这个系统还调用了哪些其他系统？" |
| **接口参数遗漏** | 没约定字段名、格式、错误码 | "接口文档对齐了吗？参数格式一致吗？" |
| **数据同步遗漏** | 没写同步方式、同步时效 | "两个系统的数据怎么保持一致？" |
| **异常处理遗漏** | 只写正常流程，没写失败怎么办 | "部分成功时怎么处理？有没有回滚机制？" |
| **幂等设计遗漏** | 没考虑重试会产生重复数据 | "接口有没有幂等设计？" |
| **上线排期遗漏** | 没确认依赖系统的上线时间 | "依赖系统的上线时间定了吗？" |
| **联调时间遗漏** | 联调预留时间太短 | "联调预留多少时间？有没有缓冲？" |

## 五、关键概念解释

### 系统依赖关系图

**产品经理需要理解**：
- 是一张标明"谁依赖谁"的图
- 包含：直接依赖 + 下游依赖 + 上游依赖
- 标注：先后顺序、数据流向、接口类型

**类比**：连锁餐厅的订单流转单

### 接口对接

**产品经理需要理解**：
- 不是"调一个接口"那么简单
- 涉及：参数协商、格式约定、错误码定义、认证方式
- 联调前必须完成接口文档对齐

**类比**：餐厅厨房窗口的菜品交接规则

### 数据同步

**产品经理需要理解**：
- 同步方式：实时同步、定时同步、手动触发
- 同步方向：单向同步、双向同步
- 同步时效：用户多久能看到数据变化

**类比**：传菜员把菜品从一个餐厅送到另一个餐厅

### 异常处理与回滚

**产品经理需要理解**：
- A成功B失败是最常见的异常
- 补偿机制：自动回滚、人工介入、用户选择
- 幂等设计：防止重试产生重复数据

**类比**：旅行社行程退改签流程

### 最终一致性

**产品经理需要理解**：
- 不是所有数据都要实时一致
- 允许短暂不一致，最终达到一致
- 用户体验设计要考虑延迟

**类比**：微信群聊——发消息后，有人立刻看到，有人几秒后才看到

### 幂等性

**产品经理需要理解**：
- 同一操作执行一次和多次效果相同
- 防止用户重复点击、网络重试产生重复数据
- 幂等键：唯一标识一个请求

**类比**：自助餐厅取餐牌——无论刷多少次，只取一份餐

### 补偿事务（TCC模式）

**产品经理需要理解**：
- Try：预扣资源（冻结积分、锁定库存）
- Confirm：确认执行（扣积分、发商品）
- Cancel：回滚操作（解冻积分、释放库存）

**类比**：旅行社行程——如果租车失败，取消酒店和机票预订

## 六、开发视角的接口设计关注点

### 开发关心的问题清单

| 关注点 | 开发视角 | 产品需要提供的信息 |
|-------|---------|-------------------|
| **系统依赖** | "这个功能涉及哪些系统？每个系统提供什么能力？" | 系统依赖关系图、调用顺序 |

## 七、联调排期经验法则

### 联调时间估算公式

```
联调时间 = 接口文档协商时间 + (接口数量 × 2天) + 缓冲时间
```

**案例参考**：
- 4个接口，联调预留10天（协商2天 + 联调8天 + 缓冲2天）
- 实际用时10天，按时上线

### 排期协调要点

| 要点 | 说明 |
|------|------|
| **依赖系统排期** | 提前确认所有依赖系统的上线时间 |
| **联调排期** | 预留足够的联调时间和缓冲 |
| **上线窗口** | 所有系统必须配合，统一上线时间 |
| **缓冲时间** | "宁可预留了没用，也不要用时发现没有" |
