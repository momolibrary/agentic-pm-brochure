# 图示需求：传统竖井模式 vs 耦合协作模式对比图

> 素材类型：图示需求
> 知识点：A9-05 业务+系统+运营耦合
> 来源：知识点地图 A9-05 图示锚点

---

## 图示目的

展示「传统竖井模式」与「耦合协作模式」的组织结构对比，帮助读者直观理解两种模式的差异。

---

## 图示类型建议

**推荐类型**：左右对比图（或上下对比图）

**风格建议**：
- 简洁、扁平化风格
- 使用一致的图标风格（建议使用 Lucide或类似图标库）
- 色彩对比：传统模式用灰色/冷色调，耦合模式用彩色/暖色调

---

## 图示内容规划

### 左侧：传统竖井模式

**结构描述**：

```
┌─────────────────────────────────────────────────────────┐
│                    传统竖井模式                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌─────────┐     ┌─────────┐     ┌─────────┐           │
│   │ 业务部门 │ ──→ │ 产品部门 │ ──→ │ IT 部门 │           │
│   │   (灰)  │     │   (灰)  │     │   (灰)  │           │
│   └─────────┘     └─────────┘     └─────────┘           │
│        │              │              │                  │
│        ↓              ↓              ↓                  │
│   「甩需求」        「整理文档」      「排队开发」          │
│                                                         │
│   ┌─────────┐     ┌─────────┐                           │
│   │ 测试部门 │ ──→ │ 运维部门 │                           │
│   │   (灰)  │     │   (灰)  │                           │
│   └─────────┘     └─────────┘                           │
│        │              │                                  │
│        ↓              ↓                                  │
│   「排队测试」        「等待发布」                          │
│                                                         │
│   关键特征：                                             │
│   • 各部门各自为政，沟通成本高                            │
│   • 需求要经过 5 个部门接力                              │
│   • 大量时间在「等待」和「往返沟通」                       │
│   • 总耗时：10-15 周                                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**视觉元素建议**：

| 元素 | 建议 |
|------|------|
| 部门方块 | 使用灰色或冷色调，表示「冷冰冰」的部门墙 |
| 箭头 | 使用单向箭头，表示「接力」模式 |
| 排队标识 | 可在 IT、测试、运维方块旁加「队列」图标（多人排队） |
| 等待时间 | 可在箭头旁标注「等待 X 周」 |

### 右侧：耦合协作模式

**结构描述**：

```
┌─────────────────────────────────────────────────────────┐
│                    耦合协作模式                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│              ┌───────────────────────┐                  │
│              │     跨职能小组        │                  │
│              │    (Squad/Team)       │                  │
│              ├───────────────────────┤                  │
│              │                       │                  │
│              │  业务   产品   IT     │                  │
│              │  (蓝)  (绿)  (橙)     │                  │
│              │                       │                  │
│              │测试   运营   设计     │                  │
│              │(紫)   (红)  (黄)      │                  │
│              │                       │                  │
│              └───────────────────────┘                  │
│                      ↕↕↕↕↕                              │
│                  「实时协作」                             │
│                      ↓                                  │
│              ┌───────────────────────┐                  │
│              │      迭代周期         │                  │
│              │     (2-3 周)          │                  │
│              └───────────────────────┘                  │
│                                                         │
│   关键特征：                                             │
│   • 业务+产品+IT+测试+运营在一个小组                      │
│   • 共享一个迭代周期（2-3 周）                            │
│   • 零往返沟通，问题当场解决                              │
│   • 总耗时：2-3 周                                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**视觉元素建议**：

| 元素 | 建议 |
|------|------|
| 小组圆框 | 使用暖色调，表示「温暖」的协作氛围 |
| 角色图标 | 使用不同颜色区分角色，但都在同一个圆框内 |
| 双向箭头 | 使用双向箭头（↕），表示「实时双向沟通」 |
| 迭代周期 | 可在下方标注「2-3 周/迭代」 |

## 对比要点标注

建议在图示下方或两侧添加对比要点：

| 对比维度 | 传统竖井模式 | 耦合协作模式 |
|----------|--------------|--------------|
| **组织结构** | 5 个独立部门 | 1 个跨职能小组 |
| **沟通方式** |接力传递 | 实时双向 |
| **需求澄清** |往返 5 次 | 当场解决 |
| **迭代周期** | 10-15 周 | 2-3 周 |
| **知识共享** | 信息衰减 | 实时同步 |

## 辅助说明文字（供作者参考）

> 「传统模式下，需求要经过五个部门的接力传递，每个部门都是一个『竖井』——各自为政、各自有排期。大量时间浪费在『等待』和『往返沟通』上。
>
> 耦合模式下，业务、产品、IT、测试、运营组成一个跨职能小组，共享一个迭代周期。需求澄清当场解决，零往返沟通。上线节奏从『三个月』缩短到『两周』。」
