## 核心类比：建筑设计图 vs 施工产物

### 主类比

**数据模型 = 建筑设计蓝图**

- 定义「是什么」：客厅、卧室、厨房是什么功能空间
- 定义「怎么连接」：客厅和阳台连通、卧室和卫生间连通
- 定义「约束规则」：承重墙不能拆、管线走向规定
- **抽象层面**：不关心具体砖用什么牌子、墙厚多少厘米

**数据库表 = 施工材料清单 / 实际建成物**

- 存储「在哪里」：砖在哪个仓库、钢筋在哪里采购
- 实现「具体值」：墙厚 24cm、用红砖还是空心砖
- **物理层面**：关心存储位置、字段类型、索引配置

---

### 类比展开表

| 数据建模概念 | 建筑类比 | 解释 |
|-------------|---------|------|
| **概念模型** | 建筑平面图（草图） | 画出大致空间划分，不精确 |
| **逻辑模型** | 建筑设计蓝图 | 精确标注各空间关系，无施工细节 |
| **物理模型** | 施工图纸 + 材料清单 | 具体：墙厚、材料、施工顺序 |
| **实体** | 功能空间（客厅） | 「是什么」的概念定义 |
| **属性** | 空间特征（面积、朝向） | 空间的描述信息 |
| **关系** | 空间连接（客厅-阳台） | 空间之间的通路 |
| **约束** | 建筑规范（承重墙） | 必须遵守的规则 |
| **表** | 实际墙体/房间 | 物理建造结果 |
| **字段** | 具体参数（墙厚 24cm） | 物理实现的参数值 |
| **索引** | 门牌号/标签 | 快速找到某个房间的方式 |

## 其他类比选项

### 类比 2：菜单 vs 食材库

| 数据模型概念 | 菜单类比 | 解释 |
|-------------|---------|------|
| **数据模型** | 菜单设计 | 定义菜品是什么、怎么组合 |
| **数据库表** | 食材库 | 具体食材存放在哪个仓库、什么库存 |
| **实体** | 菜品概念（宫保鸡丁） | 定义「这道菜是什么」 |
| **属性** | 菜品特征（价格、口味） | 菜品的描述信息 |
| **表** | 食材库存记录 | 具体食材数量、位置 |

**适用场景**：餐饮行业 PM 或更生活化场景

### 类比 3：地图概念层 vs 地图数据层

| 数据模型概念 | 地图类比 | 解释 |
|-------------|---------|------|
| **数据模型** | 地图概念层 | 定义「道路」「河流」「建筑」是什么 |
| **数据库表** | 地图数据层 | 具体坐标、像素、存储格式 |
| **实体** | 地图概念（道路） | 定义「这是什么类型」 |
| **属性** | 特征（长度、等级） | 描述信息 |
| **表** | 坐标数据记录 | 具体坐标值 |

**适用场景**：地理信息或导航系统 PM

### 类比 4：词汇定义 vs 词汇库

| 数据模型概念 | 语言类比 | 解释 |
|-------------|---------|------|
| **数据模型** | 词典 + 句法规则 | 定义「名词」「动词」「主谓宾」 |
| **数据库表** | 词汇库 | 具体单词存放位置、使用频率 |
| **实体** | 词类（名词） | 定义「这是什么类型」 |
| **属性** | 词特征（词性、释义） | 描述信息 |
| **关系** | 句法规则（主谓） | 词汇之间的关系 |

**适用场景**：NLP 或文本处理系统 PM

## 核心认知纠正点

### 误解：「表就是模型」

**真相**：
- 表是模型的**物理实现**，不是模型本身
- 一个模型可能对应多张表（分表、历史表、视图）
- 模型关注「业务语义」，表关注「存储效率」

### 误解：「数据模型是开发的事」

**真相**：
- 数据模型是**业务逻辑的抽象**
- 如果模型抽象错误（比如把「订单」和「购物车」混淆），后续功能都会歪
- 产品经理需要参与数据模型的业务语义确认

### 误解：「模型和表是一回事」

**真相**：
- 模型有三个层次：概念 → 逻辑 → 物理
- 「表」只是物理层
- 产品经理参与的是**概念层和逻辑层**，不是物理层

## TOGAF 三层模型参考

| 层次 | 内容 | 责任人 | 产品经理参与度 |
|------|------|--------|---------------|
| **概念模型** | 业务实体、关系（高层） | 业务 + 架构 | ⭐⭐⭐⭐⭐ 高度参与 |
| **逻辑模型** | 实体属性、关系细节 | 数据架构师 | ⭐⭐⭐⭐ 需要确认 |
| **物理模型** | 表结构、索引、分区 | DBA + 开发 | ⭐⭐ 了解即可 |

## DDIA 补充观点

来自《数据密集型应用系统设计》Chapter 2：

> 「数据模型不仅决定数据如何存储，还决定数据如何被访问、如何被理解。」
>
> 「关系模型、文档模型、图模型各有优劣，选择取决于**访问模式**。」

**对 PM 的启示**：
- 数据模型的选择影响系统性能和功能边界
- PM 需要理解「为什么选这个模型」，而不是直接跳到「表怎么建」
