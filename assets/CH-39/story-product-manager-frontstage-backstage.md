# 故事场景：产品经理只看前台，不看后台支撑

> 知识点：B6-03（服务蓝图）
> 用途：第 2 节「真实业务场景开场」、第 4 节误解案例
> 场景真实度：⭐⭐⭐⭐⭐（常见误解）

---

## 场景背景

### 故事设定

**人物**：
- 老李：40+ 岁产品总监，经验丰富但很少写代码
- 小想：22 岁实习生，AI 原生代产品经理
- 开发组长：技术团队负责人
- 运营同事：负责客服和订单处理

**场景**：
某电商公司的产品评审会议，讨论新功能"一键查询订单"。

---

## 故事正文

### 第一幕：功能评审的争论

会议室里，小想正在展示新功能方案。

**小想**：我设计了一个"一键查询订单"功能。用户在订单页面点击一个按钮，就能看到订单的所有信息，包括物流状态、支付状态、退款进度。这样用户不用在多个页面跳转，体验更流畅。

老李看着方案，眉头微微皱起。

**老李**：小想，这个功能的前台界面设计做得不错，按钮位置、信息展示都很清晰。但我想问一个问题：用户点击按钮后，后台发生了什么？

**小想**：后台？我设计的是前台界面，后台是技术团队负责的吧。

**开发组长叹了口气**：每次产品经理这么说，我就知道要踩坑了。

**老李**：小想，你有没有画过服务蓝图？

**小想**：服务蓝图？那是设计工具吧，我只画过用户旅程图。

**老李**：用户旅程图和服务蓝图有什么区别？

**小想**：用户旅程图是用户视角的体验流程图，服务蓝图...我不太清楚。

---

### 第二幕：服务蓝图的分析

老李打开投影仪，展示一张服务蓝图示意图。

**老李**：小想，你来看这张图。用户旅程图只看用户视角，记录用户能感知的部分。服务蓝图不仅看用户视角，还看前台、后台、内部流程三层。

```
┌─────────────────────────────────────────────────────────────────┐
│                    订单查询服务蓝图                               │
├─────────────────────────────────────────────────────────────────┤
│ 用户行为层     ──→ 点击按钮 → 等待加载 → 看到信息                  │
│ (用户做的事)                                                      │
├─────────────────────────────────────────────────────────────────┤
│ 前台交互层     ──→ 按钮响应 → 加载动画 → 信息展示                  │
│ (用户能看到的)                    ← 可视线 →                      │
├─────────────────────────────────────────────────────────────────┤
│ 后台支撑层     ──→ 订单系统查询 → 物流系统查询 → 支付系统查询        │
│ (用户看不到但支撑前台)                                             │
│                    ← 内部线 →                                     │
├─────────────────────────────────────────────────────────────────┤
│ 内部流程层     ──→ IT运维 · 数据库 · 财务系统 · 仓库系统            │
│ (系统的基础支撑)                                                   │
└─────────────────────────────────────────────────────────────────┘
```

**老李**：你设计的是"用户点击按钮 → 看到信息"，这是用户旅程图覆盖的部分。但后台发生了什么？

**小想**：后台应该会查询订单数据、物流数据、支付数据吧。

**老李**：对。你的一键查询功能，后台需要同时调用订单系统、物流系统、支付系统三个接口。这三个接口的响应时间分别是多少？

**小想愣了一下**：我不知道...

**开发组长**：订单系统平均响应 200ms，物流系统平均响应 800ms（因为是外部接口），支付系统平均响应 300ms。合计可能超过 1 秒。

**小想**：1 秒也可以接受吧？

**运营同事**：用户等待超过 3 秒就会焦虑，超过 5 秒可能直接放弃。如果物流接口偶尔超时，用户会投诉"功能不好用"。

**老李**：小想，这就是只看前台不看后台的问题。你设计的前台界面很美观，但如果后台支撑不到位，前台承诺就无法兑现。

---

### 第三幕：误解的纠正

**老李**：我再问你一个问题。你的功能显示"退款进度"，这个信息从哪里来？

**小想**：应该从退款系统获取吧。

**老李**：退款信息需要调用财务系统。财务系统的数据更新频率是多少？

**小想**：不知道...

**运营同事**：财务系统每天更新一次，不是实时的。如果用户刚申请退款，马上查询，财务系统还没更新，用户会看到"无退款记录"，然后投诉我们"数据不准"。

**小想开始意识到问题**：那我应该怎么设计？

**老李**：设计功能时，不能只看前台界面。要画出服务蓝图，确认后台支撑能否到位。

**老李指着服务蓝图**：

1. **用户行为层**：用户点击按钮，期望 3 秒内看到信息
2. **前台交互层**：界面设计、加载动画、信息展示
3. **后台支撑层**：三个接口并行调用，需要接口优化或缓存
4. **内部流程层**：财务系统更新频率需要告知用户

**老李**：产品设计要考虑三层。如果后台支撑不到位，要么优化后台，要么调整前台设计（比如显示"数据更新中"而不是实时承诺）。

---

### 第四幕：小想的反思

会议结束后，小想找到老李。

**小想**：老李，我以前真的以为产品经理只管前台界面，后台是技术团队的事。今天才发现，前台设计离不开后台支撑。

**老李**：这是产品经理常见的误解。很多产品经理只画用户旅程图，不看服务蓝图，设计的需求可能无法落地。

**小想**：那我以后设计功能，应该怎么做？

**老李**：先画服务蓝图，确认三层都能支持你的设计。如果后台支撑不到位，要么推动后台优化，要么调整前台设计。不能只看前台，不看后台。

**小想点头**：服务蓝图就像一个"全景检查器"，帮我把前台设计和后台支撑对齐。

**老李**：记住一句话：**用户旅程图只看前台，服务蓝图还看后台支撑和内部流程。**

---

## 场景复盘

### 误解分析

| 误解 | 表现 | 根因 |
|------|------|------|
| "产品经理只看前台界面，后台是技术团队的事" | 功能设计不考虑后台支撑 | 只画用户旅程图，不画服务蓝图 |

### 正确理解

| 正确理解 | 应用 |
|---------|------|
| 功能设计必须考虑后台支撑 | 设计功能前画服务蓝图，确认三层都能支持 |

### 服务蓝图的工具价值

服务蓝图帮助产品经理：
1. 看到前台界面背后的后台支撑
2. 识别服务断点（后台支撑不到位的地方）
3. 设计可落地的功能（前台承诺与后台能力匹配）

---

## 产品经理实践要点

### 功能设计流程

产品经理设计功能时：
1. 先画用户旅程图，确认用户体验流程
2. 再画服务蓝图，确认后台支撑能到位
3. 如果后台支撑不到位，推动优化或调整前台设计

### 服务断点识别

产品经理可以通过服务蓝图识别断点：
- 用户期望 vs 后台能力不匹配
- 前台承诺 vs 内部流程不到位
- 多系统接口响应时间过长
