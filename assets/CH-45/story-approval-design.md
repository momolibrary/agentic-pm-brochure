## 场景描述

公司要做一套采购审批系统，PM 老李第一次设计审批流程，以为很简单：提交 → 颅导审批 → 通过。结果上线后发现各种问题：审批卡住不动、申请人想撤回没入口、驳回后不知道能做什么……

---

## 对话脚本

### 第一幕：PM 以为审批很简单

**小想**：老李，公司要做采购审批系统，我来设计流程吧。很简单：提交 → 颅导审批 → 通过，三个状态搞定！

**老李**：三个状态？你确定够用？

**小想**：够用啊！申请人提交，领导看一眼，通过就完事。驳回的话再改改重新提交。

**老李**：那我问你几个问题：
- 申请人提交后，想撤回怎么办？
- 颅导审批超时了怎么办？比如领导出差一周没人批？
- 颅导驳回后，申请人能做什么？重新提交？申诉？放弃？
- 采购金额超过 10 万，是不是要找总经理批？

**小想**：……我没想这些。

### 第二幕：老李揭示审批复杂性

**老李**：审批流程不只是提交→通过/驳回。真实世界里的审批要考虑很多场景：

| 要素 | 你漏掉的 | 实际需要 |
|------|----------|----------|
| **撤回机制** | 没入口 | 待审批时申请人可以撤回 |
| **超时处理** | 没考虑 | 设置超时自动升级或催办 |
| **驳回后动作** | 不知道能做什么 | 明确：可重新提交、可申诉 |
| **多级审批** | 只有一级 | 金额超过阈值要找更高级领导 |
| **会签** | 没考虑 | 采购可能需要财务+采购+使用部门都同意 |
| **审批记录** | 没记录 | 记录每次审批动作、时间、意见 |

**小想**：这么多？我以为审批很简单……

**老李**：审批是把公司决策流程搬到线上。公司里找领导签字，也有很多情况：领导出差了、领导说不行、申请人说算了……这些都要考虑进去。

### 第三幕：设计完整的审批流程

**老李**：来，我们重新设计。先画状态机：

```
状态机设计：

┌─────────┐    提交     ┌─────────┐
│  草稿   │ ───────────→│  待审批 │
│ (DRAFT) │             │(PENDING)│
└─────────┘             └────┬────┘
     ↑                       │
     │                       ├────── 通过 ──→ 已通过
     │                       │
     │                       ├────── 颅回 ──→ 已驳回 → 可重新编辑
     │                       │
     │ 撤回 ←────────────────┤
     │                       │
已撤回 ←──────────────────────┘
```

**老李**：状态要完整，转换条件要清楚。然后是审批规则：

| 规则 | 设计 |
|------|------|
| **金额分级** | ≤5万：部门经理批；>5万且≤10万：总经理批；>10万：老板批 |
| **超时处理** | 待审批超过 3 天，自动催办；超过 7 天，自动升级 |
| **撤回机制** | 待审批状态可撤回，已通过不可撤回 |
| **驳回后动作** | 可重新编辑后再次提交，或放弃申请 |
| **审批记录** | 记录审批人、审批时间、审批意见、审批动作 |

**小想**：这样完整多了。还有什么要考虑的？

**老李**：还有会签。采购涉及多个部门：

| 部门 | 审批内容 |
|------|----------|
| **使用部门** | 确认采购需求是否真实 |
| **采购部** | 确认采购流程是否合规 |
| **财务部** | 确认预算是否充足 |

会签逻辑：三个部门都要通过才算通过，一个驳回就驳回。

## 核心洞察

### 审批流程设计要素清单

| 要素 | 设计要点 | 漏掉会怎样 |
|------|----------|-----------|
| **审批角色** | 按角色、按金额阈值、按部门层级 | 审批找不到人 |
| **审批层级** | 单级还是多级，条件分支 | 大事小领导批了，出问题 |
| **审批规则** | 什么情况触发审批，条件分支 | 不该批的批了，该批的没批 |
| **驳回处理** | 颅回后能做什么 | 申请人不知道怎么办 |
| **撤回机制** | 待审批时可撤回 | 申请人想改没入口 |
| **超时处理** | 自动升级或催办 | 审批卡住不动 |
| **并行会签** | 多人同时审批逻辑 | 会签规则不清 |
| **串行审批** | 按顺序审批逻辑 | 流程卡在某一级 |
| **状态机设计** | 状态、转换、异常处理 | 状态漏了，系统出 bug |

### 审批流程常见遗漏

| 遗漏 | 问题 | 修复 |
|------|------|------|
| **超时处理** | 审批卡住不动 | 设置超时自动升级 |
| **撤回机制** | 申请人想改没入口 | 待审批状态可撤回 |
| **驳回后动作** | 不知道能做什么 | 明确重新提交或放弃 |
| **审批记录** | 没历史追溯 | 记录审批动作 |
| **条件分支** | 不同情况走不同路径 | 按金额、类型分支 |
| **会签逻辑** | 多人审批逻辑不清 | 明确全会签通过才算 |

## 审批状态机模板

### 基础状态机（适用于简单审批）

| 状态 | 说明 | 允许操作 |
|------|------|----------|
| 草稿 (DRAFT) | 申请人在编辑 | 编辑、删除、提交 |
| 待审批 (PENDING) | 等待审批 | 撤回、查看 |
| 已通过 (APPROVED) | 审批通过 | 查看、执行后续动作 |
| 已驳回 (REJECTED) | 审批驳回 | 重新编辑、放弃 |
| 已撤回 (REVOKED) | 申请人撤回 | 重新编辑、删除 |

### 进阶状态机（适用于复杂审批）

额外考虑：
- 多级审批：待一级审批 → 待二级审批 → 待三级审批
- 会签状态：待会签（并行）→ 会签中 → 会签完成
- 超时状态：待审批 → 超时催办 → 超时升级
