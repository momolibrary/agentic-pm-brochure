# 类比：审批流程 → 公司决策流程

## 类比核心

**审批流程 = 公司决策流程的数字化。**

公司里大大小小的决策都要走流程：预算审批、合同审批、报销审批、采购审批……这些流程在公司里早就在运行，审批系统只是把它们"搬到线上"。

所以，**设计审批系统不是发明新流程，而是理解现有决策流程，然后数字化。**

---

## 详细类比表

| 公司决策流程 | 数字化审批流程 | 说明 |
|--------------|-----------------|------|
| **找领导签字** | **提交审批申请** | 申请人在系统里提交，代替拿着纸去找领导 |
| **领导看材料** | **审批人看详情页** | 权限控制：只有审批人能看到完整信息 |
| **领导签字同意** | **点击"通过"按钮** | 系统记录：审批人、审批时间、审批意见 |
| **领导说不行** | **点击"驳回"按钮** | 系统记录驳回原因，申请人收到通知 |
| **找大老板** | **多级审批流转** | 一级审批通过自动流转到下一级 |
| **各部门都要同意** | **会签（并行审批）** | 所有审批人都要通过，才能继续 |
| **申请撤回** | **撤回机制** | 申请人在审批完成前可以撤回 |
| **领导出差了** | **审批超时处理** | 设置超时提醒，或自动升级 |
| **事后查档案** | **审批记录追溯** | 所有审批动作都有历史记录 |

## 类比叙述文本

### 标准叙述版本

老李说："审批流程就是把公司里找领导签字的过程搬到线上。

你想想，公司里大大小小的事都要审批：预算、合同、报销、采购……以前是拿着纸质材料找领导签字，领导看了觉得没问题就签，觉得有问题就说不行。

现在有了系统，就在系统里提交，领导在线上看、在线上批。本质上是一样的——只是从线下搬到线上，效率更高，记录更清楚。

所以设计审批系统，首先要搞清楚：公司里有哪些决策流程？谁有权批什么？什么情况找大老板？各部门怎么会签？这些都在公司里早就在运行了，你只是把它们数字化。"

### 精简叙述版本

"审批流程 = 公司决策流程的数字化。找领导签字搬到线上，找大老板变成多级审批，各部门同意变成会签。设计审批系统不是发明新流程，而是理解现有决策流程，然后数字化。"

## 审批流程复杂场景类比

### 多级审批 → 找大老板

| 场景 | 类比说明 |
|------|----------|
| **金额超过阈值** | 花钱多了要找大老板签字 |
| **预算审批** | 部门经理批小钱，总经理批大钱 |
| **合同审批** | 普通合同部门经理批，重要合同总经理批 |

### 会签 → 各部门都要同意

| 场景 | 类比说明 |
|------|----------|
| **采购审批** | 财务部看预算、采购部看合规、使用部门看需求 |
| **合同审批** | 法务看条款、财务看金额、业务看内容 |
| **新产品上线** | 产品看功能、技术看可行性、运营看市场 |

### 条件分支 → 不同情况找不同人

| 场景 | 类比说明 |
|------|----------|
| **紧急采购** | 紧急情况下简化流程，直接找领导批 |
| **特殊类型** | IT 设备找 IT 部门批，办公用品找行政批 |
| **金额分级** | 小钱找部门经理，大钱找总经理，超大找老板 |

## 类比延伸：审批流程的状态机

审批流程的状态机，就像审批单的"生命周期"：

| 状态 | 类比 | 说明 |
|------|------|------|
| **草稿** | 申请人在写材料 | 还没提交，随时改 |
| **待审批** | 材料写好递给领导 | 等领导看 |
| **已通过** | 领导签字同意 | 可以执行了 |
| **已驳回** | 颅导说不行 | 要修改再重新递 |
| **已撤回** | 申请人说算了 | 自己撤回，不批了 |

**状态之间的转换，就是审批过程中可能发生的事：**

- 草稿 → 待审批：提交申请（递给领导）
- 待审批 → 已通过：审批通过（领导签字）
- 待审批 → 已驳回：审批驳回（领导说不行）
- 待审批 → 已撤回：申请撤回（申请人撤回）
- 已驳回 → 草稿：重新编辑（改好再递）
- 已撤回 → 草稿：重新编辑（改好再递）
