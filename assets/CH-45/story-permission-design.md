# 故事场景：权限矩阵设计漏洞

> 素材类型：故事场景
> 知识点：C1-02 权限系统实战
> 更新日期：2026-04-22

---

## 场景描述

公司 CRM 系统上线后，销售经理发现一个严重问题：他能看全公司的客户数据，包括其他部门的客户。原来 PM 设计权限时只考虑了"有没有权限"，没考虑"能看到哪些数据"。

---

## 对话脚本

### 第一幕：权限漏洞被发现

**销售经理**：老李，你们这个 CRM 系统有问题！我能看到财务部门的客户信息！

**老李**：什么？销售经理不应该看到财务部门的客户啊。

**销售经理**：我点开"客户管理"，能看到所有客户，包括财务部在跟进的那些。这不合适吧？

**老李**：我来查一下……原来我们只设置了功能权限，没设置数据权限。

**小想**：数据权限？我只设置了"销售经理有客户管理权限"，没管数据范围。

**老李**：这就是问题所在。权限不只是"有没有"，还有"能看到哪些"。

### 第二幕：老李揭示权限多维性

**老李**：权限系统是三维的，不只是有和无：

| 维度 | 说明 | 你漏掉的 |
|------|------|----------|
| **功能权限** | 能看到哪些功能、菜单、按钮 | 设置了 ✓ |
| **数据权限** | 能看到哪些数据范围 | 漏掉了 ✗ |
| **操作权限** | 能执行什么操作（查看/编辑/删除） | 漏掉了 ✗ |

**小想**：我以为权限就是"管理员"和"普通用户"两种。

**老李**：这是最常见的误解。真实系统的权限是多维的：

| 角色 | 功能权限 | 数据权限 | 操作权限 |
|------|----------|----------|----------|
| **普通销售** | 客户管理 | 自己的客户 | 查看、编辑 |
| **销售经理** | 客户管理 + 报表 | 本部门客户 | 查看、编辑、审批折扣 |
| **销售总监** | 客户管理 + 报表 + 配置 | 全公司客户 | 查看、编辑、删除 |
| **财务人员** | 财务报表 | 全公司财务数据 | 查看 |

**小想**：原来权限要三维设计……

### 第三幕：重新设计权限矩阵

**老李**：来，我们重新设计权限矩阵。先用六步法：

| 步骤 | 内容 | 输出 |
|------|------|------|
| **1. 识别角色** | 列出所有用户角色 | 普通销售、销售经理、销售总监、财务人员、系统管理员 |
| **2. 定义功能权限** | 每个角色能执行什么操作 | 功能权限表 |
| **3. 定义数据权限范围** | 每个角色能看到哪些数据 | 数据权限表 |
| **4. 设计权限继承规则** | 角色之间有无继承 | 销售经理继承普通销售权限 + 部门权限 |
| **5. 处理特殊角色** | 临时授权、代理审批 | 临时访客权限、代理审批权限 |
| **6. 输出权限矩阵** | 整合所有权限定义 | 权限矩阵文档 |

**老李**：然后输出权限矩阵：

| 角色 | 客户管理 | 订单管理 | 财务报表 | 系统配置 |
|------|----------|----------|----------|----------|
| **普通销售** | 本人客户 | 本人订单 | ❌ | ❌ |
| **销售经理** | 本部门客户 | 本部门订单 | ❌ | ❌ |
| **销售总监** | 全公司客户 | 全公司订单 | 查看权限 | ❌ |
| **财务人员** | ❌ | ❌ | 全公司财务 | ❌ |
| **系统管理员** | 配置权限 | 配置权限 | 配置权限 | ✅ 全部 |

**小想**：这样每个角色都知道能看到什么数据、能做什么操作了。

**老李**：还有两个要考虑：权限继承和特殊角色。

### 第四幕：权限继承和特殊角色

**老李**：权限继承规则：

| 角色 | 基础权限 | 继承权限 | 额外权限 |
|------|----------|----------|----------|
| **普通销售** | 基础权限 | 无 | 无 |
| **销售经理** | 无 | 继承普通销售权限 | 部门数据权限 + 折扣审批 |
| **销售总监** | 无 | 继承销售经理权限 | 全公司数据权限 + 删除权限 |
| **系统管理员** | 无 | 继承总监权限 | 系统配置权限 |

**老李**：特殊角色：

| 场景 | 权限设计 |
|------|----------|
| **临时授权** | 项目期间临时访问权限，到期自动收回 |
| **代理审批** | 颅导出差时，设置代理审批人 |
| **一人多角色** | 权限叠加，取最高权限 |

## 核心洞察

### 权限设计六步法

| 步骤 | 内容 | 输出 |
|------|------|------|
| **1. 识别角色** | 列出系统所有用户角色 | 角色清单 |
| **2. 定义功能权限** | 每个角色能执行什么操作 | 功能权限表 |
| **3. 定义数据权限范围** | 每个角色能看到哪些数据 | 数据权限表 |
| **4. 设计权限继承规则** | 角色之间有无继承关系 | 继承规则 |
| **5. 处理特殊角色** | 临时角色、代理角色 | 特殊角色定义 |
| **6. 输出权限矩阵** | 整合所有权限定义 | 权限矩阵文档 |

### 权限系统常见遗漏

| 遗漏 | 问题 | 修复 |
|------|------|------|
| **数据权限边界** | 只考虑功能权限 | 定义：全部、本部门、本人、无 |
| **权限继承** | 新角色从哪里继承 | 明确继承规则 |
| **临时授权** | 临时需要额外权限 | 设计临时授权流程 |
| **权限变更审计** | 不知道谁改了权限 | 记录权限变更历史 |
| **角色冲突** | 一人多角色权限逻辑 | 明确叠加或取最高 |
| **敏感操作保护** | 删除无二次确认 | 添加二次确认或审批 |

## 权限矩阵模板

### 基础权限矩阵

| 角色 | 功能模块 | 数据范围 | 操作权限 |
|------|----------|----------|----------|
| **角色 A** | 模块 X | 本人 | 查看、编辑 |
| **角色 B** | 模块 X + Y | 本部门 | 查看、编辑、审批 |
| **角色 C** | 模块 X + Y + Z | 全部 | 查看、编辑、删除、配置 |

### 数据权限范围定义

| 范围 | 说明 |
|------|------|
| **全部** | 能看到全公司数据 |
| **本部门** | 只能看到本部门数据 |
| **本人** | 只能看到自己创建的数据 |
| **无** | 看不到任何数据 |

### 操作权限定义

| 操作 | 说明 |
|------|------|
| **查看** | 能看到数据 |
| **编辑** | 能修改数据 |
| **删除** | 能删除数据 |
| **审批** | 能审批申请 |
| **配置** | 能配置系统参数 |

## 审批与权限的关联

审批流程和权限系统是耦合的：

| 审批节点 | 权限问题 |
|----------|----------|
| **谁有权审批** | 角色权限：审批人角色有审批权限 |
| **能看到哪些申请** | 数据权限：审批人能看到范围内的申请 |
| **审批后能做什么** | 操作权限：审批人能通过、驳回、撤回 |

**设计时必须一起考虑**：审批流程设计 → 每个节点涉及谁审批 → 权限系统定义谁能审批。
