## 详细类比

### 场景：汽车碰撞测试

一辆新车研发出来，车企会做哪些测试？

| 测试类型 | 测试内容 | 为什么测这个 |
|---------|---------|-------------|
| 正常驾驶测试 | 日常驾驶场景 | 确保基本功能正常 |
| 正面碰撞测试 | 64km/h正面撞击刚性墙 | 极端情况下的安全性 |
|侧面碰撞测试 | 50km/h侧面撞击 | 侧向冲击保护 |
| 追尾测试 | 后方碰撞 | 保护后排乘客 |
| 滚翻测试 | 车辆翻滚 | 极端失控场景 |
| 极端温度测试 | -40°C到+50°C环境 | 适应各种气候 |

**为什么车企不只测"正常驾驶"？**

因为汽车上路后会遇到各种意外情况：
- 高速追尾
- 冰雪路面失控翻车
- 极寒环境下启动

**如果只测正常驾驶，不测碰撞和极端场景，上市后会怎样？**

出了事故，乘客伤亡，车企要赔钱、召回、品牌受损。

---

## 映射关系

| 汽车碰撞测试 | 软件边界检查 |
|-------------|-------------|
| 正常驾驶测试 | 正常流程验证 |
| 碰撞测试 | 极端场景验证 |
| 极端温度测试 | 边界条件验证 |
| 测试发现问题 → 改进设计 | 预演发现问题 → 补充需求 |
| 不测碰撞 → 上市后出事故 | 不测边界 → 上线后系统崩溃 |
| 碰撞测试成本几百万 | 预演成本几小时 |
| 事故召回成本几十亿 | 上线修复成本几周甚至几个月 |

---

## 产品经理视角

老李对小想说：

> "你写需求只写了'正常情况'——用户点击按钮、系统正常处理。就像车企只测正常驾驶。但系统上线后会遇到什么？并发5000人同时点击、数据量突然涨到100万条、网络断了、服务器宕了...这些都是'碰撞场景'。你不测，上线后用真实用户当碰撞假人？"

**边界检查就是软件的碰撞测试。**

---

## 类比要点

1. **正常场景≠完整场景** — 只测正常驾驶，不测碰撞，车不安全；只测正常流程，不测边界，系统不稳定
2. **边界场景是事故来源** — 大多数bug不是正常场景出问题，而是边界场景：并发高、数据多、网络断、用户乱操作
3. **边界检查成本远低于事故成本** — 碰撞测试几百万 vs 事故召回几十亿；预演几小时 vs 上线修复几周
4. **边界场景列举清单** — 并发上限、数据量上限、输入长度上限、异常输入、网络中断、服务器故障

---

## 常见边界场景清单

| 边界类型 | 需要追问的问题 |
|---------|---------------|
| 数据量边界 | 数据量多大时还能正常？超了怎么办？ |
| 并发边界 | 多少人同时操作还能正常？超了排队还是拒绝？ |
| 输入边界 | 输入最长多长？最短多短？超了怎么处理？ |
| 时间边界 | 超时多久算失败？失败怎么处理？ |
| 异常边界 | 输入乱七八糟的内容怎么办？ |
| 依赖边界 | 依赖的系统挂了怎么办？网络断了怎么办？ |
