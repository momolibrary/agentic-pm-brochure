# Git-like 工作方式实战案例

> 知识点：B5-06 Git-like 工作方式
> 素材类型：故事场景
> 来源：知乎、GitHub 官方教程

---

## 场景背景

产品经理小陈负责一个电商项目的需求管理。项目运行半年后，他发现需求变更记录混乱：不知道谁改了什么、为什么改、改了之后有什么影响。这导致开发经常问"这个需求什么时候改的？"，小陈却回答不上来。

---

## 失败案例：需求变更无记录

### 场景描述

**开发小王**：小陈，这个需求的逻辑怎么变了？上周评审的时候不是说要显示原价吗？

**小陈**：嗯...业务方上周五说要改成显示折扣价，我就改了PRD。

**开发小王**：那你通知我们了吗？我代码已经按原价写的了。

**小陈**：我直接改了文档，以为你们会看...

**开发小王**：我们怎么知道你改了？你也没发通知。

**小陈**：那我怎么办？

**开发小王**：现在改要重写代码，影响排期。而且我不知道你改了哪些，还要重新看一遍PRD。

---

### 问题分析

| 问题 | 原因 |
|------|------|
| 不知道谁改了什么 | 没有变更记录 |
| 不知道什么时候改的 | 没有变更时间戳 |
| 不知道为什么改 | 没有变更原因说明 |
| 不知道改了哪些 | 没有变更内容对比 |

---

## 正确做法：Git-like 工作方式

### 什么是 Git-like 工作方式？

Git-like 工作方式是将 Git 版本管理思维应用到需求管理中：
- 每次改动有记录（像 Git commit）
- 改动原因有说明（像 commit message）
- 改动内容可追溯（像 git diff）
- 改动需要评审（像 Pull Request）

---

### Git-like 需求管理流程

```
需求变更请求（Issue） → 变更评审（PR Review） → 变更实施（Merge） → 变更记录（Commit Log）
```

#### 1. 用 Issue 追踪需求变更

**Issue 模板**：

```markdown
## 变更请求

**变更标题**: 价格显示从原价改为折扣价
**变更原因**: 业务方反馈用户更关注折扣力度
**变更来源**: 业务运营团队
**变更时间**: 2026-04-21
**影响范围**:
- 商品详情页价格展示
- 购物车价格展示
- 订单确认页价格展示

**变更内容**:
| 原逻辑 | 新逻辑 |
|--------|--------|
| 显示原价 ¥100 | 显示折扣价 ¥80 + 原价 ¥100（划线） |

**期望排期**: 下周一上线
```

#### 2. 用 PR Review 评审变更

**变更评审清单**：

| 评审项 | 检查内容 |
|--------|---------|
| 变更原因是否合理 | 业务价值清晰 |
| 影响范围是否评估 | 开发、测试、运营都已确认 |
| 技术可行性是否确认 | 技术团队已评估 |
| 排期是否协调 | 开发排期已调整 |

#### 3. 变更实施后记录

**变更记录格式**：

```markdown
## 变更日志

**变更日期**: 2026-04-21
**变更内容**: 价格显示逻辑调整
**变更原因**: 业务方反馈用户更关注折扣力度
**影响范围**: 商品详情页、购物车、订单确认页
**关联Issue**: #123
**评审通过**: 开发小王、测试小李、运营小陈
```

## Git-like 工作方式类比

### 类比一：文档修订留痕

| Git 概念 | 文档类比 |
|---------|---------|
| **版本库** | 文档的完整修订历史档案 |
| **Commit** | 每次保存文档的修改记录 |
| **分支** | 不同作者同时修改文档的副本 |
| **Pull Request** | 编辑审核修改建议 |
| **Merge** | 将多位作者的修改合并到主文档 |
| **Diff** | 文档修订痕迹对比（红色删除、绿色新增） |
| **冲突** | 两人同时修改同一段落 |

### 类比二：装修施工记录

| Git 概念 | 装修类比 |
|---------|---------|
| **项目** | 一套房子装修工程 |
| **源代码** | 设计师的图纸 |
| **Commit** | 每天的施工日志记录 |
| **分支** | 不同装修方案（现代风格分支、中式风格分支） |
| **Merge** | 将多个房间的装修方案整合 |
| **Rollback** | 发现问题，拆除重装 |
| **Push** | 向业主提交施工进度报告 |
| **Pull** | 获取业主的新要求 |

## 实战案例：Git-like 需求管理

### 场景描述

**业务方**：小陈，我们想把价格显示改成折扣价。

**小陈**：好的，我先创建一个 Issue 记录这个变更请求。

**小陈创建 Issue**：
- 标题：价格显示从原价改为折扣价
- 原因：业务方反馈用户更关注折扣力度
- 影响范围：商品详情页、购物车、订单确认页

**小陈发起变更评审**：
- 发给开发小王：评估技术可行性和排期
- 发给测试小李：评估测试范围
- 发给运营小陈：评估上线推广

**开发小王回复**：技术上可行，需要修改3个页面，排期2天。

**测试小李回复**：测试范围增加，需要增加价格对比测试用例。

**运营小陈回复**：上线推广方案不变，可以按计划推进。

**小陈汇总评审结果**：
- 评审通过
- 记录变更内容、原因、时间、影响范围
- 通知开发开始实施

## 产品经理判断方法

### 判断是否需要创建 Issue

| 情况 | 是否创建 Issue |
|------|---------------|
| 新增需求 | 是 |
| 需求逻辑变更 | 是 |
| 需求优先级调整 | 是 |
| 需求取消 | 是 |
| 文档格式调整 | 视重要性决定 |

### 判断变更影响范围

| 影响维度 | 检查内容 |
|---------|---------|
| **开发影响** | 是否需要修改代码？修改哪些模块？ |
| **测试影响** | 是否需要新增测试用例？回归范围？ |
| **运营影响** | 是否影响上线推广？用户通知？ |
| **数据影响** | 是否影响数据统计？报表调整？ |
| **兼容性** | 是否影响旧版本用户？历史数据？ |

## 误解纠正

### 误解：需求文档改了不用记录

**错误认知**：
- "改了PRD就行了，开发会看到"
- "小改动不需要记录"

**正确认知**：
- Git-like 变更管理是协作基础
- 每次改动需要 Issue 追踪
- 改动需要评审确认
- 改动记录是追溯依据
