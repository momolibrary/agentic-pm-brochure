# 评审准备实战案例

> 知识点：B5-05 评审准备
> 素材类型：故事场景
> 来源：知乎专栏、人人都是产品经理

---

## 场景背景

产品经理小张准备第一次主导需求评审会，他写好了PRD，准备了原型图，觉得准备充分。然而评审会上被开发、测试、运营连连追问，场面一度失控。

---

## 失败案例：准备不足的评审会

### 会前准备（小张的准备清单）

**小张准备的内容**：
- PRD文档（写了10页）
- 原型图（用墨刀做了交互原型）
- 功能列表（列了15个功能点）

**小张觉得准备充分，直接发会议邀请。**

### 评审会现场

**开发小王**：这个需求的业务场景是什么？为什么要做？

**小张**：业务方提的，他们说用户有这个需求。

**开发小王**：有数据支撑吗？有多少用户提过？

**小张**：（懵）这个...没有具体数据，业务方说用户反馈多。

**测试小李**：这个功能的边界条件是什么？比如用户输入超长文本会怎样？

**小张**：（更懵）这个PRD里没写...

**运营小陈**：上线后怎么推广？我们需要配合什么？

**小张**：（继续懵）推广方案还没想...

**开发小王**：这个功能涉及三个系统接口，我们需要和对方系统对接，排期可能要协调。

**小张**：什么系统？我不知道有这个依赖...

**主持人**：今天的评审先暂停，小张回去补充材料后再开第二轮。

---

## 失败原因分析

| 失败点 | 原因 |
|--------|------|
| 没有数据支撑 | 只凭业务方口头说法，没有量化数据 |
| 没有边界条件 | PRD只写了正常流程，没写异常情况 |
| 没有推广方案 | 只考虑功能实现，没考虑上线运营 |
| 没有依赖确认 | 不知道涉及哪些外部系统 |

---

## 正确做法：评审准备清单

### 会前准备清单

| 准备事项 | 具体内容 | 检查方法 |
|---------|---------|---------|
| **需求背景** | 需求来源、业务场景、解决什么问题 | 能用一句话清晰说明 |
| **数据支撑** | 用户数据、行为数据、竞品分析 | 有具体数字和图表 |
| **目标明确** | 预期效果、成功指标、上线时间 | 有可量化的指标 |
| **方案完整** | 业务流程、功能模块、交互设计 | PRD覆盖正常+异常流程 |
| **风险评估** | 技术风险、运营风险、兼容性问题 | 标注风险点和应对方案 |
| **依赖确认** | 外部系统接口、第三方服务、其他团队 | 确认依赖方排期 |
| **演示材料** | PRD文档、原型图、流程图、演示Demo | 会前一天发给参会人 |
| **预演方案** | 会议流程、时间分配、问题预案 | 在内部先预演一遍 |

---

## 不同角色关注点

| 参会角色 | 关注重点 | 产品经理准备 |
|---------|---------|-------------|
| **开发** | 技术可行性、工作量评估、技术难点 | 会前与技术骨干沟通方案 |
| **测试** | 测试点覆盖、边界条件、异常场景 | 准备测试用例要点 |
| **设计** | 交互体验、视觉一致性、设计资源 | 准备设计规范参考 |
| **运营** | 上线效果、推广配合、用户反馈 | 准备运营推广方案 |

## 常见追问及预答策略

### 1. 需求来源类

| 常见追问 | 预答策略 |
|---------|---------|
| "这个需求是哪个老板提的？" | 说明需求来源：用户反馈/数据分析/业务目标/竞品分析 |
| "为什么要做这个需求？" | 明确业务价值和用户价值，提供具体场景支撑 |

### 2. 数据支撑类

| 常见追问 | 预答策略 |
|---------|---------|
| "有数据支撑吗？" | 准备用户数据、行为数据、业务数据（有具体数字） |
| "竞品有这个功能吗？" | 提供竞品分析报告，对比优劣势 |

### 3. 可行性类

| 常见追问 | 预答策略 |
|---------|---------|
| "技术上能实现吗？" | 会前与技术骨干预沟通，评估技术可行性 |
| "开发周期多久？" | 根据功能复杂度，给出合理排期建议 |
| "有没有更简单的方案？" | 准备 Plan B，如 MVP 版本或分期迭代方案 |

### 4. 价值类

| 常见追问 | 预答策略 |
|---------|---------|
| "这个需求解决了什么问题？" | 从用户痛点出发，说明核心价值 |
| "上线后预期效果如何？" | 提供预期指标，如转化率、活跃度提升等 |

## 成功案例：充分准备的评审会

### 会前准备（产品经理小李）

**小李的准备清单**：
- ✅ 需求背景文档（用户调研报告、竞品分析）
- ✅ 数据支撑（用户反馈数量、行为数据分析图表）
- ✅ PRD文档（正常流程 + 异常流程 + 边界条件）
- ✅ 原型图（交互原型 + UI设计稿）
- ✅ 技术风险标注（标注3个风险点及应对方案）
- ✅ 依赖确认（已和对方系统团队确认排期）
- ✅ 预演（内部先和业务方预演一遍）

### 评审会现场

**开发小王**：这个需求的业务场景是什么？

**小李**：我们调研了100个用户，68%反馈找不到历史订单，导致复购率下降15%。这个功能可以提升复购率。

**开发小王**：数据很清晰。技术可行性如何？

**小李**：我已经和技术骨干老张预沟通了，他评估可行，但有两个风险点：
1. 订单数据量大，需要分页优化
2. 历史订单接口需要和订单系统对接，我已经和订单团队确认排期

**测试小李**：边界条件是什么？

**小李**：PRD第5节写了边界条件：订单数量超过100条时分页显示、订单详情加载失败时显示错误提示、无历史订单时显示空状态引导。

**运营小陈**：上线后怎么推广？

**小李**：运营方案在第8节：上线前推送通知用户、上线后首页引导文案、配合复购优惠券活动。

**主持人**：小李准备很充分，评审通过。

## 产品经理判断方法

### 判断准备是否充分

| 自检问题 | 通过标准 |
|---------|---------|
| 能用一句话说清楚需求背景吗？ | 是 |
| 有具体数字的数据支撑吗？ | 有 |
| PRD覆盖异常流程了吗？ | 覆盖 |
| 技术风险标注了吗？ | 标注 |
| 依赖方确认排期了吗？ | 确认 |
| 会前预演了吗？ | 预演 |

### 判断是否需要第二轮评审

| 情况 | 是否需要第二轮 |
|------|---------------|
| 需求有重大变更 | 需要 |
| 技术方案有调整 | 需要 |
| 参会人提出新问题 | 视问题重要性决定 |
| 评审通过 | 不需要 |
