## 核心故事框架

### 场景设定

业务部门提了一个需求：「我们要做一个会员积分兑换功能，用户积分达到一定数量可以兑换优惠券。」

**小想**（实习生）记录了需求，按照传统流程推进：

1. **第 1 周**：业务部门提需求 → 产品部门整理需求文档
2. **第 2-3 周**：需求评审会 → IT 部门排期 → 排在队列第 5 位
3. **第 4-8 周**：IT 部门开发 → 开发过程中多次澄清需求细节，往返沟通
4. **第 9-10 周**：测试部门测试 → 发现问题 → 回到开发修复
5. **第 11-12 周**：运维部门部署上线 → 等待发布窗口 → 生产环境验证
6. **第 13 周**：上线 → 运营部门发现效果不好，提改进需求

**结果**：从需求提出到上线，花了 **三个月**。

### 小想的困惑

上线后复盘会，小想不解地问：

> 「为什么这么简单一个功能，要三个月才上线？」

**老李**（产品总监）画了一张流程图：

```
业务部门 ──────(甩需求)──────→ 产品部门 ──────(整理)──────→
                                                       ↓
运营部门 ←──────(上线反馈)────── 运维部门 ←──────(部署)─────── 测试部门
                                                       ↑
                                                       │
                    IT 部门 ←──────(排期开发)──────────────┘
```

老李说：

> 「你看这张图，需求从业务部门提出来，要经过五个部门的接力。每个部门都是一个『竖井』——各自为政、各自有排期、各自有流程。
>
> 需求评审会排期 2 周，IT 开发排期 4 周，测试排期 2 周，运维部署等窗口 1 周……每一步都要『排队』。
>
> 这还不是最慢的。最慢的是：开发过程中发现需求细节不清楚，要回去问业务部门；业务部门回复了，产品部门再整理，再传给 IT……一个往返就要一周。三个月里，有一半时间是在『等待』和『往返沟通』上。」

### 老李的洞察

老李继续说：

> 「这种『接力棒』模式的问题在于：
>
> 1. **信息衰减**：需求从业务 → 产品 → IT → 测试 → 运维，每传一次就丢失一些上下文。到运维部门时，他们只知道『部署这个版本』，完全不知道这个功能是什么、为什么要做、业务场景是什么。
>
> 2. **往返延迟**：任何一个环节有问题，就要回到上游环节澄清。开发发现需求不清楚，等业务回复要 2 天；测试发现 bug，等开发修复要 3 天。这些『等待』都在浪费时间。
>
> 3. **上线时差**：需求三个月前提出，三个月后上线。三个月后业务环境可能变了，运营发现效果不好，再提改进需求，又三个月……
>
> 这种模式，业务永远跟不上变化。」

## 技术细节提炼（供作者参考）

### 传统竖井模式的代价

| 阶段 | 传统模式耗时 | 主要延迟原因 |
|------|-------------|-------------|
| 需求整理 | 1-2 周 | 产品部门等待业务部门澄清细节 |
| 需求评审 | 1-2 周 | 各部门排期冲突，凑人难 |
| 开发排期 | 2-4 周 | IT 部门队列排队 |
| 开发执行 | 4-6 周 | 开发中多次往返澄清需求 |
| 测试排期 | 1-2 周 | 测试队列排队 |
| 测试执行 | 2-3 周 | 发现问题要回开发修复 |
| 部署等待 | 1-2 周 |运维发布窗口排队 |
| **总计** | **10-15 周** | 大半时间在「等待」和「往返」 |

### 跨部门沟通的隐形成本

| 问题类型 | 传统模式处理方式 | 耗时 |
|----------|-----------------|------|
| 需求细节不清晰 | 业务 → 产品 → IT → 产品 → 业务（往返） | 3-5 天/次 |
| 技术方案有疑问 | IT → 产品 → 业务 → 产品 → IT（往返） | 3-5 天/次 |
| 测试发现问题 | 测试 → IT → 产品 → 业务（往返） | 5-7 天/次 |
| 上线后运营反馈 | 运营 → 产品 → 业务 → 产品 → IT（新需求循环） | 10+ 周 |

### 「45% 未使用功能」现象

行业研究表明：**传统模式下上线功能，45% 从未或很少被使用**。

原因：

- 需求提出时是「三个月前的业务想法」
- 上线时「业务环境已变化」
- 运营发现「用户根本不需要这个功能」

## 对比：耦合协作模式

### 理想状态

| 传统竖井模式 | 耦合协作模式 |
|-------------|-------------|
| 业务 → 产品 → IT → 测试 → 运维（接力） | 业务+产品+IT+测试+运营（一个小组） |
| 各部门有各自排期 | 共享一个迭代周期 |
| 需求细节往返澄清 5 次 | 需求细节当场澄清，零往返 |
| 总耗时 10-15 周 | 总耗时 2-3 周 |

### Spotify 模型参考

Spotify 的「Squad」模式：业务、产品、设计、开发、测试组成一个小组，共享一个目标，持续协作，不接力。
