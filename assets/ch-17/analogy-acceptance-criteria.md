# 类比：验收标准 → 签收快递的检查清单

## 类比场景

**产品经理定义验收标准，就像签收快递时的检查清单。**

---

## 详细类比

### 场景：签收快递

你网购了一个手机，快递送到家门口。签收前你会做什么？

| 检查项 | 为什么检查 |
|--------|-----------|
| ✓ 外包装完好 | 确认运输中没摔坏 |
| ✓ 数量对 | 一个手机，一套配件 |
| ✓ 型号对 | 是我下单的那个型号 |
| ✓ 配件齐全 | 充电器、数据线、说明书都有 |
| ✓ 开机正常 | 手机能正常启动 |

**全部检查完，确认无误，才签收说"做对了"。**

如果快递员说"我送到了"（做完了），但你发现：
- 外包装凹陷有水渍
- 配件少了一条数据线

你会签收吗？**不会。** 因为"送到了"不等于"送对了"。

---

## 映射关系

| 快递签收 | 产品验收 |
|---------|---------|
| 快递员说"送到了" | 开发说"做完了" |
| 检查清单（外包装/数量/型号） | 验收标准（具体条件） |
| 全部通过才签收 | 全部通过才验收 |
| 发现问题不签收 | 发现问题不上线 |

---

## 产品经理视角

老李对小想说：

> "你写需求的时候，开发说'做完了'，你知道怎么验证吗？就像签收快递，你得有一张检查清单。什么算'做对了'？什么算'有问题'？这就是验收标准。"

**验收标准就是产品经理的签收检查清单。**

---

## 类比要点

1. **"做完了"≠"做对了"** — 快递送到是"做完了"，检查通过才是"做对了"
2. **验收标准是检查清单** — 列出具体条件，一项一项验证
3. **缺验收标准 = 盲签快递** — 开发说做完就接收，上线后出Bug才发现问题
4. **产品经理负责定义标准** — 快递主人决定检查什么，产品经理决定验收什么
