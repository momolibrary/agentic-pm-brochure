## 一、验收标准（Acceptance Criteria）

### 1. Atlassian 官方指南

**链接**：[Atlassian - Acceptance Criteria](https://www.atlassian.com/agile/project-management/acceptance-criteria)

**核心内容**：
- 验收标准定义用户故事的边界，确认何时"完成"
- 常用格式：Given/When/Then（Gherkin语法）
- 最佳实践：可测试、清晰、用户视角

---

### 2. Mountain Goat Software（Mike Cohn）

**链接**：[Mountain Goat Software - Acceptance Criteria](https://www.mountaingoatsoftware.com/blog/acceptance-criteria)

**核心内容**：
- 验收标准应该可测试、可验证
- 从用户视角编写，不涉及实现细节
- 每个故事通常3-7条验收标准

---

### 3. 行业实践总结

| 要素 | 说明 |
|------|------|
| **明确性** | 条件描述清晰，无歧义 |
| **可测量** | 可通过测试验证 |
| **可追溯** | 与需求文档对应 |
| **完整性** | 覆盖关键场景 |

## 二、测试场景覆盖（Test Scenarios）

### 1. 测试场景分类

| 类型 | 说明 | 占Bug比例 |
|------|------|----------|
| **正常流程** | 用户正常操作路径 | ~10% |
| **异常流程** | 错误处理、网络异常 | ~60% |
| **边界条件** | 临界值、极端数据 | ~20% |
| **权限控制** | 无权限、跨权限操作 | ~10% |

**关键数据**：90%的线上Bug出在异常流程和边界条件。

---

### 2. 异常场景常见遗漏

| 类别 | 典型遗漏 |
|------|---------|
| **网络异常** | 断网、弱网、超时 |
| **数据异常** | 空数据、重复数据、格式错误 |
| **操作异常** | 重复点击、中断操作、非法输入 |
| **并发异常** | 多人同时操作、数据竞争 |

---

### 3. 边界值测试原则

```
示例：输入框限制1-100
- 下边界：0, 1, 2
- 上边界：99, 100, 101
- 特殊值：空值、负数、非数字
```

---

## 三、边界测试与极端条件

### 1. 边界测试定义

- **Boundary Testing**：在输入边界值处测试，最容易发现错误
- **典型场景**：最小值、最大值、刚好超限值

---

### 2. 极端条件测试

| 类型 | 示例 |
|------|------|
| **数据量极端** | 10万+条数据查询 |
| **并发极端** | 5000+用户同时操作 |
| **输入极端** | 超长文本、特殊字符、空输入 |
| **时间极端** | 超长运行、内存累积 |

---

### 3. 系统崩溃预防策略

| 策略 | 说明 |
|------|------|
| **Graceful Degradation** | 超载时保持部分功能 |
| **Circuit Breaker** | 达到阈值自动停止 |
| **Rate Limiting** | 控制请求量 |
| **Auto-scaling** | 自动扩容 |

## 四、权威书籍参考

### 1. 《Google 软件测试之道》

**作者**：James Whittaker 等
**章节**：测试文化与质量保障

**核心观点**：
- 测试不是测试人员的事，是全员的事
- 产品定义验收标准是质量保障的第一步
- "质量不是测出来的，是定义出来的"

---

### 2. 《敏捷软件开发》

**章节**：用户故事与验收标准

**核心内容**：
- INVEST原则中的"Testable"
- 验收标准与用户故事的关系
- 如何编写可验证的需求

## 五、产品经理视角要点

### 误解纠正

| 误解 | 正确理解 |
|------|---------|
| "测试是测试人员的事" | 产品定义验收标准是质量保障第一步 |
| "正常流程走通了就行" | 90% Bug出在异常流程和边界条件 |
| "数据量不需要定义" | 必须定义预期上限，否则上线崩 |

---

### 行动建议

1. **为每个需求定义验收标准**：列出5条以上验收条件
2. **系统性设计测试场景**：覆盖正常/异常/边界/权限
3. **追问边界条件**：数据量多大？并发多高？超出怎么办？
