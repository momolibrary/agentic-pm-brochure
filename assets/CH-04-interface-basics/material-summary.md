## 一、类比素材

### 1. 餐厅类比（核心类比）

**来源可信度**：高（广泛使用的经典类比）

| 餐厅组件 | 接口对应 | 解释 |
|---------|----------|------|
| 顾客 | 用户/调用方 | 提出需求的人或系统 |
| 菜单 | API文档 | 展示可用的服务和数据 |
| 服务员 | API接口本身 | 接收请求、传递响应的中介 |
| 厨房 | 服务器/数据库 | 实际处理逻辑的地方 |
| 点菜单上的选项 | 接口参数 | 菜名、份量、辣度、备注等输入项 |
| 端上来的菜 | 接口返回数据 | 按需求准备的输出结果 |

**类比故事模板**：

> 老李说："接口就像餐厅的服务窗口。你（顾客）看着菜单（API文档）点什么菜（发起请求），服务员（接口）把你的点单记下来（接收参数），传给厨房（服务器/数据库）。厨房做好饭（处理逻辑），服务员把饭端给你（返回响应）。你不需要知道厨房怎么做饭，只需要知道怎么点菜、能得到什么。"

**参数类比细化**：

> 点菜单上的每一项都是参数：
> - **菜名** = 接口路径/方法名（要什么功能）
> - **份量** = 数量参数（要多少条数据）
> - **辣度** = 选项参数（按什么条件筛选）
> - **备注** = 其他参数（特殊需求，如排序规则）
>
> 如果你没说份量，服务员可能默认给你一份；如果你没说辣度，厨师可能按标准做。接口也一样——参数没写清楚，系统就按默认值处理，可能不是你想要的。

### 2. 银行柜台类比（补充类比）

| 银行组件 | 接口对应 |
|---------|----------|
| 客户 | 用户/调用方 |
| 业务窗口 | API端点 |
| 存折/卡号 | 身份认证参数 |
| 填写的单据 | 请求参数（填写什么业务、金额等） |
| 柜员处理 | 服务端逻辑 |
| 回执/现金 | 响应数据 |

**类比要点**：银行窗口不接受"随便给我点钱"的请求，必须按格式填写单据、提供凭证。接口也一样，必须按约定格式传参数。

### 3. 快递服务类比（场景类比）

| 快递组件 | 接口对应 |
|---------|----------|
| 寄件人 | 调用方 |
| 快递公司系统 | API平台 |
| 寄件单 | 请求参数（收件人地址、物品类型、重量） |
| 物流追踪 | 接口返回状态 |
| 签收凭证 | 最终响应确认 |

**类比要点**：快递单上的每一项都是必填或可选参数。漏填地址快递就送不到，漏填物品类型可能无法处理。接口参数遗漏会导致功能异常。

## 二、分页参数遗漏案例

### 案例 1：列表数据不全（知识点地图故事锚点）

**场景描述**：

> 小想设计了一个用户管理列表页需求，开发问"接口传什么参数？"
> 小想说："就传用户ID过滤一下。"
> 开发追问："分页怎么处理？每页多少条？排序规则？"
> 小想愣住了——需求里只写了"展示用户列表"，没考虑这些参数。
>
> 上线后，列表只显示前 20 条用户，后面的数据用户看不到。小想才发现漏了分页参数。

**教训要点**：
- 列表接口必须考虑分页（page/pageSize 或 offset/limit）
- 没有分页参数，要么一次返回全部（性能爆炸），要么只返回默认数量（数据不全）
- 产品需求必须明确：是否需要分页、每页多少条、默认排序规则

### 案例 2：大数据量崩溃

**场景描述**：

> 某产品设计的报表导出功能，没有分页限制。用户点击"导出全部数据"，系统试图一次性返回 10 万条记录，请求超时、页面崩溃。
> 开发反馈："没有分页参数，接口无法控制返回量。"

**教训要点**：
- 批量数据必须设计分页或分段返回
- 产品需明确：最大导出数量限制、分段机制

### 案例 3：移动端加载慢

**场景描述**：

> 移动端商品列表，没有分页参数，每次请求返回 1000+ 商品，加载耗时 5 秒以上。
> 用户投诉体验差，排查发现接口设计时没有考虑移动端分页需求。

**教训要点**：
- 不同终端需要不同的分页策略（移动端页条数更少）
- 产品需求应区分场景设计

## 三、产品需求常见遗漏场景

### 遗漏类型汇总

| 遗漏类型 | 具体表现 | 开发追问 |
|---------|----------|----------|
| **分页遗漏** | 列表页没有说明分页规则 | "分页吗？每页多少条？页码还是游标？" |
| **排序遗漏** | 没有说明默认排序方式 | "按什么排序？升序还是降序？" |
| **筛选条件遗漏** | 只写"展示XX列表"，没写筛选维度 | "能筛选吗？按什么条件筛选？" |
| **字段遗漏** | 没列清楚返回哪些字段 | "返回哪些字段？必返还是可选？" |
| **错误处理遗漏** | 没说明异常情况 | "没数据时返回什么？报错怎么处理？" |
| **权限遗漏** | 没说明谁能调这个接口 | "所有人都能调吗？需要什么权限？" |
| **边界条件遗漏** | 没考虑极端情况 | "并发怎么处理？数据量上限？" |

### 具体场景模板

**场景 1：筛选条件遗漏**

> 开发问："这个订单列表能筛选吗？"
> 产品说："能，按状态筛选。"
> 开发追问："还有什么筛选维度？日期范围？客户类型？金额区间？"
> 产品才发现需求只写了"按状态筛选"，漏了其他业务维度。

**场景 2：字段定义遗漏**

> 开发问："用户详情接口返回哪些字段？"
> 产品说："用户基本信息。"
> 开发追问："具体有哪些字段？头像？手机号脱敏还是完整显示？注册时间格式？"
> 产品才发现需求里的"用户信息"太笼统，开发无法确定字段列表。

**场景 3：错误处理遗漏**

> 上线后，接口返回空数据时前端报错。开发说："你没说空数据怎么处理，我就按正常逻辑走了。"
> 产品才发现需求只写了"有数据时怎么展示"，漏了"无数据时怎么处理"。

## 四、开发视角的接口设计关注点

### 开发关心的问题清单

| 关注点 | 开发视角 | 产品需要提供的信息 |
|-------|---------|-------------------|
| **输入参数** | "传什么参数？每个参数什么类型？必填还是可选？" | 字段名称、类型、必填性、示例值 |
| **输出结构** | "返回哪些字段？格式是什么？" | 字段列表、字段类型、示例数据 |
| **分页设计** | "用什么分页方式？page/pageSize 还是 offset/limit？" | 分页方式、每页条数、默认值 |
| **排序规则** | "默认排序是什么？用户能自定义排序吗？" | 默认排序字段、排序方向、可排序维度 |
| **筛选条件** | "支持哪些筛选维度？筛选参数怎么传？" | 筛选字段、筛选方式（精确/模糊/范围） |
| **错误处理** | "异常情况返回什么？错误码怎么定义？" | 错误场景列表、错误码、错误提示文案 |
| **权限控制** | "谁有权限调用？需要什么认证？" | 角色权限、认证方式 |
| **性能边界** | "预估调用量？响应时间要求？" | QPS预估、响应时间期望 |

### RESTful API 设计原则摘要

**来源**：阮一峰《RESTful API 设计指南》、Google API Design Guide

| 方面 | 最佳实践 |
|------|----------|
| **URL设计** | 使用名词而非动词，复数形式表示资源集合（如 `/users` 而非 `/getUser`） |
| **HTTP方法** | GET查询、POST创建、PUT更新、DELETE删除 |
| **参数命名** | 使用驼峰命名或下划线，保持一致性 |
| **版本控制** | URL路径版本号（如 `/v1/users`）或请求头版本号 |
| **分页设计** | page/pageSize 或 offset/limit 组合，返回总数(total)和分页信息 |
| **排序设计** | sort 参数，支持字段名和方向（如 `sort=name:asc`） |
| **过滤设计** | 查询参数传递（如 `status=active&type=premium`） |
| **错误处理** | 统一错误响应格式，包含 code、message、details |
| **响应格式** | JSON为主，包含状态码、数据、分页信息等 |

### N+1 问题（性能陷阱）

**场景描述**：

> 小想设计了一个订单列表页，每一行要显示订单详情。前端调了列表接口拿到订单 ID，然后逐条调详情接口。
> 结果：列表 100 条订单，前端调了 1 次列表接口 + 100 次详情接口 = 101 次请求。
> 页面加载耗时 10 秒。老李说："这是经典的 N+1 问题。"

**教训要点**：
- 列表接口应该一次返回完整数据，避免逐条调详情
- 产品需要理解：批量获取比逐条获取效率高得多
- 设计需求时应考虑：列表需要哪些详情字段，尽量在列表接口返回
