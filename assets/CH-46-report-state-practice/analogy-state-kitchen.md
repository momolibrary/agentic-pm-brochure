# 类比：状态机设计 → 餐厅订单流程

> 知识点：C1-04 状态流转实战
> 适用章节：Section 2（状态机设计）

---

## 类比描述

设计一个订单状态机，就像设计餐厅的出餐流程。

每张订单从「顾客下单」到「结账离开」，经历多个环节。每个环节都有特定的状态，状态之间有固定的流转规则。厨师、服务员、收银员各司其职，协同完成。

---

## 对应关系

| 状态机要素 | 餐厅出餐流程 | 说明 |
|-----------|-------------|------|
| **状态定义** | 出餐环节（下单、备菜、烹饪、上菜、用餐、结账） | 订单可能处于的阶段 |
| **流转路径** | 环节衔接顺序（下单→备菜→烹饪→上菜） | 合法的状态变更路径 |
| **流转条件** | 环节触发条件（备菜完成才能烹饪） | 状态变更的前置条件 |
| **并发冲突** | 多桌同时点单（厨房调度） | 多订单同时处理 |
| **异常状态** | 退菜、换菜、投诉、厨房故障 | 非正常流程的处理 |
| **超时处理** | 菜品超时未上（催单、取消） | 状态停留过久的处理 |
| **状态流转图** | 厨房流程图 | 全流程可视化 |

## 核心洞察

### 1. 状态 ≠ 流程步骤的简单复制

**餐厅类比**：餐厅状态是「订单当前在哪」，不是「已完成哪些步骤」。

- 「烹饪中」是一个状态（订单正在厨房加工）
- 「烹饪完成」是状态变更事件（触发从「烹饪中」→「待上菜」的流转）

**状态机设计**：
- 状态是「订单当前所处阶段」
- 事件是「触发状态变更的条件」
- 不要把「事件」和「状态」混淆

> 常见错误：把「已支付」「已发货」当作事件，而非状态，导致状态机设计混乱。

### 2. 流转路径 ≠ 所有状态都能互相跳转

**餐厅类比**：
- 「下单」不能直接跳到「结账」（必须先上菜、用餐）
- 「烹饪中」不能跳回「备菜」（不能退回去重新切菜）
- 「上菜」可以跳到「退菜」（顾客不满意可退）

**状态机设计**：
- 有些流转是单向的（已发货→已签收，不可逆转）
- 有些流转需要前置条件（已支付才能发货）
- 有些流转是异常路径（正常流程之外的退单、取消）

> 常见错误：设计成「任何状态都能跳到任何状态」，导致状态机失控。

### 3. 并发冲突 ≠ 忽略就没事

**餐厅类比**：
- 同一时间多桌下单，厨房需要排队处理
- 一个厨师不能同时炒两桌的菜
- 需要调度：谁先做、谁后做、谁并行处理

**状态机设计**：
- 多订单同时请求状态变更，可能冲突
- 需要锁机制、版本控制、幂等设计
- 防止「两个线程同时发货」导致重复发货

> 常见错误：忽略并发场景，导致同一订单被两个操作同时修改，出现异常状态。

### 4. 异常状态 ≠ 正常流程的「例外说明」

**餐厅类比**：
- 顾客投诉菜品，需要「退菜」流程
- 厨房设备故障，需要「暂停接单」状态
- 菜品超时未上，需要「催单」→「取消」路径

**状态机设计**：
- 不能只设计「理想流程」，还要设计「异常处理」
- 退单、取消、超时、失败都是正式状态
- 异常状态也要有明确的流转规则

> 常见错误：只画「正常流程图」，忽略「超时取消」「异常回退」，上线后遇到异常无法处理。

## 人物对话示例

> **老李**：小想，你的订单状态机画得不错。但有个问题：如果顾客下单后30分钟没支付，订单怎么办？
>
> **小想**：呃...我没想过这个情况。
>
> **老李**：这就是「超时状态」。订单停留在「待支付」超过30分钟，应该自动流转到「已取消」。
>
> **小想**：那我再加一个「已取消」状态？
>
> **老李**：不止是加状态。你要设计：①超时时间定义 ②触发条件（30分钟无支付）③流转路径（待支付→已取消）④取消后的后续处理（库存恢复）。
>
> **小想**：原来状态机不是画几个圈和箭头就完了，还要考虑异常场景！
