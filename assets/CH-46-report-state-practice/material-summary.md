## 验收标准支撑

### 1. 设计一个数据报表功能，需要考虑哪些要素？

**类比支撑：** 餐厅点菜系统类比
- 展示字段 → 菜品展示信息（名称、价格、口味）
- 筛选条件 → 菜品分类筛选（菜系、价格区间）
- 排序规则 → 菜品排序（推荐、价格、人气）
- 分页策略 → 菜品翻页/分类展示
- 数据权限 → VIP专属菜/普通顾客可选菜
- 导出功能 → 打包带走/外卖清单
- 性能优化 → 厨房备菜策略（预切配菜）

**故事支撑：**
- 权限遗漏故事：普通客服能看到VIP客户完整信息，引发投诉
- 导出性能故事：50万条数据导出失败，服务器崩溃

### 2. 设计一个订单状态机，完整步骤是什么？

**类比支撑：** 餐厅出餐流程类比
- 状态定义 → 出餐环节（下单、备菜、烹饪、上菜）
- 流转路径 → 环节衔接顺序（下单→备菜→烹饪→上菜）
- 流转条件 → 环节触发条件（备菜完成才能烹饪）
- 并发冲突 → 多桌同时点单（厨房调度）
- 异常状态 → 退菜、换菜、投诉、厨房故障
- 超时处理 → 菜品超时未上（催单、取消）

**故事支撑：**
- 并发超时故事：同一订单两次发货 + 超时订单未取消占用库存

### 3. 报表设计中最容易被忽略的是什么？

**故事支撑：**
- 数据权限（经理看全部 vs 员工只看自己的）— 权限遗漏故事
- 大数据量下的查询性能 — 导出性能故事
- 导出时的数据量限制 — 导出性能故事

## 核心结论提炼

### C1-03 数据报表实战

> 报表设计不只是「展示数据」，要考虑六个维度：展示字段（用户视角）、筛选条件（业务语言）、排序规则、分页策略、数据权限（行级+列级）、导出限制（数量+异步）。
>
> 最容易被忽略的是：数据权限（不同角色看不同范围）、大数据量性能、导出数量限制。

### C1-04 状态流转实战

> 状态机设计六步骤：①列出所有状态 ②定义合法流转路径 ③标注流转条件 ④处理并发冲突 ⑤设计异常状态 ⑥输出状态流转图。
>
> 最容易被忽略的是：并发冲突（多操作同时触发）、超时处理（状态停留过久）、异常路径（非正常流程的处理）。
