# 类比：数据报表设计 → 餐厅点菜系统

> 知识点：C1-03 数据报表实战
> 适用章节：Section 1（报表设计要素）

---

## 类比描述

设计一个数据报表，就像设计餐厅的点菜系统。

顾客（用户）需要从众多菜品（数据）中找到自己想要的，餐厅需要提供一个清晰的菜单（报表界面），让顾客能够快速定位、筛选、选择。

---

## 对应关系

| 报表设计要素 | 餐厅点菜系统 | 说明 |
|------------|-------------|------|
| **展示字段** | 菜品展示信息（名称、价格、图片、口味） | 选择哪些信息要展示给用户 |
| **筛选条件** | 菜品分类筛选（菜系、价格区间、口味） | 用户如何缩小选择范围 |
| **排序规则** | 菜品排序（推荐、价格、人气） | 数据的呈现顺序 |
| **分页策略** | 菜品翻页/分类展示 | 数据量大时的展示方式 |
| **数据权限** | VIP专属菜/普通顾客可选菜 | 不同角色看到不同范围的数据 |
| **导出功能** | 打包带走/外卖清单 | 数据的输出形式 |
| **性能优化** | 厨房备菜策略（预切配菜、快炒区） | 大数据量下的响应速度 |

---

## 核心洞察

### 1. 展示字段 ≠ 把所有信息都放上去

**餐厅类比**：菜单上不会列出每道菜的所有原料、烹饪时间、厨师信息。只展示顾客关心的核心信息：名称、价格、口味、推荐指数。

**报表设计**：不要把数据库所有字段都展示出来。要站在用户视角，选择「用户关心什么」，而非「数据库有什么」。

> 常见错误：产品经理把数据库表的所有字段都列到报表里，导致表格臃肿，用户找不到重点。

### 2. 筛选条件 ≠ 数据库字段直接映射

**餐厅类比**：顾客不会用「食材产地」或「厨师编号」来筛选菜品。他们用「菜系」「价格区间」「辣度」这些业务维度。

**报表设计**：筛选条件要转化为「用户能理解的业务语言」，而非「数据库字段名」。

> 常见错误：筛选条件直接使用数据库字段名（如「status_code = 1」），用户根本不理解。

### 3. 数据权限 ≠ 简单的「可见/不可见」

**餐厅类比**：
- VIP会员可以看到「专属菜品」区
- 普通顾客只能看到大众菜品
- 儿童菜单单独展示给带孩子的家庭

**报表设计**：
- 经理能看到所有部门的数据
- 部门主管只能看到本部门数据
- 普通员工只能看到自己负责的数据

> 常见错误：只设计「能看/不能看」二元权限，忽略了「行级权限」（不同角色看不同数据范围）。

### 4. 性能优化 ≠ 等用户抱怨了再处理

**餐厅类比**：热门菜品预先备好半成品（预切配菜），高峰期直接快炒。不会等顾客点了才开始洗菜切菜。

**报表设计**：高频查询要预计算、缓存、建索引。不会等用户点击后才临时聚合百万条数据。

> 常见错误：报表上线后才发现大数据量查询超时，用户体验极差。

## 人物对话示例

> **老李**：小想，你设计的这个报表，筛选条件是「status_code=1」，用户知道这是「已支付」吗？
>
> **小想**：呃...数据库里就是这么存的啊。
>
> **老李**：数据库是技术语言，报表是业务语言。你得把「status_code=1」转化成「订单状态：已支付」，用户才理解。
>
> **小想**：明白了！那我再加个「支付时间」字段...
>
> **老李**：先停一下。你觉得用户关心「支付时间」吗？还是关心「下单时间」？别把数据库所有字段都搬上去。
>
> **小想**：我理解了，要站在用户视角选字段，不是从数据库搬字段。
