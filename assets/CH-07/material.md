## 一、核心概念素材

### 1.1 业务流程 vs 系统流程

| 维度 | 业务流程 | 系统流程 |
|------|----------|----------|
| 关注点 | 用户需求、商业价值 | 技术实现、系统逻辑 |
| 使用者 | 业务人员、运营人员 | 开发人员、测试人员 |
| 表达方式 | 流程图、泳道图 | 时序图、状态图 |
| 文档产出 | 业务需求文档(BRD) | 功能规格说明书(FSD) |
| 核心问题 | 用户如何完成目标 | 系统如何处理请求 |

**关键理解**：
- 业务流程是用户视角的操作步骤（如"提交审批→审批→通知")
- 系统流程是系统视角的处理步骤（如"接收请求→数据校验→写入数据库→触发通知")
- 区分两者是因为业务一步操作，系统可能要做多步处理

## 二、类比素材

### 2.1 餐厅类比（顾客点菜 vs 厨房备菜）

> 类比来源：知识点地图 A3-01/A3-02 类比需求

| 餐厅场景 | 对应概念 |
|----------|----------|
| **顾客点菜** | 业务流程（用户视角） |
| - 顾客看菜单 | 用户了解功能 |
| - 顾客下单 | 用户提交请求 |
| - 等待菜品 | 用户等待处理结果 |
| - 菜品上桌 | 用户收到结果 |
| **厨房备菜** | 系统流程（后台视角） |
| - 接收订单 | 系统接收请求 |
| - 检查食材库存 | 数据校验 |
| - 分配厨师 | 任务分发 |
| - 烹饪制作 | 数据处理 |
| - 质量检查 | 结果验证 |
| - 出菜通知 | 通知发送 |

**类比要点**：
- 顾客的"下单"只是一个动作，厨房要做很多步
- 顾客只关心菜品是否正确送达，不关心厨房内部步骤
- 但如果厨房出问题（食材不足、厨师忙），顾客体验会受影响

### 2.2 其他类比建议

| 类比场景 | 业务流程 | 系统流程 |
|----------|----------|----------|
| **快递寄件** | 用户填单→寄件→收件 | 订单录入→分拣→运输→派送→签收 |
| **银行转账** | 用户输入金额→确认→收款 | 验证账户→扣款→记账→通知收款方 |
| **医院挂号** | 患者挂号→就诊→缴费 | 记录信息→分配医生→排号→结算 |

## 三、常见遗漏素材

### 3.1 Happy Path vs Unhappy Path

| 概念 | 说明 |
|------|------|
| **Happy Path** | 理想状态下用户顺利完成的流程路径（正常流程） |
| **Unhappy Path** | 异常情况下的流程路径（逆向流程、异常流程） |

### 3.2 产品经理常见遗漏清单

| 遗漏类型 | 具体问题 | 影响 |
|----------|----------|------|
| **逆向流程遗漏** | 驳回后怎么处理？回到哪一步？ | 用户操作无法完成，流程卡死 |
| **取消/撤回遗漏** | 能否取消？取消后状态如何？ | 用户误操作无法纠正 |
| **超时处理遗漏** | 审批超时怎么办？自动通过还是退回？ | 流程无限等待 |
| **并发冲突遗漏** | 两个人同时操作怎么办？ | 数据不一致 |
| **边界条件遗漏** | 空数据、特殊字符、大量数据时怎么办？ | 系统崩溃或异常 |

### 3.3 审批流程逆向路径常见遗漏点

| 遗漏问题 | 设计要点 |
|----------|----------|
| 驳回后回到哪一步 | 需明确：回到上一节点、回到发起节点、还是指定节点 |
| 驳回后能否再次提交 | 需定义：是否允许重新提交、是否需要修改内容 |
| 驳回后状态变化 | 需明确状态流转：驳回状态→待修改状态→重新提交 |
| 驳回后通知机制 | 需考虑：是否通知发起人、通知内容是什么 |
| 多级审批的逆向处理 | 需明确：第N级驳回时，是否需要重新走第1-N级 |

## 四、系统内部处理步骤案例

### 4.1 用户"提交审批"操作触发的系统步骤

> 故事锚点：小想以为提交就完了，原来后台还有5步

| 用户视角 | 系统视角（后台处理） |
|----------|---------------------|
| 点击"提交"按钮 | 1. 接收HTTP请求 |
| | 2. 验证用户身份（鉴权） |
| | 3. 验证数据格式和完整性 |
| | 4. 检查业务规则（如审批条件） |
| | 5. 写入数据库（审批记录） |
| | 6. 更新状态（待审批） |
| | 7. 分配审批人 |
| | 8. 发送通知（消息/邮件） |
| | 9. 记录操作日志 |
| 看到"提交成功"提示 | 10. 返回响应结果 |

**关键理解**：
- 用户的一个操作可能触发系统内部多个处理步骤
- 每一步都可能出错，都需要考虑异常处理
- 产品经理不能只设计"提交成功"的提示，还要考虑每一步失败时用户看到什么

### 4.2 其他系统多步处理案例

| 用户操作 | 系统内部步骤 |
|----------|--------------|
| **登录** | 验证账号→验证密码→生成Token→记录登录日志→加载用户配置→返回首页 |
| **下单** | 验证商品→验证库存→计算价格→创建订单→扣减库存→生成支付链接→发送订单通知 |
| **支付** | 验证订单→验证金额→调用支付网关→等待支付结果→更新订单状态→发送支付通知 |

## 五、BPMN 建模入门素材

### 5.1 BPMN 核心符号

| 符号类型 | 用途 | 示例 |
|---------|------|------|
| **事件 (Event)** | 流程的开始或结束 | 开始事件 ○、结束事件 ◉ |
| **活动 (Activity)** | 具体任务或操作 | 用户审批、数据处理 |
| **网关 (Gateway)** | 决策分支点 | 排他网关 ◇、并行网关 ◇+ |
| **顺序流** | 流程走向 | → 箭头连接 |
| **消息流** | 跨池通信 | ⤳ 虚线箭头 |

### 5.2 BPMN 学习路径

```
入门 (1-2周)
├── 了解基础符号和规则
├── 学习泳道图（区分角色）
└── 工具选择（Draw.io、Lucidchart）

进阶 (2-4周)
├── 掌握网关类型和复杂分支
├── 学习子流程和异常处理
└── 实际项目练习

精通 (1-3个月)
├── 自动化流程设计
├── 与工作流引擎集成（Camunda）
└── 流程仿真和优化
```

### 5.3 推荐工具

| 工具 | 特点 | 适合人群 |
|------|------|---------|
| **Camunda Modeler** | 开源、专业、BPMN标准 | 技术型产品经理 |
| **Draw.io** | 免费、在线、简单 | 初学者 |
| **Lucidchart** | 协作友好、模板丰富 | 团队使用 |
| **ProcessMaker** | 低代码自动化 | 企业流程管理 |

## 六、TOGAF 活动定义参考

### 6.1 TOGAF 活动定义标准

> 知识点 A3-06 参考

每个流程活动需关联以下要素：

| 要素 | 说明 |
|------|------|
| 所属信息系统 | 活动在哪个系统执行 |
| 业务模块 | 属于哪个业务功能模块 |
| 操作角色 | 谁来执行这个活动 |
| 是否关键决策点 | 是否需要做出判断 |
| 输入物 | 活动执行需要什么输入 |
| 输出物 | 活动执行产生什么输出 |

### 6.2 TOGAF 流程架构层级

```
活动 (Activity)
    ↓ 连接形成
详细流程 (Detailed Process)
    ↓ 汇总形成
作业指导书 (SOP)
    ↓ 组合形成
流程架构 (Process Architecture)
    ↓ 覆盖
领域完整自闭环 (Business Domain)
```
