# 状态冲突类比素材

> CH-16 知识点 A5-04：状态冲突
> 用于帮助产品经理理解并发操作导致的状态不一致问题

## 类比一：十字路口没红绿灯

### 场景描述
想象一个十字路口，没有红绿灯也没有交警。两辆车同时从不同方向驶来，都认为自己可以先通过，结果——撞了。

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 十字路口 | 同一个订单/数据 |
| 两辆车 | 两个操作者 |
| 同时驶入 | 同时操作 |
| 没红绿灯 | 没并发控制机制 |
| 撞车 | 状态冲突/数据不一致 |

### 核心结论
红绿灯就是"并发控制机制"。它规定谁先走、谁等，避免冲突。系统里的锁、排队、版本号校验，本质就是红绿灯。

## 类比二：两个人同时改同一份文档

### 场景描述
一份需求文档，小王改了第 3 段，小李改了第 5 段，两人同时保存。结果要么小王的改动丢了，要么小李的改动丢了，要么文档变成奇怪的样子。

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 同一份文档 | 同一条数据库记录 |
| 小王和小李 | 两个并发请求 |
| 同时保存 | 同时提交更新 |
| 改动丢失 | 数据覆盖/状态不一致 |
| 文档变奇怪 | 业务逻辑混乱 |

### 解决方案对比

| 生活方案 | 技术方案 |
|---------|---------|
| 只让一个人改（锁定） | 悲观锁：改之前先锁住 |
| 改完保存时检查有没有人先改了 | 乐观锁：保存时检查版本号 |
| 最后合并两个人的改动 | 冲突合并：提示用户手动解决 |

## 类比三：银行柜台排队

### 场景描述
银行只有一个窗口，但来了 5 个人要办业务。如果没有排队，5 个人同时挤到窗口，谁也办不成。

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 一个窗口 | 一份共享资源（库存、座位） |
| 5 个人 | 多个并发请求 |
| 同时挤到窗口 | 同时抢资源 |
| 排队 | 操作排队/消息队列 |
| 号牌 | 分布式锁 |

### 核心结论
排队让"同时想做事的人"变成"一个接一个做事"。消息队列、分布式锁本质上就是排队机制。

## 类比四：秒杀抢购

### 场景描述
双十一秒杀，100 个库存，来了 1000 个人。如果每个人都直接下单，100 个库存可能被 1000 个人抢到——出现"超卖"。

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 100 个库存 | 有限资源 |
| 1000 个人 | 高并发请求 |
| 直接下单 | 无锁扣减库存 |
| 超卖 | 库存变成负数/状态异常 |
| 先锁定再扣减 | 分布式锁 + 预扣库存 |

### 技术方案简述

| 方案 | 思路 | 适用场景 |
|-----|-----|---------|
| 乐观锁 | 扣减时检查版本号，失败就提示"库存不足" | 低并发、冲突少 |
| 悲观锁 | 扣减前先锁住库存，别人等 | 高并发、冲突多 |
| 预扣 + 异步 | 先预扣库存、异步确认支付 | 秒杀、限时抢购 |
| 消息队列 | 把请求排队，一个个处理 | 流量削峰 |

## 类比使用指南

### 什么时候用哪个类比？

| 读者情况 | 推荐类比 |
|---------|---------|
| 完全不懂技术 | 十字路口红绿灯（最直观） |
| 有办公协作经验 | 文档同时修改（共鸣强） |
| 了解排队概念 | 银行排队（扩展性好） |
| 需要理解具体场景 | 秒杀抢购（贴近业务） |

### 类比讲解节奏

1. **先用最简单类比**：十字路口没红绿灯 → 两车相撞
2. **再展开技术原理**：红绿灯 = 锁，排队 = 消息队列
3. **最后回到业务场景**：两个人同时审批订单会怎样？
