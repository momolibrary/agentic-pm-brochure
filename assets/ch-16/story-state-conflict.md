## 故事一：上线后发现"状态乱了"

### 场景背景
小想负责一个订单审批系统。上线一周后，客服反馈：有几个订单的状态"很奇怪"——明明审批通过了，状态却显示"待审批"；还有订单同时被两个人审批，结果谁都不知道该听谁的。

### 故事正文

周五下午，小想正准备下班，运维老王跑过来说："小想，你们那个审批系统出问题了。"

小想一愣："什么问题？测试的时候没问题啊。"

老王打开监控页面："你看这个订单，ID 是 12847。上午 10 点 23 分，张经理审批通过了。但是状态记录显示，同一时间还有一条'驳回'的操作记录。"

小想盯着屏幕，半天没说话。

老王接着说："还有这个订单，两个人同时打开审批页面，都点了'同意'。系统记录了两条审批记录，但状态变成了'审批中'——按你们的逻辑，两个人都同意应该是'已通过'吧？"

小想这下彻底懵了。他跑去问开发，开发说："这是并发问题。你需求里没写两个人同时操作怎么处理，我们默认做了个简单的处理，结果出问题了。"

开发打开代码给小想看："你看，这里判断状态的时候，没加锁。两个人同时点，第一个人的请求刚读到'待审批'状态，第二个人的请求也读到了'待审批'状态，两个请求都认为可以审批，就都写了。结果数据库里有了两条记录，状态被第二个请求覆盖了。"

小想这才明白：原来"两个人同时操作"不是虚构场景，是真会发生的事。

### 老李复盘

周一的复盘会上，老李总结了这个教训：

"小想，你觉得这个问题出在哪？"

小想说："我以为不会有人同时操作。"

老李摇头："在多用户系统里，并发操作是常态，不是例外。只要有多个用户，就必须考虑并发。"

老李画了一张图：

```
时间线：
┌───────────────────────────────────────────────────────┐
│ 用户A: 读状态(待审批) → 判断可审批 → 写状态(已通过)     │
│ 用户B: 读状态(待审批) → 判断可审批 → 写状态(已驳回)     │
│                                                        │
│ 问题: A和B都读到了"待审批"，都认为可以审批，结果冲突    │
└───────────────────────────────────────────────────────┘
```

老李说："想象一下，这就像两个人同时改同一份文档。你改了第3段，他改了第5段，两人同时保存，结果要么你的丢了，要么他的丢了。"

小想问："那怎么解决？"

老李说："解决方法有好几种。最简单的是加锁——审批之前先把订单锁住，别人等。还有一种是版本号校验——审批时检查订单有没有被人改过，改过就提示'订单状态已变更，请重新查看'。"

老李总结："以后写需求，凡是涉及状态变更的，都要问一句：多人同时操作怎么办？"

### 知识点映射

| 故事元素 | 知识点 |
|---------|-------|
| 两个人同时审批 | 并发操作场景 |
| 状态记录冲突 | 状态冲突现象 |
| 没加锁导致问题 | 技术根因 |
| "不会有人同时操作" | 产品经理常见误解 |
| 老李画的时序图 | 并发冲突可视化 |

## 故事二：秒杀超卖事件

### 场景背景
公司搞双十一促销，设置了一个限量秒杀商品：100件库存。结果秒杀开始后，客服接到大量投诉：下单成功的用户去付款时发现库存没了。

### 故事正文

双十一凌晨，小想盯着后台监控。秒杀开始后一分钟，下单量已经破千。

但很快，客服群里炸开了锅："有用户说下单成功了，付款的时候提示库存不足！"

小想心想：设置了100件库存，怎么会库存不足？

运营老张跑过来问："你们系统是不是坏了？我这边统计显示有120个人下单成功了，但库存明明只有100件。"

小想打开数据库一看：库存变成了 -20。

开发紧急排查后说："这是超卖问题。100件库存，来了1000个请求。请求1读库存是100，扣减成99；请求2读库存还是100（因为请求1还没写完），也扣减成99……最后1000个请求都认为自己扣减成功了，结果库存变成了负数。"

开发连夜加了分布式锁：下单前先锁库存，扣减成功才释放锁。没抢到的直接提示"库存不足"。

第二天复盘会上，老李说："这就是典型的状态冲突场景——秒杀。库存也是一种'状态'，多人同时抢同一个库存，就会冲突。"

老李问小想："你觉得产品需求里应该怎么写？"

小想说："写清楚库存扣减的逻辑？"

老李点头："对，还要写清楚并发策略。比如：是先锁库存再下单，还是下单时检查库存版本号？失败的用户怎么提示？这些都要在需求里定义清楚。"

### 知识点映射

| 故事元素 | 知识点 |
|---------|-------|
| 库存变成负数 | 状态不一致表现 |
| 多请求同时扣减 | 并发冲突根因 |
| 分布式锁 | 技术解决方案 |
| 需求要定义并发策略 | 产品经理职责 |

## 故事三：双击提交的困惑

### 场景背景
用户投诉：点击"提交审批"按钮后，系统提示"提交成功"，但刷新页面发现审批状态变成了"已驳回"。

### 故事正文

客服转来一个用户投诉："我点了提交审批，明明显示成功了，怎么变成驳回状态了？是不是系统有问题？"

小想查了日志，发现用户在短时间内点了两次"提交"按钮——可能是手抖双击了。

开发解释："两次请求都成功提交了。第一次请求创建了一条'审批通过'记录，第二次请求因为订单状态已经变了（从待审批变成审批中），触发了另一个逻辑分支，结果被系统自动驳回。"

小想问："为什么会驳回？"

开发说："你需求里写'审批中的订单不能再次提交'，第二次请求检测到状态不是待审批了，就走了驳回分支。"

小想这才意识到：自己写的需求逻辑有问题。他只考虑了正常流程，没考虑用户可能的异常操作——比如双击。

老李看了日志说："这种问题很常见。解决方案有几种：前端防双击（按钮点击后禁用一段时间）、后端幂等校验（同一请求只处理一次）、或者需求里明确写'重复提交怎么处理'。"

老李总结："写状态相关的需求，要想想用户会怎么操作——不是只有正常的点一次，还有双击、刷新、回退、关闭再开……这些都要考虑。"

### 知识点映射

| 故事元素 | 知识点 |
|---------|-------|
| 用户双击提交 | 异常操作场景 |
| 第二次请求触发驳回 | 状态判断导致异常 |
| 前端防双击 | UI层面解决方案 |
| 后端幂等校验 | 技术层面解决方案 |
