# 状态机类比素材

> CH-16 知识点 A5-05：状态机
> 用于帮助产品经理理解状态机的概念和设计方法

## 类比一：自动售货机

### 场景描述
一台自动售货机的完整工作流程：

```
┌─────────────┐
│   空闲状态    │ ←─────────────────┐
└──────┬──────┘                    │
       │ 投币                       │ 退币/完成
       ▼                            │
┌─────────────┐                    │
│   已投币状态  │                    │
└──────┬──────┘                    │
       │ 选择商品                   │
       ▼                            │
┌─────────────┐                    │
│   选择商品状态 │                    │
└──────┬──────┘                    │
       │ 确认购买                   │
       ▼                            │
┌─────────────┐                    │
│   出货状态    │─────────────────────┘
└─────────────┘
```

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 空闲 → 已投币 → 选择 → 出货 | 状态流转序列 |
| 投币动作 | 触发事件（支付成功） |
| 只能选择商品（不能直接出货） | 状态转换约束 |
| 退币回到空闲 | 状态回退/取消 |
| 出货后回到空闲 | 状态流转闭环 |

### 核心结论
状态机就是"规定什么状态能跳到什么状态，中间要做什么"。自动售货机不能从"空闲"直接跳到"出货"——必须经过"投币"和"选择"。这对应系统里"待支付不能直接变成已完成"。

## 类比二：交通信号灯

### 场景描述
交通信号灯的状态循环：

```
红灯 → 绿灯 → 黄灯 → 红灯 → ...
```

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 红/绿/黄三种状态 | 系统定义的状态集合 |
| 定时器触发切换 | 事件驱动状态转换 |
| 严格顺序（红→绿→黄） | 状态转换规则 |
| 循环往复 | 状态流转闭环 |

### 特殊点说明
交通信号灯的特点是"固定顺序、定时触发"。很多系统状态机也是这样——比如订单状态：待支付→已支付→已发货→已完成，严格顺序，不会跳步。

## 类比三：电梯

### 场景描述
电梯的状态系统更复杂：

```
空闲 → 上行 → 停靠 → 门开启 → 门关闭 → 空闲
              ↓
         下行 → 停靠 → 门开启 → 门关闭 → 空闲
```

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 空闲、上行、下行、停靠、门开、门关 | 多状态定义 |
| 楼层请求触发上行/下行 | 事件触发转换 |
| 门开时不能上下行 | 状态互斥约束 |
| 同时多个楼层请求 | 并发请求处理 |

### 核心结论
电梯展示了状态机的"互斥约束"：门开着的时候不能上下行（安全约束）。系统里也有类似规则：退款进行中不能发货、审批中不能取消订单。

## 类比四：洗衣机

### 场景描述
洗衣机的状态流转：

```
待机 → 进水 → 洗涤 → 漂洗 → 脱水 → 待机
         ↓
      暂停 → 继续
         ↓
      取消 → 待机
```

### 类比映射

| 生活场景 | 系统场景 |
|---------|---------|
| 待机/进水/洗涤/漂洗/脱水 | 状态定义 |
| 暂停按钮 | 中断事件 |
| 取消回到待机 | 异常回退 |
| 按顺序进行 | 状态流转约束 |

### 特殊点说明
洗衣机展示了"暂停"和"取消"两个特殊事件。系统状态机也需要考虑这些边界：支付超时取消、审批驳回退回。

## 状态机设计核心要素（给产品经理的）

### 从类比提炼的设计要素

| 要素 | 类比来源 | 系统含义 |
|-----|---------|---------|
| 状态集合 | 信号灯的红绿黄 | 有哪些状态？每个叫什么？ |
| 转换规则 | 售货机不能跳步 | 哪个状态能跳到哪个？不能跳到哪个？ |
| 触发事件 | 投币、按钮 | 什么条件触发状态切换？ |
| 状态互斥 | 电梯门开时不能移动 | 一个时刻只能处于一个状态 |
| 边界处理 | 洗衣机暂停/取消 | 异常情况怎么处理？超时怎么办？ |

## 产品经理设计状态机的步骤

### 步骤一：列出所有状态
问自己：这个订单/审批/任务，从头到尾可能处于哪些情况？

示例：订单状态
- 待支付、已支付、已发货、已完成、已取消、退款中、已退款

### 步骤二：画出状态流转
问自己：正常流程是什么？

示例：订单正常流程
- 待支付 → 已支付 → 已发货 → 已完成

### 步骤三：标注异常分支
问自己：哪里可能出岔子？

示例：订单异常分支
- 待支付 → 已取消（用户取消/超时）
- 已支付 → 退款中 → 已退款（申请退款）
- 已发货 → 退款中 → 已退款（退货退款）

### 步骤四：定义转换条件
问自己：每个转换需要什么条件？

示例：订单转换条件
- 待支付 → 已支付：支付成功回调
- 待支付 → 已取消：用户取消 或 支付超时
- 已支付 → 已发货：商家发货操作

### 步骤五：考虑并发
问自己：两个人同时操作会怎样？

示例：订单并发场景
- 两个人同时审批同一订单
- 用户取消的同时商家发货
