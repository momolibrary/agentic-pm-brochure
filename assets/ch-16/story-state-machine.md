# 状态机设计故事素材

> CH-16 知识点 A5-05：状态机

## 故事一：老李教小想设计状态机

### 场景背景
小想负责设计一个"请假审批"功能。老李看他画的流程图，发现只有三步：提交→审批→完成。

### 故事正文

周一下午，小想拿着需求文档来找老李："李叔，你看我画的请假审批流程，对不对？"

老李看了两分钟，问："这是流程，不是状态机。状态机呢？"

小想愣住了："状态机？我以为流程图就是状态机。"

老李摇头："流程和状态机是两回事。流程说的是'操作的顺序'，状态机说的是'数据的状态和转换规则'。"

老李拿起笔，在纸上画了一个图：

```
┌─────────────────────────────────────────────────────────┐
│                    请假单状态机                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│    ┌──────────┐                                        │
│    │  待提交   │ ←──────────────────┐                  │
│    └────┬─────┘                    │                  │
│         │ 提交                      │ 撤回              │
│         ▼                           │                  │
│    ┌──────────┐                    │                  │
│    │  待审批   │ ────────────────────────┘              │
│    └────┬─────┘                                        │
│         │                                              │
│    ┌────┴────┐                                         │
│    │         │                                         │
│  审批通过   审批驳回                                    │
│    │         │                                         │
│    ▼         ▼                                         │
│ ┌────────┐ ┌────────┐                                  │
│ │ 已通过  │ │ 已驳回  │                                 │
│ └────────┘ └────┬───┘                                  │
│               │                                        │
│            重新提交                                     │
│               │                                        │
│               ▼                                        │
│          ┌──────────┐                                  │
│          │  待审批   │                                  │
│          └──────────┘                                  │
└─────────────────────────────────────────────────────────┘
```

老李说："看，这是状态机。有六个状态：待提交、待审批、已通过、已驳回、已撤销（用户撤销）、已过期（超过审批时限自动取消）。每个状态能跳到哪些状态，什么条件触发跳转，都要写清楚。"

小想看着图问："为什么已驳回还能跳回待审批？"

老李说："因为用户可能重新提交。驳回不是终点，用户修改后还能再提交。你要把这些分支都考虑进去。"

小想又问："那我画的流程图没用吗？"

老李说："有用。流程图是'业务流程'——用户怎么操作。状态机是'数据状态'——请假单在系统里是什么状态。两者要对应起来，但不能混为一谈。"

老李总结："开发说'要设计状态机'，就是让你把这些状态、转换规则、触发条件都定义清楚。他拿到这个设计，才能写出正确的代码。"

### 知识点映射

| 故事元素 | 知识点 |
|---------|-------|
| 流程图 ≠ 状态机 | 核心概念区分 |
| 六种状态定义 | 状态集合设计 |
| 状态转换规则 | 转换路径定义 |
| 触发条件说明 | 事件驱动 |
| 驳回可重新提交 | 异常分支设计 |

## 故事二：订单状态机设计实战

### 场景背景
业务部门要做一个电商订单系统。小想第一次独立负责状态设计。

### 故事正文

周三，业务方来提需求："我们要做一个订单系统，和淘宝差不多那种。"

小想问："订单有哪些状态？"

业务说："待支付、已支付、已发货、已完成、已取消。"

小想打开文档开始写。写到一半，老李路过，看了一眼问："退款呢？"

小想愣住："退款？"

老李说："用户买了东西想退，这状态怎么变？是从已支付跳到退款，还是从已发货跳到退款？退款成功了状态是什么？退款失败状态回哪？"

小想这才意识到自己漏了好多状态。他重新画了一张图：

```
订单状态机（完整版）

待支付 ──────┬──────→ 已取消（用户取消/超时）
             │
           支付成功
             │
             ▼
已支付 ──────┬──────→ 已发货
             │
           申请退款
             │
             ▼
退款中 ──────┬──────→ 已退款
             │
           退款失败
             │
             ▼
已支付（退回）

已发货 ──────┬──────→ 已完成（确认收货）
             │
           申请退款
             │
             ▼
退款中 ──────┬──────→ 已退款
             │
           退款失败
             │
             ▼
已发货（退回）
```

小想问老李："这样够了吗？"

老李又问："支付超时呢？发货超时呢？自动确认收货呢？"

小想又加了几条规则：
- 待支付 → 已取消：支付超时30分钟自动取消
- 已发货 → 已完成：发货后7天自动确认收货

老李点头："现在差不多了。但要记住，状态机设计有四个要素：状态集合、转换规则、触发条件、边界处理。你把这四个都写清楚，开发才能实现。"

小想总结了一下：

| 要素 | 内容 |
|-----|-----|
| 状态集合 | 待支付、已支付、已发货、已完成、已取消、退款中、已退款（共7个） |
| 转换规则 | 每个状态能跳到哪些状态（如已支付只能跳到已发货或退款中） |
| 触发条件 | 什么事件触发转换（如支付成功、商家发货、用户取消） |
| 边界处理 | 超时、退款失败等异常情况 |

### 知识点映射

| 故事元素 | 知识点 |
|---------|-------|
| 业务只说5个状态 | 需求挖掘不完整 |
| 老李追问退款 | 状态遗漏识别 |
| 完整状态7个 | 状态集合扩充 |
| 转换规则定义 | 状态流转约束 |
| 超时自动处理 | 边界条件设计 |

## 故事三：状态机可视化工具

### 场景背景
小想画完状态图后，想找一个可视化工具和开发共享。

### 故事正文

周五，小想拿着手绘的状态图找开发沟通。开发看了半天说："你这画的有点乱，我用 XState 画一个吧。"

小想问："XState 是什么？"

开发打开 [stately.ai/editor](https://stately.ai/editor)："这是一个在线状态机编辑器。你拖拽就能画状态图，还能导出代码给我们用。"

开发演示了一遍：
1. 创建状态节点（待支付、已支付、已发货……）
2. 连接状态之间的转换线
3. 给每个转换定义触发事件（支付成功、发货……）
4. 导出状态机定义文件

小想说："这个好用，以后我可以用这个画状态图，直接发给开发。"

老李看到后说："状态机可视化有两个好处：一是你画完能检查有没有遗漏（比如某个状态没有出口）；二是开发能直接看懂你的设计，不用反复沟通。"

老李建议："以后写需求，先在这个工具上画状态机，截图放进文档，再把状态转换规则用表格写清楚。这样开发拿到就能做。"

### 知识点映射

| 故事元素 | 知识点 |
|---------|-------|
| XState/Stately.ai | 状态机可视化工具 |
| 拖拽画状态图 | 低门槛设计方式 |
| 导出代码 | 产品→开发的衔接 |
| 检查遗漏 | 可视化的验证价值 |

## 状态机设计模板（给产品经理用）

### 基础模板

```markdown
## 状态机定义

### 1. 状态集合
| 状态名称 | 含义 | 备注 |
|---------|-----|-----|
| 待XX    | ... | ... |

### 2. 状态转换规则
| 当前状态 | 可转换到 | 触发条件 | 业务动作 |
|---------|---------|---------|---------|
| 待XX    → 已XX  | XX成功   | 发通知、改数据 |

### 3. 异常分支
| 场景 | 状态变化 | 处理方式 |
|-----|---------|---------|
| 超时 | 待XX → 已取消 | 自动处理 |

### 4. 并发场景
| 场景 | 处理策略 |
|-----|---------|
| 两人同时审批 | 乐观锁/排队/提示冲突 |
```
